# vision: keyrack prefer daemon over envvar

## the outcome world

### day-in-the-life

alice debugs an integration test. her shell has `STRIPE_API_KEY` set to a stale key from yesterday. she runs:

```bash
rhx keyrack unlock --owner ehmpath --env test --key STRIPE_API_KEY
```

keyrack unlocks the fresh key from 1password into the daemon. now she runs:

```bash
source <(rhx keyrack source --owner ehmpath --env test)
```

**today (broken):** the source command reads back her stale envvar because envvar is checked first. she's confused — "i just unlocked it, why is it still stale?"

**after fix:** the source command reads from daemon first, gets the unlocked key, and overwrites her stale envvar. it just works.

### before/after contrast

| scenario | before | after |
|----------|--------|-------|
| envvar set + daemon unlocked | envvar wins (ambient) | daemon wins (explicit intent) |
| envvar unset + daemon unlocked | daemon wins | daemon wins |
| envvar set + daemon empty | envvar wins | envvar wins |
| neither set | absent | absent |

### the "aha" moment

> "oh, unlock actually *has effect* now. when i unlock a key, that's the key i get."

the daemon represents explicit user intent. envvar is ambient state. explicit should beat ambient.

## user experience

### usecases

| usecase | goal | flow |
|---------|------|------|
| override stale envvar | use freshly unlocked credential | unlock → source → get fresh value |
| ci passthrough | no unlock needed, use injected envvar | envvar is set, no daemon → envvar used |
| local dev with rotation | rotate key, unlock new, immediate effect | unlock → next get uses new key |

### contract inputs & outputs

**keyrack get:**

```
input: --key API_KEY --env test
precedence:
  1. os.daemon (if live and has key) → granted
  2. os.envvar (if set) → granted
  3. neither → locked or absent
```

**keyrack source:**

```
input: --env test
behavior: calls keyrack get internally, so same precedence applies
output: export statements for shell
```

**keyrack unlock:**

```
input: --key API_KEY --env test
behavior: vault → daemon (unchanged, never uses envvar)
```

### timeline

```
[0] shell starts with STRIPE_API_KEY="stale"
[1] user runs: keyrack unlock --env test --key STRIPE_API_KEY
    → daemon now has: STRIPE_API_KEY="fresh"
[2] user runs: keyrack get --env test --key STRIPE_API_KEY
    → returns "fresh" (daemon wins)
[3] user runs: source <(keyrack source --env test)
    → shell now has: STRIPE_API_KEY="fresh"
```

## mental model

### how users describe it

> "keyrack unlock puts keys in a session cache. keyrack get checks the session cache first, then falls back to environment variables."

### analogies

| concept | analogy |
|---------|---------|
| daemon | clipboard — what you just copied |
| envvar | sticky note on monitor — what was there before |
| unlock | cmd+c — copy fresh value to clipboard |
| get | cmd+v — paste from clipboard if available, else read sticky note |

### terms

| user term | our term | definition |
|-----------|----------|------------|
| "session cache" | os.daemon | unlocked keys held in memory |
| "env var" | os.envvar | ambient shell environment |
| "unlock" | unlock | populate daemon from vault |
| "get" | get | retrieve from daemon (now) or envvar (fallback) |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| unlocked keys take precedence | yes | daemon checked first |
| ci passthrough still works | yes | if no daemon, envvar used |
| no surprise stale values | yes | explicit unlock > ambient envvar |

### pros

- explicit intent wins over ambient state
- unlock actually results in "use this key"
- simpler mental model: "unlock it to use it"
- ci unchanged (no daemon active → envvar used)

### cons

- behavior change (could surprise users who rely on envvar-first)
- must document the new precedence clearly

### edgecases and pit of success

| edgecase | behavior |
|----------|----------|
| daemon dead, envvar set | envvar used (fallback) |
| daemon live but key not in it | envvar used (fallback) |
| daemon live, key in it, envvar also set | daemon wins (explicit intent) |
| daemon live, key expired | key pruned, envvar used |

pit of success: unlock → get works. no need to unset envvars.

### assumptions

2. [answered] **ci environments don't run daemon** — proposed behavior handles both cases correctly
3. [answered] **envvar-first was not intentional design** — code comment reveals "ci passthrough" intent

### questions for wisher

1. [answered] is the precedence (daemon → envvar → absent) correct?
   - yes, per wish: "if os.daemon is already live and has the key we want, we should prefer it over os.envvar"

2. [answered] should we log/warn when daemon overrides envvar? or silent?
   - wisher confirmed: silent for now. add verbosity later if needed.

3. [answered] should `keyrack source` mention when it overrides an extant envvar?
   - no. source output is consumed by `source <(...)` — extra output would break shell eval.
   - source must be silent by design, not preference.

## what is awkward?

### behavior change risk

some users may rely on envvar precedence. if they set `API_KEY=test` in their shell and expect it to always win, this breaks that.

mitigation: document clearly. the new behavior is more intuitive for the unlock workflow.

### term shift: "passthrough" becomes "fallback"

the comment calls envvar "ci passthrough" which implies it's the primary path. after this change, it's a fallback.

mitigation: update comments to call it "ci fallback" not "ci passthrough".

### silent override?

when daemon overrides envvar, should we log it? this could be noisy but also useful for debug.

recommendation: silent for now. if users are confused, add opt-in verbosity later.
