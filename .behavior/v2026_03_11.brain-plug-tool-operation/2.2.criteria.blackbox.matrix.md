# blackbox criteria coverage matrix

---

## matrix 1: tool execution by brain type

| ind: brain type | ind: tool count | dep: who executes | dep: output.calls | dep: caller handles loop |
|-----------------|-----------------|-------------------|-------------------|--------------------------|
| atom | single | caller | `{ tools: [invocation] }` | yes |
| atom | multiple | caller | `{ tools: [inv1, inv2, ...] }` | yes |
| repl | single | repl via .execute() | `null` | no |
| repl | multiple | repl via .execute() | `null` | no |

**coverage:** usecases 1, 2, 3

---

## matrix 2: execution signal by outcome

| ind: execution outcome | dep: signal | dep: brain interpretation |
|------------------------|-------------|---------------------------|
| tool succeeds | `'success'` | use output normally |
| input/business constraint violated | `'error:constraint'` | try different approach |
| unexpected failure | `'error:malfunction'` | tool is broken |

**coverage:** usecases 4, 5, 6

---

## matrix 3: parallel execution with mixed results

| ind: tool 1 outcome | ind: tool 2 outcome | dep: execution 1 signal | dep: execution 2 signal | dep: brain receives |
|---------------------|---------------------|-------------------------|-------------------------|---------------------|
| success | success | `'success'` | `'success'` | both results |
| success | constraint error | `'success'` | `'error:constraint'` | both results |
| success | malfunction | `'success'` | `'error:malfunction'` | both results |
| constraint error | success | `'error:constraint'` | `'success'` | both results |
| constraint error | constraint error | `'error:constraint'` | `'error:constraint'` | both results |
| constraint error | malfunction | `'error:constraint'` | `'error:malfunction'` | both results |
| malfunction | success | `'error:malfunction'` | `'success'` | both results |
| malfunction | constraint error | `'error:malfunction'` | `'error:constraint'` | both results |
| malfunction | malfunction | `'error:malfunction'` | `'error:malfunction'` | both results |

**coverage:** usecase 7

**note:** all combinations result in brain receipt of both results. no fail-fast behavior at contract level — caller decides via Promise.all vs Promise.allSettled.

---

## matrix 4: prompt type by continuation state

| ind: prompt type | ind: has prior episode | dep: brain interpretation |
|------------------|------------------------|---------------------------|
| string | no | initial prompt, start episode |
| string | yes | follow-up prompt, continue episode |
| execution array | no | invalid (no invocations to correlate) |
| execution array | yes | tool result continuation |

**coverage:** usecases 9, 10

**gap identified:** execution array without prior episode is invalid — should this error or create fresh episode?

**recommendation:** error at supplier level. executions must correlate to invocations from a prior exchange.

---

## matrix 5: type safety flow

| ind: invocation typed | ind: execution typed | dep: compile-time safety | dep: runtime correlation |
|-----------------------|----------------------|--------------------------|--------------------------|
| `TInput` known | `TInput, TOutput` known | full type flow | exid matches |
| `TInput` known | `unknown` output | partial type flow | exid matches |
| `unknown` input | `unknown` output | no type flow | exid matches |

**coverage:** usecase 8

**note:** type safety is opt-in via generics. runtime correlation via exid is always present.

---

## gaps identified

1. **execution array without episode** — matrix 4 shows this combination is invalid but not explicitly handled in criteria. added recommendation to error.

2. **exid mismatch** — what happens if caller provides execution with wrong exid? not covered in criteria.
   - **recommendation:** supplier validates exid against queued invocations; throws if mismatch.

---

## decomposition analysis

all matrices have 2-3 independent dimensions — no decomposition needed.

the design has clean separation:
- brain type determines execution ownership (matrix 1)
- outcome determines signal (matrix 2)
- parallel execution is symmetric (matrix 3)
- prompt type is binary (matrix 4)
- type safety is opt-in (matrix 5)

no dimension explosion detected. behavioral boundaries are appropriately scoped.
