# research: external product claims

---

## claim 1: error messages should be structured for LLM consumption

**[FACT]** tool errors should return structured messages with type, description, and context.

> "When a tool fails, it should return an error message specifically designed for LLM consumption, including: An Error Type (a simple classification of the error), a Descriptive Message (a human-readable explanation of what went wrong), and Context (information about the specific input or operation that caused the error)." [1]

**relevance:** validates our `signal` field design — three-way classification (success, error:constraint, error:malfunction) gives LLM actionable error types.

[1] https://apxml.com/courses/building-advanced-llm-agent-tools/chapter-1-llm-agent-tooling-foundations/tool-error-handling

## claim 2: tool errors serve two purposes

**[FACT]** error returns serve both LLM decision-support and developer debug.

> "When a tool fails, it should return an error message specifically designed for LLM consumption, including: An Error Type (a simple classification of the error), a Descriptive Message (a human-readable explanation of what went wrong), and Context (information about the specific input or operation that caused the error)." [1]

**relevance:** our design separates signal (for LLM) from output (for details). metrics capture time for observability.

[1] https://apxml.com/courses/building-advanced-llm-agent-tools/chapter-1-llm-agent-tooling-foundations/tool-error-handling

## claim 3: input errors are a distinct category

**[FACT]** input validation failures are a common, distinct error category.

> "Input Issues represent a common scenario where the tool receives invalid, malformed, or unexpected input, such as a text string where a number is expected." [1]

**relevance:** validates `error:constraint` signal — input/business rule violations are distinct from system failures (`error:malfunction`).

[1] https://apxml.com/courses/building-advanced-llm-agent-tools/chapter-1-llm-agent-tooling-foundations/tool-error-handling

## claim 4: treat LLMs as software components with strict interfaces

**[OPIN]** 2025 best practice: treat LLMs as software components, not magic text generators.

> "Treat JSON schema as your contract, not your prompt" [2]

**relevance:** aligns with our typed contract approach — `BrainPlugToolExecution<TInput, TOutput>` with schema validation.

[2] https://medium.com/@gbalagangadhar/llm-engineering-in-2025-the-failure-modes-that-actually-matter-and-how-i-fix-them-ad1f6f1da77e

## claim 5: parallel execution requires task independence

**[FACT]** parallel tool execution is appropriate only for independent tasks.

> "Tasks that can be decomposed into independent sub tasks are strong candidates for parallel execution. When work streams do not rely on one another, multiple agents can operate simultaneously." [3]

> "By contrast, interdependent steps require sequential execution. When one action depends on the output of another, parallel execution introduces coordination risk." [3]

**relevance:** our design supports parallel tool execution via array of invocations/executions. caller decides independence.

[3] https://medium.com/@prithasaha_62327/from-linear-prompts-to-agentic-workflows-a-guide-to-sequential-parallel-and-loop-architectures-b5d394606b9d

---

## claim 6: parallel results should preserve order

**[SUMP]** parallel results should be re-ordered to match request order before return to LLM.

> "When tools are executed in parallel, their individual results (which may arrive out of order) should be collected and then re-sorted to match the agent's original requested order before return to the LLM." [4]

**relevance:** our contract uses exid correlation — order is irrelevant as long as exid matches. this is simpler than order-based correlation.

[4] https://agentic-patterns.com/patterns/parallel-tool-execution/

---

## claim 7: agentic loops require feedback observation

**[FACT]** agentic workflows iterate via action → observation → next action cycles.

> "What makes workflows truly agentic are the iteration and feedback loops built into the process. Agentic workflows involve cycles where the agent takes an action, observes the result, and uses that observation to inform the next action." [5]

**relevance:** validates BrainRepl design — repl loops internally via tool.execute(), observes results, continues until complete.

[5] https://fme.safe.com/guides/ai-agent-architecture/ai-agentic-workflows/

---

## claim 8: read vs write operations need different execution strategies

**[OPIN]** classify operations as read-only vs stateful for execution strategy.

> "The system solves this by classify operations as read-only or stateful, apply different execution strategies to each. Smart Concurrency Control: Parallel for read operations... Sequential for write operations." [3]

**relevance:** our design leaves execution strategy to caller/repl. the contract doesn't distinguish read/write — caller knows their tools.

[3] https://medium.com/@prithasaha_62327/from-linear-prompts-to-agentic-workflows-a-guide-to-sequential-parallel-and-loop-architectures-b5d394606b9d

---

## claim 9: tool loop termination needs explicit conditions

**[FACT]** agentic loops need explicit termination conditions.

> "The multi-agent loop agent pattern repeatedly executes a sequence of specialized subagents until a specific termination condition is met." [6]

**relevance:** our BrainRepl design uses output.complete or explicit schema fulfillment as termination. clear exit conditions.

[6] https://docs.cloud.google.com/architecture/choose-design-pattern-agentic-ai-system

---

## open questions discovered

**[KHUE]** should we support read/write classification in tool definitions?

the research suggests read-only tools can run in parallel safely, while stateful tools need sequential execution. our current design doesn't capture this distinction.

**recommendation:** out of scope for v1. caller knows their tools. could add `tool.sideEffects: 'none' | 'read' | 'write'` in future if needed.
