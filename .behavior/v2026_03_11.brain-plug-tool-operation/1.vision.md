# vision: brain plug tool operation

## the outcome world

### before: the tool call cliff

today, when a brain returns tool calls, the developer hits a cliff:

```ts
// get tool calls from brain
const result = await brain.ask({
  prompt: 'lookup customer by email',
  plugs: { tools: [stripeCustomerLookup] },
  schema: { output: z.object({ customer: customerSchema }) },
  role,
});

// brain defers to tool
if (result.calls?.tools) {
  const invocation = result.calls.tools[0]!;

  // execute the tool
  const customer = await stripe.customers.list({ email: invocation.input.email });

  // now what? 🤔
  // - how do i feed this result back?
  // - what shape does the brain expect?
  // - how do i correlate with the exid?
  // - how do i track execution metrics?
  // - how do i signal success vs failure?
  // - how do i continue the episode?
}
```

developers must:
- construct provider-specific message formats manually
- juggle exid correlation across different sdk shapes
- lose type safety when they compose tool results
- implement their own metric trackers
- figure out error signal semantics
- figure out episode continuation semantics

### after: the tool operation bridge

with `BrainPlugToolExecution` and `BrainPlugToolDefinition.execute`, the cliff becomes a bridge:

```ts
// define tool with execute operation (repl only)
const stripeCustomerLookup: BrainPlugToolDefinition<{ email: string }, Customer, 'repl'> = {
  slug: 'stripe-customer-lookup',
  name: 'Stripe Customer Lookup',
  description: 'lookup a customer by email in stripe',
  schema: { input: z.object({ email: z.string().email() }) },
  execute: async ({ invocation }, context) => {
    const customers = await context.stripe.customers.list({ email: invocation.input.email });
    if (!customers.data[0]) {
      throw new BadRequestError('customer not found');
    }
    return customers.data[0];
  },
};

// BrainRepl handles the loop automatically
const result = await repl.act({
  prompt: 'lookup customer alice@example.com and summarize their account',
  plugs: { tools: [stripeCustomerLookup] },
  schema: { output: z.object({ summary: z.string() }) },
  role,
});

// result.output is final - repl executed tools internally
return result.output;
```

for BrainAtom (manual execution, no .execute on definition):

```ts
// define tool for atom (no execute operation)
const stripeCustomerLookup: BrainPlugToolDefinition<{ email: string }, Customer, 'atom'> = {
  slug: 'stripe-customer-lookup',
  name: 'Stripe Customer Lookup',
  description: 'lookup a customer by email in stripe',
  schema: { input: z.object({ email: z.string().email() }) },
  // no execute - atoms don't use it
};

// get tool calls from brain
const result = await atom.ask({
  prompt: 'lookup customer by email',
  plugs: { tools: [stripeCustomerLookup] },
  schema: { output: z.object({ customer: customerSchema }) },
  role,
});

// brain defers to tool
if (result.calls?.tools) {
  const invocation = result.calls.tools[0]!;

  // caller executes the tool with time measurement
  const timeBefore = Date.now();
  let output: Customer;
  let signal: 'success' | 'error:constraint' | 'error:malfunction' = 'success';

  try {
    output = await stripe.customers.list({ email: invocation.input.email });
  } catch (error) {
    output = { error: error.message };
    signal = isConstraintError(error) ? 'error:constraint' : 'error:malfunction';
  }
  const timeAfter = Date.now();

  // compose the execution result - typed, correlated, measured, signaled
  const execution: BrainPlugToolExecution<typeof invocation.input, Customer> = {
    exid: invocation.exid,
    slug: invocation.slug,
    input: invocation.input,
    output,
    signal,
    metrics: { cost: { time: asIsoDuration({ milliseconds: timeAfter - timeBefore }) } },
  };

  // continue the episode with the tool result
  const final = await atom.ask({
    prompt: [execution],  // <- tool executions as prompt continuation (always array)
    on: { episode: result.episode },
    plugs: { tools: [stripeCustomerLookup] },
    schema: { output: z.object({ customer: customerSchema }) },
    role,
  });

  // now we have the final output
  return final.output;
}
```

### the "aha" moment

the value clicks when developers realize:

> "i can pass tool results back to the brain the same way i pass prompts - and the brain continues naturally"

> "BrainRepl handles the tool loop for me - i just plug in tools with execute functions"

> "atoms are stateless - i handle tool execution. repls are agentic - they execute tools via .execute()"

the `prompt: string | BrainPlugToolExecution[]` union makes tool results a first-class continuation mechanism. no special apis. no format juggle. just compose and continue.

## user experience

### usecases

#### usecase 1: single tool call → response (BrainAtom)

the simplest case: brain calls one tool, caller executes, brain responds.

```ts
const lookupCustomer = async (input: { email: string }) => {
  // 1. initial ask with tools (atom tools have no execute)
  const result = await atom.ask({
    prompt: `lookup customer: ${input.email}`,
    plugs: { tools: [stripeCustomerLookup] },
    schema: { output: z.object({ summary: z.string() }) },
    role,
  });

  // 2. if brain defers to tool, caller executes and continues
  if (result.calls?.tools) {
    const executions = await Promise.all(
      result.calls.tools.map(inv => executeToolInvocation(inv))
    );

    return await atom.ask({
      prompt: executions,  // always array
      on: { episode: result.episode },
      plugs: { tools: [stripeCustomerLookup] },
      schema: { output: z.object({ summary: z.string() }) },
      role,
    });
  }

  // 3. or brain answered directly
  return result;
};
```

#### usecase 2: parallel tool calls → response (BrainAtom)

brain calls multiple tools, caller executes in parallel, brain synthesizes.

```ts
const analyzeCustomer = async (input: { customerId: string }) => {
  const result = await atom.ask({
    prompt: `analyze customer ${input.customerId} - get their invoices, subscriptions, and recent charges`,
    plugs: { tools: [getInvoices, getSubscriptions, getCharges] },
    schema: { output: z.object({ analysis: analysisSchema }) },
    role,
  });

  if (result.calls?.tools) {
    // caller executes all tool calls in parallel
    const executions = await Promise.all(
      result.calls.tools.map(invocation => executeToolInvocation(invocation))
    );

    // continue with all executions
    return await atom.ask({
      prompt: executions,  // <- array of executions
      on: { episode: result.episode },
      plugs: { tools: [getInvoices, getSubscriptions, getCharges] },
      schema: { output: z.object({ analysis: analysisSchema }) },
      role,
    });
  }

  return result;
};
```

#### usecase 3: agentic tool loop (BrainRepl with execute)

brain iteratively calls tools until task is complete. BrainRepl handles the loop internally via tool.execute().

```ts
// define tools with execute operations (repl-only)
const fileTools: BrainPlugToolDefinition<any, any, 'repl'>[] = [
  {
    slug: 'read-file',
    name: 'Read File',
    description: 'read contents of a file',
    schema: { input: z.object({ path: z.string() }) },
    execute: async ({ invocation }, context) => {
      return await context.fs.readFile(invocation.input.path, 'utf-8');
    },
  },
  {
    slug: 'write-file',
    name: 'Write File',
    description: 'write contents to a file',
    schema: { input: z.object({ path: z.string(), content: z.string() }) },
    execute: async ({ invocation }, context) => {
      await context.fs.writeFile(invocation.input.path, invocation.input.content);
      return { written: true };
    },
  },
  {
    slug: 'run-tests',
    name: 'Run Tests',
    description: 'run the test suite',
    schema: { input: z.object({}) },
    execute: async ({ invocation }, context) => {
      const result = await context.shell.exec('npm run test');
      return { passed: result.exitCode === 0, output: result.stdout };
    },
  },
];

// BrainRepl loops internally via tool.execute()
const refactorCode = async (input: { file: string; goal: string }) => {
  const result = await repl.act({
    prompt: `refactor ${input.file}: ${input.goal}`,
    plugs: { tools: fileTools },
    schema: { output: z.object({ complete: z.boolean(), summary: z.string() }) },
    role,
    context: { fs, shell },  // passed to tool.execute()
  });

  // result.output is final - repl handled all tool calls
  return result.output;
};
```

### contract inputs & outputs

#### BrainPlugToolDefinition<TInput, TOutput, TBrainType>

```ts
// base fields shared by both atom and repl
interface BrainPlugToolDefinitionBase<TInput = unknown> {
  slug: string;                    // unique identifier
  name: string;                    // display name
  description: string;             // what the tool does (for brain)
  schema: {
    input: z.Schema<TInput>;       // zod schema for input validation
  };
}

// atom tools: no execute (caller handles execution)
type BrainPlugToolDefinition<TInput, TOutput, 'atom'> = BrainPlugToolDefinitionBase<TInput>;

// repl tools: execute is available (repl handles execution)
type BrainPlugToolDefinition<TInput, TOutput, 'repl'> = BrainPlugToolDefinitionBase<TInput> & {
  execute?: (
    input: { invocation: BrainPlugToolInvocation<TInput> },
    context: TContext,
  ) => Promise<TOutput>;
};
```

the type parameter `'atom' | 'repl'` mirrors `BrainOutput<TOutput, TBrainType>`:
- `'atom'` — no execute field; caller handles tool execution
- `'repl'` — optional execute field; repl calls it automatically when present

#### BrainPlugToolExecution<TInput, TOutput>

```ts
interface BrainPlugToolExecution<TInput = unknown, TOutput = unknown> {
  // correlation
  exid: string;                    // from invocation.exid - links result to request
  slug: string;                    // tool identifier - for validation

  // data
  input: TInput;                   // what the tool was called with
  output: TOutput;                 // what the tool returned

  // signal
  signal: 'success' | 'error:constraint' | 'error:malfunction';

  // observability
  metrics: {
    cost: {
      time: IsoDuration;           // how long the tool took
    };
  };
}
```

#### signal semantics

| signal | definition | http analogy | anthropic `is_error` |
|--------|------------|--------------|---------------------|
| `'success'` | tool completed normally | 2xx | `false` |
| `'error:constraint'` | tool hit a business/input constraint | 4xx | `true` |
| `'error:malfunction'` | tool broke unexpectedly | 5xx | `true` |

#### error type exports

rhachet exports synonyms for helpful-errors types with tool-context semantics:

```ts
// rhachet re-exports
// todo: lift into helpful-errors
export { BadRequestError as ConstraintError } from 'helpful-errors';
export { UnexpectedCodePathError as MalfunctionError } from 'helpful-errors';
```

usage in tool.execute():

```ts
execute: async ({ invocation }, context) => {
  const customer = await context.stripe.customers.list({ email: invocation.input.email });

  // constraint: expected failure (4xx)
  if (!customer.data[0]) {
    throw new ConstraintError('customer not found', { email: invocation.input.email });
  }

  // malfunction: unexpected failure (5xx) - or just let it throw naturally
  if (!customer.data[0].id) {
    throw new MalfunctionError('customer has no id', { customer: customer.data[0] });
  }

  return customer.data[0];
}
```

signal map:
- `ConstraintError` → `signal: 'error:constraint'` (4xx, expected failure)
- `MalfunctionError` → `signal: 'error:malfunction'` (5xx, unexpected failure)
- any other error → `signal: 'error:malfunction'` (5xx, unexpected failure)

examples:

```ts
// success - customer found
{
  signal: 'success',
  output: { id: 'cus_123', email: 'alice@example.com' },
}

// error:constraint - customer not found (expected failure mode)
{
  signal: 'error:constraint',
  output: { message: 'no customer with email bob@example.com' },
}

// error:malfunction - stripe api is down (unexpected)
{
  signal: 'error:malfunction',
  output: { message: 'ECONNREFUSED: stripe api unreachable' },
}
```

#### prompt union type

```ts
type BrainAskPrompt<TPlugs extends BrainPlugs> =
  TPlugs extends { tools: BrainPlugToolDefinition[] }
    ? string | BrainPlugToolExecution[]
    : string;
```

when tools are plugged, prompt accepts:
- `string` — initial prompt or follow-up question
- `BrainPlugToolExecution[]` — tool results continuation (always array, even for single tool)

### how BrainRepl executes tools internally

BrainRepl handles the agentic loop via tool.execute():

```ts
// inside BrainRepl.act()
while (true) {
  const result = await this.inference({ prompt, episode });

  // if brain is done, return final output
  if (result.output?.complete) return result;

  // if brain needs tools, execute them via tool.execute()
  if (result.calls?.tools) {
    const executions = await Promise.all(
      result.calls.tools.map(async (invocation) => {
        const tool = plugs.tools.find(t => t.slug === invocation.slug);

        // execute the tool
        const timeBefore = Date.now();
        let output: TOutput;
        let signal: 'success' | 'error:constraint' | 'error:malfunction' = 'success';

        try {
          output = await tool.execute({ invocation }, context);
        } catch (error) {
          output = { error: error.message };
          signal = error instanceof BadRequestError
            ? 'error:constraint'
            : 'error:malfunction';
        }

        const timeAfter = Date.now();

        // wrap in BrainPlugToolExecution
        return {
          exid: invocation.exid,
          slug: invocation.slug,
          input: invocation.input,
          output,
          signal,
          metrics: { cost: { time: asIsoDuration({ milliseconds: timeAfter - timeBefore }) } },
        };
      })
    );

    // continue with executions
    prompt = executions;
    episode = result.episode;
    continue;
  }

  // unexpected state
  throw new UnexpectedCodePathError('brain neither completed nor called tools');
}
```

### why atom vs repl type distinction

the type system enforces the architectural difference:

| aspect | BrainAtom | BrainRepl |
|--------|-----------|-----------|
| purpose | invoke atomic brain exchanges | coordinate tool use into coherent responses |
| tool execution | outputs tool invocations (caller handles) | chains tool use via .execute() |
| tool definition type | `BrainPlugToolDefinition<I, O, 'atom'>` | `BrainPlugToolDefinition<I, O, 'repl'>` |
| execute field | not present (type omits it) | required (repl calls it) |
| output.calls | `{ tools } \| null` (may have invocations) | always `null` (repl executed internally) |

```ts
/**
 * atom tools: no execute
 * .why = atoms invoke atomic brain exchanges; tool execution is caller's job
 */
const atomTool: BrainPlugToolDefinition<Input, Output, 'atom'> = {
  slug: 'my-tool',
  // no execute field - atoms output tool invocations for caller to handle
};

/**
 * repl tools: execute required
 * .why = repls coordinate tool use into coherent responses; execute enables chained tool use
 */
const replTool: BrainPlugToolDefinition<Input, Output, 'repl'> = {
  slug: 'my-tool',
  execute: async ({ invocation }, context) => { ... },  // required for repl to chain
};

/**
 * repl tools fit into atom tools (execute is inaccessible)
 * .why = define tools once with execute, use with both atoms and repls
 */
const tool: BrainPlugToolDefinition<Input, Output, 'repl'> = {
  slug: 'my-tool',
  execute: async ({ invocation }, context) => { ... },
};

// use with repl: repl calls tool.execute() internally
await repl.act({ plugs: { tools: [tool] } });

// use with atom: atom outputs invocations, execute is inaccessible by design
const atomTools: BrainPlugToolDefinition<Input, Output, 'atom'>[] = [tool];  // ✅ fits
await atom.ask({ plugs: { tools: atomTools } });
// atom.ask returns calls: { tools: [invocation] } - caller handles execution
```

#### BrainOutput.calls type constraint

the `AsBrainOutputCallsFor` conditional type will be updated to also consider `TBrainGrain`:

```ts
// current: only considers TPlugs
export type AsBrainOutputCallsFor<TPlugs extends BrainPlugs> =
  TPlugs extends { tools: BrainPlugToolDefinition[] }
    ? { tools: BrainPlugToolInvocation[] } | null
    : null;

// proposed: also considers TBrainGrain
export type AsBrainOutputCallsFor<
  TPlugs extends BrainPlugs,
  TGrain extends BrainGrain,
> = TGrain extends 'repl'
  ? null  // repl always returns null (executes tools internally)
  : TPlugs extends { tools: BrainPlugToolDefinition[] }
    ? { tools: BrainPlugToolInvocation[] } | null
    : null;
```

this ensures:
- atom callers check `output.calls` and handle tool execution when present
- repl callers know `output.calls` is always null (repl executed tools via .execute())

## mental model

### how users would describe this

> "when the brain asks me to run a tool, i execute it and hand back the result. the brain picks up where it left off."

> "tool execution is just another kind of prompt continuation. string prompt to start, execution prompt to continue."

> "atoms are stateless - i handle tool execution. repls are agentic - they execute tools via .execute()"

> "the type system tells me: atom tools don't have execute, repl tools can."

### analogies

#### the waiter analogy

- **brain** = chef who designs the dish
- **tool definition** = menu item description
- **tool.execute** = waiter knows how to fetch the ingredient (repl only)
- **tool invocation** = chef's order to the waiter
- **tool execution** = waiter fetches ingredients and reports back
- **continuation** = chef continues to cook with the ingredients

the waiter doesn't need to understand the recipe - just fetch what's requested and report time.

#### the callback analogy

familiar to javascript developers:

```ts
// traditional callback
doThing((err, result) => {
  // continue with result
});

// brain tool callback
const invocation = await brain.ask({ ... });
const execution = await executeAndMeasure(invocation);
await brain.ask({ prompt: execution, on: { episode } });
```

the execution IS the callback - typed, measured, and composable.

### terminology map

| user term | rhachet term | provider terms |
|-----------|--------------|----------------|
| "tool request" | `BrainPlugToolInvocation` | anthropic: `tool_use`, openai: `tool_calls[n]` |
| "tool response" | `BrainPlugToolExecution` | anthropic: `tool_result`, openai: `{role: "tool"}` |
| "tool succeeded" | `signal: 'success'` | anthropic: `is_error: false` |
| "tool failed (4xx)" | `signal: 'error:constraint'` | anthropic: `is_error: true` |
| "tool broke (5xx)" | `signal: 'error:malfunction'` | anthropic: `is_error: true` |
| "conversation" | `BrainEpisode` | anthropic: message array, openai: messages array |
| "continue" | `on: { episode }` | both: append to messages |

## evaluation

### how well does it solve the goals?

| goal | solution | score |
|------|----------|-------|
| clear guidance for tool results | `BrainPlugToolExecution` contract with typed fields | ✅ |
| easy composition with brain exchanges | `prompt: string \| execution` union | ✅ |
| type safety | generics `<TInput, TOutput>` + `'atom' \| 'repl'` type param | ✅ |
| metric trackers | `metrics: { cost: { time } }` | ✅ |
| error signal | `signal: 'success' \| 'error:constraint' \| 'error:malfunction'` | ✅ |
| provider agnostic | unified contract, suppliers translate | ✅ |
| episode continuation | `on: { episode }` with execution prompt | ✅ |
| automatic tool execution | `BrainPlugToolDefinition.execute` for BrainRepl | ✅ |
| type-level atom/repl distinction | `BrainPlugToolDefinition<I, O, 'atom' \| 'repl'>` | ✅ |

### pros

1. **type safety end-to-end** — generics flow from invocation through execution to continuation
2. **provider agnostic** — same contract works for anthropic, openai, xai, etc
3. **composable** — single tool, parallel tools, loops - all work naturally
4. **observable** — metrics captured at execution boundary
5. **explicit error signal** — `signal` distinguishes success, constraint errors, and malfunctions
6. **type-level enforcement** — atom tools omit execute; repl tools require it
7. **tool reuse** — repl tools fit into atom tools (execute becomes inaccessible)
8. **minimal api surface** — two types, one union extension
9. **symmetry** — input/output mirrors invocation structure

### cons

1. **metrics simplicity** — only `time` initially; may need `cash`, `tokens` later
2. **context injection** — tools with `execute` need context via closure or factory pattern

### edge cases & pit of success

#### edge case: tool execution fails

```ts
// use signal to communicate failure type
try {
  const result = await executeTool(invocation);
  execution = { exid, slug, input, output: result, signal: 'success', metrics };
} catch (error) {
  execution = {
    exid,
    slug,
    input,
    output: { error: error.message },
    signal: error instanceof BadRequestError ? 'error:constraint' : 'error:malfunction',
    metrics,
  };
}
```

the brain sees the signal and can reason accordingly:
- `success`: use output normally
- `error:constraint`: try different approach
- `error:malfunction`: know tool is broken, maybe retry or give up

#### edge case: exid mismatch

```ts
// pit of success: type system catches mismatch at build time
const execution: BrainPlugToolExecution = {
  exid: 'wrong-id',  // would fail correlation at runtime
  // ...
};
```

future: could add runtime validation in supplier to fail fast on mismatched exids.

#### edge case: parallel execution with partial failure

```ts
const executions = await Promise.allSettled(
  invocations.map(inv => executeToolInvocation(inv))
);

// map to executions with appropriate signals
const mapped = executions.map((result, i) => {
  if (result.status === 'fulfilled') {
    return result.value;  // already has signal: 'success'
  }
  return {
    exid: invocations[i].exid,
    slug: invocations[i].slug,
    input: invocations[i].input,
    output: { error: result.reason.message },
    signal: 'error:malfunction',
    metrics: { cost: { time: 'PT0S' } },
  };
});

// continue with all executions - brain sees which succeeded/failed
await brain.ask({ prompt: mapped, on: { episode } });
```

## open questions & assumptions

### assumptions

1. **suppliers can translate** — we assume anthropic/openai suppliers can translate `BrainPlugToolExecution` to their respective formats (`tool_result` / `{role: "tool"}`)

2. **exid is sufficient correlation** — we assume the provider-assigned exid uniquely identifies the tool call within an episode

3. **time is the primary metric** — we assume execution time is the most important metric; cash/tokens are less relevant for tool execution (that's brain inference cost)

4. **output is json-serializable** — we assume tool outputs can be serialized for the brain to consume

5. **context injection via closure or factory** — tools with `execute` capture their context at definition time

### questions for wisher

1. **metrics expansion** — should `metrics` include more than `time`? e.g., `{ bytes, retries }`?

2. **validation** — should suppliers validate that `execution.exid` matches a queued invocation, or trust callers?

3. **stream** — how does this interact with stream responses? does the execution happen mid-stream or post-stream?

## what is awkward?

### the prompt union

```ts
prompt: string | BrainPlugToolExecution[]
```

two shapes in one field: string to start, array to continue with tool results.

alternatives considered:
- `prompt` + `toolResults` as separate fields — more explicit but more api surface
- `prompt` for strings only, `continue` for executions — semantic split but two entry points

**decision**: union keeps the api minimal. always-array for executions removes single-vs-many ambiguity.

### metrics feels minimal

```ts
metrics: {
  cost: {
    time: IsoDuration;
  };
}
```

only time? feels sparse compared to `BrainOutputMetrics` which has `size`, `cost.cash`, `cost.time`, etc.

**rationale**: tools are external to the brain. we can't know their token/cash costs. time is universally measurable. users can extend with their own wrappers.

### slug redundancy

`BrainPlugToolExecution` includes both `exid` (correlation) and `slug` (tool name). the exid should be sufficient for correlation, so slug feels redundant.

**rationale**: slug enables defensive code - suppliers can validate that the execution matches the expected tool type, not just the correlation id.

### no explicit type link between invocation and execution

the current design doesn't enforce that `execution.input` matches `invocation.input` at the type level. users could theoretically mismatch.

**mitigation**: provide helper function:

```ts
const toExecution = <TInput, TOutput>(
  invocation: BrainPlugToolInvocation<TInput>,
  output: TOutput,
  signal: 'success' | 'error:constraint' | 'error:malfunction',
  metrics: { cost: { time: IsoDuration } },
): BrainPlugToolExecution<TInput, TOutput> => ({
  exid: invocation.exid,
  slug: invocation.slug,
  input: invocation.input,
  output,
  signal,
  metrics,
});
```

this helper guarantees type flow from invocation to execution.

---

## summary

`BrainPlugToolExecution` and `BrainPlugToolDefinition.execute` bridge the gap between tool invocation and continuation. they provide:

- **typed contract** for tool results with `<TInput, TOutput>` generics
- **type-level atom/repl distinction** via `BrainPlugToolDefinition<I, O, 'atom' | 'repl'>`
- **correlation** via `exid` from the original invocation
- **explicit signal** via `signal: 'success' | 'error:constraint' | 'error:malfunction'`
- **observability** via `metrics.cost.time`
- **composability** via `prompt: string | execution | execution[]` union
- **coordinated execution** via required `execute` on repl tool definitions

the design prioritizes:
- minimal api surface (two types, one union extension)
- provider agnosticism (suppliers translate to their formats)
- type safety (generics flow through the round trip, atom/repl constrain shape)
- observability (execution time and signal captured at boundary)
- architectural clarity (atoms output tool use, repls coordinate tool use)

the core distinction:
- **BrainAtom**: invokes atomic brain exchanges; outputs tool invocations for caller to handle
- **BrainRepl**: coordinates tool use into coherent responses; chains tool execution via .execute()

tool execution results continue episodes as easily as string prompts start them.
