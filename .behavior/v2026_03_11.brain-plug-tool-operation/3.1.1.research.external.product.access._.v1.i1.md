# research: external product access

---

## summary

this feature requires translation to/from provider-specific tool result formats. no direct remote access is needed at the contract level — suppliers handle provider communication.

## external product 1: anthropic messages api

### what is it?

anthropic's messages api for claude models, with tool use via `tool_use` and `tool_result` content blocks.

### contract

**tool_use block (from model):** [1]
```json
{
  "type": "tool_use",
  "id": "toolu_01abc123",
  "name": "get_weather",
  "input": { "location": "San Francisco" }
}
```

**tool_result block (to model):** [1]
```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_01abc123",
  "content": "65 degrees"
}
```

**error signal:** [2]
- set `is_error: true` on tool_result block
- include error message in content

### best practices

- match `tool_use_id` exactly for correlation [1]
- set `is_error: true` for failures so model can reason about errors [2]
- provide clear error messages in content for model interpretation [2]

### citations

[1] anthropic tool use documentation: https://platform.claude.com/docs/en/agents-and-tools/tool-use/implement-tool-use

> "When you receive a tool_use response from Claude, you should: extract the id (a unique identifier for this particular tool use block used to match up the tool results later)"

[2] anthropic tool use error flow: https://platform.claude.com/docs/en/agents-and-tools/tool-use/implement-tool-use

> "when your tool encounters an error, send a clear and informative error message back to Claude within the content of the tool_result block, and for Anthropic, set is_error: true"

## external product 2: openai chat completions api

### what is it?

openai's chat completions api with function call via `tool_calls` and `role: "tool"` messages.

### contract

**tool_calls (from model):** [3]
```json
{
  "role": "assistant",
  "tool_calls": [
    {
      "id": "call_abc123",
      "type": "function",
      "function": {
        "name": "get_weather",
        "arguments": "{\"location\": \"San Francisco\"}"
      }
    }
  ]
}
```

**tool message (to model):** [3]
```json
{
  "role": "tool",
  "tool_call_id": "call_abc123",
  "content": "65 degrees"
}
```

### best practices

- match `tool_call_id` exactly for correlation [3]
- use `strict: true` for reliable schema adherence [3]
- set `parallel_tool_calls: false` to prevent multiple simultaneous calls if needed [3]

### citations

[3] openai function call documentation: https://platform.openai.com/docs/guides/function-calling

> "When you append a message with role 'tool', include the tool_call_id and content with the results"

> "Set strict to true to ensure function calls reliably adhere to the function schema"

## external product 4: helpful-errors package

### what is it?

npm package with standardized error types for observable and actionable error flow.

### contract

**BadRequestError (4xx semantics):** [5]
```ts
throw new BadRequestError('user does not exist for uuid', { userUuid });
```

**UnexpectedCodePathError (5xx semantics):** [5]
```ts
throw new UnexpectedCodePathError('no tire pressures found', { tires });
```

### best practices

- use `BadRequestError` when request is logically invalid (caller's fault) [5]
- use `UnexpectedCodePathError` when code reaches impossible state (our fault) [5]
- include context metadata for debug [5]

### citations

[5] helpful-errors npm package: https://www.npmjs.com/package/helpful-errors

> "leverage the BadRequestError to make it clear when your logic successfully rejected a request"
> "The UnexpectedCodePath error is probably the most common type of error you'll throw... this is definitely an unexpected code path for our application"

## signal map: rhachet to provider format

| rhachet signal | anthropic | openai |
|----------------|-----------|--------|
| `'success'` | `is_error: false` (default) | normal content |
| `'error:constraint'` | `is_error: true` | error in content |
| `'error:malfunction'` | `is_error: true` | error in content |

note: openai has no explicit error flag — errors are encoded in content. anthropic has `is_error` boolean that maps to both constraint and malfunction signals.

---

## access pattern

no direct remote access at contract level:
- `BrainPlugToolExecution` is a data contract
- suppliers translate to/from provider formats
- callers execute tools locally and compose results

the contract provides a uniform shape; suppliers handle provider-specific translation.
