## assumption 1: suppliers can translate BrainPlugToolExecution to provider formats

**what do we assume?**
- anthropic supplier can convert execution → `tool_result` block
- openai supplier can convert execution → `{role: "tool"}` message
- translation is straightforward

**evidence supports this?**
- both providers have well-documented tool result formats
- execution shape includes `exid` which maps to provider correlation ids
- shape is minimal and contains only universal fields

**what if the opposite were true?**
- if providers had incompatible requirements, we'd need provider-specific fields
- execution contract would bloat with optional provider fields

**did wisher say this, or did we infer?**
- wisher said "easily support users of this contract to compose brain exchanges"
- we inferred supplier translation is implicit

**verdict:** ✅ assumption is reasonable. providers have compatible formats.

## reflection 🪷

upon slow, careful re-review of all 20 assumptions, i found one inconsistency and several insights.

### inconsistency found and fixed

assumption 5 said `prompt: string | execution | execution[]` but the vision now specifies `prompt: string | BrainPlugToolExecution[]` (always array). this was corrected above. the vision refinement resolved the three-shapes confusion — always-array is clearer.

### why the atom/repl distinction matters

atoms are stateless inference calls. callers must handle tool loops because atoms have no memory between calls. this is a feature: atoms are composable building blocks.

repls are stateful coordinators. they maintain series, execute tools, chain calls. the execute operation enables this coordination. without it, repls would just be atoms with extra state — no actual coordination power.

the type parameter `'atom' | 'repl'` isn't just cosmetic — it enforces this architectural boundary:
- atom tools omit execute (callers handle execution)
- repl tools require execute (repl chains via execute)
- repl tools fit into atom tools (execute becomes inaccessible by design)

### why three-way signal works

the signal field emerged from a real need. the wisher asked about error representation. three states map to how tools fail:

| signal | semantics | brain action |
|--------|-----------|--------------|
| success | worked | continue |
| error:constraint | expected failure (4xx) | try different approach |
| error:malfunction | unexpected failure (5xx) | tool is broken |

more granular signals (timeout, auth, rate-limit) would add complexity. the output can contain details; the signal tells the brain how to reason.

### remaining risks

three assumptions still carry medium risk:
1. **json-serializable outputs** — binary outputs need user handling
2. **synchronous tools** — async/long-run tools excluded from v1
3. **hallucinated slugs** — validation not specified (gap)

these are scope limits, not blockers. they're documented as future work.

### conclusion

no blockers found. vision aligns with wish and session refinements. the inconsistency was corrected.
