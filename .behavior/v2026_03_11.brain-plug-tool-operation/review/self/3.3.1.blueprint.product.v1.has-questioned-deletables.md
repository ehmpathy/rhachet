## conclusion

### what i found

**deleted:** ConstraintError and MalfunctionError exports from sdk.brains.ts

these were aliases for BadRequestError and UnexpectedCodePathError from helpful-errors. i questioned: "if we deleted these and had to add them back, would we?" the answer was no. users can import directly from helpful-errors. the aliases add a layer of indirection without meaningful semantic benefit.

**retained:** all other components

each remaining component serves the core wish:
- `BrainPlugToolExecution` — the typed contract for tool results (core deliverable)
- `BrainPlugToolDefinition.execute` — enables repl to run tools internally (usecase.3)
- prompt union type — enables tool results as continuation (usecase.10)
- test factories and tests — required for proof of behavior

i considered whether `genMockedBrainPlugToolInvocation.ts` could be deleted (inlined in tests). i would add it back for consistency with the extant genMocked* pattern. the pattern is established and valuable.

i considered whether `BrainPlugToolDefinition.execute` could be deleted. callers could handle tool loops manually. but the vision explicitly requires repl to execute tools via .execute() for the agentic loop usecase. if deleted, usecase.3 would not be satisfied.

### the simplest version

the blueprint after this review is the simplest version that satisfies the wish and criteria:
- one new domain object
- type extensions to extant domain objects
- test coverage mapped to criteria

no optimization was added prematurely. no abstraction beyond what is requested.
