## second pass: found an issue

re-read the blueprint line by line with fresh eyes.

**issue found: name field is required, not optional**

blueprint line 73 shows:
```
├── [+] name: string
```

no `?` suffix - this means `name` is **required**.

extant shape is `{ slug, description, schema }`. new shape adds required `name` field. extant code that constructs tool definitions would fail type check until `name` is added.

**is this a real backwards compat concern?**

examined the codebase: BrainPlugToolDefinition is used in rhachet brains, an unreleased package. there are no external consumers. the "breaking" change affects only internal code which will be updated in the same PR.

**should name be optional instead?**

considered making it `name?: string` for backwards compat. rejected because:
1. vision explicitly shows `name: string` as required (line 282)
2. display name is meaningful - tools without names are confusing
3. no external consumers to break

**verdict: not a backwards compat PROBLEM, but worth noting**

the change is intentionally breaking within the package. extant tool definitions will be updated to include name. this is acceptable for an unreleased package.
