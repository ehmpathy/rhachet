## assumption 1: BrainPlugToolExecution extends DomainLiteral

**the assumption:** execution results should be DomainLiteral (immutable, identity by all properties)

**what if the opposite?** could be a plain interface or a DomainEntity with exid as unique key

**evidence:**
- vision defines execution as a value object: same exid+slug+input+output = same execution
- no mutable state after creation
- no lifecycle beyond the request-response

**verdict:** DomainLiteral is correct. execution results are immutable values, not entities with lifecycle.

---

## assumption 2: prompt union is always array for executions

**the assumption:** `prompt: string | BrainPlugToolExecution[]` — single execution must be wrapped in array

**what if the opposite?** could be `string | BrainPlugToolExecution | BrainPlugToolExecution[]`

**evidence:**
- vision explicitly says "always array for tool results"
- parallel tool calls are common (multiple executions at once)
- single-vs-many ambiguity is a source of bugs
- providers (anthropic, openai) accept arrays of tool results

**verdict:** always-array is correct. removes ambiguity, matches provider contracts.

---

## assumption 3: execute field conditional on TBrainGrain

**the assumption:** `execute?: TBrainGrain extends 'repl' ? ... : never`

**what if the opposite?** execute could be available for atoms too

**evidence:**
- vision clearly separates atom vs repl:
  - atoms: "invoke atomic brain exchanges; tool execution is caller's job"
  - repls: "coordinate tool use into coherent responses; execute enables chained tool use"
- atom callers need control over tool execution (parallel, retry, etc.)
- repl handles the loop internally

**verdict:** conditional execute is correct. reflects architectural difference between atom and repl.

---

## assumption 4: signal uses three-state enum

**the assumption:** `signal: 'success' | 'error:constraint' | 'error:malfunction'`

**what if the opposite?** could be boolean `success: true|false`, or include more states

**evidence:**
- vision maps to http semantics (2xx, 4xx, 5xx)
- anthropic uses `is_error: boolean` — we are more expressive
- constraint vs malfunction distinction helps brain reason:
  - constraint: "try different approach"
  - malfunction: "tool is broken"

**what exceptions?** could add `'error:timeout'` or `'error:quota'`. but these are malfunctions (unexpected failures). three states are sufficient for now.

**verdict:** three-state signal is correct. more expressive than boolean, sufficient for current usecases.

---

## assumption 5: metrics only includes time

**the assumption:** `metrics: { cost: { time: IsoDuration } }`

**what if the opposite?** could include `tokens`, `bytes`, `retries`, `cash`

**evidence:**
- vision notes: "only time initially; may need cash, tokens later"
- tool execution is external to brain inference — no tokens
- time is universally measurable
- users can extend with wrappers if needed

**verdict:** time-only is correct for MVP. extensible if needed later.

## assumption 7: AsBrainOutputCallsFor returns null for repl

**the assumption:** `TGrain extends 'repl' ? null : ...`

**what if the opposite?** repl could return calls if it wants caller to handle

**evidence:**
- vision: "repl returns final output with no tool calls — repl executed tools via .execute()"
- repl handles the loop internally
- if repl returns calls, it would break the agentic abstraction

**verdict:** null for repl is correct. repl's contract is to handle tools internally.

## assumption 8: prompt union unconditional on TPlugs

**the assumption:** `prompt: string | BrainPlugToolExecution[]` accepts executions even without tools plugged

**what if the opposite?** could be conditional: only accept executions when TPlugs has tools

**evidence:**
- vision shows execution continuation requires tools plugged
- if no tools plugged, executions would be meaningless

**issue found:** the prompt type should be conditional on TPlugs

**fix needed:** update prompt type to:
```ts
type AsBrainPromptFor<TPlugs> = TPlugs extends { tools: BrainPlugToolDefinition[] }
  ? string | BrainPlugToolExecution[]
  : string;
```

## assumption 9: context parameter is unknown

**the assumption:** `execute: (input, context: unknown) => Promise<TOutput>`

**what if the opposite?** context could be typed via a generic TContext

**evidence:**
- tools need access to services (e.g., stripe, fs)
- `unknown` requires unsafe casts in tool implementations

**is this based on evidence or habit?** habit. we defaulted to unknown for simplicity.

**could simpler approach work?** yes, but adding TContext increases generic complexity (4 type params)

**verdict:** keep unknown for now. tools can use type guards or casts. note as future improvement.
