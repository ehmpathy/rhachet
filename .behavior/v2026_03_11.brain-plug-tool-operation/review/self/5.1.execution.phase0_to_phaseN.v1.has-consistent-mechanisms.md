# self-review: has-consistent-mechanisms

## approach

searched for related codepaths and compared new mechanisms against extant patterns:
1. domain object structure (interface + class + DomainLiteral)
2. test factory pattern (genMocked* convention)
3. metrics structure (cost.time pattern)
4. conditional type pattern (As* prefix, extends ternary)

## findings

### domain object pattern: consistent with BrainPlugToolInvocation

compared `BrainPlugToolExecution.ts` against `BrainPlugToolInvocation.ts`:

| aspect | BrainPlugToolInvocation | BrainPlugToolExecution |
|--------|-------------------------|------------------------|
| file structure | interface + class | interface + class |
| inheritance | `extends DomainLiteral<Self>` | `extends DomainLiteral<Self>` |
| implements | `implements Self` | `implements Self` |
| JSDoc style | `.what`, `.why`, `.note` | `.what`, `.why`, `.note` |
| generic defaults | `<TInput = unknown>` | `<TInput = unknown, TOutput = unknown>` |
| shared fields | exid, slug, input | exid, slug, input (plus output, signal, metrics) |

the new domain object follows the same structure exactly.

### metrics structure: consistent with BrainOutputMetrics

compared `BrainPlugToolExecution.metrics` against `BrainOutputMetrics`:

| aspect | BrainOutputMetrics | BrainPlugToolExecution.metrics |
|--------|-------------------|-------------------------------|
| time field | `cost.time: IsoDuration` | `cost.time: IsoDuration` |
| cash field | `cost.cash: { total, deets }` | not present (intentional) |
| size field | `size: { tokens, chars }` | not present (intentional) |

rationale for minimal metrics: tools are external to the brain. we cannot measure their token/cash costs. time is universally measurable. this matches vision section "metrics feels minimal" which acknowledges the trade-off.

### conditional type pattern: consistent with As* types

compared `AsBrainPromptFor` against extant `As*` types in BrainOutput.ts:

| type | pattern |
|------|---------|
| `AsBrainOutputSeriesFor<T>` | `T extends 'repl' ? BrainSeries : ...` |
| `AsBrainOutputCallsFor<T, G>` | `G extends 'repl' ? null : ...` |
| `AsBrainOutputOutputFor<T, P>` | `P extends { tools: ... } ? T \| null : T` |
| `AsBrainPromptFor<P>` | `P extends { tools: ... } ? string \| Execution[] : string` |

all use:
- `As*For` suffix convention
- `extends` ternary conditional
- explicit fallback branches

the new type follows the same pattern.

### test factory pattern: consistent with genMocked* convention

compared `genMockedBrainPlugToolExecution.ts` against `genMockedBrainOutputMetrics.ts`:

| aspect | extant pattern | new factory |
|--------|---------------|-------------|
| function name | `genMocked{DomainObject}` | `genMockedBrainPlugToolExecution` |
| input param | `input?: { field?: Type; ... }` | `input?: { exid?: string; ... }` |
| defaults | `input?.field ?? defaultValue` | `input?.exid ?? '__mock_exid__'` |
| return | `new DomainObject({ ... })` | `new BrainPlugToolExecution({ ... })` |
| JSDoc | `.what` and `.why` | `.what` and `.why` |

the new factory follows the same structure exactly.

### no duplicate mechanisms found

searched for potential duplications:

| mechanism | search | result |
|-----------|--------|--------|
| signal pattern | `grep "success.*error"` | only in new BrainPlugToolExecution (new concept) |
| time metrics | `grep "metrics.*cost.*time"` | only in tests and new execution (uses extant IsoDuration) |
| conditional types | `grep "type As"` | only in BrainAtom.ts and BrainOutput.ts (follows pattern) |

no duplicate utilities, factories, or patterns found.

## questions considered

**Q: could `BrainPlugToolExecution.metrics` reuse `BrainOutputMetrics`?**

no. BrainOutputMetrics includes size (tokens, chars) and cash cost which are brain-specific. tool execution only measures time. different shape, different purpose.

**Q: could `AsBrainPromptFor` be combined with other `As*` types?**

no. it serves a different purpose (prompt input) vs the others (output shape). separation keeps types focused.

**Q: could `genMockedBrainPlugToolExecution` reuse `genMockedBrainPlugToolInvocation`?**

no. execution has additional fields (output, signal, metrics) that invocation does not have. they share some fields but are distinct domain objects.

## conclusion

all new mechanisms follow extant patterns. no duplicates found. the implementation is consistent with codebase conventions.
