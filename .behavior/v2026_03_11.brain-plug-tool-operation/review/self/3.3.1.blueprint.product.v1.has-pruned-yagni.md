## conclusion

### what i found

every component in the blueprint traces to an explicit requirement in wish, vision, or criteria. no features were added "for future flexibility" or "while we're here".

the most likely yagni candidate was `genMockedBrainPlugToolInvocation` — a new factory not explicitly requested. i retained it because:
1. it follows extant genMocked* pattern in the codebase
2. test coverage is required per criteria
3. factory enables composable test setup without duplication

### what was already pruned

in the deletables review, two components were deleted:
- ConstraintError export — users can import BadRequestError directly
- MalfunctionError export — users can import UnexpectedCodePathError directly

these were "nice to have" synonyms without explicit requirement.

### why no other components warrant deletion

each component maps 1:1 to requirements:
- BrainPlugToolExecution — wish core deliverable
- generics — wish explicit ask
- execute conditional — vision explicit architecture
- signal enum — vision explicit design choice
- metrics.cost.time — wish explicit field
- AsBrainOutputCallsFor TGrain — vision type constraint
- AsBrainPromptFor — vision prompt union semantics
- genMocked* factories — extant pattern for test coverage

the blueprint is minimum viable for the declared behavior.

## second pass: fresh eyes

re-read the blueprint line by line. questioned each component again.

**one item i scrutinized harder: genMockedBrainPlugToolInvocation**

this factory creates test invocations. but BrainPlugToolInvocation already exists and is simple - just exid, slug, input. could tests construct invocations inline?

verdict: factory is still justified because:
1. the extant pattern uses genMocked* factories consistently
2. inline construction requires the class import and default knowledge
3. tests read cleaner with `genMockedBrainPlugToolInvocation({ slug: 'my-tool' })`

**what i did NOT find**

no abstractions "for future flexibility":
- no generic TContext on execute (kept as `unknown`)
- no metrics beyond time (no tokens, cash, bytes)
- no helper utilities beyond the two factories

no features "while we're here":
- no error type exports (ConstraintError, MalfunctionError were deleted in deletables review)
- no convenience wrappers like `toExecution(invocation, output)`
- no validation utilities

the blueprint restrains itself to exactly what is needed.
