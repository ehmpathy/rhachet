## requirement 3: "prompt: string | BrainPlugToolExecution"

**who said this?** the wisher

**alternatives considered:**
1. **separate `toolResults` field** — `{ prompt: string, toolResults?: execution[] }`
   - pros: explicit, no union confusion
   - cons: larger api surface, two places to check

2. **separate `continue` method** — `brain.continue({ executions })`
   - pros: semantic clarity
   - cons: breaks the simple ask/act pattern

3. **union on prompt** — `prompt: string | execution | execution[]`
   - pros: minimal api surface, continuation feels natural
   - cons: three shapes in one field may surprise users

**evidence for union approach:**
- `on: { episode }` already handles continuation context
- tool results are conceptually "the next prompt" from the brain's perspective
- provider apis already treat tool results as message content

**what if we used separate field instead?**
- clearer contract but more api surface
- would need to handle `prompt + toolResults` ambiguity

**verdict:** ✅ union approach is defensible, but worth noting as awkward (vision does this)

## requirement 4: "BrainPlugToolExecution shape: { exid, input, output, metrics }"

**who said this?** the wisher specifies:
```
exid: // continuation id from brain atom
input: // what it was called with
output: // what the tool returned
metrics: { cost: { time } }
```

**questions about the shape:**

### is `input` necessary?

the invocation already contains the input. why duplicate?

**rationale:** symmetry and self-containment. the execution is a complete record of what happened. callers don't need to correlate back to invocation for context.

**verdict:** ✅ keep `input` for self-contained records

### is `slug` necessary? (added in vision, not in wish)

vision adds `slug` which wasn't in wish.

**rationale:** defensive validation. suppliers can verify execution matches expected tool type, not just correlation id.

**should validate with wisher?** yes — `slug` was added beyond wish scope

### is `metrics.cost.time` sufficient?

wish specifies only `time`. vision notes this feels minimal.

**alternatives:**
- `metrics: { cost: { time, cash? }, size?: { bytes? } }`
- keep minimal, let users extend

**verdict:** ✅ start minimal, but flag for wisher input on expansion

## requirement 7: "signal: 'success' | 'error:constraint' | 'error:malfunction'" (added via session)

**who said this?** human (wisher) in session refinement

**rationale:**
- tools can fail in two distinct ways: expected (4xx) vs unexpected (5xx)
- brain should know the difference to decide how to proceed
- anthropic has `is_error` boolean — we add granularity

**evidence for three-way signal:**
- `success` maps to `is_error: false`
- `error:constraint` = 4xx, maps to `is_error: true` — brain might try different approach
- `error:malfunction` = 5xx, maps to `is_error: true` — brain knows tool is broken

**what if we used boolean instead?**
- lose distinction between expected and unexpected failures
- brain can't reason about retry vs abandon

**verdict:** ✅ justified. three-way signal adds value over boolean.

## requirement 8: "BrainPlugToolDefinition.execute" (added via session)

**who said this?** human (wisher) in session refinement

**rationale:**
- BrainRepl needs to execute tools automatically in agentic loop
- without .execute, repl can't chain tool calls
- caller shouldn't need to handle tool loop for repls

**evidence:**
- repl.act() should be fire-and-forget: pass tools, get final result
- repl internally calls tool.execute() to chain
- atoms don't need this — callers handle tool execution

**what if we didn't add .execute?**
- repls would need callers to handle tool loops (defeats purpose)
- no distinction between atom and repl tool usage

**verdict:** ✅ justified. execute enables repl's coordinated tool use.

## requirement 9: "type-level atom/repl distinction" (added via session)

**who said this?** human (wisher) in session refinement

**rationale:**
- atoms output tool invocations (caller handles)
- repls execute tools internally (calls always null)
- type system should enforce this difference

**evidence for `'atom' | 'repl'` type parameter:**
- mirrors `BrainOutput<TOutput, TBrainType>`
- atom tools omit `execute` field
- repl tools require `execute` field
- repl tools fit into atom tools (execute becomes inaccessible)

**what if we used single type?**
- lose compile-time enforcement
- callers wouldn't know if they need to handle tool loops

**verdict:** ✅ justified. type-level distinction enforces architectural boundary.

## requirement 10: "ConstraintError and MalfunctionError exports" (added via session)

**who said this?** human (wisher) in session refinement

**rationale:**
- tool authors need semantic error types
- map to signal values: ConstraintError → error:constraint, MalfunctionError → error:malfunction
- re-export from helpful-errors with tool-context names

**evidence:**
- `ConstraintError` = `BadRequestError` (4xx semantics)
- `MalfunctionError` = `UnexpectedCodePathError` (5xx semantics)
- tool.execute() throws these, supplier maps to signal

**what if we didn't export these?**
- users would import helpful-errors directly (fine, but less semantic)
- signal map less obvious

**verdict:** ✅ justified. semantic exports clarify intent. `// todo: lift into helpful-errors` noted.

## requirement 11: "prompt: string | BrainPlugToolExecution[]" (refined via session)

**who said this?** human (wisher) in session refinement

**original:** `prompt: string | BrainPlugToolExecution`
**refined:** `prompt: string | BrainPlugToolExecution[]` (always array)

**rationale:**
- removes single-vs-many ambiguity
- parallel tool calls return array, single tool returns array of one
- simpler mental model: string to start, array to continue

**verdict:** ✅ justified. always-array removes edge case confusion.
