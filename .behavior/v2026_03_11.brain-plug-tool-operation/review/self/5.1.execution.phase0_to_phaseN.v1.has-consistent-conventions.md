# self-review: has-consistent-conventions

## approach

searched for extant name conventions and patterns via grep and file reads, then verified new names follow them:
1. domain object names (Brain*, BrainPlug* prefix)
2. conditional type names (As*For suffix)
3. field names (exid, slug, input, output, metrics)
4. test file names (*.test.ts suffix)
5. factory names and mock value patterns

## findings

### domain object names: consistent

**search**: `grep "extends DomainLiteral" src/domain.objects/` found 37 domain objects.

**extant tool-related pattern**: `BrainPlugTool{Role}` where Role describes the object's purpose in the tool flow.

| extant | what it represents |
|--------|--------------------|
| `BrainPlugToolDefinition` | what the tool is (declaration) |
| `BrainPlugToolInvocation` | when brain asks for tool (request) |

| new | what it represents | follows pattern? |
|-----|--------------------|------------------|
| `BrainPlugToolExecution` | what caller returns (result) | yes: `BrainPlugTool` + `Execution` |
| `BrainPlugToolExecute` | type alias for execute function | yes: `BrainPlugTool` + verb |

**verification**: the three tool-related domain objects form a coherent lifecycle:
- Definition → what the tool is
- Invocation → brain requests it
- Execution → caller returns result

the semantic progression mirrors extant patterns like BrainExchange (one request-response) vs BrainEpisode (chain of exchanges).

### signal values: new vocabulary, justified

**search**: `grep "'[a-z]+:[a-z]+'" src/` found only BrainPlugToolExecution uses colon-separated signals.

`signal: 'success' | 'error:constraint' | 'error:malfunction'` introduces new vocabulary.

**justification from vision**:
- `success` = tool completed normally (2xx analogy)
- `error:constraint` = business/input constraint (4xx analogy)
- `error:malfunction` = unexpected break (5xx analogy)

**questions considered**:

Q: why not use extant error types like `BadRequestError` names?
A: signals are data values, not error classes. the brain needs to reason about outcome type, not catch exceptions.

Q: why colon separator (`error:constraint`) instead of camelCase (`errorConstraint`)?
A: colon creates a namespace prefix pattern. `error:*` signals are clearly a family. this aids pattern matching and is consistent with namespaced identifier patterns used elsewhere (e.g., `node:fs`).

Q: is `signal` the right field name?
A: searched codebase for `status`, `state`, `result` - these are overloaded with other meanings. `signal` is fresh vocabulary that clearly conveys "outcome indicator".

## no divergences found

all new names follow extant conventions:
- domain object name: `BrainPlugToolExecution` follows `BrainPlugTool{Role}` pattern
- conditional type name: `AsBrainPromptFor` follows `As{Subject}For` pattern
- field names: all reuse extant vocabulary (exid, slug, input, output, metrics, cost.time)
- test file name: follows `{DomainObject}.test.ts` pattern
- factory name: follows `genMocked{DomainObject}` pattern
- mock values: follow `__mock_{name}__` pattern
- signal values: new vocabulary but justified and does not conflict

no new terms conflict with or shadow extant terms.

## conclusion

new types, fields, files, and factories follow extant name conventions. signal vocabulary is new but justified by semantic need and follows namespaced identifier pattern. no divergences found.
