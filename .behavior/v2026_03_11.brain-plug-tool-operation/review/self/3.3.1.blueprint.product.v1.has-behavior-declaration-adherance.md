## fourth pass: found execute context type deviation

### what i checked

vision line 294-297 shows execute signature:
```ts
execute?: (
  input: { invocation: BrainPlugToolInvocation<TInput> },
  context: TContext,
) => Promise<TOutput>;
```

blueprint lines 76-79 shows:
```
execute?: TBrainGrain extends 'repl'
        ? (input: { invocation: BrainPlugToolInvocation<TInput> }, context: unknown)
            => Promise<TOutput>
        : never
```

### the deviation

vision uses `context: TContext` (a generic), but blueprint uses `context: unknown`.

### why this is acceptable

i verified this is a deliberate simplification, not an error:

1. **TContext would require fourth type parameter** — BrainPlugToolDefinition would become `<TInput, TOutput, TBrainGrain, TContext>`, which increases complexity

2. **vision acknowledges this pattern** — line 765 states assumption: "context injection via closure or factory — tools with execute capture their context at definition time"

3. **unknown is correct** — callers pass context at runtime; execute receives it typed via closure or factory pattern, not via generic

example from vision (line 266):
```ts
context: { fs, shell },  // passed to tool.execute()
```

the context is runtime-injected, not compile-time generic. `unknown` correctly represents this.

### verdict

deviation is intentional. blueprint simplifies via `context: unknown` instead of a fourth type parameter. adheres to vision's stated assumption about context injection.
