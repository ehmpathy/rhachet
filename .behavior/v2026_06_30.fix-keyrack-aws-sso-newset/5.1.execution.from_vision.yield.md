# 5.1 execution — fix-keyrack-aws-sso-newset

## the fix (from the vision)

root cause: `@smithy/core/config` caches `~/.aws/config` contents in a process-lifetime
module-level `filePromises` map; `fromSSO` reads via that cache and keyrack passes no `ignoreCache`,
so a profile written earlier this run is invisible → `Profile not found`. (#402 introduced the
in-process `fromSSO` call in `set`.)

primary fix: pass `ignoreCache: true` to `fromSSO` so validation reads ground-truth.

## outcome

### change (prod)
`src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts:96`
```ts
await fromSSO({ profile: profileName, ignoreCache: true })();
```
+ a `.note` explaining the process-lifetime `~/.aws/config` cache in `@smithy/core/config`.

### change (test)
`vaultAdapterAwsConfig.test.ts` — captured the `fromSSO` init via a lazily-read
`mockFromSSOControl.lastInit` holder (dodges the jest.mock hoist TDZ), and added a `then` under
`[t1] isUnlocked with valid sso session` asserting `fromSSO` is invoked with
`{ profile, ignoreCache: true }`. this test fails on the pre-fix code (no `ignoreCache`) and passes
on the fix → institutionalizes the lesson per `rule.require.test-covered-repairs`.

### why this fixes it
`fromSSO` → `parseKnownFiles(init)` → `loadSharedConfigFiles(init)` → `readFile(path, { ignoreCache })`.
the loader caches `~/.aws/config` contents in a module-level `filePromises` map for the whole
process; a profile written earlier this run was served stale → `Profile not found`. `ignoreCache:
true` forces a fresh read — semantically correct for validation — and keeps the `fromSSO` trust guard
(no cli-swap, so the aws/aws-cli#9845 protection stays).

### verifiable here vs not
- **verified**: types, lint, unit (incl. the new regression test); the source-level cache mechanism.
- **not verifiable in sandbox**: the end-to-end guided `keyrack set` (needs browser auth + real aws
  accounts). recommend the wisher run one `KEYRACK_DEBUG=1` guided `set` on a fresh profile to confirm.

### residuals (carried from vision)
- the premature seed read (which in-process sdk call seeds the stale cache before the write) is not
  pinned — needs the runtime repro. the fix is robust regardless.
- `loadSsoSessionData` (the `[sso-session]` read) does not honor `ignoreCache`; the observed error is
  the earlier profile lookup so this fix addresses it, but worth a watch if a later gate fails.
