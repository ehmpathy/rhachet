# 1. vision — fix-keyrack-aws-sso-newset

> the wish: "why did this happen? the auth completed successfully but then we got this error"
> + (clarified by wisher): "on retry it works"

this is a **defect-fix** wish. the contract surface (`rhx keyrack set --key AWS_PROFILE --vault
aws.config`, guided tty setup) does not change — its **reliability on first run** does.

this wish is a **recurrence** of the 06_25 `fix-sso-set` defect, but with a sharper signal: the
error text is now `Profile X was not found` (config-lookup), **not** `sso session not valid`
(token validation). that distinction reframes the root cause.

---

## the outcome world

### day-in-the-life

a human onboards a fresh aws sso profile via guided setup:

```
➜ rhx keyrack set --owner ehmpath --env test --key AWS_PROFILE --vault aws.config
```

they pick domain → authenticate in browser → pick account → role → name it. the profile is
written to `~/.aws/config`, verified, and vaulted — **on the first try.** no
`✋ ConstraintError: sso session not valid`, no detour, no re-run.

### before / after

| | before (today) | after (wished) |
|---|---|---|
| first guided `set` on a fresh profile | ✋ `sso session not valid; run aws sso login` (real cause: `fromSSO` → `Profile X was not found`) | ✓ verified + vaulted |
| recovery ritual | re-run the exact same command (works 2nd time) | none needed |
| profile already present | ✓ works | ✓ works (unchanged) |

### the "aha"

> "i authenticated seconds ago, it wrote the profile, then said the profile didn't exist — but
> the very next run with the same command worked. now it just works the first time."

the value clicks when the **first** attempt succeeds — the whole point of guided setup is that the
user just authenticated and the CLI just proved the profile works (`aws sso login` + `aws sts
get-caller-identity` both succeeded in the same run). failing right after that is the surprise we delete.

## user experience

### usecases / goals

- **onboard** a brand-new aws sso profile for an env (`test`/`prep`/`prod`) in one guided pass
- **refill** an env's keys without a re-run ritual
- goal: a vaulted `AWS_PROFILE` key whose sso session is provably valid, on first run

### contract inputs & outputs (unchanged)

inputs:
- `rhx keyrack set --owner <owner> --env <env> --key AWS_PROFILE --vault aws.config` (guided, tty)
- `rhx keyrack fill --owner <owner> --env <env>` (loops guided set per key)

outputs (the guided tree):
- domain → with sso prior? → which sso login? → account → role → name → vault → verify
- success: `✓ written to ~/.aws/config` → `✓ unlock / ✓ get / ✓ relock`
- failure: a `ConstraintError` **only when the session is genuinely unrecoverable**, with a
  truthful, actionable hint

### timeline (wished)

```
🔐 keyrack set AWS_PROFILE
   ├─ which sso domain?        → pick
   ├─ with sso prior?          → clear / reuse
   ├─ which sso login?         → ✓ authenticated
   ├─ which account?           → pick
   ├─ which role?              → pick
   ├─ what should we call it?  → name
   ├─ lovely, lets vault it now... → ✓ written to ~/.aws/config
   └─ perfect, now lets verify...
        ├─ ✓ unlock        ← no "Profile not found" here on first run
        ├─ ✓ get
        └─ ✓ relock
```

## mental model

### how a user describes it to a friend

> "keyrack walks me through aws sso setup, logs me in, saves the profile, then checks the login
> works before vaulting it."

they do **not** think: "the in-process aws node-sdk and the aws cli disagree about whether a
profile that was written earlier in this same process exists." that's our problem to hide.

### analogy

a clerk who watches you sign the register, then turns to a photocopy of the register from *before*
you signed and says "i don't see your name." the cli read the live register; the sdk read a stale
copy. retry works because by then your name is already in the original.

### terms

| user says | we say |
|---|---|
| "log in" | `aws sso login` / sso device authorization |
| "save the profile" | `setupAwsSsoProfile` writes `~/.aws/config` (v2 `sso-session` block) |
| "check it works" | `validateSsoSession` → sdk `fromSSO` → `export-credentials` → `sts get-caller-identity` |
| "it failed" | `ConstraintError: sso session not valid` (surfacing an underlying `fromSSO` → `Profile X was not found`) |

## evaluation

### how well it solves the goal

solves it fully **iff** the root cause is the in-process CLI-vs-SDK divergence on a freshly-written
profile (see groundwork). a correct fix makes first-run guided setup succeed without changing the
contract, while preserving a true `ConstraintError` when the session genuinely can't be established.

### pros

- deletes a confusing, repeatable failure on the happy path
- removes a superstition ("just run it again") from team muscle memory
- aligns validation with the source that demonstrably works in-process (the CLI)
- keeps failfast: a genuine invalid/timed-out session must still `ConstraintError`

### cons / tradeoffs

- the fix lives at the validation seam (`validateSsoSession`); a wrong fix could mask a *real*
  invalid-session signal → must preserve true-negative behavior
- if we switch the first gate from sdk `fromSSO` to the cli, we *might* lose the "cache cannot be
  trusted" protection that `fromSSO` was added for (`vaultAdapterAwsConfig.ts:93-101`, cites
  aws/aws-cli#9845). but this is **not yet established** — the validation has two further cli gates
  (`export-credentials` `:194`, `sts` `:227`), and it is unproven that those miss the remote-revocation
  case #9845 describes. so "the `fromSSO` gate must be preserved" is itself a requirement to
  **re-justify or drop** in diagnosis, not a given (see Q2). the fix must keep a *trustworthy*
  validation — whether that requires `fromSSO` specifically is open.

### edgecases & pit-of-success

- **fresh profile, valid login** → must succeed (the bug)
- **fresh profile, login truly failed / timed out** → must still `ConstraintError` (failfast preserved)
- **profile already present** → unchanged ✓ (this is why retry works today)
- **`--exid` (non-guided) path** → **immune by construction, narrows fix scope**. `set` with
  `--exid` requires the profile to pre-exist (`checkProfileExists`, `vaultAdapterAwsConfig.ts:590-598`)
  and never writes a fresh profile mid-run, so it can never hit the written-this-run defect. the bug
  *and* the fix are therefore **guided-setup-only** — the fix must not perturb the `--exid` branch
- **username mismatch / domain collision** → extant clear-and-reauth path unchanged
- **mock mode (`AWS_SDK_MOCK=1`)** → sdk validation skipped (`vaultAdapterAwsConfig.ts:167`); fix
  must not break this branch
- **`keyrack fill` multi-key loop** → *engineer-added consideration, not in the wish* (the wish is a
  standalone `set`). called out because if the cause involves in-process SDK state, a fill loop (many
  sets in one process) is the worst case and the strongest place the memoization theory could bite;
  a seam-level fix covers it for free, so no fill-specific work is implied

## open questions & assumptions

### assumptions made

1. the trigger is **"profile freshly written this run"**, not any logout ritual. evidence: wisher
   confirms "on retry it works", and on retry the profile already exists on disk (written by the
   failed first attempt) so guided setup takes the "profile exists & equivalent" branch
   (`setupAwsSsoWithGuide.ts:385-396`). **confirmed by wisher (Q5):** the retry was the *identical*
   command, *same profile name*, immediate, with no logout — so the sole variable is *written this
   run* vs *present at process start*. assumption #1 holds.
2. the failure surfaces at `vaultAdapterAwsConfig.ts:603` (`set` → `validateSsoSession`, info level),
   which rethrows as the `ConstraintError` at `:606-611`.
3. the underlying throw is sdk `fromSSO({ profile })` → `Profile X was not found`
   (`vaultAdapterAwsConfig.ts:90`), classified as `CredentialsProviderError` → `{ valid: false }`
   at `:172-181` — confirmed by the wish's `🐈 ... gate invalid` log (`gate: fromSSO`).

### triaged questions

tagged [answered] (settled now), [research] (needs runtime repro), or [wisher] (only the human can answer).

- **Q1 — why does in-process sdk `fromSSO` report `Profile not found` for a profile the cli just
  used successfully in the same process?** → **[answered] via installed source.** the mechanism is a
  **process-lifetime in-memory cache of the config file contents** in
  `@smithy/shared-ini-file-loader/dist-cjs/readFile.js`:
  ```js
  exports.filePromises = {};                      // module-level, process-lifetime
  const readFile = (path, options) => {
    if (fileIntercept[path] !== undefined) return fileIntercept[path];
    if (!filePromises[path] || options?.ignoreCache) {
      filePromises[path] = fsReadFile(path, "utf8");   // caches the PROMISE, keyed by path
    }
    return filePromises[path];
  };
  ```
  `fromSSO` → `parseKnownFiles` → `loadSharedConfigFiles` → `readFile(path, { ignoreCache })`, and
  keyrack calls `fromSSO({ profile })` with **no** `ignoreCache`, so the cache is used. so if
  `~/.aws/config` is read via the sdk **before** the profile is written, the stale (profile-less)
  contents are served for the rest of the process → `Profile not found`. the cli is immune (each
  `aws` call is a fresh subprocess). **this also answers "would an in-process retry work?" → NO**
  (same cached promise); only a fresh process clears it — which is exactly the wisher's successful
  re-run.
  - residual **[research]**: locate *which* in-process sdk config read seeds the stale cache before
    the write in a standalone `set` (the cache existence is proven; the seed-read site is not yet
    pinned). candidate dimensions still worth a glance: the fragmented `@aws-sdk/*` tree (06_25's
    multiple `@aws-sdk/core` copies) could mean more than one `filePromises` cache instance.

- **Q2 — which fix?** → **primary candidate now source-confirmed.** the chain
  `fromSSO(init)` → `parseKnownFiles(init)` → `loadSharedConfigFiles(init)` → `readFile(path, {
  ignoreCache: init.ignoreCache })` threads `init.ignoreCache` end-to-end (verified in
  `@aws-sdk/credential-provider-sso@3.972.54` `:119` and `@smithy/core@3.25.1/config` `:261-330`).
  so the minimal fix is **one line** at `vaultAdapterAwsConfig.ts:90`:
  `fromSSO({ profile: profileName, ignoreCache: true })()` — forces a fresh read, bypasses the stale
  `filePromises` cache. it is *semantically correct* (validation wants ground-truth, not a snapshot)
  and keeps the `fromSSO` trust guard (unlike the cli-swap option), so it does **not** trade away the
  #9845 protection.
  - **caveat 1**: symptom-targeted — the premature seed read still exists; pair with a solve-at-cause
    pass to find/remove it (residual of Q1).
  - **caveat 2**: `loadSsoSessionData` (the `[sso-session]` read at `fromSSO` `:130`) does **not**
    honor `ignoreCache`; the observed error is the earlier profile lookup so this fix addresses it,
    but a later sso-session gate could still serve stale data in an edge.
  - **alternative**: validate via cli `sts`/`export-credentials` (fresh subprocess) — also works, but
    trades away the in-process `fromSSO` trust guard. the `ignoreCache` one-liner is preferable.

- **Q3 — does the wisher want a code fix now, or first a confirmed diagnosis?** → **[wisher].** the
  06_25 route shipped diagnostics precisely because the fix shape was unconfirmed; we now have a
  sharper signal but still no in-sandbox repro.

- **Q4 — `aws --version` (cli) on the wisher's machine?** → **[wisher].** rules in/out a cli-side
  parser quirk vs an sdk-side one.

- **Q5 — did the successful retry use the *identical* command and *same profile name*?** →
  **[answered] — wisher confirmed:** "no logout in between, just direct retry immediately." same
  command, same profile name, immediate re-run. this **locks assumption #1**: the sole difference
  between the failed run and the successful retry is *profile written this run* vs *profile already on
  disk at process start*. no other variable changes (no logout, no name change, no `--exid`).

- **Q6 — is this a *recurrence* or a *regression introduced by the prior fixes*?** →
  **[research] — partially answered now via git.** checked the commits: **#402** (`10d8e63`, "reuse
  sso session for same domain+user") explicitly "store awsSsoUsername in KeyrackKeyHost.meta on
  keyrack set" via a new "validate-before-login flow", and changed `vaultAdapterAwsConfig.ts` by ~283
  lines. that maps directly to the current `set` step at `:600-612` — "capture awsSsoUsername from ARN
  BEFORE roundtrip" via `validateSsoSession(profileName, { onInvalidLevel: 'info' })`, which is the
  exact line that throws `Profile not found`. **this is a confirmed regression introduced by
  #402**: the #402 diff changes `set`'s return from `{ mech, exid }` to `{ mech, exid, meta: {
  awsSsoUsername: stsResult.username } }`, where `stsResult` is the **new** post-write
  `validateSsoSession` capture. before #402, `set` ran no in-process `fromSSO` on a freshly-written
  profile; #402 added it to source the username. the profile *write* did not move (still in guided
  setup); #402 only *added* a post-write in-process validation. **diff now read** (`git show 10d8e63 -- vaultAdapterAwsConfig.ts`):
  #402 reworked `validateSsoSession`/`unlock`/`isUnlocked` and added a `.note` on
  `validateSsoTokenWithAwsSdk` — "call `aws sso login --profile X` BEFORE this function … only then
  can `fromSSO` find the cached profile *credentials*." key insight: that mitigation targets **absent
  cached credentials**, but this wish's error is `Profile not found` = a **config-section lookup**
  failure, a *distinct* class the login-first note does not address. so #402 both (a) added the
  in-process `fromSSO` call in `set` and (b) mitigated only the credentials case, which left the
  config-lookup case exposed. residual [research]: the precise sdk-internal reason the config section
  is missed in-process (needs repro).

### what must we validate with the wisher before proceeding

- confirm Q2 tradeoff (cli validation vs `fromSSO` cache-distrust) is acceptable — this is the one
  genuine blocker on fix shape
- a repro run with `KEYRACK_DEBUG=1` (which gate + precise `fromSSO` error) is a **strong
  preference, not a hard gate**: the wish log already shows `gate: fromSSO` and "retry works", so
  the cli-validation fix is justifiable on current evidence; the repro would *confirm* memoization
  vs config-lookup and refine the fix, rather than unblock it

## groundwork

### external research

- **aws sso (cli + node sdk)** — referenced indirectly (this is aws sso plumbing). verified the
  code paths and how each validates:
  - cli proves the profile in-run: `aws sso login` (`setupAwsSsoProfile.ts:101`) + `aws sts
    get-caller-identity` (`:123`) — both succeed per the wish (`✓ written to ~/.aws/config`).
  - sdk validates the session: `fromSSO({ profile })` (`vaultAdapterAwsConfig.ts:87-91`) — throws
    `Profile X was not found` (the defect).
  - codebase already documents why `fromSSO` exists (cli cache cannot be trusted), citing
    aws/aws-cli#9845 (`vaultAdapterAwsConfig.ts:93-101`).
  - **not runtime-verified**: the precise sdk-internal reason it misses a same-process freshly-written
    profile — needs repro (diagnosis stone).

### internal research

extant behavior this fix extends — verified by reading:

- **the failure seam**: `validateSsoSession` first gate = sdk `fromSSO`
  (`vaultAdapterAwsConfig.ts:155-188`); `set` calls it at `:603` and rethrows as `ConstraintError`
  at `:606-611` with `hint: run: aws sso login --profile X`.
- **return contract to conform to**: `{ valid: true; username } | { valid: false }`
  (`vaultAdapterAwsConfig.ts:155-158`).
- **the cli path that works in-process**: `aws sso login` + `aws sts get-caller-identity`
  (`setupAwsSsoProfile.ts:101,123`); also used as gates 2+3 of `validateSsoSession`
  (`vaultAdapterAwsConfig.ts:194,227`).
- **prior-suspect ruled out — env contamination**: `initiateAwsSsoAuth` scopes `AWS_CONFIG_FILE`
  to the **child** env only and cleans up the temp dir (`setupAwsSsoProfile.ts:388,453-456`);
  `listAwsSsoAccounts` scopes `AWS_PROFILE` to the **child** env only (`listAwsSsoAccounts.ts:18`).
  neither mutates `process.env`. so the in-process `fromSSO` reads the real `~/.aws/config`, which
  rules out "AWS_CONFIG_FILE points to the temp file."
- **temp-session token cache**: browser auth runs under a temp `keyrack-auth` sso-session
  (`setupAwsSsoProfile.ts:370-388`); the real profile uses its own `sso-session` block
  (`:82-93`). this was the 06_25 hypothesis; the new `Profile not found` error points at config
  **lookup**, not token cache — so this is likely a secondary factor, not the cause.
- **mock-mode branch to preserve**: `AWS_SDK_MOCK` skip (`vaultAdapterAwsConfig.ts:167`).
- **diagnostics already in place** (06_25): `emitSsoGateInvalid` logs `gate` + `ssoCacheFiles`
  (`vaultAdapterAwsConfig.ts:134-153`) — the `🐈 ... gate invalid` line in the wish is this.

### contrast with the 06_25 yield (what changed)

- 06_25 saw `sso session not valid` and theorized a **temp-session token-cache-key** mismatch,
  unable to repro. this wish shows `Profile X was not found` — a **config-lookup** failure at
  the same `fromSSO` gate, plus the decisive new datum "retry works." that reframes the cause from
  "token cached under wrong session key" to "in-process sdk can't see a profile the cli can."

## what is awkward?

- **two sources of truth for "does this profile exist / is its session valid?"** — the cli (used
  throughout setup) and the in-process sdk (used only for the first validation gate). they disagree
  exactly on a profile written earlier in the same process. the design fights the user's mental
  model: "i just made it, the tool just used it, and now the tool says it doesn't exist."
- **the `fromSSO` gate is a double-edged sword** — it was added because cli cache can't be trusted
  (aws/aws-cli#9845), but it's the very gate that misfires here. removing it to fix this bug risks
  reintroducing the bug it was added to prevent. the fix must validate *trustworthily*, not just
  *successfully*.
- **diagnosis before fix shape** — the exact sdk-internal cause is still not runtime-confirmed.
  this vision is honest about that; to commit a fix shape before a repro (or wisher sign-off on the
  cli-validation tradeoff) risks a symptom mask.
- **the fix-layer assumption (solve-at-cause tension)** — this vision leans toward a fix at the
  *validation-logic* layer (route the first gate through the cli). but if the true cause is the
  **fragmented `@aws-sdk/*` dependency tree** (06_25's lockfile "find 2": multiple `@aws-sdk/core`
  copies), then a cli-validation change is a *symptom mask*, not a cause fix — `fromSSO` would stay
  broken for any other in-process caller. per `rule.require.solve-at-cause`, diagnosis must pick the
  *layer* (logic vs dependency tree) before the fix shape, else we paper over a deps defect with a
  validation workaround. the vision must not pre-bias toward the logic-layer fix just because it is
  smaller.
