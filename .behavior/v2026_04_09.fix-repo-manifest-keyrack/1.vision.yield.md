# vision: `rhachet repo compile`

## the outcome world

### before

role authors maintain a fragile rsync incantation in package.json:

```json
"build:complete:dist": "rsync -a --prune-empty-dirs --exclude='.test' --exclude='.route' --exclude='.scratch' --exclude='.behavior' --exclude='*.test.*' --include='*/' --include='briefs/**' --include='skills/**' --include='inits/**' --include='templates/**' --include='**/readme.md' --include='**/boot.yml' --include='**/keyrack.yml' --exclude='*' src/ dist/"
```

when rhachet adds a new artifact type (like keyrack.yml), every role package must update their rsync. forget to update? your role ships without keyracks. users hit cryptic errors. you push a patch release.

### after

role authors use a single command:

```json
"build:complete:dist": "npx rhachet repo compile --from src --into dist"
```

rhachet reads the repo manifest, finds registered roles, and copies each role's artifacts. rhachet owns the knowledge of what role artifacts matter. when rhachet adds keyrack.yml support, extant roles automatically include it on next build. no package.json changes. no forgotten artifacts.

### the "aha" moment

> "wait, i don't have to maintain that rsync anymore? and new artifact types just work?"

the value clicks when a role author realizes:
1. they can delete 3 lines of rsync gibberish
2. they'll never forget to include a new artifact type
3. rhachet evolves; their build command stays stable

## user experience

### usecase: build a role package

**goal:** prepare role for npm publish

**contract:**
```bash
npx rhachet repo compile --from src --into dist
```

**behavior:**
1. discover roles via same internal logic as `rhachet repo introspect`
2. for each role's directory (e.g., `.agent/repo=.this/role=mechanic/`):
   - apply globs to find artifacts (briefs/**, skills/**/*.sh, etc.)
   - copy matched files to `--into` with same relative structure
3. apply default exclusions (test files, hidden dirs, etc)

**what gets included (per role):**

rhachet discovers each role's registered directories (from the same internal logic as introspect), then applies suffix globs:

| artifact | source | globs |
|----------|--------|-------|
| briefs | role's registered briefs dirs | `**/*.md`, `**/*.min` |
| skills | role's registered skills dirs | `**/*.sh`, `**/*.jsonc`, `**/template/**`, `**/templates/**` |
| inits | role's registered inits dirs | `**/*.sh`, `**/*.jsonc` |
| readme | role directory | `readme.md` |
| boot | role directory | `boot.yml` |
| keyrack | role directory | `keyrack.yml` |

users can use full path globs in `--include`/`--exclude` because they know their own directory structure.

**what gets excluded (by default):**
- `.test/**` — test fixtures
- `__test_*/**` — test assets
- `.route/**` — behavior routes
- `.scratch/**` — scratch files
- `.behavior/**` — behavior definitions
- `*.test.*` — test files
- `.*` — hidden directories

### usecase: customize inclusions

**goal:** include additional artifacts (e.g., data files)

```bash
npx rhachet repo compile --from src --into dist --include 'data/**/*.json'
```

### usecase: customize exclusions

**goal:** exclude vendor-specific files

```bash
npx rhachet repo compile --from src --into dist --exclude 'vendor/**'
```

### usecase: mixed include/exclude overrides

**goal:** include test skills but exclude test inits

```bash
npx rhachet repo compile --from src --into dist --include 'skills/**/*.test.sh' --exclude 'inits/**/*.test.sh'
```

note: by default, `*.test.*` files are excluded. this example shows how to selectively override that for skills while the exclusion remains for inits.

### timeline

1. role author runs `npm run build` (which includes tsc + rhachet repo compile)
2. rhachet repo compile reads `rhachet.repo.yml` to find declared roles
3. for each role, rhachet repo compile copies its artifacts to dist/
4. author runs `npm publish`
5. consumers install and get complete roles with all artifacts

## mental model

### how users describe it

> "it's like rsync but it knows what role files are"

> "rhachet repo compile is to roles what tsc is to typescript — it knows the file types"

### symmetry with `rhachet repo introspect`

| command | what it does |
|---------|--------------|
| `rhachet repo introspect` | discovers roles from `.agent/` → writes `rhachet.repo.yml` |
| `rhachet repo compile` | discovers roles from `.agent/` → copies artifacts to dist/ |

both commands share the same internal discovery logic. no order of operations required — introspect and compile can run independently.

### analogies

| concept | analogy |
|---------|---------|
| rhachet repo compile | tsc for role artifacts |
| default includes | .gitignore but inverted (allowlist) |
| --include/--exclude | like rsync, familiar pattern |

### terms

| user term | rhachet term |
|-----------|--------------|
| "build the role" | `rhachet repo compile` |
| "role files" | "role artifacts" |
| "the dist" | "compilation target" |

## evaluation

### how well does it solve the goals?

| goal | solution |
|------|----------|
| don't forget keyrack.yml | eliminated — rhachet owns the include list |
| don't maintain rsync | eliminated — one command replaces it |
| extensible for future artifacts | solved — rhachet adds new types centrally |

### pros

- **pit of success** — correct by default, hard to mess up
- **single source of truth** — rhachet defines what matters
- **forward compatible** — new artifact types auto-include
- **familiar contract** — --include/--exclude like rsync

### cons

- **new dependency** — roles depend on rhachet at build time (but they already do at runtime)
- **magic defaults** — user must trust rhachet's choices (but can override)

### edgecases

| edgecase | handling |
|----------|----------|
| custom artifact type | use --include |
| conflicting include/exclude | exclude wins (explicit beats default) |
| empty src dir | no-op, exit 0 |
| missing --into | fail-fast, require explicit target |
| --into outside repo | fail-fast, security boundary |
| no package.json in cwd | fail-fast, must run from package root |
| no .agent/ directory found | fail-fast, require .agent/ structure |
| no roles discovered in .agent/ | fail-fast, no roles to compile |
| role artifact path absent | fail-fast with clear error per role |
| extant dist/ contents | preserve — add files, don't delete (tsc output survives) |
| empty dirs after copy | prune them (match rsync --prune-empty-dirs) |

## what is awkward?

### the name "compile"

"compile" suggests transformation (like tsc). but this is really "copy with smart defaults". alternatives:
- `rhachet build` — but conflicts with tsc build step
- `rhachet dist` — clearer intent, "prepare for distribution"
- `rhachet pack` — npm uses this, familiar
- `rhachet sync` — too close to rsync

**recommendation:** keep `compile` — it's part of a build pipeline, and "compiling" role artifacts into distributable form is conceptually accurate.

### --from and --into vs positional

rsync uses positional: `rsync src/ dist/`
we use named: `--from src --into dist`

named is more explicit but more typing. tradeoff feels acceptable for a build command run rarely.

### what if user forgets rhachet repo compile?

they'll publish a role without artifacts. but this is the same failure mode as forgetting rsync today — no regression. we could add a `prepublishOnly` check but that's a separate concern.
