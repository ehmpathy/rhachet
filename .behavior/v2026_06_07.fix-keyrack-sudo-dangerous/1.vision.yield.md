# vision: fix keyrack source dangerous output

## the outcome world

### before: two problems

**problem 1: cryptic error message**

```bash
➜ rhx keyrack source --owner admin --env sudo --key GITHUB_TOKEN
not granted: ehmpathy.sudo.GITHUB_TOKEN (blocked)

hint: use --lenient if partial results are acceptable
```

the human sees "(blocked)" but has no idea WHY it's blocked. are they missing a credential? is something misconfigured? the hint suggests `--lenient` but that's not the fix.

**problem 2: no way to bypass**

```bash
➜ rhx keyrack source --owner admin --env sudo --key GITHUB_TOKEN --allow-dangerous
[commander error] error: unknown option '--allow-dangerous'
```

`keyrack get` supports `--allow-dangerous`, but `keyrack source` doesn't. the human is stuck.

### after

```bash
➜ rhx keyrack source --owner admin --env sudo --key GITHUB_TOKEN

🔐 keyrack
   └─ ehmpathy.sudo.GITHUB_TOKEN
      ├─ status: blocked 🚫
      │  ├─ its dangerous to use long lived tokens via replica mechanisms
      │  └─ detected github classic pat (ghp_*)
      └─ tip: --allow-dangerous if you must
```

the human immediately understands:
1. the key IS set (it's not absent or locked)
2. it's blocked because it's a dangerous long-lived token
3. they can bypass with `--allow-dangerous` if they must

### the "aha" moment

the value clicks when a human runs `eval $(rhx keyrack source ...)` and instead of mysterious silence or a cryptic "(blocked)" message, they get actionable context about what's wrong and how to fix it.

## user experience

### usecases

| goal | command | when blocked... |
|------|---------|-----------------|
| source for shell | `eval $(rhx keyrack source --env sudo --key X)` | see why blocked, how to fix |
| check key status | `rhx keyrack get --env sudo --key X` | see why blocked, how to fix |
| allow dangerous | `rhx keyrack source --env sudo --key X --allow-dangerous` | bypasses firewall |

### contract

`keyrack source` should:

1. **support `--allow-dangerous`** — pass through to grant logic, same as `keyrack get`
2. **emit same stderr as `keyrack get`** when keys are not granted:
   - use `formatKeyrackGetOneOutput` for single-key errors
   - use `formatKeyrackGetAllOutput` for multi-key errors
   - preserve `--lenient` hint in multi-key mode (different fix than `--allow-dangerous`)
   - exit code 2 on any not-granted (unchanged)
3. **acceptance tests** to prove and prevent regression:
   - [case] blocked key shows formatted output with reasons and `--allow-dangerous` tip
   - [case] blocked key with `--allow-dangerous` bypasses firewall and grants
   - update extant snapshots for new output format

### timeline

1. human runs `rhx keyrack source --env sudo --key X`
2. if blocked: sees full tree output explaining WHY
3. decides: fix the underlying issue OR use `--allow-dangerous`

**perfectly** — the fix is surgical: reuse extant formatting instead of inventing a new one.

### pros

- **consistency**: same output format across get/source/unlock commands
- **actionable**: blocked keys show WHY and HOW to fix
- **no new code**: reuses extant `formatKeyrackGetOneOutput` / `formatKeyrackGetAllOutput`
- **maintainable**: one place to update output format

### cons

- **none identified** — this is strictly better than current behavior

## open questions & assumptions

### assumptions

1. `keyrack source` is intended for `eval $(...)` usage
2. stderr output is acceptable for error context (doesn't pollute env vars)
3. extant `formatKeyrackGetOneOutput` / `formatKeyrackGetAllOutput` are the canonical formats

all assumptions validated by code inspection.

### questions answered

**Q1: why are there alternate codepaths for stdout when key not granted?** [answered]

A: there shouldn't be. the source command at lines 566-577 invented its own minimal format:
```typescript
console.error(`not granted: ${slug} (${a.status})`);
```

instead of the extant formatters. this was likely a shortcut that lost context.

**Q2: should --lenient hint be preserved?** [answered]

A: yes. the acceptance test at `keyrack.source.cli.acceptance.test.ts:223-241` explicitly verifies this behavior. the hint addresses a different issue (partial results) than `--allow-dangerous` (firewall bypass).

**Q3: what happens if --allow-dangerous is used with a non-dangerous key?** [answered]

A: the key grants normally. the flag only affects blocked status keys — it tells the firewall to allow through. if the key isn't blocked, the flag has no effect.

**Q4: should --allow-dangerous apply to all keys in multi-key mode?** [answered]

A: yes. same behavior as `keyrack get`. the flag is a global bypass for the current invocation.

**Q5: do we need acceptance tests for the new behavior?** [answered]

A: yes. the current acceptance tests at `keyrack.source.cli.acceptance.test.ts` don't cover `--allow-dangerous` or blocked status. we must add tests for:
- blocked key shows formatted output with reasons and `--allow-dangerous` tip
- blocked key with `--allow-dangerous` bypasses firewall and grants
this prevents regression and proves the fix works.

**key finding**: source command at line 549-559 passes `allow: {}` to grant logic, while get command passes `allow: { dangerous: opts.allowDangerous }`. this is the root cause.

contracts to conform to:
- `KeyrackGrantAttempt` discriminated union (status: granted | absent | locked | blocked)
- `formatKeyrackGetOneOutput` for single key
- `formatKeyrackGetAllOutput` for multiple keys

vocab:
- `blocked` = firewall rejected the value
- `reasons` = array of strings explaining why
- `--allow-dangerous` = bypass flag

stdouts to match:
```
🔐 keyrack
   └─ slug
      ├─ status: blocked 🚫
      │  ├─ reason 1
      │  └─ reason 2
      └─ tip: --allow-dangerous if you must
```

## what is awkward?

### no awkwardness

the fix is clean:
1. add `--allow-dangerous` option to source command (copy from get command, line 355-358)
2. pass `allow: { dangerous: opts.allowDangerous }` to grant calls (lines 549-559)
3. replace custom error format (lines 566-577) with extant formatters
4. done

the extant formatters already handle all status types (granted, blocked, locked, absent). we're just wiring them up instead of reinventing a worse version.

### potential concern: output length

the full tree format is longer than `not granted: slug (blocked)`. but this is a feature, not a bug — the extra context is exactly what was missing.

for `eval $(...)` usage: error output goes to stderr, not stdout. the shell only sees export statements on stdout. so verbosity on stderr doesn't affect functionality.
