# blocker.1: Manual try/catch used instead of HelpfulError.wrap for error observability

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.prefer.helpful-error-wrap.md

**locations**:
- src/contract/cli/invokeKeyrack.ts:780

The code catches a JSON parse error and manually constructs a BadRequestError with metadata. Per the rule, try/catch for making errors more observable should use HelpfulError.wrap (or UnexpectedCodePathError.wrap etc.) which automatically instantiates the error for maximum observability and consistent handling. This is a manual pattern that the rule says to avoid in favor of the helper.

**snippet**:
```ts
try {
  secrets = JSON.parse(rawJson);
} catch {
  throw new BadRequestError('malformed secrets JSON', {
    source,
    hint: 'ensure the input is valid JSON object',
  });
}
```

# blocker.2: Constraint and error cases use process.exit(2) + console.error instead of proper error classes with context

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.require.failloud.md

**locations**:
- src/contract/cli/invokeKeyrack.ts:470
- src/contract/cli/invokeKeyrack.ts:310
- src/contract/cli/invokeKeyrack.ts:560
- src/contract/cli/invokeKeyrack.ts:630
- src/contract/cli/invokeKeyrack.ts:710
- src/contract/cli/invokeKeyrack.ts:800

Multiple constraint/malfunction cases (keys not granted, missing manifests, blocked tokens, invalid custom paths) output via console.error or console.log and process.exit(2) without throwing ConstraintError/BadRequestError (or equivalents with metadata/hints). Per rule, errors must use proper classes with full context for observability, distinguishing caller vs server fix, and exit codes. This bypasses failloud requirements and is a blocker. Should throw new ConstraintError(msg, { metadata, hint }) instead so callers get structured errors.

**snippet**:
```ts
if (!isLenient && notGranted.length > 0) {
  if (opts.key) {
    console.error(formatKeyrackGetOneOutput({ attempt: notGranted[0]! }));
  } else {
    console.error(formatKeyrackGetAllOutput({ attempts: notGranted }));
    console.error(
      'hint: use --lenient if partial results are acceptable',
    );
  }
  process.exit(2);
}
```

# blocker.3: Eval scenarios in tests return evalStatus but do not verify it (silent pass-through risk)

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.test/pitofsuccess.errors/rule.forbid.failhide.md

**locations**:
- blackbox/cli/keyrack.source.cli.acceptance.test.ts:130
- blackbox/cli/keyrack.source.cli.acceptance.test.ts:240

In some 'when' blocks the async setup returns { sourceStatus, evalStatus, stdout } but the corresponding 'then' blocks only assert sourceStatus and stdout contents without asserting evalStatus. Per test rule.forbid.failhide, tests must verify on every code path; absent verification (like not checking evalStatus or using only snapshot) creates false confidence and is forbidden. This is a blocker as it risks silent passes if bash eval fails.

**snippet**:
```ts

  expect(result.stdout).toContain(`KEY1=${envValue1}`);
  expect(result.stdout).toContain(`KEY2=${envValue2}`);
});
// ... (no expect(result.evalStatus).toEqual(0) and similar in case3 [t1])
```
