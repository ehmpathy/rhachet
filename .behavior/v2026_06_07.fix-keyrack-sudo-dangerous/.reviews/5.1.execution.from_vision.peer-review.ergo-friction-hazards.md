# blocker.1: Invalid --env values silently succeed with empty output

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- blackbox/cli/keyrack.source.cli.acceptance.test.ts:305
- src/contract/cli/invokeKeyrack.ts:820

When --env does not match any env in the manifest (e.g. 'nonexistent'), the source command exits 0 with empty stdout and no error or hint on stderr. The code performs no validation of opts.env against KEYRACK_VALID_ENVS or the manifest before fetching attempts, and the test explicitly asserts this is 'not an error'. This hides typos/mistakes from users, provides no actionable message telling what went wrong or how to fix (e.g. 'env not defined in keyrack.yml'), and leaves a friction path without clear communication or proper error handling. Every blocked/invalid state must produce clear errors that are snapped.

**snippet**:
```ts
      then('exits with status 0 (no keys defined = no keys needed)', () => {
        // when env doesn't exist in manifest, there are no keys to source
        // this is not an error - the command succeeds with empty output
        expect(result.status).toEqual(0);
      });

      then('stdout is empty (no keys defined)', () => {
        expect(result.stdout.trim()).toEqual('');
      });
```

**locations**:

Multiple user-faced operations (set, del, list) output error messages, tips, and '✋' indicators via console.log followed by process.exit(2). This is inconsistent with the source command (which uses console.error for errors before exit 2) and with the requirement for clear, consistent error output.             console.log('');

              "   └─ tip: run 'npx rhachet keyrack init --at <path>' first",

# blocker.3: Error message location inconsistent (combined stderr+stdout checks)

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- blackbox/cli/keyrack.source.cli.acceptance.test.ts:280
- blackbox/cli/keyrack.source.cli.acceptance.test.ts:380

In validation of error cases for missing required options (commander) and mutual exclusivity, tests combine result.stderr + result.stdout to find the message and only snapshot stderr. This indicates errors are not reliably/consistently emitted to stderr across invocations. Output consistency requires errors on stderr (so stdout can be empty for source/eval safety) and every error path must be clearly snapped without ambiguity about where the message appears.

**snippet**:
```ts
      then('error mentions --env required', () => {
        const output = result.stderr + result.stdout;
        expect(output).toMatch(/(--env.*required|required.*'--env)/i);
      });

      then('stderr matches snapshot', () => {
        expect(asSnapshotSafe(result.stderr)).toMatchSnapshot();
      });
```
