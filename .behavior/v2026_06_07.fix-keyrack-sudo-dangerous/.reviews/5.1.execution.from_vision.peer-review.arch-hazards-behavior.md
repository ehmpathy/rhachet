# blocker.1: Race condition: daemon kill without synchronization

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- blackbox/cli/keyrack.source.cli.acceptance.test.ts:9

The describe-level beforeAll calls killKeyrackDaemonForTests() synchronously with no await, no returned promise, and no subsequent wait for the kill to complete or for the process to actually exit. Per the rule, this creates a race condition where subsequent test setups (which write manifests and invoke the binary) may run while a prior daemon instance is still terminating or lingering. This can lead to intermittent failures, shared state interference across test cases, and hard-to-debug production-like incidents in the blackbox suite. The code assumes kill completes instantly and has no side effects, violating predictable behavior under all conditions.

**snippet**:
```ts
  // kill any stale daemon to ensure fresh daemon with current code
  beforeAll(() => killKeyrackDaemonForTests());
```

**locations**:
- src/contract/cli/invokeKeyrack.ts:530

The order of attempts returned by getAllKeyrackGrantsByRepo (or the single-key array) can vary based on manifest parse order, extends merging (see cross-org case), object key iteration, or dao implementation details. Per the rule, given the same inputs this can produce different outputs (different stdout ordering), creating non-determinism.
