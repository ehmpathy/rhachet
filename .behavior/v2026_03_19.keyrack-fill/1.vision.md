# 1.vision: rhx keyrack fill

## the outcome world

### before

every repo that needs credentials writes its own adhoc fill command:

```bash
# .agent/repo=.this/role=any/skills/keyrack.fill.sh
REQUIRED_KEYS=("SQUARESPACE_EMAIL" "SQUARESPACE_PASSWORD" "SQUARESPACE_TOTP_SECRET")

for KEY in "${REQUIRED_KEYS[@]}"; do
  # check if configured in user's keyrack
  # check if configured in ehmpath keyrack
  # prompt user for value
  # set in user's keyrack
  # set in ehmpath keyrack
  # unlock both
  # verify roundtrip
done
```

this pattern repeats across repos. each custom fill:
- hardcodes the key list (duplicates keyrack.yml)
- hardcodes owner names (ehmpath, default)
- hardcodes prikey paths
- manually orchestrates set → unlock → get verification

### after

```bash
rhx keyrack fill --env test --owner default --owner ehmpath --prikey ~/.ssh/ehmpath
```

one command. reads from the repo's keyrack.yml. fills all keys for all specified owners. verifies roundtrip automatically.

### the "aha" moment

**"wait, i don't need to write another adhoc fill command?"**

the moment a user realizes they can just declare keys in keyrack.yml and run `rhx keyrack fill` — no custom commands — is when the value clicks. the manifest becomes the source of truth. the fill command is the universal executor.

## user experience

### usecases

| goal | command |
|------|---------|
| fill test keys for default owner | `rhx keyrack fill --env test` |
| fill test keys for multiple owners | `rhx keyrack fill --env test --owner default --owner ehmpath --prikey ~/.ssh/ehmpath` |
| fill prod keys (user only) | `rhx keyrack fill --env prod` |
| refresh a specific key | `rhx keyrack fill --env test --key CLOUDFLARE_API_TOKEN` |
| refresh all keys | `rhx keyrack fill --env test --refresh` |

### contract inputs & outputs

**inputs:**
- `--env <test|prod|all>` — which environment to fill (required)
- `--owner <name>` — which owner keyrack to fill (repeatable, default: `default`)
- `--prikey <path>` — prikey to consider for owner manifest decryption (repeatable, extends discovered prikeys)
- `--key <name>` — specific key to fill (optional, default: all keys)
- `--refresh` — re-prompt even if already configured

**outputs:**
- tree output with each key × owner result (keyrack CLI style)
- exit 0 if all keys verified
- exit 1 if any key fails roundtrip

**stdout within stdout** — fill wraps each `keyrack set` invocation's full stdout within a treebucket (the `├─`...`└─` frame). the sub-command's output flows through unmodified, just indented within the bucket.

the inner loop is owners, so the user can repeat the same action (paste value, or complete sso flow) twice in succession before the next key.

## mental model

### how would users describe this to a friend?

> "rhx keyrack fill reads my repo's keyrack.yml and walks me through setting up each credential. i just run one command and it prompts me for each key, stores it for each owner i specify, and verifies everything works."

### analogies

| concept | analogy |
|---------|---------|
| keyrack.yml | package.json dependencies — declares what you need |
| rhx keyrack fill | npm install — installs what's declared |
| owners | profiles — same key, different storage contexts |
| prikey | ssh key — proves you can access an owner's keyrack |

### terms alignment

| user might say | we say | meaning |
|----------------|--------|---------|
| "set up my credentials" | "fill the keyrack" | populate keys for an env |
| "my personal keyrack" | "owner=default" | the user's own keyrack |
| "the bot's keyrack" | "owner=ehmpath" | shared keyrack for agents |
| "make sure it works" | "roundtrip verification" | set → unlock → get |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| no more adhoc fill commands per repo | ✓ manifest-driven |
| fill multiple owners in one run | ✓ inner loop on owners |
| auto-discover prikey for each owner | ✓ compare discovered + supplied prikeys |
| verify roundtrip | ✓ set → unlock → get sequence |
| idempotent (skip already configured) | ✓ check before prompt |
| refresh capability | ✓ --refresh and --key flags |

### pros

- **manifest is source of truth** — keyrack.yml declares keys, fill command executes
- **no duplication** — keys declared once, not in yml AND in adhoc command
- **multi-owner support** — one command fills multiple keyracks
- **prikey auto-selection** — figures out which prikey works for which owner
- **roundtrip verification** — catches misconfigurations immediately

### cons

- **interactive** — requires user presence to enter values
- **no automation path (yet)** — can't pipe values from 1password cli, etc
- **vault inference may surprise** — if no vault prescribed, fallback to os.secure

### fill vs set

`keyrack set` is the low-level primitive. `keyrack fill` is the high-level orchestrator.

| aspect | `keyrack set` | `keyrack fill` |
|--------|---------------|----------------|
| key discovery | user specifies `--key` | reads from keyrack.yml |
| vault | required `--vault` flag | inferred from manifest |
| owners | single `--owner` | multi-owner (repeatable `--owner`) |
| roundtrip | no verification | yes: set → unlock → get |
| prikey | manual `--prikey` | auto-discovery (tries pool) |
| idempotent | no (always sets) | yes (skips already configured) |

**when to use which:**
- `keyrack set` — configure a single key for a single owner with explicit vault
- `keyrack fill` — fill all keys for all owners from manifest, with verification

**fill dispatches to extant primitives** — no new low-level operations needed:

```
for each key in manifest:
  for each owner:
    keyrack set --key $key --vault (inferred) --owner $owner
    keyrack unlock --key $key --owner $owner
    keyrack get --key $key --owner $owner  # verify roundtrip
```

### edgecases and pit of success

| edgecase | pit of success |
|----------|----------------|
| owner's host manifest requires unknown prikey | fail-fast: "no available prikey for owner=X" |
| key already configured for one owner but not another | fill only missing, skip configured |
| extends cycle in keyrack.yml | fail-fast: detected during manifest expansion |
| no keys match specified --env | warn: "no keys found for env=test" |
| user enters empty value | fail-fast: "value cannot be empty" |

## open questions & assumptions

### assumptions

1. **interactive is acceptable** — fill requires user presence to enter values
2. **owners specified, not discovered** — user explicitly lists --owner flags (no "fill all owners" auto-mode)
3. **prikey pool is finite** — discovered prikeys + supplied --prikey flags is a bounded set
4. **vault fallback to os.secure** — when key has no prescribed vault and we can't infer, default to os.secure

### questions answered via triage

| question | answer |
|----------|--------|
| `--owner` default to `default`? | yes — single-owner is common case |
| `--owner @all` mode? | no for v1 — too complex (requires prikey for each owner) |
| failed owner decryption = blocker? | yes — fail-fast pattern, consistent with keyrack |
| `--refresh` scope? | all specified owners — if user says --refresh, they want refresh |
| prikey discovery mechanism? | ssh-agent first, then ~/.ssh/id_ed25519 etc, converts to age identity |
| how extends works? | keys merged from extends, supports bare strings or objects with grade |

## what is awkward?

### `--prikey` flag feels disconnected

the --prikey flag extends the set of prikeys to consider, but there's no explicit mapping of "this prikey is for that owner." the system tries each prikey against each owner's host manifest until one works.

this is flexible but might confuse users: "why am i providing prikeys without saying which owner they're for?"

**mitigation:** clear messaging like "trying prikeys for owner=ehmpath... ~/.ssh/ehmpath ✓"

### inner loop on owners means repeated prompts

for the same key, user enters the value once per owner. if the value is the same across owners (common case), this is redundant.

but if values differ per owner (e.g., different api tokens for user vs agent), separate prompts are correct.

**tradeoff:** accept the redundancy for correctness. future: `--same-value` flag to copy first owner's value to others.

### --env required but could be inferred

if keyrack.yml only declares env.test keys, --env test is the only valid choice. requiring it feels unnecessary.

**tradeoff:** explicit is better than magic. keeps the command predictable.

## summary

`rhx keyrack fill` transforms credential setup from "write a custom adhoc command per repo" to "run one command that reads your manifest." the manifest becomes declarative truth; fill is the executor. multi-owner support with prikey auto-selection enables the "one command fills all" experience.
