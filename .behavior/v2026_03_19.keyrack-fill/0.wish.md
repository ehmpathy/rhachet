wish =

we want to support a `keyrack fill` command which enables a user to easily fill the keyracks of one or more owners at a time, based on the repo manifest they run the command in

for example

if they run `keyrack fill --env test --owner default --owner ehmpath --prikey $path/to/ehmpath/prikey` from a repo which requires

```
org: ehmpathy

extends:
  - .agent/repo=ehmpathy/role=mechanic/keyrack.yml

env.prod:
  # - AWS_PROFILE
  # - OPENAI_API_KEY

env.test:
  - CLOUDFLARE_ACCOUNT_ID
  - CLOUDFLARE_API_TOKEN
```

then, what we want keyrack to do is

for each key that is prescribed by the repo manifest 

go ahead and set them into the prescribed storage class (if not prescribed; infer the vault and mechanism as needed; fallback to replica os.secure when not prescribed and not inferrable)


so, 

- for each key,
    - for each owner,   
        - set it 
        - unlock it
        - get it 

to verify that it is roundtrip usable (get after set)

and do the owner on the inner loop, so that if the user can repeat the steps required to set their key with that vault (e.g., paste the same password, or repeat the same sso steps, etc - dependent on the vault type)

just make it super clear which owner its for that they are asked to set into

---

and the --prikey; we should expect a random set of --prikeys to consider

each --owner will have their host manifest encrypted with whatever --prikey they chose. (could be discoverable, could be one of the ones specied in --prikey). we need to compare each discovered and supplied prikey against the host manifest to figure out which one is available to use. (and if none, as always, failfast)

but the point is, they're not ordered in any relationship when supplied. the --prikey repeatable flag just extends the set of prikeys we should consider (ontop of the discovered ones)
