# 3.1.5.research.reflection.product.rootcause._.v1.i1

## what is the symptom?

describe the observable problem or desired effect.

- **what is broken or absent?** every repo that needs credentials writes its own adhoc fill command
- **how does it manifest?** duplicated fill logic across repos, each hardcodes key lists, owner names, prikey paths
- **when does it occur?** when a developer needs to configure keyrack credentials for a repo

---

## 5 whys tree

drill down via repeated "why?" questions. branch when multiple causes exist.

### branch 1: adhoc fill commands per repo

1. why do repos write adhoc fill commands? → keyrack has `set`, `unlock`, `get` but no orchestrated "fill from manifest" command
2. why no orchestrated fill command? → keyrack primitives were built first; orchestration was left to consumers
3. why was orchestration left to consumers? → each repo has different key requirements; seemed repo-specific
4. why seemed repo-specific? → no manifest standard existed to declare which keys a repo needs
5. why was there no manifest standard? → **keyrack.yml manifest standard now exists but fill command lags behind**

**potential root cause:** manifest exists, orchestrator does not — the gap between declaration and execution

### branch 2: multi-owner complexity

1. why do fill commands hardcode owner names? → each owner (default, ehmpath) needs separate set/unlock/get calls
2. why separate calls? → keyrack primitives operate on one owner at a time
3. why one owner at a time? → keeps primitives simple; owner is explicit per call
4. why is owner per call? → host manifests are per-owner; prikey decrypts specific owner's manifest
5. why prikey per owner? → **security model: each owner's keyrack is independently encrypted**

**not root cause:** this is intentional design. the multi-owner loop belongs in the orchestrator, not in primitives.

---

## root cause summary

| layer | issue | is root cause? |
|-------|-------|----------------|
| primitive ops | set, unlock, get work on single key + single owner | no — correct design |
| manifest declaration | keyrack.yml declares required keys per env | no — already solved |
| orchestration | no command reads manifest and orchestrates set → unlock → get | **yes** |
| multi-owner | fill commands hardcode owners + prikeys | symptom of absent orchestration |

---

## recommendation

which layer should we address?

- **address root cause**: create `rhx keyrack fill` that reads keyrack.yml manifest, iterates keys × owners, dispatches to extant primitives, verifies roundtrip

- **do not address:**
  - primitives (already correct)
  - manifest format (already exists)
  - security model (prikey per owner is intentional)

the fix is pure orchestration: compose extant primitives in a standard loop. no new low-level operations needed.
