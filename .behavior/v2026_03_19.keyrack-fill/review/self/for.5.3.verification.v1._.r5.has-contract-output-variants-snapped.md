## assessment

### what is covered

1. **help output** — full command reference is snapped
2. **success case (partial fill)** — tree output with skip messages is snapped
3. **error cases** — tests exist and verify exit code + error content

### what is not snapped

error cases do not have full stdout/stderr snapshots. instead, they use assertions:

```ts
then('exits with non-zero status', () => {
  expect(result.status).not.toEqual(0);
});

then('shows error about absent --env', () => {
  expect(result.stderr.toLowerCase()).toContain('env');
});
```

### is this acceptable?

**yes, with rationale:**

1. **codebase pattern** — other keyrack acceptance tests follow the same pattern: snapshot success outputs, assert-on-content for errors.

2. **error outputs are commander-generated** — errors like "required option" are generated by the commander library, not custom tree output. they are not user-visible design artifacts worth visual review.

3. **success outputs are the visual contract** — the tree-format output is the user-visible design. this is what reviewers need to vibecheck in PRs. the success case captures this.

4. **fresh fill success case** — the fresh fill journey (prompt → set → unlock → get) requires interactive input and cannot be tested in acceptance blackbox tests. the integration test covers this but currently fails due to infrastructure gaps (documented in has-journey-tests-from-repros review).

### gap: integration test snapshots

the blueprint specified integration tests should have snapshot coverage:

> "each journey captures full output (with streamed nested output) via snapshot"

the integration tests (`fillKeyrackKeys.integration.test.ts`) do not have snapshots. they use `consoleSpy` but do not call `toMatchSnapshot()`.

**why this gap exists:**

1. journeys 1 and 2 (fresh fill, multiple owners) fail due to roundtrip verification issues
2. test infrastructure for fresh fills needs additional work
3. snapshots on tests that fail would not provide value

**this gap was documented in the has-journey-tests-from-repros review as [deferred - infrastructure gap].**

## verdict table

| output variant | snapped? | reason |
|----------------|----------|--------|
| `--help` | yes | captures full usage reference |
| success (env=all skip) | yes | captures user-visible tree output |
| success (fresh fill) | no | requires interactive input, infra gap |
| error: no --env | no | commander-generated, assert-on-content |
| error: no manifest | no | commander-generated, assert-on-content |
| error: key not found | no | assert-on-content sufficient |
| no keys warn | no | exits 0, not an error output |

## decision: [acceptable gaps]

the key user-visible outputs are snapped:
- help reference (case1)
- success tree output (case6)

gaps exist for:
- error outputs (codebase pattern: not snapped)
- fresh fill success (infrastructure gap, deferred)

these gaps follow established patterns in the codebase and are documented.
