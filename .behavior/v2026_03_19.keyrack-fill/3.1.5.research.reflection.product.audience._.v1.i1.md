# 3.1.5.research.reflection.product.audience._.v1.i1

## who is the audience?

### primary: repo maintainers who need credentials configured

developers who maintain repos that depend on external services (cloudflare, aws, etc). they need to configure credentials for multiple owners (themselves + shared bot accounts like ehmpath).

**specific context:**
- work across multiple repos with different credential requirements
- manage credentials for both personal (owner=default) and shared (owner=ehmpath) keyracks
- currently write adhoc fill commands per repo — duplication and maintenance burden

### secondary: new developers who join the team

new team members who clone a repo and need to configure credentials before they can run tests or deploy. they inherit the keyrack.yml manifest but have no adhoc commands to guide them.

**specific context:**
- unfamiliar with which keys are required
- may not know which vaults or mechanisms are expected
- benefit from guided, interactive setup

### tertiary: automation systems (future)

CI/CD pipelines or scripts that may eventually invoke `keyrack fill` non-interactively (e.g., via piped values from 1password CLI). not primary for v1 but shapes design decisions.

---

## why do they care?

### pain relieved

| audience | pain point |
|----------|------------|
| repo maintainers | write adhoc fill commands for every repo — duplication |
| repo maintainers | hardcode key lists, owner names, prikey paths — drift from manifest |
| repo maintainers | manually orchestrate set → unlock → get — tedious |
| new devs | "how do I set up credentials?" — no guidance |
| new devs | trial-and-error to discover which keys are needed |

### goal enabled

| audience | goal |
|----------|------|
| repo maintainers | one command fills all keys for all owners from manifest |
| repo maintainers | verify roundtrip works before they continue |
| new devs | get to "functional setup" faster |
| new devs | trust the process (guided, verified) |

### frustration eliminated

- "why do I have to copy-paste this same pattern across repos?"
- "did I set this key for the bot account too, or just myself?"
- "is this credential actually functional, or did I mistype it?"

## how much do they care?

| audience | priority | urgency | stakes |
|----------|----------|---------|--------|
| repo maintainers | high | medium | time saved per repo, reduced errors |
| new devs | medium | low | smoother first experience |
| automation (future) | low | low | unblocked but not v1 scope |

repo maintainers care most — they feel the duplication pain repeatedly. new devs benefit but may not realize it (smooth is invisible).

---

## what do they care about?

### success criteria

| audience | judges success by |
|----------|-------------------|
| repo maintainers | "I ran one command, all keys set, all verified — done" |
| repo maintainers | "I didn't have to remember which owner needed which prikey" |
| new devs | "it told me what to do and confirmed it worked" |

### acceptable tradeoffs

- **accept:** interactive prompts (requires user presence)
- **accept:** inner loop on owners (repeat same value if needed)
- **reject:** silent failures or partial success without clear output
- **reject:** vault inference that surprises (must be predictable)

### delighted vs disappointed

| delighted | disappointed |
|-----------|--------------|
| "it just worked for both owners" | "I had to re-run for the bot account separately" |
| "it told me exactly which key failed" | "it said '1 key failed' but which one?" |
| "it skipped keys I already had" | "it made me re-enter all values" |

## how will they experience it?

### touchpoints

1. **discover** — see `keyrack fill` mentioned in repo README or CLAUDE.md
2. **invoke** — run `rhx keyrack fill --env test`
3. **prompted** — for each key, enter value (paste, SSO flow, etc)
4. **verified** — see "✓ set → unlock → get" confirmation per key
5. **done** — see summary "N/N keys verified"

### friction points

| potential friction | mitigation |
|--------------------|------------|
| "which prikey unlocks ehmpath?" | prikey auto-discovery + clear output |
| "I pasted the wrong value" | fail-fast on empty, roundtrip verification catches issues |
| "do I need --refresh or not?" | default skips already-set, explicit --refresh for rotation |

### smooth experience

```
$ rhx keyrack fill --env test --owner default --owner ehmpath

🔐 keyrack fill (env: test, keys: 3, owners: 2)

🔑 key 1/3, CLOUDFLARE_API_TOKEN
   ├─ owner default: ✓ set → unlock → get
   └─ owner ehmpath: ✓ set → unlock → get

🔑 key 2/3, CLOUDFLARE_ACCOUNT_ID
   ├─ owner default: ✓ already set, skip
   └─ owner ehmpath: ✓ set → unlock → get

🔑 key 3/3, AWS_PROFILE
   ├─ owner default: ✓ set → unlock → get
   └─ owner ehmpath: ✓ set → unlock → get

🔐 keyrack fill complete (3/3 keys verified)
```

user knows exactly what happened, which keys for which owners, and that all is verified.
