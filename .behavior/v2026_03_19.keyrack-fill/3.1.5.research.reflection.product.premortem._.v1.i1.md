# 3.1.5.research.reflection.product.premortem._.v1.i1

## premortem: imagine we shipped and it failed

project forward: we shipped this, and it failed badly. why?

list the most plausible failure scenarios:

1. **prikey resolution confused users** — user passed `--prikey ~/.ssh/ehmpath` but it didn't work because the owner's host manifest was encrypted with a different key. error message was unclear, user gave up.

2. **vault inference surprised users** — key had no prescribed vault, system inferred `os.secure`, but user expected `os.plain`. credentials stored in wrong vault, hard to debug.

3. **partial success left state inconsistent** — fill set 2/3 keys, then failed on 3rd. user assumed all failed and re-ran, now has duplicate entries or unexpected state.

---

## what obvious signs did we ignore?

look back from the imagined failure — what signs were present that we dismissed?

- the `--prikey` flag is disconnected from `--owner` — no explicit association, relies on trial-and-error against host manifests
- vault inference has only one rule (AWS_PROFILE → aws.iam.sso), all else falls back to os.secure — users unfamiliar with this default
- the inner loop on owners means repeated prompts — users may enter wrong value for second owner and not notice until verification fails

---

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| users always have prikey for each owner accessible | user may have prikey on different machine, or in yubikey not plugged in | medium |
| vault inference to os.secure is acceptable | user's org may prefer os.plain or custom vault | low |
| interactive mode is sufficient for v1 | user may need to fill keys in headless CI where interactive prompts block | low |
| users will read error messages carefully | rushed user may miss "key X not found" and file a bug | medium |

---

## what is the nightmare scenario?

describe the worst plausible outcome.

- **what would make us deeply regret this?** fill appears to succeed (exit 0) but roundtrip verification silently skipped a key. user discovers key is absent days later in production incident.

- **what would cause the most damage?** fill writes credentials to wrong vault, user cannot unlock them later. credentials are "lost" and must be regenerated from external provider.

- **what would be hardest to recover from?** fill corrupts host manifest on set operation. user loses access to all keys in that manifest, not just the one set.

---

## what mitigations to consider?

| risk | mitigation | cost of mitigation |
|------|------------|--------------------|
| prikey resolution unclear | clear log output: "trial prikey ~/.ssh/ehmpath against owner=ehmpath... ✓ decrypted" | low — just better logs |
| vault inference surprises | warn when vault inferred: "no vault prescribed for KEY, infer os.secure" | low — log message |
| partial success state | atomic: if any key fails verification, report which ones succeeded vs failed in summary | low — track in loop |
| roundtrip silently skips | verification is mandatory, not optional; fail-fast if get returns empty after set | zero — already in design |
| host manifest corruption | daoKeyrackHostManifest.set is transactional (backup before write) | already extant in code |
| credentials in wrong vault | require vault in manifest for keys where vault choice matters; only infer for simple cases | medium — shifts responsibility to manifest author |
