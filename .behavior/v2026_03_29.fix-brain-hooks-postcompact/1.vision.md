# vision = support claude cli PreCompact and PostCompact hooks

## the outcome world

### day-in-the-life: after

vlad's role has a `trust-but-verify` hook that only fires after compaction. when context compacts, the hook questions the assumptions the bot made during compaction — never trust anyone, even yourself. the compaction summary is a claim that should be verified.

this hook doesn't run on every SessionStart (that would be noisy). it runs *only* on PostCompact — when compaction just happened.

### before/after contrast

| before | after |
|--------|-------|
| can't distinguish compaction from session start | can target compaction specifically |
| hooks fire on all boots or none | hooks can fire on specific boot triggers |
| no compaction-specific behaviors | roles can add verification, logging, state-saves for compaction |

### "aha" moment

> "i can run something *only* when compaction happens — not on every session start."

the aha is precision. SessionStart is the hammer; PostCompact/PreCompact are the scalpels.

## user experience

### usecase 1: trust-but-verify after compaction

**goal:** question assumptions the brain made after compaction

**contract:**

```yaml
hooks:
  onBrain:
    onBoot:
      - command: npx rhachet run --init postcompact.trust-but-verify.sh
        timeout: PT30S
        filter:
          what: PostCompact   # only after compaction
```

**timeline:**

```
t0: session starts
    → SessionStart fires → briefs boot (as usual)
    → PostCompact hook does NOT fire (no compaction happened)

t1: user works for 90 min, context fills

t2: context compacts
    → SessionStart fires → briefs re-boot (as usual)
    → PostCompact fires → verification hook runs
    → bot questions its own compaction assumptions

t3: user continues with verified context
```

## mental model

### how users would describe this

> "i can filter my onBoot hooks to fire on specific triggers — session start, before compaction, after compaction, or all of them."

### analogies

**filter as event selector:**

just like onTool uses `filter.what` to select which tools trigger the hook, onBoot uses `filter.what` to select which boot triggers fire it.

| hook | filter.what values |
|------|-------------------|
| onTool | `Bash`, `Write`, `*` (tool names) |
| onBoot | `SessionStart`, `PostCompact`, `PreCompact`, `*` (boot triggers) |

the pattern is consistent: `filter.what` answers "what triggers this hook?"

### key insight: SessionStart includes compaction

claude code fires SessionStart on both:
- new session start
- after compaction

so SessionStart is the broad event. PostCompact is the narrow event that fires *only* on compaction.

```
triggers:
  new session    → SessionStart fires
  compaction     → SessionStart fires AND PostCompact fires
  pre-compaction → PreCompact fires (then SessionStart + PostCompact)
```

this means:
- hook on SessionStart = fires on new session AND compaction
- hook on PostCompact = fires ONLY on compaction
- PostCompact is a subset of SessionStart triggers

### terms: rhachet vs claude code

| rhachet | claude code | notes |
|---------|-------------|-------|
| `onBoot` + no filter | `SessionStart` | backwards compat: same as prior behavior |
| `onBoot` + `filter.what=SessionStart` | `SessionStart` | explicit: fires on new session + compaction |
| `onBoot` + `filter.what=PostCompact` | `PostCompact` | compaction-only hook |
| `onBoot` + `filter.what=PreCompact` | `PreCompact` | pre-compaction hook |
| `onBoot` + `filter.what=*` | all three events | explicit wildcard for all triggers |
| `onTool` | `PreToolUse`/`PostToolUse` | current pattern |
| `onStop` | `Stop` | current pattern |

## evaluation

### how well does it solve the goals?

| goal | solved? | how |
|------|---------|-----|
| compaction-specific hooks | ✓ | filter.what=PostCompact targets compaction only |
| pre-compaction hooks | ✓ | filter.what=PreCompact targets pre-compact only |
| backwards compatible | ✓ | no filter = SessionStart only (same as prior) |
| consistent mental model | ✓ | filter.what works same as onTool |

### pros

1. **precision** — can target exact boot triggers instead of all-or-nothing
2. **consistent pattern** — filter.what already works for onTool; same pattern for onBoot
3. **backwards compatible** — prior onBoot hooks without filter continue to work
4. **extensible** — future boot events can be added without contract changes

### cons

1. **filter overload** — filter.what means "tool name" for onTool but "boot trigger" for onBoot
2. **discovery** — users must know valid boot trigger values
3. **translation complexity** — adapter emits multiple claude hooks from one rhachet hook

### pit of success

| edge case | solution |
|-----------|----------|
| user forgets filter | defaults to SessionStart (backwards compat, safe) |
| user typos filter value | fail-fast validation at sync time |
| multiple roles with same hook | namespace isolation via author field |

## what is awkward?

### filter.what semantics diverge

for onTool, `filter.what` is a tool name pattern: `Bash`, `Write|Edit`, `*`

for onBoot, `filter.what` would be a boot trigger: `SessionStart`, `PostCompact`, `*`

these are conceptually different:
- onTool.filter.what = "which tool?"
- onBoot.filter.what = "which boot trigger?"

**mitigation:** this is still the same field answering "what triggers this hook?" — just applied to different event types.

### translation complexity

the adapter must:
1. for onBoot without filter → emit SessionStart only (backwards compat)
2. for onBoot with filter.what=* → emit all boot events
3. for onBoot with filter.what=X → emit hook only for X

this is more complex than the current 1:1 map.

**mitigation:** the common cases (no filter, specific filter) remain simple. only explicit wildcard triggers multi-emit. complexity is in the adapter, not the role definition.

## proposed contract changes

### BrainHookFilter — no changes needed

the contract already supports this:

```ts
interface BrainHookFilter {
  what: string;                    // already accepts any string
  when?: 'before' | 'after';       // still applies to onTool only
}
```

`filter.what` already accepts any string. the contract is unchanged — only the **adapter** (translateHook) needs to interpret `filter.what` for onBoot events.

### translateHook changes

```ts
// current
const EVENT_MAP: Record<BrainHookEvent, string> = {
  onBoot: 'SessionStart',
  onTool: 'PreToolUse',
  onStop: 'Stop',
};

// proposed
const translateHookToClaudeCode = (input: { hook: BrainHook }) => {
  // for onBoot, filter.what determines the event(s)
  if (input.hook.event === 'onBoot') {
    // backwards compat: no filter = SessionStart only (prior behavior)
    const bootTrigger = input.hook.filter?.what ?? 'SessionStart';

    // explicit wildcard = all boot events
    if (bootTrigger === '*') {
      return [
        { event: 'SessionStart', entry: ... },
        { event: 'PreCompact', entry: ... },
        { event: 'PostCompact', entry: ... },
      ];
    }

    // specific trigger = single event
    return [{ event: bootTrigger, entry: ... }];
  }
  // prior logic for onTool, onStop
  ...
};
```
