## what the vision got wrong (corrected by groundwork)

the vision's top suspect was: "`credential-provider-sso@3.972.54` is ~69 minors stale vs
`client-sso@3.1041.0`; align it up to 3.1041.x." **this is impossible and the comparison was
apples-to-oranges.** verified:

- `@aws-sdk/credential-provider-sso` only publishes on a `3.972.x` track — latest is `3.972.55`
  (`npm view`). there is NO `3.1041.x`/`3.1075.x` for it. the current pin `3.972.54` is **one
  patch** behind its own latest, not 69 minors.
- it is a **`packages-internal`** package (homepage:
  `github.com/aws/aws-sdk-js-v3/tree/main/packages-internal/credential-provider-sso`). aws does
  NOT version it in lockstep with `client-sso`. they are siblings; `client-sso` does not depend
  on it.
- the **public** wrapper `@aws-sdk/credential-providers` (plural) IS lockstep at `3.1075.0`, and
  it consumes the internal one via `^3.972.55`. `fromSSO` is the public api exported there.

## what is actually wrong (real candidates — unverified as the cause)

### find 1 — direct dependency on an INTERNAL package
the code imports `fromSSO` directly from `@aws-sdk/credential-provider-sso` (singular, internal)
at `vaultAdapterAwsConfig.ts:1`, and pins it directly in `package.json:102`. the documented
public path is `@aws-sdk/credential-providers` (plural). a direct dep on the internal package
is what lets its version float independently of the rest of the sdk.

### find 2 — fragmented `@aws-sdk` tree (lockfile evidence)
`pnpm-lock.yaml` resolves MULTIPLE versions of the same packages:
- `@aws-sdk/core`: **3.943.0, 3.974.8, 3.974.22** (three copies)
- `@aws-sdk/client-sso`: 3.1041.0, 3.943.0
- `@aws-sdk/credential-provider-sso`: 3.972.54, 3.943.0
- `@aws-sdk/token-providers`: 3.1071.0, 3.943.0
- `@aws-sdk/nested-clients`: 3.997.22, 3.943.0

multiple `@aws-sdk/core` / `token-providers` instances are a credible cause for in-process
`fromSSO` to read/write the sso token cache inconsistently with the cli — far more plausible
than "version number mismatch". (the whole `3.943.0` set is a parallel old tree some other dep
pulls; the keyrack tree itself still splits core 3.974.8 vs 3.974.22.)

## CORRECTED ROOT CAUSE (supersedes the dependency-tree theory)

after a read of `setupAwsSsoWithGuide.ts`, the real cause is documented **in the code itself**
at lines 386-388:

> equivalent profile found - still need to login with ACTUAL profile so the token is cached
> under the correct sso-session name (**initiateAwsSsoAuth uses a temp sso-session, which has
> different cache key**)

### the mechanism

the guided flow authenticates the browser via `initiateAwsSsoAuth` (`setupAwsSsoWithGuide.ts:172`)
which logs in under a **temp sso-session** named `keyrack-auth` with `AWS_CONFIG_FILE=<temp>`
(`setupAwsSsoProfile.ts:371-388`). the aws sso TOKEN gets cached keyed by that temp session name,
not the real profile's session name.

then the two branches diverge:

| path | branch | follow-up login | result |
|------|--------|----------------|--------|
| profile is FRESH | `setupAwsSsoWithGuide.ts:375-384` → `setupAwsSsoProfile` | `aws sso login --profile X` inside setup (`setupAwsSsoProfile.ts:101`) | ✋ in-process `fromSSO` then finds no token under session X's key → "sso session not valid" |
| profile EXISTS & equivalent | `setupAwsSsoWithGuide.ts:385-396` | explicit `aws sso login --profile X` (`:391`) | ✓ token now cached under session X → `fromSSO` reads it |

so the trigger is exactly the fresh-write↔fail correlation the vision observed — but the cause is
the **temp-session token-cache-key mismatch**, NOT a version skew. the dependency-tree note was
a detour (`credential-provider-sso@3.972.x` is its own internal track; not stale).

### why "consistently" + why logout "fixes" it

- the cli `aws sso login --profile X` appears to dedup against the still-valid temp-session
  registration for the same start_url and skip a fresh token write under session X's key — so the
  in-process `fromSSO` (which keys strictly by session X) finds no token. deterministic → fails
  every first time.
- `aws sso logout` wipes ALL cached tokens; the retry then hits the EXISTS-branch (profile was
  written on the failed attempt) which does its own explicit `aws sso login --profile X`. that is
  why logout "helps" — it is a distractor, as predicted.

### still needs wisher repro to confirm

i cannot reproduce (needs browser auth + real aws accounts), so the cli-dedup mechanism above is
a strong hypothesis, not proven. the fix shape depends on a wisher repro.
