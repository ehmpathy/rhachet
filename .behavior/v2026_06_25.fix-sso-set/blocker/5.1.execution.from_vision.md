### root cause (corrected — code-documented)

the guided flow browser-auths via `initiateAwsSsoAuth` under a **temp sso-session**
(`keyrack-auth`, `setupAwsSsoProfile.ts:371-388`). the sso token caches under that temp key,
not the real profile's session key.

the code itself documents this at `setupAwsSsoWithGuide.ts:386-388`:
> "still need to login with ACTUAL profile so the token is cached under the correct sso-session
> name (initiateAwsSsoAuth uses a temp sso-session, which has different cache key)"

- FRESH profile path (`setupAwsSsoWithGuide.ts:375-384`) → `setupAwsSsoProfile` runs
  `aws sso login --profile X` (`:101`) and `aws sts get-caller-identity` (`:123`, PASSES).
- then `vaultAdapterAwsConfig.set` re-validates at `:535` via `validateSsoSession`, whose FIRST
  gate is in-process sdk `fromSSO` (`:89`) → **FAILS** → `ConstraintError` (`:537`).

so the failure is a genuine cli-vs-sdk divergence: at the same instant, cli `sts` says valid,
sdk `fromSSO` says invalid — because the token sits under the temp session's cache key, not
session X's, and the post-setup `aws sso login --profile X` appears to dedup against the still
valid temp token rather than a fresh write for session X. deterministic → "consistently". the
EXISTS branch (`:385-396`) works because by the retry the cache is clean (logout wiped it), so
its explicit login writes a token under X. → logout is a distractor, as the vision predicted.

### why this is unverifiable here

- which fix is correct depends on how aws keys the sso token cache (by session-name vs
  start_url) and whether `aws sso login --profile X` dedups — this varies by aws-cli version and
  is only observable on the wisher's machine with real auth.
- i must not ship an unverified guess: it could be a non-fix, or regress the deliberate
  `fromSSO`-validation intent from PR #391 (which moved AWAY from cli validation because the cli
  cache can lie — `vaultAdapterAwsConfig.ts:94-101`, cites aws/aws-cli#9845).
