# 1. vision — fix-sso-set

> the wish: "why does it consistently fail after the first set until i run logout?"
> plus: "also, why that random newline"

this is a **defect-fix** wish, not a new-feature wish. so the "vision" is a world where an
extant contract (`rhx keyrack fill` / `rhx keyrack set AWS_PROFILE`) stops misbehaving.
the contract surface does not change — its **reliability** does.

---

## the outcome world

### day-in-the-life

a human onboards a new aws sso profile:

```
➜ rhx keyrack fill --owner orgone --env prod
```

they pick domain → account → role → name it. the profile is written to `~/.aws/config`,
verified, and vaulted. **on the first try.** no `✋ ConstraintError`. no detour to
`aws sso logout`. no re-run.

### before / after

| | before (today) | after (wished) |
|---|---|---|
| first `fill` on a fresh profile | ✋ `sso session not valid; run aws sso login` | ✓ verified + vaulted |
| recovery ritual | `aws sso logout` → re-run `fill` | none needed |
| profile already present | ✓ works | ✓ works (unchanged) |
| guided-setup output tail | stray blank line after `✓ relock` | clean tree, no orphan newline |

### the "aha"

> "i ran `keyrack fill` once and it just... worked. i didn't have to log out and try again."

the value clicks when the **first** attempt succeeds — because the whole point of guided
setup is that the user just authenticated seconds ago. failing immediately after a fresh,
successful login is the surprise we delete.

## user experience

### usecases / goals

- **onboard** a brand-new aws sso profile for an env (`prod`/`prep`/`test`) in one pass
- **refill** an env's keys without a logout ritual interrupting the flow
- goal: a vaulted `AWS_PROFILE` key whose sso session is provably valid, on first run

### contract inputs & outputs (unchanged)

inputs:
- `rhx keyrack fill --owner <owner> --env <env>`
- `rhx keyrack set AWS_PROFILE ...` (guided, tty)

outputs:
- the guided tree (domain → sso login → account → role → name → vault → verify)
- success: `🔐 keyrack fill complete (N/N keys verified)`
- failure: a `ConstraintError` **only when the session is genuinely unrecoverable**, with a
  truthful, actionable hint

### timeline (wished)

```
set the key
  └─ 🔐 keyrack set AWS_PROFILE
       ├─ which sso domain?   → pick
       ├─ with sso prior?     → clear / reuse
       ├─ which sso login?    → ✓ authenticated
       ├─ which account?      → pick
       ├─ which role?         → pick
       ├─ what should we call it? → name
       ├─ lovely, lets vault it now... → ✓ written to ~/.aws/config
       └─ perfect, now lets verify...
            ├─ ✓ unlock
            ├─ ✓ get
            └─ ✓ relock          ← no orphan blank line after this
get after set, to verify → ✓ unlock / ✓ get
```

## mental model

### how a user describes it to a friend

> "keyrack walks me through aws sso setup and saves the profile. it logs me in, then checks
> the login actually works before saving."

they do **not** think: "there are two credential resolvers (cli vs sdk) and they disagree about
a freshly-minted sso-session token cache." that's our problem to hide.

### analogy

a turnstile that opens for the person who *just* tapped their badge. today it occasionally
rejects them and makes them tap-out and tap-in again. the fix: trust the tap that just happened.

### terms

| user says | we say |
|---|---|
| "log in" | `aws sso login` / sso device authorization |
| "save the profile" | `setupAwsSsoProfile` writes `~/.aws/config` (v2 `sso-session` block) |
| "check it works" | `validateSsoSession` (sdk `fromSSO` → export-credentials → `sts get-caller-identity`) |
| "it failed" | `ConstraintError: sso session not valid` |

## evaluation

### how well it solves the goal

solves it fully **iff** the root cause is the fresh-profile validation divergence (see open
questions). a correct fix makes first-run guided setup succeed without changing the contract.

### pros

- deletes a confusing, repeatable failure on the happy path
- removes a superstition ("you must logout first") from team muscle memory
- a genuine `ConstraintError` still fires when the session truly can't be established —
  we keep failfast, we just stop crying wolf
- output cleanup is low-risk and independently shippable

### cons / tradeoffs

- the fix likely lives at the cli-vs-sdk seam (`validateSsoSession`); getting it wrong could
  mask a *real* invalid-session signal → must preserve true-negative behavior
- if the cause is timing/propagation, a naive retry could paper over it rather than solve at cause

### edgecases & pit-of-success

- **fresh profile, valid login** → must succeed (the bug)
- **fresh profile, login truly failed / timed out** → must still `ConstraintError` (failfast preserved)
- **profile already present** → unchanged ✓
- **username mismatch / domain collision** → extant clear-and-reauth path unchanged
- **mock mode (`AWS_SDK_MOCK=1`)** → sdk validation skipped; fix must not break this branch
  (`vaultAdapterAwsConfig.ts:112`)

## open questions & assumptions

### assumptions made

1. the trigger is **"profile freshly written this run"**, not `aws sso logout`. evidence: the
   wish logs show every `✓ written to ~/.aws/config` (new) → ✋ fail, and every
   `✓ profile already in ~/.aws/config` (present) → ✓ pass. logout is a **red herring** —
   the retry succeeds because the profile now pre-exists.
2. the failure surfaces at `vaultAdapterAwsConfig.ts:535-541` (`set` → `validateSsoSession`).
3. `validateSsoSession` validates via the **node aws sdk** `fromSSO()` first
   (`vaultAdapterAwsConfig.ts:89`), whereas `setupAwsSsoProfile` proves the profile works via
   the **cli** (`aws sso login` + `aws sts get-caller-identity`, `setupAwsSsoProfile.ts:101,123`).

### triaged questions

each tagged [answered] (settled now), [research] (needs runtime repro), or [wisher] (only the
human can answer).

- **Q3 — is the orphan newline only `:585`, or also `fillKeyrackKeys.ts` tree lines?** →
  **[answered].** read `fillKeyrackKeys.ts:245-267`: it emits the correct tree bucket
  (open 245-247, prefixed `set` body 250-263, close 266-267). the ONLY stray emission is
  `console.log('')` at `vaultAdapterAwsConfig.ts:585`. the fix = delete that one line. (verify
  the standalone-`set` gap before deletion.)

- **Q5 — installed aws-sdk + aws-cli versions?** → **[answered] (sdk) / [wisher] (cli).**
  `package.json`: `@aws-sdk/client-sso` `3.1041.0` vs `@aws-sdk/credential-provider-sso`
  `3.972.54` — **mismatched ~69 minor versions**. aws-sdk-js v3 packages normally move in
  lockstep; this skew is a prime suspect for why in-process `fromSSO` mishandles a freshly
  written `sso-session` token the cli accepts. user's `aws --version` is still [wisher].

- **Q1 — why does sdk `fromSSO()` reject a `sso-session` profile the cli just accepted?** →
  **[research].** candidate causes, now ranked by the "consistently fails" signal (deterministic
  first, since a race would be intermittent):
  1. aws-sdk internal version skew (Q5) — top suspect
  2. sdk vs cli sso-token-cache key/format divergence for v2 `sso-session` blocks
  3. in-process `process.env.AWS_*` contamination (the flow already mutates aws env:
     `setupAwsSsoProfile.ts:388`, `listAwsSsoAccounts.ts:18`)
  4. sts-cache propagation race — least likely (would be intermittent, not "consistent")

- **Q2 — fix at the validation seam, or upstream?** → **[research].** if Q1 resolves to the
  version skew, the fix is upstream (align the `@aws-sdk/*` versions), not a logic change.

- **Q4 — does a plain re-run WITHOUT `aws sso logout` also succeed?** → **[wisher].** the clean
  test that confirms/refutes the logout-is-a-distractor assumption (#1).

- **Q7 — does the wisher want a code fix, or just the explanation?** → **[wisher].** the wish's
  mood is interrogative ("why..."); confirm scope before the execution stone.

- **Q8 — does the wisher also have v1 (inline) profiles, not just v2?** → **[wisher].** a fix must
  support both regardless (`getAwsSsoProfileConfig` handles both, `:209-225`).

## what is awkward?

- **two sources of truth for "is this session valid?"** — the cli (used during setup) and the
  sdk (used during validation). they can disagree, and the disagreement is exactly the bug.
  the design fights the user's mental model: "i just logged in and it worked, why does saving
  say i'm not logged in?"
- **logout-as-folk-remedy** — the user learned a ritual (`aws sso logout`) that doesn't address
  the cause; it just resets state so the next run hits the "profile already present" happy path.
  any fix should make the ritual unnecessary *and* ideally make its irrelevance obvious.
- **diagnosis vs vision ordering** — the exact root cause is not yet runtime-confirmed. this
  vision is honest about that; committing to a fix shape before the diagnosis stone confirms the
  cli-vs-sdk seam would risk solving a symptom (e.g. blanket retry) instead of the cause.
- **bundling** — the "random newline" is a cosmetic, low-risk, independently-shippable fix
  riding along with a subtle reliability bug. worth keeping them as separate commits even if one
  branch.
