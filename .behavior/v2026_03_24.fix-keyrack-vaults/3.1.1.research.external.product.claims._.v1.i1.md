# research.external.product.claims: keyrack 1password integration

## summary

this research categorizes claims relevant to keyrack's 1password vault integration, labelled as [FACT], [SUMP], [KHUE], or [OPIN].

---

## claim.1: op cli secret reference format

**[FACT]** the 1password CLI uses `op://vault/item/field` URI format for secret references.

> "a secret reference is a URI: `op://<vault-name>/<item-name>/[section-name/]<field-name>`"
> — [1] 1Password Developer Docs, Secret Reference Syntax [official]

this is documented API behavior, not subject to change without versioned deprecation.

---

## claim.2: op read retrieves secrets

**[FACT]** the `op read` command retrieves secret values from 1password.

> "to retrieve the contents of a specific field, use `op read`."
> — [2] 1Password Developer Docs, read command reference [official]

documented CLI contract.

---

## claim.3: service accounts enable headless auth

**[FACT]** service accounts allow non-interactive authentication via `OP_SERVICE_ACCOUNT_TOKEN`.

> "service accounts require 1password CLI version 2.18.0 or later."
> — [3] 1Password Developer Docs, Use service accounts with CLI [official]

documented feature with version requirement.

---

## claim.4: op cli exit codes are limited

**[FACT]** op CLI uses only exit codes 0 (success) and 1 (failure).

> "currently, it seems like only two exit codes are used (0 and 1), where 0 indicates success and 1 indicates a failure."
> — [4] 1Password Community, exit codes discussion [practitioner]

keyrack must parse error messages to distinguish constraint vs malfunction errors.

---

## claim.5: biometric auth improves ux

**[SUMP]** users prefer biometric authentication for sensitive operations.

> "you can authenticate 1password CLI the same way you unlock your device, like with your fingerprint, face, Apple Watch, Windows Hello PIN, or device user password."
> — [5] 1Password Developer Docs, App Integration [official]

assumption: users find biometric more convenient than master password entry. may vary by user preference.

---

## claim.6: secrets should never persist in version control

**[FACT]** secrets in version control are a security anti-pattern.

> "secrets in Git repositories are the most hazardous anti-pattern — a developer might commit a Kubernetes Secret because 'it is just the prep password'."
> — [6] Cycode, Secrets Management Best Practices [practitioner]

> "in 2023, GitHub reported a scan of over 1.4 billion commits and detection of over 15 million secrets."
> — [6] Cycode, Secrets Management Best Practices [practitioner]

industry consensus across all security guidance.

---

## claim.7: ephemeral credentials reduce attack surface

**[FACT]** ephemeral credentials with automatic expiration are more secure than static keys.

> "systems should automatically request credentials with a defined time for expiration rather than rely on static keys."
> — [7] Doppler, Secrets Management for Ephemeral Environments [practitioner]

> "just-in-time dynamic secrets create temporary, automatically expired secrets for ephemeral access."
> — [8] Akeyless, Top Secrets Management Tools [practitioner]

convergent guidance from multiple practitioners.

---

## claim.8: op run conceals secrets in output

**[FACT]** `op run` conceals secrets printed to stdout by subprocesses.

> "if a subprocess used with `op run` prints a secret to stdout, the secret will be concealed by default."
> — [9] 1Password Developer Docs, Secrets Environment Variables [official]

documented security feature.

---

## claim.9: service accounts support least privilege

**[FACT]** service accounts can be scoped to specific vaults for least privilege access.

> "authenticate with a 1password service account to follow the principle of least privilege. you can scope service account access to specific vaults."
> — [9] 1Password Developer Docs, Secrets Environment Variables [official]

documented capability.

---

## claim.10: op cli is preferred over connect server for local dev

**[SUMP]** the CLI is better suited for local development than 1password connect server.

> "connect server is recommended for automated pipelines and servers, while the CLI is more suited for individual developers."
> — [10] 1Password Community, Connect Server discussion [practitioner]

assumption based on practitioner guidance. connect server may be viable for some workflows.

---

## claim.11: roundtrip validation catches broken exids early

**[SUMP]** validation of exid via `op read` at set time provides better UX than failure at unlock time.

no external citation — this is a design assumption for keyrack. fail-fast principle suggests early validation is better, but adds latency to set operation.

---

## claim.12: 9h default expiry is appropriate

**[KHUE]** what is the optimal default expiry time for daemon-cached secrets?

no external citation — keyrack design choice. 9h covers a workday but may be too long for highly sensitive keys or too short for extended sessions.

---

## claim.13: ubuntu apt installation is preferred

**[SUMP]** apt-based installation is preferred over manual binary download for op CLI on ubuntu.

> "to install 1password CLI via APT on Debian- and Ubuntu-based distributions, you need to add the key for the 1password apt repository."
> — [11] 1Password Developer Docs, CLI Get Started [official]

assumption: users prefer package manager for automatic updates. some may prefer manual control.

---

## claim.14: app integration is preferred auth method

**[SUMP]** 1password desktop app integration is the preferred authentication method for CLI.

> "you can use the 1password desktop app integration to quickly and securely sign in to 1password CLI."
> — [5] 1Password Developer Docs, App Integration [official]

assumption based on official positioning. some users may prefer explicit `op signin`.

---

## claim.15: keyrack should not create items in 1password

**[OPIN]** keyrack should only read from 1password, never create or modify items.

no external citation — design opinion. 1password is a refed vault, not owned. keyrack stores pointers, not secrets. this maintains separation of concerns.

---

## claim.16: daemon memory is sufficiently secure

**[SUMP]** in-memory storage in the daemon process is secure enough for ephemeral secrets.

no external citation — assumption. memory can be accessed by root, memory dumps, or malware. for most use cases, process isolation is sufficient.

---

## claim.17: exit code 2 for constraint errors is appropriate

**[OPIN]** use of exit code 2 for constraint errors (user must fix) vs 1 for malfunction errors (transient/config) is good practice.

no external citation — design opinion aligned with keyrack's error semantics. provides actionable signal to callers.

## key insights

2. **assumptions need validation** — user preferences for biometric, apt install, 9h expiry
3. **exit code limitation is a fact** — keyrack must work around op CLI's limited exit codes
4. **ephemeral credentials are industry consensus** — not just an opinion
