# vision: keyrack ephemeral and 1password vaults

## the outcome world

### before

coco needs `STRIPE_ADMIN_API_KEY` for local dev, e.g., to setup some test creds. the key is in 1password. today:

1. open 1password
2. find the key
3. copy it
4. `export STRIPE_ADMIN_API_KEY=sk_live_...` in terminal
5. run the service
6. key lives in bash history forever
7. key dies when terminal closes — repeat steps 1-6

the problem: **sensitive keys shouldn't persist locally.**

| option | problem |
|--------|---------|
| `keyrack set --vault os.secure` | persists to disk — SSH key protection insufficient for this sensitivity |
| `keyrack set --vault os.direct` | persists to disk in plaintext |
| `export KEY=...` | bash history pollution, manual copy, drift when 1password rotates |

what coco wants:
1. **ephemeral lifespan** — keys live only in daemon memory, die with session
2. **remote source of truth** — 1password holds the key, keyrack points at it
3. **sync on unlock** — when 1password rotates, keyrack fetches the new value

### after

two paths for sensitive keys — both ensure ephemeral lifespan in keyrack:

**path 1: direct to daemon**

for keys coco enters manually (e.g., a key not yet in 1password):

```sh
$ rhx keyrack set --key STRIPE_ADMIN_API_KEY --vault os.daemon --env prod

enter secret for STRIPE_ADMIN_API_KEY: ********

🔐 keyrack set (org: ehmpathy, env: prod)
   └─ ehmpathy.prod.STRIPE_ADMIN_API_KEY
      ├─ mech: EPHEMERAL_VIA_SESSION
      ├─ vault: os.daemon
      └─ expires in: 9h
```

key lives in daemon memory only. no disk writes. no bash history. session logout = gone.

**path 2: through 1password**

for keys that live in 1password (source of truth):

```sh
# point keyrack at your 1password item
$ rhx keyrack set --key STRIPE_ADMIN_API_KEY --vault 1password --env prod

enter 1password uri (e.g., op://vault/item/field): op://prod-keys/stripe-admin/credential

🔐 keyrack set (org: ehmpathy, env: prod)
   └─ ehmpathy.prod.STRIPE_ADMIN_API_KEY
      ├─ mech: PERMANENT_VIA_REFERENCE
      ├─ vault: 1password
      ├─ exid: op://prod-keys/stripe-admin/credential
      │
      └─ verify roundtrip...
         ├─ ✓ op read
         └─ ✓ stored
```

keyrack stores the pointer, not the secret. unlock fetches from 1password → daemon:

```sh
$ rhx keyrack unlock --env prod

🔓 keyrack unlock
   └─ ehmpathy.prod.STRIPE_ADMIN_API_KEY
      ├─ vault: 1password
      ├─ exid: op://prod-keys/stripe-admin/credential
      └─ expires in: 9h

# biometric prompt from 1password → key flows to daemon
```

**path 2b: op cli not installed**

```sh
$ rhx keyrack set --key STRIPE_ADMIN_API_KEY --vault 1password --env prod

🔐 keyrack set
   └─ ✗ op cli not found

   to install on ubuntu:

   1. add 1password gpg key:
      curl -sS https://downloads.1password.com/linux/keys/1password.asc | \
        sudo gpg --dearmor --output /usr/share/keyrings/1password-archive-keyring.gpg

   2. add apt repository:
      echo "deb [signed-by=/usr/share/keyrings/1password-archive-keyring.gpg] \
        https://downloads.1password.com/linux/debian/$(dpkg --print-architecture) stable main" | \
        sudo tee /etc/apt/sources.list.d/1password.sources.list

   3. install op cli:
      sudo apt update && sudo apt install 1password-cli

   4. verify installation:
      op --version

   5. authenticate:
      - preferred: enable "integrate with 1password cli" in 1password app settings
      - alternative: op signin

$ echo $?
2
```

failfast with exit code 2 (constraint error). user must install op cli before retry.

### the "aha" moment

> "sensitive keys never touch disk. they either go straight to daemon or flow through 1password. either way, they die with my session — and when 1password rotates a key, i get the new value on next unlock."

## user experience

### usecase 1: sensitive key direct to daemon

**goal:** set a sensitive key with ephemeral lifespan, no disk persistence

**when:** key not yet in 1password, or coco prefers manual entry

**inputs:**
- `--key KEY_NAME` — the key to set
- `--vault os.daemon` — ephemeral storage
- `--env prod|test|...` — target environment

**outputs:**
- key stored in daemon memory only
- 9h default expiry
- no host manifest entry

**timeline:**
1. `keyrack set --key X --vault os.daemon --env prod` → prompt → stored in daemon
2. `keyrack get --key X --env prod` → returns value
3. session ends or 9h passes → key gone

### usecase 2: sensitive key through 1password

**goal:** keyrack points at 1password, 1password is source of truth

**when:** key lives in 1password, coco wants sync on unlock

**inputs (set):**
- `--key KEY_NAME` — the key to set
- `--vault 1password` — 1password storage
- prompt for exid (secret reference uri)

**outputs (set):**
- host manifest entry with `vault: 1password`, `exid: op://...`
- no secret stored in keyrack — only the pointer
- roundtrip validated via `op read $exid`

**inputs (unlock):**
- `--env prod|test|...` — target environment
- 1password cli must be installed and authed

**outputs (unlock):**
- 1password prompted for biometric/auth
- secret fetched and cached in daemon (ephemeral)
- standard keyrack get flow works

**timeline:**
1. `keyrack set --key X --vault 1password --env prod` → prompt for exid → pointer stored
2. `keyrack unlock --env prod` → 1password auth → secret in daemon
3. `keyrack get --key X --env prod` → returns value from daemon
4. secret never written to keyrack storage
5. next session: unlock fetches fresh value from 1password (picks up rotations)

### usecase 3: ci with 1password service accounts

**goal:** ci pulls secrets from 1password without human interaction

**inputs:**
- `OP_SERVICE_ACCOUNT_TOKEN` env var set in ci
- host manifest already has 1password exids

**outputs:**
- unlock pulls from 1password automatically
- no prompts (service account token auth)

**timeline:**
1. ci job starts with `OP_SERVICE_ACCOUNT_TOKEN`
2. `keyrack unlock --env test` → 1password service account auth
3. secrets flow to daemon → tests run

**ref:** [1password service accounts + cli](https://developer.1password.com/docs/service-accounts/use-with-1password-cli/) — requires cli 2.18.0+

## mental model

### the core insight

> **keyrack is not a vault.** keyrack is a firewall, dispatcher, and broker. vaults are the source of truth for keys. keyrack knows where keys live and how to get them to the daemon for your session.

the daemon is the session keyrack — what your sessions unlock and use. but the daemon is not the source of truth. the vaults are.

### what keyrack does

| role | description |
|------|-------------|
| **firewall** | controls which keys are accessible, to whom, in which env |
| **dispatcher** | routes unlock requests to the correct vault |
| **broker** | daemon holds unlocked keys for your session |

### vault types

| type | description | examples |
|------|-------------|----------|
| **owned** | vaults owned by keyrack | os.secure, os.direct, os.daemon |
| **refed** | vaults refed by keyrack | 1password, aws.iam.sso |

### how users describe it

> "keyrack doesn't store my secrets — it knows where they live. when i unlock, it fetches from the vault and holds them in daemon memory for my session."

### analogies

| component | analogy |
|-----------|---------|
| keyrack | concierge who knows which safe holds your valuables |
| daemon | your pocket — holds keys for your session, empties when you leave |
| 1password | bank vault in another city — keyrack has the address |
| os.secure | safe in your house — local but encrypted |
| os.direct | drawer in your desk — local, plaintext |

### terms

| user says | keyrack says |
|-----------|--------------|
| "ephemeral key" | key in daemon (session-scoped) |
| "my 1password key" | 1password vault + exid |
| "secret reference" | exid (external id) |
| "where is it?" | `op://vault/item/field` format |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| no local persistence | yes | os.daemon = memory only, 1password = pointer only |
| ephemeral lifespan | yes | 9h default, dies on logout |
| 1password as source of truth | yes | set stores pointer, unlock fetches |
| sync on rotation | yes | each unlock fetches fresh from 1password |
| no bash history pollution | yes | prompts via stdin |
| ci compatibility | yes | OP_SERVICE_ACCOUNT_TOKEN |

### pros

1. **no local persistence** — sensitive keys never written to disk
2. **ephemeral by default** — daemon keys die with session
3. **1password sync** — rotations picked up on next unlock
4. **consistent ux** — same `set/unlock/get` flow as other vaults

### cons

1. **truly ephemeral** — logout = keys gone, must unlock again
2. **1password requires op cli** — extra setup step
3. **1password requires auth** — biometric prompt on unlock (or service account)
4. **no offline mode** — 1password needs connectivity

### edgecases and pit of success

| edgecase | pit of success |
|----------|----------------|
| op cli not installed | failfast exit 2 (constraint), guide user through ubuntu install steps |
| op cli not authenticated | failfast exit 2 (constraint) with `op signin` hint |
| invalid exid format | validate roundtrip via `op read $exid` on set |
| daemon not active | auto-start daemon on set |
| session ends mid-work | unlock again to refetch |

## open questions & assumptions

### assumptions

1. **os.daemon needs no host manifest entry** — pure ephemeral, no persistence
2. **1password set stores only exid** — no secret flows through set
3. **1password unlock fetches to daemon** — standard unlock flow
4. **op cli is the interface** — not 1password connect api
5. **biometric auth is acceptable ux** — users prefer it for security
6. **ubuntu only for guided install** — op cli install guidance targets ubuntu machines only

### triaged questions

#### [answered] os.daemon: should it persist to host manifest?

**answer:** no manifest entry. the wish explicitly says "without persistent storage in any vault" — pure ephemeral, daemon memory only.

#### [answered] os.daemon: what about `unlock`?

**answer:** os.daemon keys are already in daemon. `unlock --env prod` reports them as "already unlocked" and skips. no re-prompt.

#### [answered] 1password: exid validation?

**answer:** validate roundtrip on set via `op read $exid`. confirms:
1. op cli is installed and authenticated
2. exid is valid and reachable
3. secret is retrievable (roundtrip works)

fail-fast on set if any step fails. better ux than a broken exid found at unlock time.

#### [answered] 1password: guided setup?

**answer:** v1 = manual exid entry. guided picker is future enhancement (scope creep for initial implementation).

### external research needed

1. **[research] op cli install best practices** — verify ubuntu install steps, confirm app integration is preferred auth method
2. **[answered] op cli + service accounts** — `op read` works with `OP_SERVICE_ACCOUNT_TOKEN` (cli 2.18.0+). service accounts included on all plans (Individual, Family, Teams, Business) with rate limits per tier. [ref](https://developer.1password.com/docs/service-accounts/get-started/)
3. **[research] op cli auth modes** — document biometric vs master password vs service account

## what is awkward?

### os.daemon: "set" without persistence?

the word "set" implies persistence. but `os.daemon` set is ephemeral.

**resolution:** this is intentional — os.daemon is the ephemeral vault. "set" means "place in daemon memory." docs clarify the ephemeral nature.

### 1password: "set" that doesn't write a secret?

`keyrack set --vault 1password` doesn't write a secret — it stores a pointer.

**resolution:** prompt for exid, not secret. the contract is consistent (set stores a value), but what's stored differs (pointer vs secret value).

### 1password: refed vault behavior

keyrack can't create or delete items in 1password — 1password is a refed vault, not owned.

**resolution:** `keyrack del` removes the pointer from keyrack (the host manifest entry). the 1password item remains untouched. this differs from owned vaults (os.*) where keyrack manages the secret directly.

### unlock flow for os.daemon keys

if keys are set directly to os.daemon, what happens on `keyrack unlock --env prod`?

**resolution:** unlock reports them as "already unlocked" with source vault os.daemon. no re-prompt needed — they're already in daemon.
