# research.external.product.references: keyrack 1password integration

## summary

this research enumerates references that ground the keyrack 1password vault integration. each lesson establishes terms, concepts, or patterns we leverage.

---

## lesson.1: secret reference URI syntax

**term:** secret reference URI

the 1password CLI uses a structured URI format for secret references.

> "a secret reference is a URI that looks like this: `op://<vault-name>/<item-name>/[section-name/]<field-name>`"
> — [1] 1Password Developer Docs, Secret Reference Syntax [official]

**keyrack leverage:** the `exid` field in host manifest stores this URI format for 1password vault entries.

---

## lesson.2: op read command

**term:** op read

the `op read` command retrieves secret values from 1password.

> "to retrieve the contents of a specific field, use `op read`."
> — [2] 1Password Developer Docs, read command reference [official]

> "you can use `op read` with secret references directly in your executable or with environment variables."
> — [3] 1Password Developer Docs, Load secrets into executables [official]

**keyrack leverage:** unlock invokes `op read $exid` to fetch secrets from 1password into daemon memory.

---

## lesson.3: service account authentication

**term:** OP_SERVICE_ACCOUNT_TOKEN

service accounts enable non-interactive authentication for CI/CD.

> "service accounts require 1password CLI version 2.18.0 or later."
> — [4] 1Password Developer Docs, Use service accounts with CLI [official]

> "each service account has a service account token that you can provide as an environment variable for authentication."
> — [5] 1Password Developer Docs, Service Accounts overview [official]

**keyrack leverage:** CI jobs set `OP_SERVICE_ACCOUNT_TOKEN` env var, keyrack unlock detects and uses it automatically.

---

## lesson.4: biometric app integration

**term:** app integration, biometric auth

the 1password desktop app enables biometric authentication for CLI.

> "you can use the 1password desktop app integration to quickly and securely sign in to 1password CLI. you can authenticate 1password CLI the same way you unlock your device."
> — [6] 1Password Developer Docs, App Integration [official]

> "1password accounts can only be accessed through 1password CLI after the user provides explicit biometric authorization."
> — [7] 1Password Developer Docs, App Integration Security [official]

**keyrack leverage:** preferred auth method for local dev. users get biometric prompt on `keyrack unlock`.

---

## lesson.5: exit code semantics

**term:** exit codes, error categorization

op CLI uses limited exit codes; keyrack must interpret error messages.

> "1password CLI now correctly exits with code 1 instead of 0 when it encounters server error codes that are not recognized by the CLI."
> — [8] 1Password CLI Release Notes [official]

> "currently, it seems like only two exit codes are used (0 and 1), where 0 indicates success and 1 indicates a failure."
> — [9] 1Password Community, exit codes discussion [practitioner]

**keyrack leverage:** parse error messages to distinguish constraint (exit 2) vs malfunction (exit 1).

---

## lesson.6: apt installation pattern

**term:** apt repository, GPG key

ubuntu installation uses the apt repository with signed packages.

> "to install 1password CLI via APT on Debian- and Ubuntu-based distributions, you need to add the key for the 1password apt repository."
> — [10] 1Password Developer Docs, CLI Get Started [official]

> "add the GPG key via curl and gpg dearmor, then configure apt sources."
> — [11] 1Password Support, Install Linux [official]

**keyrack leverage:** when op CLI not found, keyrack displays ubuntu install instructions.

---

## lesson.7: subprocess security with op run

**term:** op run, environment variable injection

`op run` provides secure subprocess execution with secrets as env vars.

> "1password CLI loads the specified secrets, then runs the provided command in a subprocess with the secrets made available as environment variables only for the duration of the process."
> — [12] 1Password Developer Docs, run command [official]

> "if a subprocess used with `op run` prints a secret to stdout, the secret will be concealed by default."
> — [13] 1Password Developer Docs, Secrets Environment Variables [official]

**keyrack leverage:** alternative pattern for future enhancement — `op run` instead of `op read` + manual env export.

---

## lesson.8: connect server vs CLI tradeoffs

**term:** connect server, REST API

1password offers both CLI and connect server for different use cases.

> "connect server is recommended for automated pipelines and servers, while the CLI is more suited for individual developers."
> — [14] 1Password Community, Connect Server discussion [practitioner]

> "a 1password connect server allows you to securely access items in your company's apps and cloud infrastructure via a private REST API."
> — [15] 1Password Developer Docs, Connect Get Started [official]

**keyrack leverage:** keyrack uses CLI, not connect server. CLI is simpler for local dev and CI.

---

## lesson.9: secrets management anti-patterns

**term:** anti-patterns, version control secrets

common mistakes to avoid with secrets.

> "secrets in plain text make it much easier for an attacker to get into the target system."
> — [16] OWASP Secrets Management Cheat Sheet [academic]

> "secrets in Git repositories are the most hazardous anti-pattern."
> — [17] Cycode, Secrets Management Best Practices [practitioner]

> "in 2023, GitHub reported detection of over 15 million secrets across 1.4 billion scanned commits."
> — [17] Cycode, Secrets Management Best Practices [practitioner]

**keyrack leverage:** validates design — keyrack stores pointers (exids), not secrets.

---

## lesson.10: ephemeral credentials pattern

**term:** ephemeral credentials, just-in-time secrets

ephemeral credentials reduce attack surface.

> "systems should automatically request credentials with a defined time for expiration rather than rely on static keys."
> — [18] Doppler, Secrets Management for Ephemeral Environments [practitioner]

> "just-in-time dynamic secrets create temporary, automatically expired secrets for ephemeral access."
> — [19] Akeyless, Top Secrets Management Tools [practitioner]

> "dynamic secrets eliminate vulnerability windows by automatic expiration of credentials on-demand."
> — [20] Infisical, Secrets Management Best Practices [practitioner]

**keyrack leverage:** validates os.daemon design — 9h default expiry, keys die with session.

---

## lesson.11: principle of least privilege

**term:** least privilege, scoped access

scope access to only required secrets.

> "follow the principle of least privilege with 1password service accounts, which support access restrictions to only required items."
> — [21] 1Password Developer Docs, CLI Best Practices [official]

> "authenticate with a 1password service account to follow the principle of least privilege. you can scope service account access to specific vaults."
> — [13] 1Password Developer Docs, Secrets Environment Variables [official]

**keyrack leverage:** recommend scoped service accounts for CI, not full-access tokens.

---

## lesson.12: field types and item structure

**term:** field types, item create

1password items have structured field types.

> "use the fieldType with assignment statements and the type with an item JSON template."
> — [22] 1Password Developer Docs, Item Fields [official]

> "to create a new item in your 1password account and assign information to it, use the `op item create` command."
> — [23] 1Password Developer Docs, Create Items [official]

**keyrack leverage:** keyrack only reads from 1password (refed vault), does not create items.

---

## lesson.13: secret rotation patterns

**term:** secret rotation, version control

how secrets are rotated in 1password.

> "when you rotate a secret in 1password, applications that use the secret reference will automatically get the new value on the next read."
> — [1] 1Password Developer Docs, Secret Reference Syntax [official]

**keyrack leverage:** validates 1password sync design — each unlock fetches fresh value, picks up rotations.

---

## lesson.14: owned vs refed vault distinction

**term:** owned vault, refed vault, source of truth

keyrack distinguishes vault ownership for different behaviors.

> "keyrack is not a vault. keyrack is a firewall, dispatcher, and broker. vaults are the source of truth for keys."
> — [24] keyrack identity brief, internal [internal]

**keyrack leverage:** 1password is a refed vault — keyrack stores pointer (exid), not secret. del removes pointer, not 1password item.
