# research.reflection.product.rootcause: keyrack ephemeral and 1password vaults

## what is the symptom?

**observable problem**: developer cannot use sensitive keys for local development without either:
1. disk persistence (os.secure, os.direct)
2. bash history pollution (`export KEY=...`)
3. manual copy-paste from 1password

**how it manifests**:
- developer opens 1password, copies key, pastes to terminal
- key appears in bash history
- key dies when terminal closes
- developer repeats steps 1-4 each session
- if 1password rotates key, developer uses stale value until they notice

**when it occurs**:
- every time developer needs a sensitive key (prod api keys, admin tokens)
- every new terminal session
- every time 1password rotates a secret

---

## 5 whys tree

### branch 1: disk persistence is unacceptable

1. **why** can't dev store sensitive keys in os.secure or os.direct?
   → because both persist to disk, and disk = attack surface for sensitive keys

2. **why** is disk an attack surface?
   → because local machine compromise (malware, theft) exposes all disk contents

3. **why** can't we encrypt disk contents?
   → we do (os.secure uses ssh-key encryption), but encryption-at-rest is insufficient for highly sensitive keys

4. **why** is encryption-at-rest insufficient?
   → because ssh key is unlocked in memory, so encrypted secrets are decryptable while machine is on

5. **why** does this matter?
   → **root cause: sensitive keys have security requirements that exceed disk persistence**

### branch 2: 1password copy-paste is manual and error-prone

1. **why** does dev copy-paste from 1password?
   → because keyrack doesn't fetch from 1password automatically

2. **why** doesn't keyrack fetch from 1password?
   → because 1password is an external system, keyrack only has owned vaults

3. **why** is copy-paste bad?
   → because it pollutes bash history and requires manual work each session

4. **why** is manual work bad?
   → because it doesn't pick up rotations automatically

5. **why** don't rotations propagate?
   → **root cause: no automated sync from 1password source of truth to keyrack session**

### branch 3: no ephemeral vault option

1. **why** can't dev set a key that dies with session?
   → because all keyrack vaults persist to disk or file

2. **why** do vaults persist?
   → because vault design assumed persistence was always desired

3. **why** was persistence assumed?
   → because most keys (api keys, db creds) are stable and reused across sessions

4. **why** doesn't this cover all cases?
   → because some keys (admin tokens, sensitive prod keys) should not persist

5. **why** should they not persist?
   → **root cause: ephemeral keys need ephemeral vault, but keyrack lacked one**

---

## root cause summary

| layer | issue | is root cause? |
|-------|-------|----------------|
| symptom | dev copy-pastes keys, adds to history | no |
| surface | keyrack owned vaults persist to disk | no |
| deeper | no 1password integration | no |
| deeper | no ephemeral vault option | **yes** |
| root | sensitive keys require ephemeral lifespan, but no vault supported it | **yes** |
| root | 1password is source of truth, but keyrack couldn't reference it | **yes** |

---

## recommendation

**address root causes**:

1. **ephemeral vault**: add `os.daemon` vault that stores keys in daemon memory only. no disk writes. keys die with session. addresses: "sensitive keys require ephemeral lifespan"

2. **1password integration**: add `1password` vault that stores exid (pointer) in host manifest. unlock fetches from 1password → daemon. addresses: "1password is source of truth but keyrack couldn't reference it"

both root causes are addressed by this behavior. no follow-up needed.
