# research.reflection.product.audience: keyrack ephemeral and 1password vaults

## who is the audience?

### primary: developer with sensitive keys

the developer who needs `STRIPE_ADMIN_API_KEY` or similar high-value secrets for local development.

- **role**: backend engineer, platform engineer, devops
- **context**: local machine, ssh terminal, needs prod-level keys for debug or test setup
- **constraints**: cannot store sensitive keys on disk, cannot pollute bash history

### secondary: ci system with 1password service account

automated pipelines that need secrets from 1password.

- **role**: github actions, jenkins, circleci workflows
- **context**: headless environment, no human interaction possible
- **constraints**: must authenticate via OP_SERVICE_ACCOUNT_TOKEN, no biometric prompts

### tertiary: security engineer who audits key access

the person who reviews how secrets flow through systems.

- **role**: security team, compliance officer
- **context**: audit logs, access reviews, incident response
- **constraints**: needs to trace where keys came from, how long they lived

---

## why do they care?

### developer pain points

| pain | relief |
|------|--------|
| bash history pollution | keyrack prompts via stdin, no echo |
| copy-paste from 1password | keyrack unlock fetches automatically |
| keys persist on disk | os.daemon = memory only |
| forgot to rotate | each unlock fetches fresh from 1password |
| different key per terminal | daemon = session-scoped, consistent |

### ci pain points

| pain | relief |
|------|--------|
| manual secret copy to ci | service account pulls from 1password |
| secrets drift between ci and 1password | single source of truth in 1password |
| credential exposure in logs | keyrack conceals, 1password conceals |

### security pain points

| pain | relief |
|------|--------|
| keys on disk = breach target | ephemeral = no secret to steal |
| unclear key provenance | host manifest shows vault + exid |
| no expiry on local keys | 9h default, dies on logout |

---

## how much do they care?

| audience | priority | urgency | stakes |
|----------|----------|---------|--------|
| developer | high | medium | daily workflow friction, potential security incident if misstep |
| ci system | high | low | automation reliability, must work without human intervention |
| security | medium | low | compliance posture, incident response clarity |

**priority** = how important is this to their role
**urgency** = how soon do they need it
**stakes** = what happens if we get it wrong

---

## what do they care about?

### developer success criteria

- **must work**: set → unlock → get works without friction
- **must be safe**: no disk writes for sensitive keys
- **must be fast**: biometric prompt is acceptable, lengthy setup is not
- **must be consistent**: same key in all terminals via daemon

### developer tradeoffs

| would accept | would reject |
|--------------|--------------|
| biometric prompt on unlock | master password every time |
| 9h expiry | 1h expiry |
| op cli install once | custom 1password server setup |
| manual exid entry | guided picker (nice to have, not required) |

### ci success criteria

- **must work headless**: OP_SERVICE_ACCOUNT_TOKEN = no prompts
- **must be reliable**: unlock should not flake
- **must be auditable**: logs show which keys were accessed

### security success criteria

- **ephemeral by default**: no persistent storage for sensitive keys
- **clear provenance**: host manifest shows where key lives
- **minimal blast radius**: key expires, dies with session

---

## how will they experience it?

### developer journey

1. **first use**: install op cli, run `keyrack set --vault 1password`, prompted for exid
2. **daily use**: `keyrack unlock --env prod`, biometric prompt, keys in daemon
3. **access keys**: `keyrack get --key X`, returns value instantly from daemon
4. **new session**: must unlock again (ephemeral)
5. **rotation**: next unlock gets fresh value from 1password (transparent)

### ci journey

1. **setup once**: configure host manifest with 1password exids
2. **each run**: job sets OP_SERVICE_ACCOUNT_TOKEN, runs `keyrack unlock`
3. **tests run**: `keyrack get` returns secrets from daemon
4. **job ends**: daemon dies, keys gone

### touchpoints

| moment | experience |
|--------|------------|
| op cli not installed | clear error with ubuntu install instructions |
| invalid exid | immediate failure on set, not delayed to unlock |
| biometric prompt | 1password native ui, user is familiar |
| key expired | hint to unlock again |
| rotation picked up | transparent, user sees fresh value |

### friction points to avoid

| friction | mitigation |
|----------|------------|
| confuse exid format | prompt shows example `op://vault/item/field` |
| forget which vault | confirmation shows vault=1password |
| wonder if key rotated | unlock always fetches fresh |
| wonder where key is | status shows vault and exid |

---

## summary

the primary audience is a developer who needs sensitive keys without disk persistence or bash history pollution. the secondary audience is ci that needs headless secret access. both care about reliability and security. the experience should be minimal friction: install op cli once, enter exid once, unlock each session via biometric, keys flow automatically.
