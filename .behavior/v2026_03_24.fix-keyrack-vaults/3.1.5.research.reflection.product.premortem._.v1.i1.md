# research.reflection.product.premortem: keyrack ephemeral and 1password vaults

## premortem: imagine we shipped and it failed

project forward: we shipped this, and it failed badly. why?

### failure scenario 1: op cli install friction killed adoption

users tried `keyrack set --vault 1password`, saw "op cli not found", read 4-step ubuntu instructions, gave up. they went back to copy-paste from 1password app. the feature exists but no one uses it.

### failure scenario 2: ephemeral lifespan caused data loss

developer worked for 10 hours, keys expired at 9h, daemon purged keys mid-request. error messages confused them. they lost work because downstream service failed without the key. they switched back to os.secure (persistent) to avoid this.

### failure scenario 3: roundtrip validation broke ci

ci jobs set keys via `keyrack set --vault 1password`, roundtrip validation called `op read`, 1password rate limited the ci at high concurrency. ci failed intermittently. team reverted to env vars.

---

## what obvious signs did we ignore?

- **op cli is not always installed**: most developers don't have it, even if they use 1password app
- **biometric prompts disrupt flow**: every unlock = context switch to 1password app
- **9h is arbitrary**: some sessions run longer, some run shorter
- **rate limits exist**: 1password has api rate limits, especially for service accounts
- **exid format is unfamiliar**: `op://vault/item/field` is not obvious to non-1password-cli users

---

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| developers will install op cli | they won't, adoption zero | medium |
| biometric prompt is acceptable | users find it annoying, skip 1password vault | low |
| 9h expiry is sufficient | keys expire mid-work, user loses flow | medium |
| roundtrip validation is instant | network latency or rate limits cause friction | low |
| exid format is intuitive | users enter wrong format, fail on set | medium |
| service account works in ci | token expires, ci fails silently | low |
| op cli version >= 2.18.0 | older versions installed, service account fails | low |

---

## what is the nightmare scenario?

**secret leakage via debugging**

developer runs `keyrack get --key STRIPE_ADMIN_API_KEY --env prod`, copies output to debug log, pushes to github. the secret was ephemeral in keyrack, but now it's in version control.

**why this is bad**:
- keyrack did its job (ephemeral in daemon)
- but user behavior leaked the secret
- we can't prevent user from echoing output
- 1password rotates key, but old value is now in git history

**mitigation limitation**: keyrack cannot prevent user from copy-paste or redirect to file. we can only document best practices.

---

## what mitigations to consider?

| risk | mitigation | cost of mitigation |
|------|------------|--------------------|
| op cli install friction | one-liner install command, detect platform | low (documentation) |
| 9h expiry too short | configurable ttl via flag or env var | medium (new feature) |
| roundtrip validation latency | cache validation result, skip on repeated set | medium (complexity) |
| exid format confusion | show example in prompt, validate format before op call | low (validation) |
| rate limits in ci | batch unlock instead of per-key, use connect server | high (arch change) |
| secret echoed to stdout | warn user, add `--quiet` mode, mask output | low (ux polish) |
| op cli version too old | check version on first use, warn if < 2.18.0 | low (validation) |

---

## summary

the most likely failures are:
1. adoption blocked by op cli install friction
2. 9h expiry causes surprise key loss
3. exid format confusion causes bad ux on set

mitigations are mostly low cost (documentation, validation, warnings). the nightmare scenario (secret leaked via user behavior) cannot be fully prevented — keyrack can only make ephemeral storage safe, not user actions.
