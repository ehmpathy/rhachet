# vision: fix keyrack error helper

## the outcome world

### before

a bot runs `rhx keyrack get --key SOME_API_KEY --owner ehmpathy --env prod` and gets:

```
credential 'ehmpathy.prod.SOME_API_KEY' is locked. unlock it first.
fix: rhx keyrack unlock --env prod --key SOME_API_KEY
```

the bot tries the suggested fix. it fails or unlocks the wrong owner's keys. wild goose chase ensues. the bot doesn't realize `--owner ehmpathy` was required. minutes wasted.

### after

same scenario, but now:

```
credential 'ehmpathy.prod.SOME_API_KEY' is locked. unlock it first.
fix: rhx keyrack unlock --owner ehmpathy --env prod --key SOME_API_KEY
```

the bot copies the fix verbatim. it works. done.

### the "aha" moment

the error message becomes copy-paste-able. no guesswork about which owner to use. the fix command is complete and correct.

## user experience

### usecases

1. **bot gets a locked key error** - sees exact command to run, runs it, proceeds
2. **human gets a locked key error** - copies the fix, runs it, proceeds
3. **absent key error** - same pattern: `rhx keyrack set --owner ehmpathy --key ... --env ...`

### contract inputs & outputs

| field | before | after |
|-------|--------|-------|
| `fix` for locked | `rhx keyrack unlock --env X --key Y` | `rhx keyrack unlock --owner Z --env X --key Y` |
| `fix` for absent | `rhx keyrack set --key Y --env X` | `rhx keyrack set --owner Z --key Y --env X` |

### timeline

1. call `getKeyrackKeyGrant()` with owner context
2. key not found in daemon
3. infer status as 'locked' or 'absent'
4. return error with fix command that includes `--owner` from context

## mental model

### how users describe it

"when keyrack tells me to unlock, the command it gives me actually works"

### analogies

like git telling you `git push --set-upstream origin my-branch` instead of just `git push`. the suggestion is complete.

### terms

| user says | we say |
|-----------|--------|
| "the fix command" | `fix` field in `KeyrackGrantAttempt` |
| "owner" | `context.owner` - the identity scope for keys |

## evaluation

### how well does it solve the goals?

perfectly. bots no longer chase ghosts trying to figure out which owner to unlock.

### pros

- zero additional user input needed
- fixes are copy-paste-able
- bots succeed on first try
- consistent with unlock CLI contract

### cons

- none significant
- slightly longer error messages (acceptable)

### edgecases

| case | behavior |
|------|----------|
| owner is null/default | omit `--owner` flag (current behavior preserved) |
| owner is specified | include `--owner $owner` in fix |

## open questions & assumptions

### assumptions

- `context.owner` is always populated correctly when an owner is in play
- the fix should mirror the invocation pattern that would succeed

### questions

- should we also add `--owner` to the `blocked` status fix? (less clear since blocked is about credential format, not unlock)

### must validate

### must research

- none

### uncomfortable tradeoffs
