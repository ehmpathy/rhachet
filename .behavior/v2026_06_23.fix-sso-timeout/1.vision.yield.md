# vision: fix sso timeout

## the outcome world

### before

a mechanic runs `rhx git.repo.test --what integration --mode apply`. keyrack unlock kicks off, which triggers `aws sso login`. the browser prompt pops up and... human goes to grab coffee.

two minutes later the aws cli times out. keyrack throws `UnexpectedCodePathError: aws sso login failed`. the mechanic sees a "malfunction" — code 1 — and interprets this as "something broke, let me find a workaround."

the mechanic burns 50 tokens trying alternative approaches. maybe it tries a different unlock strategy. maybe it starts running tests without credentials. maybe it invents a mock. the human returns to find wasted work and a confused robot.

### after

same scenario. human grabs coffee. aws cli times out.

now keyrack throws a `ConstraintError` with exit code 2:

```
✋ keyrack unlock
   └─ aws sso login timed out
      ├─ status: blocked 🚫
      │  └─ human did not respond to browser sso prompt
      ├─ robot: halt and escalate
      │  ├─ do not proceed
      │  ├─ do not find workarounds
      │  └─ do not skip credentials
      └─ fix: rhx keyrack unlock --owner ehmpath --env test
```

the mechanic sees exit code 2 — a constraint error — and knows: this is not a malfunction i can fix. this is a human constraint. i must halt and escalate.

### the "aha" moment

the value clicks when the robot immediately stops trying to work around the missing credentials and instead surfaces a clear action for the human. no wasted tokens. no creative hacks. no silent failures.

## user experience

### usecases

| actor | goal | trigger |
|-------|------|---------|
| mechanic | run integration tests | `rhx git.repo.test --what integration` |
| mechanic | run acceptance tests | `rhx git.repo.test --what acceptance` |
| mechanic | unlock credentials | `rhx keyrack unlock --owner ehmpath --env test` |

### contract inputs & outputs

**input**: aws sso login attempt via `initiateAwsSsoAuth()`

**output on timeout**: `ConstraintError` with:
- exit code: 2
- message: clear statement that human didn't respond
- hint: explicit instructions to halt and escalate
- stderr: loud, unmistakable error output

### timelines

1. mechanic invokes keyrack unlock (directly or via git.repo.test)
2. keyrack calls `aws sso login --profile ...`
3. browser prompt opens, verification code shown
4. **timeout path**: human doesn't respond within ~2 minutes
5. aws cli exits with non-zero code
6. keyrack throws `ConstraintError` with loud stderr
7. mechanic halts and asks human to fulfill the prompt
8. human runs unlock, completes browser flow
9. mechanic resumes work

## mental model

### how would users describe this to a friend?

> "when sso times out, it stops and tells me to go auth instead of trying to work around it."

### analogies

- **bouncer at a club**: "you need to show your ID. i'm not letting you in. i'm not finding a back door. show your ID."
- **door lock**: "the key is in the human's pocket. robot waits at the door. robot does not pick the lock."

|--------|----------|
| ConstraintError | "human needs to do something" |
| MalfunctionError | "something broke" |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| stop wasted tokens | yes | mechanic halts immediately on exit code 2 |
| clear messaging | yes | stderr says exactly what to do |
| no workarounds | yes | ConstraintError semantics = "stop trying" |
| correct error class | yes | ConstraintError (human-fix) not MalfunctionError (system-fix) |

### pros

- zero ambiguity — robot knows this requires human action
- token efficient — no creative problem-solving attempts
- loud — error is impossible to miss
- semantically correct — exit code 2 = constraint, not malfunction

### cons

- requires aws cli timeout to trigger (can't preemptively detect distraction)
- human still needs to return and run unlock

### edge cases

| edge case | handling |
|-----------|----------|
| human closes browser without auth | same timeout error |
| network failure during auth | different error path (MalfunctionError) |
| aws cli not installed | different error path (ConstraintError) |
| valid sso but wrong account | different error path (BadRequestError after auth) |

## groundwork

### external research

- **aws cli sso login behavior**: the aws cli spawns a browser and waits for device auth. on timeout, it exits with code 1 and outputs "The SSO authorization request timed out" or similar. there's no `--timeout` flag in current aws cli v2.
- source: aws cli documentation and observed behavior

### internal research

**contracts to conform to:**
- `ConstraintError` from `helpful-errors` — has `code.exit = 2`
- error semantics: ConstraintError = caller must fix (human action), MalfunctionError = server must fix

**files to modify:**
- `src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts:414-420` — `initiateAwsSsoAuth` currently throws `UnexpectedCodePathError` on failure

**implementation note:**
must pattern-match aws cli stderr to distinguish error types:
- output contains "timed out" or "authorization request" → ConstraintError (human didn't respond)
- output contains network/connection errors → MalfunctionError (infra issue)
- other errors → MalfunctionError (unknown failure)

**vocab/patterns to reuse:**
- extant `ConstraintError` usage in codebase for "human must do X" scenarios
- extant error output format from mechanic skills

**stdout patterns:**
- keyrack errors already go to stderr
- exit code semantics: 0 = success, 1 = malfunction, 2 = constraint

## what is awkward?

### timing uncertainty

we can't tell if the human walked away vs is just slow. the timeout is the signal, but it's imperfect. a slow but attentive human might feel rushed by a 2-minute timeout.

### no pre-timeout warning

there's no "hey, still waiting..." output. the human returns to a hard error, not a gentle reminder.

### error message length

the "HALT — do not..." repetition is intentionally loud but might feel aggressive in normal usage. tradeoff: loudness vs friendliness.

### single point of change

the fix is narrowly scoped to `initiateAwsSsoAuth`. if other code paths call aws sso login (e.g., `setupAwsSsoProfile.ts:99`), they also need updating. need to audit all `aws sso login` invocations.
