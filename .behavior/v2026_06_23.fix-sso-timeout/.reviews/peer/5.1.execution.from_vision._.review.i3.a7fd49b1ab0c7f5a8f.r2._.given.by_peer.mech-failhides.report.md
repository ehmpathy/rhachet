# blocker.1: Broad try/catch hides file read errors in config file operations

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts (doesAwsProfileExist)
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts (getAwsSsoProfileConfig)
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts (listAwsSsoStartUrls)
- src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts (checkProfileExists)

Multiple functions in setupAwsSsoProfile.ts and vaultAdapterAwsConfig.ts use try/catch blocks without error type checking. The catch clauses unconditionally return a fallback value (false, null, []) on any error, including permission errors, malformed files, etc. This violates the failhide rule: errors must not be silently hidden; catches must either rethrow unexpected errors or have an explicit allowlist. This pattern leads to silent failures and hours of pointless debug.

**snippet**:
```ts
// from doesAwsProfileExist
let configContent = '';
try {
  configContent = await fs.readFile(configPath, 'utf-8');
} catch {
  return false;
}
```

# nitpick.1: Potential failhide in finally block of initiateAwsSsoAuth

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts (initiateAwsSsoAuth finally block)

In initiateAwsSsoAuth, the finally block runs fs.rm on tempDir. If this deletion fails (e.g., permissions), the error is silently ignored because the promise has already been resolved or rejected. This can hide cleanup failures. Consider using a separate try/catch or ensuring errors are at least logged.

**snippet**:
```ts
  } finally {
    // cleanup temp config
    await fs.rm(tempDir, { recursive: true, force: true });
  }
```

# nitpick.2: Broad error catch in validateSsoSession sts call fallback

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts (validateSsoFunction, third try/catch)

In validateSsoSession, the third try/catch block catches all errors that have a 'code' property and returns {valid: false}. This catch-all may hide errors that are not actual session expiration (e.g., network errors, invalid JSON parsing). The rule requires that catches either have an allowlist or rethrow unexpected errors.

**snippet**:
```ts
    } catch (error) {
      // allow expected errors: command failed = session expired or profile invalid
      if (error instanceof UnexpectedCodePathError) throw error;
      if (error instanceof ConstraintError) throw error;

      if (error instanceof Error && 'code' in error) return { valid: false };
      throw error;
    }
```
