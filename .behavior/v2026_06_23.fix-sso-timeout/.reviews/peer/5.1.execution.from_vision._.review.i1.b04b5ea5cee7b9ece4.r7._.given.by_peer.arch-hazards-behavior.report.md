# blocker.1: Race condition on shared config file write in setupAwsSsoProfile

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts: readModifyWrite section (approx lines 40-80)
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts: config write (approx line 70)

The function reads the shared AWS config file, modifies it in memory, and then writes it back. Two concurrent invocations can both read the original content, causing one instance's changes to be lost (lost update). This violates the race condition prohibition under behavior hazards.

Fix: use file locking (e.g., proper locking via a lockfile) or an atomic write pattern (write to temp file then rename). Or design the operation to be idempotent and safe under concurrency.

**snippet**:
```ts
  // read current config or start fresh
  let configCurrent = '';
  try {
    configCurrent = await fs.readFile(configPath, 'utf-8');
  } catch {
    // file doesn't exist, start fresh
  }

  // ... modify configCurrent based on overwrite logic ...

  // append to config
  const configNew = configCurrent.trimEnd() + '\n' + profileConfig;
  await fs.writeFile(configPath, configNew, 'utf-8');
```

# blocker.2: Unexpected default return value in validateSsoSession (vault adapter)

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts: validateSsoSession function, near end of function

When parsing the username from the STS ARN, the code falls back to the string 'unknown' if the last part of the ARN is missing or undefined. This produces a hardcoded placeholder that can mislead callers into believing the username is valid when it is actually unknown. The behavior is not documented and the caller should be aware that the username may be inaccurate. This violates the 'unexpected defaults' clause of behavior hazards.

Fix: either return the username as nullable (e.g., string | null) and document null meaning 'unknown', or always require a valid username and throw if it cannot be parsed.

**snippet**:
```ts
    const arnParts = identity.Arn.split('/');
    const username = arnParts[arnParts.length - 1] ?? 'unknown';
    return { valid: true, username };
```

# blocker.3: Non-idempotent operation when overwrite is false in setupAwsSsoProfile

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts: profile existence check (approx line 40-45)

When `overwrite` is falsy (or not provided), calling `setupAwsSsoProfile` with the same `profileName` twice will succeed on the first call but throw a `BadRequestError` on the second call. This violates the idempotency principle: the same operation produces different results on retry. While this may be intentional, the function does not document itself as non-idempotent nor does it provide a clear idempotency contract (e.g., guard clause with 'if exists return extant' pattern). 

Fix: either make the function idempotent by returning the existing profile on duplicate (like a `findsert`), or explicitly document in the JSDoc that the function is not idempotent when `overwrite` is false.

**snippet**:
```ts
  if (profileExists && !input.overwrite) {
    throw new BadRequestError(
      `profile '${input.profileName}' already exists in ${configPath}. choose a different name or set overwrite: true.`,
    );
  }
```
