**locations**:
- src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts

The function `validateSsoSession` in the orchestrator `vaultAdapterAwsConfig` uses `identity.Arn.split('/')` and then accesses `arnParts[arnParts.length - 1]` to extract the username. This is decode-friction because it requires mental simulation of the split and positional array access. # blocker.3: Inline decode-friction in `checkProfileExists`: regex pattern construction requiring mental simulation

**rule**: rule.forbid.inline-decode-friction

**locations**:
- src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts

The helper `checkProfileExists` in the orchestrator file builds a regex pattern dynamically and tests it. This involves mental simulation to understand the regex and its purpose. The logic should be extracted to a named transformer (e.g., `doesProfileSectionExist`) to keep the orchestrator narrative clear.

Example location: vaultAdapterAwsConfig.ts, top-level function.

**snippet**:
```ts
const checkProfileExists = (profileName: string): boolean => {
  const configPath = join(homedir(), '.aws', 'config');
  if (!existsSync(configPath)) return false;
  const content = readFileSync(configPath, 'utf-8');
  const profilePattern = new RegExp(
    `^\\[profile\\s+${profileName}\\]|^\\[${profileName}\\]`,
    'm',
  );
  return profilePattern.test(content);
};
```
