# blocker.1: Failhide: empty catch swallows config file read errors

**rule**: rule.forbid.failhide.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts

The try/catch block reading the AWS config file has an empty catch that silently ignores all errors, including permission failures or corrupt file errors. This hides real errors, making debugging difficult. The rule requires that try/catch blocks either carefully handle specific errors and throw the rest, or be avoided entirely. Prefer letting the error propagate or handling specific file-not-found case with an allowlist.

**snippet**:
```ts
  let configCurrent = '';
  try {
    configCurrent = await fs.readFile(configPath, 'utf-8');
  } catch {
    // file doesn't exist, start fresh
  }
```

# blocker.2: Failhide: isAwsCliInstalled catches all errors and returns false

**rule**: rule.forbid.failhide.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts

The isAwsCliInstalled function uses try/catch to catch any error from `aws --version` and returns false. This hides real errors (e.g., permission errors, corrupt aws binary) by treating them the same as 'not installed'. The function should either throw a proper error or use a more targeted check that distinguishes installation status from other failures.

**snippet**:
```ts
export const isAwsCliInstalled = (): boolean => {
  try {
    execSync('aws --version', { stdio: 'pipe' });
    return true;
  } catch {
    return false;
  }
};
```

# blocker.3: Failhide: overly broad catch in validateSsoSession accepts any Error with 'code' property

**rule**: rule.forbid.failhide.md

**locations**:
- src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts

In vaultAdapterAwsConfig.ts, the third try/catch in validateSsoSession catches errors from `aws sts get-caller-identity` and returns `{valid: false}` for any error that has a 'code' property. This is not a restricted condition; nearly all errors have a code. Such broad handling can hide unexpected errors (e.g., JSON parse failures, node runtime errors) as session expiry, leading to silent failures and false confidence. The rule requires only errors matching very restricted conditions to be handled; all others must be rethrown.

**snippet**:
```ts
  } catch (error) {
    // rethrow our own error types (code defects, invalid requests)
    if (error instanceof UnexpectedCodePathError) throw error;
    if (error instanceof ConstraintError) throw error;

    // allow expected errors: command failed = session expired or profile invalid
    // .note = aws cli exits non-zero for expired sessions and invalid profiles
    if (error instanceof Error && 'code' in error) return { valid: false };
    throw error;
  }
```
