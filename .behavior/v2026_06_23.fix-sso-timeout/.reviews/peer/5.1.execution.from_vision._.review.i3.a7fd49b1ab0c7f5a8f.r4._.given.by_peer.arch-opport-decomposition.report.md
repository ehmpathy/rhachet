# blocker.1: setupAwsSsoProfile orchestrator contains decode-friction inline logic

**rule**: .agent/repo=ehmpathy/role=architect/briefs/practices/rule.forbid.decode-friction-in-orchestrators.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts:13-99

The `setupAwsSsoProfile` function is an orchestrator that sets up an AWS SSO profile. It contains inline logic that requires mental simulation to understand: raw execSync calls, file I/O (fs.readFile/fs.writeFile), string parsing with removeSection helper, and config string building. According to the rule, orchestrators must not contain decode-friction; each line should tell *what* happens, not *how*. These operations should be extracted to named leaf operations (transformers for compute, communicators for I/O).

**snippet**:
```ts
export const setupAwsSsoProfile = async (input: { ... }): Promise<{...}> => {
  // ensure aws cli is installed
  try {
    execSync('aws --version', { stdio: 'pipe' });
  } catch {
    throw new BadRequestError(...);
  }

  // compute config path
  const configPath = path.join(os.homedir(), '.aws', 'config');

  // ensure .aws directory exists
  const awsDir = path.join(os.homedir(), '.aws');
  await fs.mkdir(awsDir, { recursive: true });

  // read current config or start fresh
  let configCurrent = '';
  try {
    configCurrent = await fs.readFile(configPath, 'utf-8');
  } catch {
    // file doesn't exist, start fresh
  }

  // check if profile already exists
  const profileHeader = `[profile ${input.profileName}]`;
  const sessionHeader = `[sso-session ${input.profileName}]`;
  const profileExists = configCurrent.includes(profileHeader);

  if (profileExists && !input.overwrite) {
    throw new BadRequestError(...);
  }

  // helper to remove a section from config
  const removeSection = (config: string, header: string): string => {
    const headerIndex = config.indexOf(header);
    if (headerIndex === -1) return config;
    const sectionStart = headerIndex;
    const nextSection = config.indexOf('\n[', headerIndex + 1);
    const sectionEnd = nextSection === -1 ? config.length : nextSection;
    return config.slice(0, sectionStart) + config.slice(sectionEnd);
  };

}
```

# nitpick.2: triggerSsoLogin contains decode-friction with output parsing

**rule**: .agent/repo=ehmpathy/role=architect/briefs/practices/rule.forbid.decode-friction-in-orchestrators.md

**locations**:
- src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts:90-110

The internal function `triggerSsoLogin` spawns a process and parses its output to detect timeout vs other failures. This logic (regex on output buffer) requires mental simulation and mixes error handling with process management. It should be extracted to a named operation like `executeSsoLoginCommand` that encapsulates the I/O and returns structured results.

**snippet**:
```ts
const triggerSsoLogin = async (profileName: string): Promise<void> => {
  return new Promise((accept, reject) => {
    const child = spawn('aws', ['sso', 'login', '--profile', profileName], { stdio: 'pipe' });
    let outputBuffer = '';
    child.stdout?.on('data', (data: Buffer) => { outputBuffer += data.toString(); });
    child.stderr?.on('data', (data: Buffer) => { outputBuffer += data.toString(); });
    child.on('close', (code) => {
      if (code === 0) { accept(); }
      else {
        const isTimeout = /timed out|authorization.*timed out|authorization request/i.test(outputBuffer);
        if (isTimeout) { reject(new ConstraintError(...)); }
        else { reject(new MalfunctionError(...)); }
      }
    });
  });
};
```
