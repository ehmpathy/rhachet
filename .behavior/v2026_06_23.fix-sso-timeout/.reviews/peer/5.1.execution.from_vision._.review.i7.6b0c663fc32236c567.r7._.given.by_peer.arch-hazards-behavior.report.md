# blocker.1: Hidden side effects in setupAwsSsoProfile – function writes config, runs CLI commands, validates identity

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts:28-118

The function `setupAwsSsoProfile` has multiple hidden side effects beyond its name: it mutates `~/.aws/config` (writes profile configuration), spawns `aws sso login` and `aws sts get-caller-identity` processes, and may throw errors based on those side effects. The caller cannot see these side effects from the signature or name. This violates the behavior hazard rule against hidden side effects (section .how 4). The function should either be renamed to reflect its full scope (e.g., `setupAndValidateAwsSsoProfile`) or split into smaller functions with explicit side effects.

**snippet**:
```ts
export const setupAwsSsoProfile = async (input: {
  profileName: string;
  ssoStartUrl: string;
  ssoRegion: string;
  ssoAccountId: string;
  ssoRoleName: string;
  defaultRegion?: string;
  overwrite?: boolean;
}): Promise<{ profileName: string; configPath: string }> => {
  // ... writes config file
  await fs.writeFile(configPath, configNew, 'utf-8');

  // ... spawns aws sso login
  execSync(`aws sso login --profile "${input.profileName}"`, { stdio: 'pipe' });

  // ... spawns aws sts get-caller-identity
  execSync(`aws sts get-caller-identity --profile "${input.profileName}"`, { stdio: 'pipe' });

  return { profileName: input.profileName, configPath };
};
```

# blocker.2: Non-idempotent operation in setupAwsSsoProfile – duplicate calls with overwrite=true trigger additional AWS login

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts:83-113

Calling `setupAwsSsoProfile` twice with the same input and `overwrite: true` will overwrite the config file (fine) but also trigger a new `aws sso login` flow each time. AWS SSO login creates a new device authorization session, potentially generating multiple cached tokens for the same profile. This violates idempotency requirements (section .how 7): the operation produces different results on retry (additional cached sessions) and may cause duplicate side effects. The function should guard re-entry (e.g., check if a valid session already exists) or be marked non-idempotent with clear documentation.

**snippet**:
```ts
  // append to config
  const configNew = configCurrent.trimEnd() + '\n' + profileConfig;
  await fs.writeFile(configPath, configNew, 'utf-8');

  // trigger sso login to validate (suppress output — guide already showed auth info)
  try {
    execSync(`aws sso login --profile "${input.profileName}", { stdio: 'pipe' });
  } catch (error) { ... }

  // validate the profile works
  try {
    execSync(`aws sts get-caller-identity --profile "${input.profileName}", { stdio: 'pipe' });
  } catch { ... }

  return { profileName: input.profileName, configPath };
```

# blocker.4: Non-idempotent operation in vaultAdapterAwsConfig.set – repeated calls may trigger multiple guided setups and artifact creation

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts:209-290

The `set` operation is not idempotent: if called twice, it may re-run the guided setup (if interactive), re-execute the roundtrip validation (unlock, get, relock), and potentially create duplicate SSO tokens or leak temporary config files. There is no guard against re-entry (e.g., check if profile already stored). This violates idempotency requirements (section .how 7). The function should either be idempotent (skip if already set) or be explicitly documented as non-idempotent.

**snippet**:
```ts
  set: async (input) => {
    // no guard against re-entry
    if (!profileName && process.stdin.isTTY) {
      const mechAdapter = getMechAdapter(mech);
      const result = await mechAdapter.acquireForSet({ keySlug: input.slug });
      profileName = result.source;
    }
    // ... calls unlock, get, relock each time
    return { mech, exid: profileName };
  },
```
