# blocker.1: Empty try/catch silently ignores errors (failhide) in config file reads

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts (setupAwsSsoProfile, doesAwsProfileExist, getAwsSsoProfileConfig, listAwsSsoStartUrls)

Multiple try/catch blocks with empty catch clauses silently swallow all errors when reading ~/.aws/config. This violates the forbid-failhide rule because any error (e.g., permission denied, disk failure) is hidden, making debugging extremely difficult. All reads of the config file should either let the error propagate or have a specific allowlist of expected errors (e.g., file not found) and rethrow others.

The following locations all have this pattern: reading config file and catching all errors without any handling.

# blocker.2: Overly broad catch in validateSsoSession hides unexpected errors (failhide)

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt2.md

**locations**:
- src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts (validateSsoSession function)

In vaultAdapterAwsConfig.ts, the third try/catch block in validateSsoSession catches all errors that have a 'code' property and returns { valid: false }. This goes beyond the allowed pattern: it does not have a restricted allowlist of expected exceptional conditions. Errors like network timeouts (code 'ENOTFOUND') or permission denied will be hidden as 'invalid session' instead of being rethrown. The rule requires that only very restricted conditions be handled; throwing for all others ensures failfast behavior.

The specific code (lines ~159-171) should be changed to either rethrow all unexpected errors or check specifically for exit codes that indicate expired/invalid (e.g., exit code 255, specific error message).

**snippet**:
```ts
    } catch (error) {
      // rethrow our own error types (code defects, invalid requests)
      if (error instanceof UnexpectedCodePathError) throw error;
      if (error instanceof ConstraintError) throw error;

      // allow expected errors: command failed = session expired or profile invalid
      // .note = aws cli exits non-zero for expired sessions and invalid profiles
      if (error instanceof Error && 'code' in error) return { valid: false };
      throw error;
    }
```

# nitpick.1: Prefer HelpfulError.wrap over manual try/catch + rethrow for error wrapping

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.prefer.helpful-error-wrap.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts (catch for sso login and sts)
- src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts (validateSsoTokenWithAwsSdk and export-credentials catches)

Multiple places manually catch an error and throw a new error with metadata (e.g., throwing UnexpectedCodePathError('aws sso login failed', { profileName })). The rule suggests using HelpfulError.wrap (or specific subclass) instead, which automatically preserves the original error cause and provides consistent metadata. This is a minor preference, not a blocker, but improves observability.

Affected locations: setupAwsSsoProfile.ts (sso login catch, sts catch) and vaultAdapterAwsConfig.ts (validateSsoTokenWithAwsSdk catch, export-credentials catch).

**snippet**:
```ts
    try {
      await validateSsoTokenWithAwsSdk(profileName);
    } catch (error) {
      if (error instanceof Error && (error.name === 'CredentialsProviderError' || error.message.includes('expired') || error.message.includes('invalid'))) {
        return { valid: false };
      }
      throw new UnexpectedCodePathError(
        'validateSsoTokenWithAwsSdk failed with unexpected error',
        { profileName, error },
      );
    }
```
