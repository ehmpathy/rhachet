# blocker.1: unclear error messages for sso login and sts failures

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts:91-96

When `aws sso login` or `aws sts get-caller-identity` fails, the code throws `UnexpectedCodePathError` with minimal details (e.g., 'aws sso login failed'). This violates the friction-hazards rule because the error does not tell the user what went wrong or how to fix it. The user is left to guess whether the SSO session expired, credentials are wrong, or network is down. The errors should be more descriptive and include actionable hints.

**snippet**:
```ts
  // trigger sso login to validate (suppress output — guide already showed auth info)
  try {
    execSync(`aws sso login --profile "${input.profileName}"`, {
      stdio: 'pipe',
    });
  } catch {
    throw new UnexpectedCodePathError('aws sso login failed', {
      profileName: input.profileName,
    });
  }
```
