# nitpick.1: Decode-friction in getAwsSsoProfileConfig and listAwsSsoStartUrls

**rule**: .agent/repo=ehmpathy/role=architect/briefs/practices/rule.forbid.decode-friction-in-orchestrators.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts:109-120

- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts:195-210

These functions contain inline string parsing logic (regex, section extraction) that constitutes decode-friction. While they are technically get operations, they are still leaf operations that should use named transformers for clarity. Extract parsing helpers (e.g., getSection, getValue) into separate transformers to make intent explicit.

**snippet**:
```ts
  // helper to extract a section by header
  const getSection = (header: string): string | null => {
    const headerIndex = configContent.indexOf(header);
    if (headerIndex === -1) return null;
    const sectionStart = headerIndex + header.length;
    const nextSection = configContent.indexOf('\n[', sectionStart);
    const sectionEnd = nextSection === -1 ? configContent.length : nextSection;
    return configContent.slice(sectionStart, sectionEnd);
  };

  // helper to parse value from section
  const getValue = (section: string, key: string): string => {
    const match = section.match(
      new RegExp(`(?:^|\\n)${key}\\s*=\\s*(.+)`, 'm'),
    );
    return match?.[1]?.trim() ?? '';
  };
```
