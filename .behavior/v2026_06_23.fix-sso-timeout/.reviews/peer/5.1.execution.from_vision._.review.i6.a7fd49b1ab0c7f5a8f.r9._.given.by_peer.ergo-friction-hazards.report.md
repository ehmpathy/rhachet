# blocker.1: unclear error messages for sso login and sts failures

**rule**: /agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- src/domain.operations/keyrack/adapters/mechanisms/aws.sso/setupAwsSsoProfile.ts

In setupAwsSsoProfile.ts, the catch blocks for `aws sso login` and `aws sts get-caller-identity` throw `UnexpectedCodePathError` with only a generic message and the profile name. These errors do not tell the user what went wrong or how to fix it. For example, a network issue or permission failure would leave the user clueless. According to the friction hazard rule, every user-faced error must be actionable, i.e., explain the cause and suggest a fix. These need to provide context (e.g., 'aws sso login failed — check your network or open the sso portal') and a hint.

**snippet**:
```ts
  try {
    execSync(`aws sso login --profile "${input.profileName}"`, {
      stdio: 'pipe',
    });
  } catch {
    throw new UnexpectedCodePathError('aws sso login failed', {
      profileName: input.profileName,
    });
  }

  // validate the profile works
  try {
    execSync(`aws sts get-caller-identity --profile "${input.profileName}"`, {
      stdio: 'pipe',
    });
  } catch {
    throw new UnexpectedCodePathError(
      'aws sts get-caller-identity failed after sso login',
      { profileName: input.profileName },
    );
  }
```

# blocker.2: unexpected error thrown in validateSsoSession without actionable hint

**rule**: /agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts

In vaultAdapterAwsConfig.ts, the `validateSsoSession` function catches errors from `validateSsoTokenWithAwsSdk` and `execAsync` for export-credentials. When an unexpected error occurs (not matching known expired/invalid patterns), it throws a `UnexpectedCodePathError` with no hint for the user. This is a friction hazard because a user hitting such an error (e.g., network failure, aws cli version incompatibility) will see no guidance on how to proceed. The rule requires error messages to tell the user what went wrong and how to fix it.

**snippet**:
```ts
    // unknown error - fail fast
    throw new UnexpectedCodePathError(
      'validateSsoTokenWithAwsSdk failed with unexpected error',
      { profileName, error },
    );
```
