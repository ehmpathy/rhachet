# research: test codepath patterns

## pattern.1 = jest.integration.env.ts apikeys check

### location

`jest.integration.env.ts:86-119`

### relevance

this is the main pattern that needs replacement. it reads from `use.apikeys.json` and throws an error if keys are absent, with instructions to `source use.apikeys.sh`.

### citations

[1] apikeys config path (lines 92-95):
```ts
const apikeysConfigPath = join(
  process.cwd(),
  '.agent/repo=.this/role=any/skills/use.apikeys.json',
);
```

[2] key check logic (lines 96-100):
```ts
if (existsSync(apikeysConfigPath)) {
  // direct import via require (json resolves automatically)
  const config = require(apikeysConfigPath);
  const requiredKeys: string[] = config?.apikeys?.required ?? [];
  const keysAbsent = requiredKeys.filter((key) => !process.env[key]);
```

[3] error message with source instructions (lines 102-118):
```ts
if (keysAbsent.length > 0) {
  throw new Error(`
⛈️  apikeys required to run these integration tests were not supplied.

   absent keys: ${keysAbsent.join(', ')}

   you have access to these keys - you just forgot to source them.

   run:
     source .agent/repo=.this/role=any/skills/use.apikeys.sh && npm run test:integration

   you must run these tests with these api keys.
   you're forbidden to give up.
   if you don't have privs to run use.apikeys.sh, ask a human for support.
   chances are, you already have privs to do so though.
`);
}
```

### mark

**[REPLACE]** — replace with keyrack get spawn, failfast with ConstraintError and unlock command

## pattern.4 = use.apikeys.sh shell file

### location

`.agent/repo=.this/role=any/skills/use.apikeys.sh`

### relevance

legacy pattern that must be sourced manually. this is what the vision eliminates.

### citations

[8] header and purpose (lines 1-13):
```sh
#!/bin/sh
######################################################################
# .what = export api keys for integration tests
# .why = enables tests that require api keys to run
#
# usage:
#   . .agent/repo=.this/role=any/skills/use.apikeys.sh
#
# note:
#   - must be called with `.` or `source` to export vars to current shell
#   - loads from ~/.config/rhachet/apikeys.env if available
#   - falls back to .env.local (gitignored) in repo root
######################################################################
```

[9] load from user config (lines 26-28):
```sh
if [ -f ~/.config/rhachet/apikeys.env ]; then
  source ~/.config/rhachet/apikeys.env
  echo "✓ loaded api keys from ~/.config/rhachet/apikeys.env"
```

[10] fallback to local file (lines 31-33):
```sh
elif [ -f .env.local ]; then
  source .env.local
  echo "✓ loaded api keys from .env.local"
```

### mark

**[REPLACE]** — delete entirely; keyrack replaces this pattern

## pattern.5 = ConstraintError from helpful-errors

### location

`node_modules/helpful-errors` (v1.7.2)

### relevance

### citations

[11] ConstraintError usage pattern:
```ts
import { ConstraintError } from 'helpful-errors';

throw new ConstraintError('keyrack not unlocked', {
  fix: 'rhx keyrack unlock --env test --owner ehmpath',
});
```

### mark

**[REUSE]** — use ConstraintError for failfast in jest.*.env.ts

**key insight**: all four apikeys-related artifacts will be replaced. the new pattern:
1. spawn `rhx keyrack get --for repo --env test --json --owner ehmpath`
2. parse JSON response
3. if keys unlocked → inject into process.env
4. if keyrack locked → throw ConstraintError with unlock command
5. if keys absent → throw ConstraintError with set commands
6. CI environments get keys via os.envvar passthrough (no unlock needed)
