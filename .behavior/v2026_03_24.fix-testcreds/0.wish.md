wish =

currently, this repo uses an adhoc .apikeys.json and use.apikeys.sh pattern to source api keys

see jest.integration.env.ts for the pain it causes in usage

---

instead, we now have the keyrack which we can use

the keyrack enables the jest.integration.env.ts to automatically unlock the keyrack via shell spawn cli and grab the keys it needs via json

if the keyrack is not unlocked, it can just failfast and tell the caller to unlock the keyrack

---

it should support getting the keys from the `--owner ehmpath` by default, since typically ehmpaths will work here

---

that way, no longer will anyone have to remember to `source .agent/.../use.apikeys.sh` which themselves just pull from a hardcoded keys file

instead, the keys will be fetched from the keyrack

and if the keyrack is not unlocked, then the test can failfast and say 'hey, unlock your keyrack so the tests can run' via `rhx keyrack unlock --owner ehmpath`

and then retry

and boom

all the required keys, available, done deal.

---

also, `rhx keyrack unlock` should remind users to run `rhx keyrack fill` if it detects any keys that are absent

---

the goal = eliminate .agent/.../use.apikeys.sh

and elimiante all reference to it from package.json and jest.*.ts files
