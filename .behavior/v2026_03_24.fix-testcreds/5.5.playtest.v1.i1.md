# playtest: keyrack for test credentials

## prerequisites

- rhachet installed (`./node_modules/.bin/rhx` available)
- ssh key at `~/.ssh/ehmpath` (for keyrack unlock)
- keyrack secrets populated for owner=ehmpath

## sandbox

no file operations required — this playtest verifies test runner behavior

---

## happy paths

### 1. verify legacy files deleted

```bash
ls -la .agent/repo=.this/role=any/skills/use.apikeys.*
```

**expected:** `ls: cannot access '.agent/repo=.this/role=any/skills/use.apikeys.*': No such file or directory`

**pass if:** files do not exist
**fail if:** either `use.apikeys.sh` or `use.apikeys.json` exists

---

### 2. verify keyrack.source SDK is in place

check that both jest env files use keyrack.source():

```bash
grep -n "keyrack.source" jest.integration.env.ts jest.acceptance.env.ts
```

**expected:**
- `jest.integration.env.ts:97:keyrack.source({ env: 'test', owner: 'ehmpath' });`
- `jest.acceptance.env.ts:47:keyrack.source({ env: 'test', owner: 'ehmpath' });`

**pass if:** keyrack.source() call is present in both files with correct params
**fail if:** still uses old use.apikeys.json pattern or keyrack.source() is missing

**note:** the "locked keyrack" error only triggers when keyrack.yml declares required keys in env.test. currently keys are commented out in mechanic keyrack.yml, so tests will pass even with locked keyrack. this is expected behavior — keyrack.source() in strict mode only fails when there ARE keys that are not granted

---

### 3. verify keyrack unlocked behavior

unlock the keyrack:

```bash
rhx keyrack unlock --env test --owner ehmpath
```

then run integration tests:

```bash
npm run test:integration
```

**expected:** tests run with keys injected into process.env

**pass if:** tests execute (may pass or fail based on other factors, but keys are available)
**fail if:** ConstraintError about keyrack, or absent API key errors

---

### 4. verify acceptance tests same behavior

with keyrack still unlocked:

```bash
npm run test:acceptance
```

**expected:** tests run with keys injected

**pass if:** no keyrack-related errors
**fail if:** ConstraintError about keyrack

---

### 5. verify CI passthrough (env var override)

with keyrack relocked:

```bash
rhx keyrack relock --owner ehmpath
```

then run integration tests with all required keys as env vars:

```bash
OPENAI_API_KEY=test ANTHROPIC_API_KEY=test XAI_API_KEY=test npm run test:integration -- --testPathPattern="keyrack" --testPathIgnorePatterns="acceptance"
```

**expected:** integration tests start without keyrack unlock prompt because keyrack prefers env var passthrough (os.envvar vault)

**pass if:** tests start without keyrack error (tests may fail for other reasons since keys are fake)
**fail if:** keyrack error despite env vars present

---

## edgey paths

### absent keys

to simulate absent keys, check keyrack status without unlock:

```bash
rhx keyrack relock --owner ehmpath
rhx keyrack get --for repo --env test --owner ehmpath
```

**expected:** output shows each key with status:
- `absent` for keys not in vault
- `locked` for keys in vault but not unlocked
- tip for each absent key: `rhx keyrack set --key <KEY_NAME> --env test`

**pass if:** absent keys show `set` command hint
**fail if:** no hints provided for absent keys

---

## pass/fail criteria

- ✓ pass if:
  - legacy `use.apikeys.*` files are deleted
  - jest.integration.env.ts uses `keyrack.source({ env: 'test', owner: 'ehmpath' })`
  - jest.acceptance.env.ts uses `keyrack.source({ env: 'test', owner: 'ehmpath' })`
  - tests run when keyrack is unlocked
  - CI env vars work as passthrough

- ✗ fail if:
  - legacy files still exist
  - jest.*.env.ts files still use old use.apikeys.json pattern
  - keyrack.source() calls are missing or have wrong params
  - env var passthrough broken
