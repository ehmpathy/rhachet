# vision: keyrack for test credentials

## the outcome world

### day-in-the-life: before

```bash
# developer opens terminal, runs tests
npm run test:integration

# ✋ error hits:
#   apikeys required to run these integration tests were not supplied.
#   absent keys: OPENAI_API_KEY, ANTHROPIC_API_KEY, XAI_API_KEY
#   run: source .agent/repo=.this/role=any/skills/use.apikeys.sh && npm run test:integration

# developer sighs, runs the source command
source .agent/repo=.this/role=any/skills/use.apikeys.sh && npm run test:integration

# ✓ works... this time

# next terminal session: same dance
# forgot to source? same error
# new contributor? confusion: "where do i get these keys?"
```

### day-in-the-life: after

```bash
# developer opens terminal, runs tests
npm run test:integration

# scenario A: keyrack already unlocked (from earlier work)
# → tests just run, keys fetched automatically

# scenario B: keyrack not unlocked
# ✋ failfast (ConstraintError):
#   keyrack not unlocked. unlock to run integration tests:
#     rhx keyrack unlock --env test --owner ehmpath  # we expect ehmpaths to work in this repo
#
# developer runs the unlock command once per session
rhx keyrack unlock --env test --owner ehmpath  # we expect ehmpaths to work in this repo

# tests just work from now on
npm run test:integration
```

### the aha moment

the aha moment clicks when:
1. you run tests in a fresh terminal
2. they just work (keyrack still unlocked from before)
3. you realize you haven't thought about API keys in weeks

no more `source` incantations. no more "oh right, forgot to source". the keyrack session persists, tests just run.

### before/after contrast

| dimension | before (use.apikeys.sh) | after (keyrack) |
|-----------|------------------------|-----------------|
| auth frequency | every terminal session | once per session (daemon persists) |
| error message | tells you to source a shell command | tells you to unlock keyrack |
| key discovery | hunt for ~/.config/rhachet/apikeys.env | keyrack knows what keys exist |
| new contributor | "where do i put my keys?" | keyrack get shows `keyrack set` commands |
| ci/cd | manual env var setup | keyrack passthrough via os.envvar (no unlock needed) |
| key rotation | manually update .env files | keyrack pulls from vault |

## user experience

### usecases

| usecase | flow |
|---------|------|
| run integration tests | `npm run test:integration` → keyrack provides keys automatically |
| first-time setup | tests fail → error says "unlock keyrack" → user unlocks → keys flow |
| absent key | keyrack get shows `rhx keyrack set` command for each absent key |
| ci/cd | keyrack passthrough via os.envvar — no unlock needed |

### contract inputs & outputs

**jest.integration.env.ts** uses:
- input: `rhx keyrack get --for repo --env test --json --owner ehmpath`  <!-- we expect ehmpaths to work in this repo -->
- output: JSON array with status of each key

when keys are unlocked:
```json
[
  { "status": "unlocked", "slug": "ehmpathy.test.OPENAI_API_KEY", "secret": "sk-..." },
  { "status": "unlocked", "slug": "ehmpathy.test.ANTHROPIC_API_KEY", "secret": "sk-..." }
]
```

when keys are locked or absent:
```json
[
  { "status": "locked", "slug": "ehmpathy.test.OPENAI_API_KEY", "fix": "rhx keyrack unlock --env test --key OPENAI_API_KEY" },
  { "status": "absent", "slug": "ehmpathy.test.XAI_API_KEY", "fix": "rhx keyrack set --key XAI_API_KEY --env test" }
]
```

the fix commands are provided directly — excellent ergonomics.

### user timeline

1. **new developer joins**
   - clones repo, runs tests
   - sees "keyrack not unlocked" error with exact unlock command
   - runs `rhx keyrack set` for each absent key (commands provided in error)
   - runs unlock command
   - tests pass

2. **daily workflow**
   - open terminal
   - run tests
   - (keyrack already unlocked from earlier? → tests just work)
   - (keyrack locked? → unlock once, tests work rest of session)

3. **key rotation**
   - admin rotates keys in vault
   - developers: no action needed
   - keyrack pulls fresh keys on next unlock

## evaluation

### how well does it solve the goals?

| goal | score | notes |
|------|-------|-------|
| eliminate `use.apikeys.sh` | ✓ | replaced by keyrack get |
| eliminate manual `source` step | ✓ | jest.integration.env.ts auto-fetches |
| better error messages | ✓ | tells user exactly what command to run |
| new contributor experience | ✓ | clear unlock flow + fill guidance |
| ci/cd support | ✓ | keyrack init handles env setup |

### pros

- **one unlock per session** — no more per-terminal source commands
- **secure by default** — keys never in plaintext files in repo
- **discoverable** — keyrack knows which keys exist, can prompt for missing
- **standard pattern** — same keyrack used for mechanic tokens, test creds, etc.

### cons

- **daemon dependency** — keyrack daemon must be running
- **unlock step** — still need to unlock (but once per session, not per terminal)
- **learning curve** — new concept for developers used to .env files

### pit of success edgecases

| edgecase | handling |
|----------|----------|
| keyrack not installed | error: "install rhachet to run tests" |
| keyrack locked | failfast with unlock command |
| key absent from vault | error with `rhx keyrack set` command |
| wrong owner | clear error about which owner to use |
| daemon not running | keyrack unlock starts it |

## what is awkward?

### feels off

- **the owner flag** — having to specify `--owner ehmpath` feels like it should be inferrable from context
- **unlock once per session** — still a manual step, though much better than per-terminal

### fights mental model

- developers expect `.env` files — keyrack is a different paradigm
- "why do i need to unlock?" — friction until they understand the security model

### uncomfortable tradeoffs

- **security vs convenience** — keyrack adds a step, but that step is security
- **daemon dependency** — if daemon crashes, keys become inaccessible (until re-unlock)

### mitigations

| awkwardness | mitigation |
|-------------|------------|
| owner flag | default to `ehmpath` in this repo's keyrack.yml |
| prikey flag | auto-discover from `~/.ssh/$owner` (no `--prikey` needed) |
| unlock step | excellent error message with exact command |
| .env mental model | brief explanation in error output |
