# vision: keyrack daemon home-scoped caching

## the outcome world

### day-in-the-life

**test author writes integration tests with isolated HOME directories:**
```ts
describe('keyrack unlock', () => {
  given('[case1] fresh keyrack manifest in home A', () => {
    const scene = useBeforeAll(async () => {
      const tempHome = await genTestTempHome(); // e.g., /tmp/test-aaa
      return { home: tempHome };
    });

    when('[t0] keyrack unlock is called first time', () => {
      then('prompts for configuration', async () => {
        const result = await invokeKeyrackCli({
          args: ['unlock', '--owner', 'test-owner'],
          env: { HOME: scene.home },
        });
        expect(result.stdout).toContain('configure...');
      });
    });

    when('[t1] keyrack unlock is called second time', () => {
      then('uses cached state from home A', async () => {
        const result = await invokeKeyrackCli({
          args: ['unlock', '--owner', 'test-owner'],
          env: { HOME: scene.home },
        });
        expect(result.stdout).toContain('configured ✓');
      });
    });
  });

  given('[case2] fresh keyrack manifest in home B', () => {
    const scene = useBeforeAll(async () => {
      const tempHome = await genTestTempHome(); // e.g., /tmp/test-bbb (different!)
      return { home: tempHome };
    });

    when('[t0] keyrack unlock is called first time', () => {
      then('prompts for configuration (not polluted by case1)', async () => {
        // even though case1 already configured 'test-owner',
        // this HOME is different → fresh cache slot → prompts again
        const result = await invokeKeyrackCli({
          args: ['unlock', '--owner', 'test-owner'],
          env: { HOME: scene.home },
        });
        expect(result.stdout).toContain('configure...');
      });
    });

    when('[t1] keyrack unlock is called second time', () => {
      then('uses cached state from home B', async () => {
        const result = await invokeKeyrackCli({
          args: ['unlock', '--owner', 'test-owner'],
          env: { HOME: scene.home },
        });
        expect(result.stdout).toContain('configured ✓');
      });
    });
  });
});
```

both tests pass. case2 is not polluted by case1's state because each HOME gets its own daemon cache slot. no daemon restart. no manual cache clear. no flaky failures from stale state.

### before/after contrast

| aspect | before | after |
|--------|--------|-------|
| daemon scope | (session, owner) | **(session, home, owner)** |
| socket path | `keyrack.$SESSIONID.$owner.sock` | `keyrack.$SESSIONID.$HOMEHASH.$owner.sock` |
| test isolation | broken — daemon ignores HOME | works — separate daemon per (session, home) |
| session isolation | yes | **yes (kept)** |
| test setup | restart daemon between tests | just set HOME → auto-spawns daemon |
| daemon lifetime | lives forever until killed | **auto-terminates when all keys expire** |
| flaky tests | common — stale cache pollution | rare — isolated by design |
| cleanup | manual kill | automatic — daemons self-terminate |

### the "aha" moment

> "wait, i just set `HOME=/tmp/test-xyz` and the daemon treats it as a fresh environment? no daemon restart, no cache flush, none of that?"

the value clicks when a test author realizes they can run 50 parallel tests, each with their own HOME, and the daemon keeps them all isolated automatically.

## user experience

### usecases

**1. integration test isolation**
- goal: test keyrack operations without pollution from other tests
- contract: set `HOME` env var → daemon uses home-scoped cache
- timeline: instant — no setup, no teardown

**2. isolated dev environments**
- goal: test keyrack changes without polluting your real HOME
- contract: set `HOME=/tmp/dev-sandbox` → isolated daemon for sandbox work
- timeline: sandbox work stays sandboxed, real HOME unaffected

**3. ci/cd parallelism**
- goal: run multiple keyrack-dependent jobs in parallel
- contract: each job gets unique HOME → no contention
- timeline: scales with job count

### contract inputs & outputs

**input: CLI request**
```
keyrack unlock --owner ehmpath
  └─ HOME=/tmp/test-abc (from environment)
```

**daemon model: separate daemon per (session, home)**
```
CLI detects HOME=/tmp/test-abc
  └─ connects to daemon socket: keyrack.$SESSIONID.$HOMEHASH.$owner.sock
     (where HOMEHASH = hash(home)[:8])
  └─ if no daemon exists → spawns one
  └─ daemon auto-terminates when all keys expire
```

**result: fully isolated daemons (session + home)**
```
session X + /tmp/test-abc → daemon A (socket: keyrack.$X.a1b2c3d4.$owner.sock)
session X + /tmp/test-xyz → daemon B (socket: keyrack.$X.e5f6g7h8.$owner.sock)
session X + /home/vlad    → daemon C (socket: keyrack.$X.c9d0e1f2.$owner.sock)
session Y + /home/vlad    → daemon D (socket: keyrack.$Y.c9d0e1f2.$owner.sock)

each daemon:
  - owns its own cache (no shared state)
  - auto-terminates on max key TTL
  - fully isolated by (session, home, owner)
```

### timeline: test author flow

1. **minute 0**: write test with `HOME=tempDir`
2. **minute 0**: run test — it passes
3. **minute 0**: run test again — still passes
4. **minute 0**: run 10 tests in parallel — all pass
5. **never**: debug cache pollution issues

## mental model

### how users describe it to a friend

> "each HOME directory gets its own keyrack daemon. set HOME to a temp directory → fresh daemon spawns → serves your test → auto-terminates when keys expire. zero cleanup needed."

### analogies

**docker containers**: each HOME spawns its own daemon process, like each container has its own init process. container dies → process dies. HOME's keys expire → daemon dies.

**ssh-agent per terminal**: like a separate ssh-agent per terminal session. each agent holds its own keys, dies when session ends.

**systemd user services**: like a user service that auto-stops after idle. daemon is a service scoped to HOME, auto-terminates when no longer useful.

### terms: user vs internal

| user says | we say |
|-----------|--------|
| "isolated environment" | "separate daemon per HOME" |
| "fresh state" | "new daemon spawned for this HOME" |
| "shared state" | "same daemon (same HOME)" |
| "test pollution" | "daemon reuse across HOMEs" (the bug this feature solves) |

## evaluation

### how well does it solve the goals?

| goal | score | notes |
|------|-------|-------|
| test isolation | ✅ excellent | just set HOME, done |
| backwards compat | ✅ excellent | extant usage unchanged |
| performance | ✅ excellent | still cached, just more granular |
| simplicity | ✅ excellent | no new flags or config |

### pros

- **zero friction**: no new CLI flags, no config files, no manual daemon management
- **full process isolation**: each HOME = separate daemon process (not shared cache slots)
- **deterministic**: same HOME always connects to same daemon
- **parallel-friendly**: unlimited concurrent test environments, each with own daemon
- **self-cleanup**: daemons auto-terminate on max key TTL — no orphan processes

### cons

- **more processes**: N concurrent HOMEs = N daemon processes
  - mitigation: each daemon is lightweight (~10-25MB for typical node process)
  - mitigation: auto-terminate on max key TTL prevents unbounded growth

- **socket management**: each daemon needs unique socket identifier
  - mitigation: socket name includes home hash (deterministic, collision-free)

- **debugging complexity**: "which daemon am i connected to?"
  - mitigation: daemon logs include home hash
  - mitigation: `keyrack status` shows current daemon identity

### daemon lifecycle (required for this feature)

**note: architecture decision**
the original proposal suggested one daemon with home-scoped cache keys (`owner:hash(home)`). during vision refinement, we chose **separate daemons per HOME** instead. rationale:
- simpler mental model (one HOME = one daemon)
- full process isolation (no shared state)
- natural lifecycle (daemon dies when its HOME's keys expire)

home-scoped caching means separate daemons per HOME. daemons must self-terminate to prevent runaway processes.

**architecture: one daemon per (session, home)**
- each (session, home) gets its own daemon process
- daemon socket includes session ID + home hash (e.g., `keyrack.$SESSIONID.$HOMEHASH.$owner.sock`)
- keeps session isolation (security)
- adds home isolation (test isolation)
- full process isolation between test environments

**daemon auto-terminate when all keys expire**
- daemon tracks TTL of each cached key (from manifest)
- when all keys expired → daemon terminates
- no idle timeout needed — TTL is the natural lifecycle bound
- rationale: daemon with only expired keys serves no purpose

**example lifecycle**
```
t=0:     test starts, HOME=/tmp/test-abc
t=0:     keyrack unlock → spawns daemon for /tmp/test-abc
t=0:     daemon caches unlocked key (TTL from manifest)
t=TTL:   key expires → daemon terminates
```

**acceptance criteria**
```
given a daemon with cached keys
  when all cached keys reach their TTL
    then daemon terminates itself

given a terminated daemon for some HOME
  when keyrack unlock is called for that HOME
    then a fresh daemon spawns
    then user is prompted to configure (no stale state)
```

### edge cases & pit of success

| edge case | behavior | why it's safe |
|-----------|----------|---------------|
| HOME unset | use process HOME (fallback) | backwards compat |
| HOME is symlink | expand to real path before hash | consistent daemon identity |
| HOME changes mid-request | each request uses its own HOME | connects to correct daemon |
| same HOME, different owners | same daemon, different cache keys | owner scopes within daemon |
| HOME contains special chars | hash handles any string | no socket name injection |
| daemon crashes | next request spawns fresh daemon | no orphan state |
| all keys expire | daemon auto-terminates | no zombie processes |
| HOME deleted while daemon runs | daemon still terminates on TTL | natural cleanup |

### what could be awkward?

**scenario**: user runs 100 parallel tests, each with unique HOME → 100 daemon processes.

**fix**: that's expected. each daemon is lightweight, auto-terminates when keys expire. worst case is 100 processes until keys TTL, then all gone.

**scenario**: user wonders why two terminals with same HOME share state.

**fix**: that's the feature — same HOME = same daemon = shared state. document this clearly.

**scenario**: user wants to force-kill all daemons.

**fix**: `keyrack daemon kill-all` or `pkill -f keyrack-daemon` for emergency cleanup.
