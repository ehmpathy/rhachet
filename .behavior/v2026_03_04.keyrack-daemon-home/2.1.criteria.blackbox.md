# blackbox criteria: keyrack daemon home-scoped cache

# usecase.1 = test isolation via HOME

given a fresh HOME directory (e.g., /tmp/test-aaa)
  when keyrack unlock is called for owner X with that HOME
    then the user is prompted to configure (key not found)
      sothat fresh environments always start unconfigured
  when keyrack unlock is called again for owner X with that HOME
    then the cached key is returned (no prompt)
      sothat state persists within a HOME

given two different HOME directories (e.g., /tmp/test-aaa and /tmp/test-bbb)
  when keyrack unlock configures owner X in HOME A
    then HOME B still prompts for configuration when keyrack unlock is called
      sothat different HOMEs have fully isolated state
  when keyrack unlock is called in HOME A after HOME B was configured
    then HOME A returns its own cached state (not HOME B's)
      sothat HOMEs never share or pollute each other's cache

given the same HOME directory from two terminals
  when keyrack unlock configures owner X in terminal 1
    then terminal 2 sees the cached key when it calls keyrack unlock
      sothat same HOME = shared state (expected behavior)


# usecase.2 = daemon auto-termination

given a HOME with cached keys
  when all keys reach their TTL
    then the daemon terminates itself
      sothat no zombie processes accumulate

given a terminated daemon for some HOME
  when keyrack unlock is called for that HOME
    then a fresh daemon spawns
    then the user is prompted to configure (no stale state)
      sothat daemon termination is transparent to user


# usecase.3 = backwards compatibility

given HOME environment variable is unset
  when keyrack unlock is called
    then the process HOME is used as fallback
      sothat extant usage without explicit HOME continues to work

given a daemon for owner X at HOME /path/a
  when keyrack unlock is called for owner Y at HOME /path/a
    then owner Y gets its own cache entry (separate from owner X)
      sothat multiple owners can coexist within same HOME


# usecase.4 = edge cases

given HOME is a symlink to a real path
  when keyrack unlock is called
    then the daemon uses the expanded real path
      sothat symlinks and real paths connect to the same daemon

given HOME contains special characters (spaces, unicode, etc.)
  when keyrack unlock is called
    then the daemon spawns correctly and caches work
      sothat no path injection or socket name issues occur

given a daemon crashes unexpectedly
  when keyrack unlock is called for that HOME
    then a fresh daemon spawns
    then the user is prompted to configure (no stale state)
      sothat crashes are recovered gracefully

given 100 parallel tests with unique HOMEs
  when all tests call keyrack unlock concurrently
    then 100 separate daemons spawn and operate independently
      sothat parallel test execution scales without contention


# usecase.5 = observability

given a user wants to know which daemon they are connected to
  when keyrack status is called
    then the current daemon identity is shown (with home hash)
      sothat "which daemon?" is straightforward to debug

given a user wants to force-kill all daemons
  when pkill -f keyrack-daemon is called (or keyrack daemon kill-all)
    then all daemon processes terminate
      sothat emergency cleanup is possible


# usecase.6 = session isolation (security)

given two different login sessions with the same HOME
  when keyrack unlock is called from session A
    then session B cannot access session A's daemon
      sothat cross-session access is prevented (security)

given a user in login session A
  when keyrack unlock caches a key
    then the key is only accessible from session A
      sothat session isolation is maintained (extant security behavior)
