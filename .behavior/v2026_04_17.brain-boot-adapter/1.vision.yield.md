# vision: brain-boot-adapter

## key architectural change

**BrainCli is promoted to a first-class peer of BrainAtom and BrainRepl.**

| capability | what it does | discovery |
|------------|--------------|-----------|
| `BrainAtom` | single API call (stateless) | `getBrainAtomsByAnthropic()` |
| `BrainRepl` | tool-use loop (stateful session) | `getBrainReplsByAnthropic()` |
| `BrainCli` | CLI spawn (interactive terminal) | `getBrainClisByAnthropic()` |

all three are brain capabilities. `getAvailableBrains()` discovers and returns `{ atoms, repls, clis }`.

**why this matters**:
- programmatic access to CLI brains (not just `rhx enroll` command)
- consistent discovery pattern across all brain types
- `genBrainCliForClaudeCode({ slug })` follows same factory pattern as atoms/repls
- enables composition: orchestrators can discover CLI, set config, spawn — all via code

## the outcome world

### day-in-the-life with this in place

vlad starts his day. opens 20 parallel claude code sessions—some across different repos, some multiple sessions in the same repo. each session boots with ~60k tokens of role briefs (mechanic, architect, ergonomist, keyrack, etc).

**before**: hooks land at position 23 (after cwd/env). each session creates its own cache. each `/compact` recreates cache. no reuse.

**after**: CLAUDE.md loads at position 2 (before cwd/env). with ~50k bootable tokens:

1. **(primary) ~$0.25 saved per compaction** — after `/compact`, CLAUDE.md portion hits cache instead of full price
2. **(secondary) ~$0.25 saved per enrollment** — brains with same roles share cache, regardless of workspace
3. **(tertiary) reliability** — hook outputs >50K chars get persisted to disk with only 2KB preview in context (see `refs/claude-code.hook-output-limit.md`). mechanic role alone was 167KB. briefs in hooks may silently fail to reach context. CLAUDE.md bypasses this limit entirely.

### cache events vs continuation

**key insight**: per-turn costs are the same for hooks and CLAUDE.md. within a session, prior turns are cached as prefix regardless of where briefs live.

savings happen at **cache events** — when cache must be created:
- **session start** — first API call
- **after `/compact`** — context reset, re-reads CLAUDE.md
- **after 5-min gap** — cache TTL expired (rare with active sessions)

### before/after contrast

| dimension | before (hooks) | after (boots) |
|-----------|----------------|---------------|
| brief location | SessionStart stdout | `CLAUDE.md` via `CLAUDE_CONFIG_DIR` |
| prompt position | position 23 (after dynamic) | position 2 (before dynamic) |
| per-turn (continuation) | cached | cached |
| per cache event | separate cache per session | shared cache across sessions |
| cache ttl | 5min per session | 5min shared |

**savings estimate**: ~100k token boot total (system + roles), but only ~50% are static bootable from roles (briefs & skills). the other ~50% are dynamic onBoot hooks (e.g., .behaviors, .goals) or already cached brain-cli system prompts. so ~50k tokens migratable to CLAUDE.md.

- ~$0.25 per compaction event saved (50k tokens moved)
- 20 sessions × 20 compactions/day = 400 events
- 400 events × $0.25 = **~$100/day savings**

this is just from within-thread compaction cache hits alone — sessions hit shared cache after compaction instead of each session creates its own.

**bottom line**: **~$0.25 saved per compaction event**. compaction cache hits are the main measurable advantage. ~$100/day = ~$3k/month = ~$36k/year for heavy users.

**critical: CLAUDE.md, not rules/**

| location | injection behavior | with 60k tokens |
|----------|-------------------|-----------------|
| `CLAUDE.md` | once at start, re-injected after /compact | 60k once |
| `.claude/rules/*.md` | re-injected on EVERY tool call | 60k × 30 tools = 1.8M tokens |

per [issue #32057](https://github.com/anthropics/claude-code/issues/32057): one user reported 93k tokens (46% of context) consumed by rule re-injection in ~30 tool calls. **60k in rules/ = context explosion. use CLAUDE.md.**

### the "aha" moment

> "wait... we've been paid for 20 separate caches when we could have paid for 1? just because hooks land after the cwd diverges?"

that's the click. prompt cache is prefix match. hooks land at position 23 (after cwd/env at position 12), so each session's unique cwd breaks the prefix. CLAUDE.md lands at position 2, before the divergence — all sessions share one cache.

### technical proof: prompt assembly order

per [dbreunig's analysis of claude code prompt assembly](https://www.dbreunig.com/2026/04/04/how-claude-code-builds-a-system-prompt.html):

```
position 8:  cache boundary marker
...
position 12: environment info (cwd, platform, shell) ← DYNAMIC
...
position 22: git status snapshot ← DYNAMIC
position 23: append system prompt ← SessionStart hook output lands HERE
```

the cache boundary is at position 8. CLAUDE.md and rules/ land in the cacheable zone (positions 1-7). SessionStart hook output lands at position 23—after all dynamic content.

any difference in positions 12-22 (different cwd, different git state) breaks the prefix match. our 60k at position 23 never gets a chance to share cache because the prefix already diverged.

**the fix**: move content from position 23 (hook stdout) to positions 1-7 (CLAUDE.md/rules files).

## user experience

### usecases and goals

| usecase | before | after |
|---------|--------|-------|
| `rhx enroll claude --roles mechanic,architect` | writes brief-echo hook to settings.json | generates scoped config (CLAUDE.md + settings.json) |
| `rhx init claude` | links roles, adds brief-echo hook | links roles, generates default config (CLAUDE.md + settings.json) |
| fresh session start | hook echoes 60k tokens (position 23) | claude reads CLAUDE.md (position 2, before env) |
| add a role mid-session | re-run enroll, restart session | re-run enroll, restart session (same) |
| different role combos per config | separate settings.json per combo | separate config-X/CLAUDE.md per combo |

**note**: the brief-echo hook (`rhachet roles boot`) is REPLACED by CLAUDE.md generation, not supplemented. other hooks (route.drive, behavior guards, etc.) remain in settings.json.

### contract inputs & outputs

**init** (setup default config):
```
rhx init claude --roles mechanic,architect
├─ links roles to .agent/repo=*/role=*/
└─ creates .agent/.brain/claude/config/scope=default/
   ├─ CLAUDE.md (briefs from specified roles)
   ├─ settings.json (hooks config)
   └─ .credentials.json (symlink to ~/.claude/.credentials.json)
```

**enroll** (spawn brain with config via BrainCli):
```
# use default config
rhx enroll claude
├─ discovers BrainCli via getAvailableBrains() → { atoms, repls, clis }
├─ calls cli.config.set({ roles, scope: 'default' }, context)
└─ calls cli.spawn({ scope: 'default' }, context) → CLAUDE_CONFIG_DIR=...scope=default claude

# use custom config (created on-demand)
rhx enroll claude --roles +architect,-driver
├─ discovers BrainCli via getAvailableBrains()
├─ calls cli.config.set({ roles: [...], scope: $hash }, context)
│  └─ creates .agent/.brain/claude/config/scope=$hash/
└─ calls cli.spawn({ scope: $hash }, context) → CLAUDE_CONFIG_DIR=...scope=$hash claude
```

**programmatic usage** (BrainCli as peer to BrainAtom/BrainRepl):
```typescript
const { clis } = await getAvailableBrains({}, context);
const cli = getBrainCliBySlug({ slug: 'claude/code' }, { clis });
await cli.config.set({ roles: ['mechanic', 'architect'], scope: 'default' }, context);
cli.spawn({ scope: 'default' }, context);  // spawns interactive session — scope passed explicitly
```

### spawn mechanism change

**current** (from `src/domain.operations/enroll/enrollBrainCli.ts:29-35`):
```bash
claude --config-sources local --config <configPath>
# loads config file, CLAUDE.md location not controlled
```

**new** (via BrainCli.spawn()):
```typescript
// BrainCli encapsulates spawn mechanism — no in-memory state, scope passed explicitly
cli.spawn({ scope: 'default' }, context);  // internally: CLAUDE_CONFIG_DIR=<configUri> claude
```

**why BrainCli**: spawn mechanism is brain-specific (env vars, args, etc.). encapsulated in `BrainCli.spawn()` — orchestrators don't need to know `CLAUDE_CONFIG_DIR`.

**why env var internally**: no `--config-dir` CLI flag exists per claude code docs. must use `CLAUDE_CONFIG_DIR`.

### adapter architecture

**BrainCli as first-class peer**: `BrainCli` is a peer to `BrainAtom` and `BrainRepl`. brain packages export `getBrainClisByAnthropic()` with same pattern as `getBrainAtomsByAnthropic()`. discovery via `getAvailableBrains()` returns `{ atoms, repls, clis }`.

this **deprecates** the extant `BrainHooksAdapter`. instead, suppliers define their own `genBrainCliFor*` factories that return a `BrainCli` capability.

**BrainCli interface**:
```typescript
interface BrainCli {
  slug: string;
  config: {
    get(): { uri: string; scope: string; roles: RoleSlug[] } | null;
    set(input: { roles: RoleSlug[]; scope: string }, context: ContextCli): Promise<void>;
  };
  spawn(input?: { config?: { uri: string } }): void;
}
```

**adapter interfaces** (published by rhachet):

| adapter | responsibility | example impl |
|---------|----------------|--------------|
| `BrainCliConfigBootsAdapter` | read/write boot content | writes `CLAUDE.md` |
| `BrainCliConfigHooksAdapter` | read/write hooks config | writes `settings.json` |
| `BrainCliConfigAuthAdapter` | read/write credentials | symlinks `.credentials.json` |

**why separate adapters**: each brain CLI has its own:
- boot file format (claude uses `CLAUDE.md`, cursor may use `.cursor/rules/`, etc.)
- hooks format (claude uses `settings.json`, others may differ)
- auth mechanism (claude symlinks credentials, others may differ)
- config directory convention (claude uses `CLAUDE_CONFIG_DIR`, others may differ)

**plugin architecture**: suppliers implement adapters + define factories. rhachet publishes `setBrainCliConfig` orchestrator that suppliers call with their adapters.

```
rhachet core (published)
├─ BrainCli interface
├─ BrainCliConfigBootsAdapter interface
├─ BrainCliConfigHooksAdapter interface
├─ BrainCliConfigAuthAdapter interface
└─ setBrainCliConfig({ roles, scope }, { adapters }) — orchestrator

rhachet-brains-anthropic (supplier)
├─ getBrainClisByAnthropic() → BrainCli[]  — discoverable export
├─ genBrainCliForClaudeCode({ slug }) → BrainCli
│  └─ config.set() internally calls setBrainCliConfig with adapters
├─ genClaudeCodeBootsAdapter — writes CLAUDE.md
├─ genClaudeCodeHooksAdapter — writes settings.json
└─ genClaudeCodeAuthAdapter — symlinks .credentials.json

rhachet-brains-opencode (supplier)
├─ getBrainClisByOpencode() → BrainCli[]
├─ genBrainCliForOpencode({ slug }) → BrainCli
└─ adapters for opencode-specific formats
```

**key design decisions**:
1. factory takes only `{ slug }` — context passed at `config.set()` time (matches BrainAtom/BrainRepl pattern)
2. `cli.config.set({ roles, scope }, context)` triggers adapter chain via `setBrainCliConfig`
3. suppliers define factories, rhachet defines interfaces — same pattern as atoms/repls
4. CLIs are discoverable via `getAvailableBrains()` — programmatic access, not just CLI-invoked

**init time** (when user runs `rhx init claude`):

```
input:
  --roles mechanic,architect

output:
  .agent/repo=X/role=Y/  # symlinks
  .agent/.brain/claude/config/scope=default/CLAUDE.md  # briefs from specified roles
```

### full directory structure

**linked roles vs enrolled roles**:
- **linked roles** = all roles in `.agent/repo=*/role=*/` (superset, accumulated over time via `rhx init claude --roles`)
- **enrolled roles** = roles in a specific scope's config (subset selected for enrollment)

**scope** = which roles are enrolled:
- `scope=default` → roles from last `rhx init claude --roles X` (may be subset of linked)
- `scope=$hash` → custom role set (from `rhx enroll --roles`), hash of final roles

**key insight**: `rhx enroll claude` (no `--roles`) uses the DEFAULT config, not all linked roles. see `.agent/repo=.this/role=any/briefs/define.init-enroll-brain-config.md` for full relationship.

```
/your/project/
  .agent/.brain/claude/config/
    scope=default/            # default config — COMMITTED to repo
      .credentials.json       # symlink → ~/.claude/.credentials.json (gitignored globally)
      settings.json           # generated
      CLAUDE.md               # briefs from default roles
    scope=$hash1/             # custom role set — NOT committed
      .gitignore              # ignores all files in this dir
      .credentials.json       # symlink
      settings.json           # generated
      CLAUDE.md               # briefs from custom role set
    scope=$hash2/             # another custom config — NOT committed
      .gitignore
      .credentials.json       # symlink
      settings.json           # generated
      CLAUDE.md               # different briefs
```

**default vs enrolled**:
- `scope=default` = from `rhx init claude`, used via `CLAUDE_CONFIG_DIR=.agent/.brain/claude/config/scope=default`
- `scope=$hash` = from `rhx enroll --roles`, used via `CLAUDE_CONFIG_DIR=.agent/.brain/claude/config/scope=$hash`

**no more `.claude/`**: rhachet-managed configs all live in `.agent/.brain/claude/config/`. user can still have `.claude/CLAUDE.md` for non-rhachet purposes.

**credential symlinks**: on linux, credentials are file-based. each scoped config dir symlinks to global `~/.claude/.credentials.json` for shared auth.

**settings.json**: generated by `rhx enroll` with hooks config (route.drive, behavior guards, etc.) — same as today, just in scoped dir.

**gitignore strategy**:
- `scope=default/` — NO .gitignore, entire dir committed (team shares default config)
- `scope=$hash/` — has .gitignore, entire dir ignored (custom enrollments are local)

### what gets committed

| dir | file | tracked | why |
|-----|------|---------|-----|
| `scope=default/` | `CLAUDE.md` | yes | default briefs, team shares |
| `scope=default/` | `settings.json` | yes | default hooks config |
| `scope=default/` | `.credentials.json` | no | symlink, gitignored globally |
| `scope=$hash/` | `*` | no | custom enrollments are local |

### after clone / new machine

default config (`scope=default/`) is committed, including the `.credentials.json` symlink → `~/.claude/.credentials.json`.

new team members just need to have authenticated with claude once (which creates `~/.claude/.credentials.json`). the committed symlink already points there.

no `rhx init` needed — clone and go.

### timelines

| action | time | cache state |
|--------|------|-------------|
| first session starts | t=0 | cache miss, creates 60k prefix cache |
| second session starts | t=30s | cache hit, reads 60k prefix |
| request in session 1 | t=1m | cache hit |
| request in session 2 | t=1m | cache hit (same prefix) |
| no requests for 6min | t=7m | cache expired |
| next request | t=7m | cache miss, recreates prefix |

cache stays warm indefinitely while sessions are active in parallel.

## mental model

### how users describe it to a friend

> "you know how claude code has CLAUDE.md for instructions? we generate that file with all our team's briefs and skills. it's like a precompiled rulebook that claude loads on startup—and because it's a file, not a hook, claude can cache it."

### analogies and metaphors

| metaphor | explanation |
|----------|-------------|
| **compiled vs interpreted** | hooks = interpreted (re-read every time). CLAUDE.md = compiled (read once, cached) |
| **CDN vs origin** | hooks = always hit origin server. CLAUDE.md = served from edge cache |
| **docker layer cache** | same content in different layers = cache miss. CLAUDE.md puts briefs in the base layer |
| **static site generation** | hooks = server-rendered. CLAUDE.md = pre-rendered static |
| **scoped envrc** | like direnv's `.envrc` but for claude configs — enter scope, get briefs |

### terms: users vs us

| user term | our term |
|-----------|----------|
| "the boot file" | `CLAUDE.md` in scoped config dir |
| "my roles" | enrolled role set |
| "the config folder" | `.agent/.brain/claude/config/scope=$hash/` |
| "cached startup" | prefix cache hit |
| "warm sessions" | sessions that share cache prefix |
| "scoped config" | `CLAUDE_CONFIG_DIR=.agent/.brain/claude/config/scope=$hash claude` |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| verify cache claim | ✅ yes | confirmed: hooks after dynamic content = no cache |
| move briefs to cacheable location | ✅ yes | CLAUDE.md is in prefix (position 2) |
| respect `rhx enroll` pattern | ✅ yes | generates boot file alongside settings file |
| support different roles per config | ✅ yes | each config gets its own CLAUDE.md |
| maintain hook functionality | ✅ yes | hooks still work for truly dynamic content |

### pros

- **~$0.25 saved per compaction** on ~50k bootable tokens
- **faster cold starts** (cache hit = less process time)
- **no behavioral change** for users (same commands, same roles)
- **backward compatible** (hooks still exist for dynamic needs)
- **composable** (scoped configs work naturally)
- **simplifies `repo introspect`** — drops `assertRegistryBootHooksDeclared` check (`src/domain.operations/manifest/assertRegistryBootHooksDeclared.ts:61-97`), since `BrainCliConfigBootsAdapter` replaces onBoot hooks for bootable roles
- **BrainCli as peer** — CLIs are now discoverable like atoms/repls via `getAvailableBrains()` for programmatic access

### cons

- **file generation** introduces a build step (enroll must write file)
- **stale boot files** if user manually edits linked roles without re-enroll
- **disk footprint** slight increase (scoped config dirs)
- **migration** prior users need to re-enroll to get boot files

### pit of success: edgecases

| edgecase | guard |
|----------|-------|
| user forgets to re-enroll after link new role | `rhx init claude` regenerates default config |
| CLAUDE.md gets out of sync with roles | include hash/version marker, warn on mismatch |
| user manually edits CLAUDE.md | add header comment: "# GENERATED - do not edit manually" |
| multiple configs with overlap roles | each config has isolated CLAUDE.md |
| very large role sets (>200k tokens) | warn if CLAUDE.md exceeds recommended CLAUDE.md size |

## open questions & assumptions

### assumptions made

1. **claude code reads `.claude/rules/*.md` in prefix position** — verified: positions 1-7 (before cache boundary at position 8)
2. **SessionStart hooks land after dynamic content** — verified: position 23 (after env info at 12 and git status at 22)
3. **prefix cache TTL is 5 minutes** — per anthropic docs
4. **20 parallel sessions is representative** — based on vlad's usage
5. **60k tokens is typical brief size** — assumed, needs measurement
6. **cache is shared across sessions with identical prefix in same repo** — verified (note: different repos = different prefix)

### hidden dependencies

1. **prompt assembly order stability** — this solution depends on claude code's current prompt assembly order. if anthropic changes CLAUDE.md position to load after dynamic content, cache benefit disappears. no SLA exists for this.
2. **CLAUDE_CONFIG_DIR behavior** — verified empirically (2026-04-17), but undocumented. anthropic could change behavior.
3. **users will re-enroll** — stale CLAUDE.md is a real risk. freshness check hook is essential mitigation.
4. **migration required** — users must re-enroll after upgrade to brain-boot-adapter. old settings.json without CLAUDE.md = no briefs loaded.

### upgrade behavior

`rhx upgrade` must regenerate the default brain's CLAUDE.md after role upgrades:

```
rhx upgrade --roles mechanic,architect
├─ upgrades role packages
└─ regenerates .agent/.brain/claude/config/scope=default/CLAUDE.md
   (picks up new briefs from upgraded roles)
```

**why**: upgraded roles = updated briefs. without regeneration, users get stale CLAUDE.md until they manually re-run `rhx init claude`.

**scoped configs**: only default config is auto-regenerated. custom scoped configs (`scope=$hash`) require user to re-run `rhx enroll --roles` to pick up changes.

### boot order optimization

**published roles must boot BEFORE repo=.this roles** in CLAUDE.md.

why: prefix cache works on exact match. if published role content (stable) comes first, it's more likely to hit cache even when repo=.this content (local, frequently edited) changes.

```
# CLAUDE.md order
1. published roles (ehmpathy/role=mechanic, ehmpathy/role=architect) — stable
2. repo=.this roles (.this/role=any, .this/role=custom) — local, changes often
```

this maximizes cache prefix length. if local roles change, only the suffix is invalidated.

### resolved: enrolled configs live outside .claude/

**the question was**: does `CLAUDE_CONFIG_DIR` fully override or merge with project-level `.claude/`?

**solution**: sidestep entirely by putting enrolled configs in `.agent/.brain/claude/config/scope=$hash/`

2. follows established `.agent/` pattern
3. namespaced by brain — future-proof for cursor, etc.
4. clear separation: `.claude/` = user/project, `.agent/.brain/` = rhachet-managed

## what is awkward?

### feels off or forced

- **two systems**: we now have both hooks AND boot files. hooks for dynamic, boot for static. users must understand when to use which.
- **regeneration ceremony**: user must remember to re-enroll when roles change. not automatic.
- **config proliferation**: each scoped config needs its own dir in `.agent/.brain/claude/config/`

### fights the mental model

- **"i added a role, why isn't it loaded?"** — because CLAUDE.md wasn't regenerated
- **"where do my briefs come from?"** — used to be obvious (hook echoes them). now it's a file in config dir generated earlier.
- **"which config am i in?"** — must use `CLAUDE_CONFIG_DIR` explicitly or wrapper command

### uncomfortable tradeoffs

| tradeoff | tension |
|----------|---------|
| cache vs freshness | CLAUDE.md could be stale if user forgets to regenerate |
| simplicity vs cost | hooks are simpler but expensive. boot files add complexity but save 90% |
| single source of truth | roles declared in manifests, but briefs materialized in CLAUDE.md |

### mitigations

- **auto-regenerate**: `rhx init claude` and `rhx enroll --roles` always regenerate CLAUDE.md
- **freshness check**: on session start, hook can compare CLAUDE.md hash vs current roles and warn if stale
- **header comment**: generated file header explains provenance
- **launcher command**: `rhx claude --config frontend` wraps `CLAUDE_CONFIG_DIR` complexity

## out of scope (deferred)

the following are explicitly deferred from this behavior:

### migration path for extant users

users with extant hooks-based configs (`rhachet roles boot` in settings.json) will need to manually re-run `rhx init claude` to migrate to boot files. no automatic detection or migration of legacy configs is planned.

**why deferred**: migration complexity is high, and new users benefit immediately. extant users can re-init at their convenience.

### `rhx enroll --mode plan` preview mode

no preview mode for `rhx enroll`. users cannot preview what config would be created before spawning.

**why deferred**: the operation is idempotent (always overwrites) and low-risk. preview mode adds complexity without significant safety benefit.

### config freshness check hook

the vision mentions "freshness check: on session start, hook can compare CLAUDE.md hash vs current roles and warn if stale" as a mitigation but does not specify the implementation.

**why deferred**: the staleness risk is mitigated by `rhx upgrade` auto-regenerating the default config. users who want fresh configs can re-run init/enroll.

### config version tracking

no `enrollment.yaml` or version tracking for enrolled configs. config.set() always overwrites.

**why deferred**: version tracking adds complexity. always-overwrite is simpler and sufficient for v1.
