# Proposal: Native Boot Instructions

## Why This Matters

**Hooks don't get cached.** SessionStart hook output lands late in the prompt structure — after dynamic content like cwd, environment info, and git state. Even with identical hook output across sessions, any byte difference earlier in the prompt breaks the cache prefix. Result: 60k tokens re-sent and re-billed on every request across all 20 parallel sessions.

Native instruction loading (CLAUDE.md, .claude/rules/) is positioned earlier in the prompt, in the cacheable zone. Same content, dramatically lower cost.

# Boot Migration Handoff

## TL;DR

Move 60k boot briefs from SessionStart hooks to CLAUDE.md. Use `CLAUDE_CONFIG_DIR` for scoped configs. Empirically verified to work.

---

## Why Hooks Don't Get Cached

Claude Code's prompt assembly order (from [How Claude Code Builds a System Prompt](https://www.dbreunig.com/2026/04/04/how-claude-code-builds-a-system-prompt.html)):

```
Position 8:  Cache Boundary Marker
Position 12: Environment Info (cwd, platform, shell) ← DYNAMIC
Position 22: Git Status Snapshot ← DYNAMIC
Position 23: Append System Prompt ← SessionStart hook output lands HERE
```

SessionStart hook output lands AFTER dynamic content. Any difference in cwd/environment earlier in the prompt breaks the cache prefix. With 20 sessions across different repos/worktrees, you get 20 separate caches — the hook content is never shared.

**CLAUDE.md is different.** It's injected as a `<system-reminder>` in messages, positioned BEFORE environment info. From [Multi-Tiered Prompt Caching Feature Request #19436](https://github.com/anthropics/claude-code/issues/19436):

```
1. System prompt (static, shared globally)
2. CLAUDE.md (user config) ← Your 60k HERE, before env divergence
3. Skills, environment, hooks (session-specific)
4. User messages
```

Same CLAUDE.md across sessions = shared cache, even with different cwds.

---

## Why CLAUDE.md, Not Rules

Both use `<system-reminder>` tags, but they inject differently:

| Location | Injection behavior |
|----------|-------------------|
| `CLAUDE.md` | Once at start, re-injected after `/compact` |
| `.claude/rules/*.md` | Re-injected on EVERY tool call |

From [Issue #32057](https://github.com/anthropics/claude-code/issues/32057): One user reported **93K tokens (46% of context)** consumed by rule re-injection in a session with ~30 tool calls.

**60k in rules = context explosion. Use CLAUDE.md.**

# Scoped Claude Configs Setup

## Goal

Run different Claude Code sessions with different CLAUDE.md briefs (e.g., frontend vs backend) while sharing global auth credentials.

## Verified Behavior (2026-04-17)

- `CLAUDE_CONFIG_DIR` controls where Claude Code looks for `CLAUDE.md`
- On Linux, credentials are file-based: `$CLAUDE_CONFIG_DIR/.credentials.json`
- Symlink credentials to global `~/.claude/.credentials.json` for shared auth

---

## Directory Structure

```
/your/project/
  .claude/
    config-frontend/
      .gitignore             # self-contained, ignores credentials
      .credentials.json      # symlink → ~/.claude/.credentials.json
      settings.json          # symlink → ~/.claude/settings.json
      CLAUDE.md              # 60k frontend briefs (tracked)
    config-backend/
      .gitignore
      .credentials.json      # symlink
      settings.json          # symlink
      CLAUDE.md              # 60k backend briefs (tracked)
```

---

## Setup Script

Creates scoped config dirs with:
- Self-contained `.gitignore` (ignores everything except CLAUDE.md)
- Symlinked credentials from global
- Empty CLAUDE.md ready for briefs

```bash
#!/bin/bash
# scripts/setup-claude-scopes.sh

set -e

SCOPES="${@:-frontend backend shared}"
GLOBAL_CLAUDE="$HOME/.claude"

if [[ ! -f "$GLOBAL_CLAUDE/.credentials.json" ]]; then
  echo "Error: No credentials at $GLOBAL_CLAUDE/.credentials.json"
  echo "Run 'claude' and login first."
  exit 1
fi

for scope in $SCOPES; do
  dir=".claude/config-$scope"
  mkdir -p "$dir"
  
  # Self-contained gitignore
  cat > "$dir/.gitignore" << 'EOF'
# Ignore everything except briefs
*
!.gitignore
!CLAUDE.md
EOF
  
  # Symlink credentials from global
  ln -sf "$GLOBAL_CLAUDE/.credentials.json" "$dir/.credentials.json"
  
  # Symlink global settings
  [[ -f "$GLOBAL_CLAUDE/settings.json" ]] && \
    ln -sf "$GLOBAL_CLAUDE/settings.json" "$dir/settings.json"
  
  # Create CLAUDE.md if not exists
  [[ -f "$dir/CLAUDE.md" ]] || touch "$dir/CLAUDE.md"
  
  echo "✓ $dir"
done

echo ""
echo "Add your briefs to each .claude/config-*/CLAUDE.md"
```

**Usage:**
```bash
# Default scopes
./scripts/setup-claude-scopes.sh

# Custom scopes
./scripts/setup-claude-scopes.sh api worker ingestion
```

# Cache Savings Clarified

## Key Insight

Moving 60k briefs from SessionStart hooks to CLAUDE.md does **NOT** save on every turn. It saves on **cache events** — moments when the cache must be created or refreshed.

---

## How Claude Code Caching Works

### Within a Session (Continuation)
```
Turn 1: [system] [CLAUDE.md] [user msg 1] [assistant 1]
Turn 2: [system] [CLAUDE.md] [user msg 1] [assistant 1] [user msg 2] [assistant 2]
                 └──────────────── cached prefix ─────────────────┘
```

**Prior turns are cached as prefix.** Each new turn only adds tokens at the end. The 60k briefs (whether from hooks or CLAUDE.md) are part of this cached prefix.

**Both hooks and CLAUDE.md benefit from this.** No difference in per-turn costs within a session.

### Cache Events (Creation/Refresh)
Cache must be created at:
- **Session start** — first API call
- **After `/compact`** — context reset, re-reads CLAUDE.md
- **After 5-min gap** — cache TTL expired

**This is where hooks vs CLAUDE.md differs.**

---

## The Actual Difference

### Hooks (SessionStart)
- Hook output lands at position 23 in prompt (after cwd, env info)
- Different cwd = different prefix = **separate cache per session**
- 20 sessions across different repos/worktrees = **20 separate caches**

### CLAUDE.md
- Loaded before cwd/env info diverges
- Same CLAUDE.md across sessions = **shared cache**
- 20 sessions = **1 shared cache**

---

## Revised Savings Calculation (Opus 4.5, 60k tokens)

### Per Cache Event

| Scenario | Hooks (20 sessions) | CLAUDE.md (20 sessions) | Savings |
|----------|---------------------|-------------------------|---------|
| All sessions start | 20 × $1.125 = $22.50 | 1 × $1.125 = $1.125 | $21.38 |
| All sessions compact | 20 × $1.125 = $22.50 | 1 × $1.125 = $1.125 | $21.38 |
| Cache expires (5-min gap) | 20 × $1.125 = $22.50 | 1 × $1.125 = $1.125 | $21.38 |

### Daily Savings (Estimated)

| Cache events/day | Hooks cost | CLAUDE.md cost | Savings |
|------------------|------------|----------------|---------|
| 5 (light use) | $112.50 | $5.63 | ~$107/day |
| 10 (moderate) | $225.00 | $11.25 | ~$214/day |
| 20 (heavy, many compacts) | $450.00 | $22.50 | ~$428/day |

### What This Is NOT

- **NOT** $0.81 saved per turn
- **NOT** $810/day at 1000 requests
- Per-turn costs are the same — continuation is cached either way

---

## When Cache Events Happen

1. **Session start** — unavoidable, once per session
2. **Compaction** — after `/compact`, context resets
3. **5-min idle gap** — if ALL sessions go quiet for 5+ min, cache expires

With 20 parallel sessions actively working, the 5-min TTL refreshes constantly. Main events are starts and compactions.

### Estimating Your Cache Events

```
cache_events_per_day = 
  session_starts + 
  compactions + 
  (gaps_over_5min × 1)  # shared, so just 1 per gap
```

Example: 20 sessions, each starts once, compacts twice:
- Hooks: 20 starts + 40 compacts = 60 events × $1.125 = $67.50
- CLAUDE.md: 1 start + 1 compact = 2 events × $1.125 = $2.25
- Savings: ~$65/day

---

## Summary

| Metric | Hooks | CLAUDE.md |
|--------|-------|-----------|
| Per-turn (continuation) | Cached | Cached |
| Per cache event | $1.125 × N sessions | $1.125 × 1 |
| Cross-session sharing | No | Yes |

**The win is cross-session cache sharing at cache events, not per-turn savings.**

## Verification

Check API dashboard after a day of usage:

```
cache_creation_input_tokens  → should be ~60k per cache event
cache_read_input_tokens      → should be 60k × (turns - events)
```

If `cache_creation` is high relative to total turns, you're paying for many cache events. Migration helps.

If `cache_read` dominates, most turns are cached continuations. Migration helps less (but still helps at events).
