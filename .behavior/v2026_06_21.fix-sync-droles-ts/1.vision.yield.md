# vision: include .ts files in skill distribution

## the outcome world

### before

```
# developer creates a skill that dispatches to typescript
$ cat my-skill.sh
exec npx tsx "$SCRIPT_DIR/my-skill.ts" "$@"

# publishes the role package
$ npm publish

# consumer installs and tries to use it
$ rhx my-skill --env prod
Error: Cannot find module '.../my-skill.ts'
```

the `.sh` entrypoint was distributed, but the `.ts` file it references wasn't.

### after

```
# same skill setup
$ cat my-skill.sh
exec npx tsx "$SCRIPT_DIR/my-skill.ts" "$@"

# publishes the role package
$ npm publish

# consumer installs and uses it
$ rhx my-skill --env prod
✅ works — both .sh and .ts were distributed
```

### the "aha" moment

skill authors can write complex logic in typescript with full type safety, editor support, and npm package access — while keeping the `.sh` entrypoint for shell ergonomics. the distribution "just works".

## user experience

### usecases

| actor | goal | contract |
|-------|------|----------|
| skill author | write skill logic in typescript | create `foo.sh` that calls `npx tsx "$SCRIPT_DIR/foo.ts"` |
| role publisher | include all skill artifacts | `rhachet repo compile` includes `.ts` alongside `.sh` |
| skill consumer | invoke skill | `rhx foo` works because both files are symlinked |

### what it looks like

**skill author creates collocated files:**

```
src/domain.roles/observer/skills/
├── query.database.sh       # entrypoint
└── query.database.ts       # implementation
```

**the .sh dispatches to .ts:**

```bash
#!/usr/bin/env bash
# query.database.sh
exec npx tsx "$SCRIPT_DIR/query.database.ts" "$@"
```

**compile includes both:**

```
$ rhachet repo compile --from src --into dist

📦 Compile artifacts for 1 role(s)...
   └─ observer: 4 file(s)
      ├─ domain.roles/observer/skills/query.database.sh
      ├─ domain.roles/observer/skills/query.database.ts   # ← now included
      └─ ...
```

**symlinks work:**

```
.agent/repo=ghlitch/role=observer/skills/
├── query.database.sh → node_modules/.../query.database.sh
└── query.database.ts → node_modules/.../query.database.ts
```

### timeline

1. author writes `.sh` + `.ts` pair
2. `rhachet repo compile` includes both in dist/
3. `npm publish` ships both files
4. consumer runs `rhachet init --roles ghlitch`
5. both files symlinked into `.agent/`
6. `rhx query.database` works

## mental model

### how users describe it

> "shell skills can call typescript files. rhachet makes sure both files ship together."

### analogies

- like a package.json's `main` pointing to `index.js` — you expect `index.js` to exist
- like rsync with `--include '*.ts'` — just expanding what gets copied

### terminology

| user says | we say |
|-----------|--------|
| "the typescript file" | collocated .ts |
| "the shell wrapper" | .sh entrypoint |
| "distribute" / "ship" | compile + publish |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| `.ts` files included in compile | ✅ add to DEFAULT_ARTIFACT_INCLUSIONS.skills and .inits |
| `.ts` files symlinked into .agent/ | ✅ follows from compile (symlinks point to dist/) |
| no change for extant skills | ✅ additive, `.sh` still works |

### pros

- **minimal change**: one line addition to inclusions
- **backwards compatible**: extant `.sh`-only skills unaffected
- **enables richer skills**: full typescript power for complex logic
- **no build step for skills**: `npx tsx` handles compilation at runtime

### cons

- **runtime dependency**: consumer needs `tsx` available
- **larger package size**: `.ts` source files included in dist
- **not compiled**: unlike the rest of the package, skills aren't transpiled

### edgecases and pit of success

| edgecase | handling |
|----------|----------|
| `.ts` without `.sh` pair | included anyway (valid: direct `npx tsx` invocation) |
| `.test.ts` files | excluded by extant DEFAULT_ARTIFACT_EXCLUSIONS |
| nested `.ts` in templates/ | already included via `**/template/**` pattern |

## open questions & assumptions

### assumptions

1. `npx tsx` is the standard way to invoke typescript in skills
2. consumers have `tsx` installed (or will via peerDependency)
3. `.ts` files should be included unconditionally (not just when paired with `.sh`)

### internal research

**current inclusion patterns** (`getAllArtifactsForRole.ts:16-19`):

```ts
const DEFAULT_ARTIFACT_INCLUSIONS = {
  briefs: ['**/*.md', '**/*.min'],
  skills: ['**/*.sh', '**/*.jsonc', '**/template/**', '**/templates/**'],
  inits: ['**/*.sh', '**/*.jsonc'],

```

**change required**: add `'**/*.ts'` to both `skills` and `inits` arrays (wish scopes to `domain.roles/**/*` which includes both)

**symlink behavior** (`roles link` command): symlinks whatever is in the compiled dist/. no separate logic needed — fixing compile fixes symlinks.

## what is awkward?

### shipping source files

most npm packages ship compiled `.js`, not `.ts` source. skills are an exception — they're designed for direct execution via `npx tsx`. this might surprise developers expecting compilation.

**mitigation**: document the pattern clearly in role package README.

### tsx as implicit dependency

consumers need `tsx` available. if they don't have it, the error message comes from node/npx, not rhachet.

**mitigation**: could add detection in `rhx` to surface friendly error when `.ts` skill fails to run.

### inits included too

the wish focuses on skills, but we extend `.ts` support to inits as well because:

- inits use the same `.sh` -> `npx tsx .ts` pattern
- the code change is trivial (add pattern to both arrays)
- omit inits would create artificial asymmetry

this is a consistent extension, not an explicit wish requirement.
