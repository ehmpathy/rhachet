# vision: generic context for BrainAtom and BrainRepl

## the outcome world

### day-in-the-life: before

```ts
// rhachet-brains-xai/src/domain.operations/atom/genBrainAtom.ts
export const genBrainAtom = (input: { slug: XaiBrainAtomSlug }): BrainAtom => {
  return new BrainAtom({
    slug: input.slug,
    // ...
    ask: async (askInput, context?: Empty) => {
      // ❌ context is Empty — no way to receive credentials
      // forced to use environment variables
      const openai = new OpenAI({
        apiKey: process.env.XAI_API_KEY, // 👎 implicit, hard to test
        baseURL: 'https://api.x.ai/v1',
      });
      // ...
    },
  });
};
```

brain suppliers must:
- hardcode credential sources (env vars, config files)
- or wrap their brain with custom factories
- or fight typescript errors with `as any`

the friction compounds:
- no type safety for supplier-specific context
- no pit-of-success for credential injection
- suppliers invent ad-hoc patterns that don't compose

### day-in-the-life: after

```ts
// rhachet-brains-xai/src/domain.operations/atom/genBrainAtom.ts

// supplier declares which supplies it needs to supply the brain
type ContextBrainSupplierXai = ContextBrainSupplier<'xai', BrainSuppliesXai>;

// factory binds context type at return type
export const genBrainAtom = (input: { slug: XaiBrainAtomSlug }): BrainAtom<ContextBrainSupplierXai> => {
  return new BrainAtom({
    slug: input.slug,
    // ...
    ask: async (askInput, context?: ContextBrainSupplierXai) => {
      // ✅ supplier supplies may be absent — must handle explicitly

      // option A: fail fast if supplies absent
      const creds = await (context?.['brain.supplier.xai']?.creds() ?? BadRequestError.throw('brain.supplier.xai required'));

      // option B: fallback to env var if supplies absent
      const creds = await (context?.['brain.supplier.xai']?.creds() ?? { XAI_API_KEY: process.env.XAI_API_KEY });

      const openai = new OpenAI({ apiKey: creds.XAI_API_KEY, baseURL: 'https://api.x.ai/v1' });
      // ...
    },
  });
};
```

```ts
// consumer: skill that uses xai brain
const atom = genBrainAtom({ slug: 'xai/grok/code-fast-1' });

// with context — consumer provides credentials
const result = await atom.ask(
  { prompt: 'analyze this code', schema: { output: analysisSchema }, role: {} },
  genContextBrainSupplier('xai', {
    creds: async () => ({ XAI_API_KEY: await vault.get('XAI_API_KEY') }),
  }),
);

// without context — brain falls back or fails fast; types are backwards compatible
const result = await atom.ask(
  { prompt: 'analyze this code', schema: { output: analysisSchema }, role: {} },
);
```

### the "aha" moment

> "the context is just there — i declare what my brain needs, and callers provide it. no env vars, no globals, no ceremony."

the aha: **brains become dependency-injected citizens** just like daos and services.

## user experience

### usecases

| who | goal | how |
|-----|------|-----|
| brain supplier | accept credentials at runtime | declare concrete context type on `ask`/`act` |
| skill author | inject credentials into brain | use `genContextBrainSupplier` |
| actor author | compose brain + context | pass context through `actor.ask`/`actor.act` |

### contract inputs & outputs

#### supplier declares context type

```ts
// brain supplier package declares what supplies it needs
export interface BrainSuppliesXai {
  creds: () => Promise<{ XAI_API_KEY: string }>;
}

// supplier declares which supplies it needs to supply the brain
type ContextBrainSupplierXai = ContextBrainSupplier<'xai', BrainSuppliesXai>;

// factory binds context type at return type
export const genBrainAtom = (input: { slug: XaiBrainAtomSlug }): BrainAtom<ContextBrainSupplierXai> => {
  return new BrainAtom({
    slug: input.slug,
    // ...
    ask: async (askInput, context?: ContextBrainSupplierXai) => {
      const creds = await (context?.['brain.supplier.xai']?.creds() ?? BadRequestError.throw('brain.supplier.xai required'));
      // ...
    },
  });
};
```

#### consumer provides context

```ts
// get brain from factory
const atom = genBrainAtom({ slug: 'xai/grok/code-fast-1' });

// skill or actor provides what the brain needs
const contextXai = genContextBrainSupplier('xai', {
  creds: async () => ({
    XAI_API_KEY: await secretManager.get('XAI_API_KEY'),
  }),
});

const result = await atom.ask(
  { prompt: 'hello', schema: { output: z.string() }, role: {} },
  contextXai,
);
```

### timeline

1. brain supplier author defines supplies interface (e.g., `interface BrainSuppliesXai { creds: ... }`)
2. brain supplier author defines context type alias (e.g., `type ContextBrainSupplierXai = ContextBrainSupplier<'xai', BrainSuppliesXai>`)
3. brain supplier factory binds context type at return type (e.g., `genBrainAtom(...): BrainAtom<ContextBrainSupplierXai>`)
4. consumer gets brain from factory (e.g., `const atom = genBrainAtom({ slug: 'xai/grok/code-fast-1' })`)
5. consumer creates context via `genContextBrainSupplier('xai', { creds: ... })`
6. consumer passes context to brain invocation
7. brain implementation handles absent supplies (fail fast or fallback)

## evaluation

### how well does it solve the goals?

| goal | solved? | how |
|------|---------|-----|
| suppliers can accept credentials | ✅ | `BrainAtom<TContext>` / `BrainRepl<TContext>` |
| type safety for supplier context | ✅ | `ContextBrainSupplier<TSlug, TSupplies>` |
| pit of success for context construction | ✅ | `genContextBrainSupplier(slug, supplies)` |
| backwards compatible | ✅ | `TContext = Empty` default |

### pros

- **type safety**: suppliers declare needs, consumers provide, typescript verifies
- **testability**: context is explicit, easily mockable
- **composability**: context flows through actor → brain naturally
- **backwards compatible**: extant brains with `Empty` context still work

### cons

- **verbosity**: consumers must construct context explicitly
- **learning curve**: new pattern for brain suppliers to adopt

### edge cases and pit of success

| edge case | pit of success |
|-----------|----------------|
| supplier forgets to type context | defaults to `Empty`, no break |
| consumer forgets to pass context | typescript error if brain requires it |
| wrong supplier slug | typescript error, key mismatch |
| async credentials | `supplies` supports async factories |

## open questions & assumptions

### assumptions

1. **single supplier per brain**: a brain has one supplier slug (e.g., 'xai')
2. **context is optional by default**: `TContext = Empty` preserves backwards compat
3. **namespaced keys**: `brain.supplier.<slug>` avoids collisions
4. **interface-level generics**: `TContext` lives on the interface (`BrainAtom<TContext>`, `BrainRepl<TContext>`) — implementations bind at definition time

### design decisions

**interface-level generics**

the interface declares the generic:
```ts
interface BrainAtom<TContext = Empty> {
  ask: <TOutput, TPlugs>(input, context?: TContext) => ...
}
```

factories bind the context type at return type:
```ts
export const genBrainAtom = (input: { slug: XaiBrainAtomSlug }): BrainAtom<ContextBrainSupplierXai> => {
  return new BrainAtom({ ... });
};
```

**why interface-level**: a brain supplier knows what context it needs at definition time, not per-call. the factory pattern makes this natural — bind once at the return type.

### resolved questions

1. **multiple supplier contexts**: supported via intersection types
   ```ts
   ContextBrainSupplier<'xai', XaiSupplies> & ContextBrainSupplier<'anthropic', AnthropicSupplies>
   ```
2. **namespace pattern**: wisher specified `brain.supplier.<slug>` in the wish — not a question
3. **context flow through actor**: yes, automatic flow — context passes through `actor.ask`/`actor.act` to the brain

## what is awkward?

### the `brain.supplier.<slug>` key

the namespaced key feels verbose:

```ts
context['brain.supplier.xai'].creds()
```

but alternatives have tradeoffs:
- `context.xai.creds()` — collision risk with other context keys
- `context.supplier.xai.creds()` — still verbose, less explicit

the verbosity is a feature: it's self-documenting and collision-free.

### context passthrough in actor

if actor.ask/actor.act need to pass context to the underlying brain, we may need to thread context through multiple layers. this could feel like "prop drilling".

mitigation: the pattern is explicit and traceable. the alternative (implicit globals) is worse.

### two ways to invoke brains

after this change, brains can be invoked:
1. directly: `brain.ask(input, context)`
2. via actor: `actor.ask({ prompt }, context)` (if we thread context)

this duality is acceptable — actors are orchestration, direct invocation is escape hatch.

## summary

this change transforms brains from **environment-coupled** to **context-injected** resources.

the before/after contrast:

| before | after |
|--------|-------|
| `BrainAtom` / `BrainRepl` | `BrainAtom<TContext>` / `BrainRepl<TContext>` |
| `context?: Empty` | `context?: TContext` (interface-level generic) |
| env vars, globals | typed context injection |
| ad-hoc supplier patterns | `ContextBrainSupplier` standard |
| no factory | `genContextBrainSupplier` pit of success |

the value: **brains become first-class dependency-injected citizens**.
