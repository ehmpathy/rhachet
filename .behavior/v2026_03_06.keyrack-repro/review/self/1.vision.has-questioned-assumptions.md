# self-review: has-questioned-assumptions

## assumption 1: the daemon is the correct place to fix this

**what do we assume?** that a daemon notification from `set` is the right solution

**evidence?**
- `get` only reads from daemon (by design — see comment in getKeyrackKeyGrant.ts line 14)
- daemon is the cache layer; vault is source of truth
- notify daemon on set closes the loop

**what if the opposite were true?**
- alternative: `get` could read directly from vault files
- but this would bypass the security model (daemon requires unlock first)
- the design intentionally separates vault access from key retrieval

**did the wisher say this?** no, the wisher described the symptom. i diagnosed the root cause and proposed daemon notification as the fix.

**verdict**: assumption is valid based on architecture.

---

## assumption 2: daemon is reachable at set time

**what do we assume?** that a check of `isDaemonReachable` is sufficient

**evidence?**
- daemon runs as a background process
- socket path is deterministic from owner
- if daemon not active, set still succeeds (vault is written)

**what if daemon is not reachable?**
- set writes to vault, skips daemon notification
- user must call unlock later to sync
- this is acceptable — daemon is a cache, not source of truth

**counterexamples?**
- ci/cd where daemon never starts: set works, unlock needed before get
- this is the current behavior, so no regression

**verdict**: assumption is reasonable. defensive fallback exists.

## assumption 4: input.secret is the only signal that daemon needs notification

**what do we assume?** that we should only notify daemon when input.secret is provided

**evidence?**
- if no secret provided, no value to cache in daemon
- set without secret just updates manifest, not vault content

**what if opposite were true?**
- a notify to daemon without a secret would be pointless
- daemon caches secrets, not metadata

**verdict**: assumption is correct.

## assumption 5: 9h expiry is appropriate for set-triggered grants

**what do we assume?** that keys set via `set` should use same 9h expiry as unlock

**did wisher say this?** no, i inferred from unlock behavior

**what if different expiry were better?**
- shorter: more secure but might expire mid-session
- longer: less secure
- configurable: adds complexity

**counterexamples?**
- ci/cd pipelines might want shorter expiry (1h)
- but unlock also uses 9h, so consistency is maintained

**issue found**: if user called `unlock --expires 1h` before `set`, they might expect daemon keys to expire in 1h. my fix hardcodes 9h. this creates a mismatch.

**mitigations**:
- any mismatch is resolved on next `unlock` (which reloads all keys with its expiry)
- the immediate goal (get works after set) is achieved
- configurable `--expires` can be added to `set` if users need control

**verdict**: acceptable tradeoff. the fix works for the common case. edge cases are recoverable via unlock.

## assumption 7: the fix will not break extant behavior

**evidence?**

- daemon notification is best-effort after vault write

**counterexamples?**

- now set still succeeds but might log an error
- this is not a break in the contract

**verdict**: assumption is valid. fix is additive.

## conclusion

one issue found: **expiry mismatch** (assumption 5). if unlock used custom expiry, my 9h hardcode creates inconsistency. but this is recoverable via next unlock, so it's acceptable.

no critical blockers. the fix follows the architecture (vault = truth, daemon = cache) and reuses extant patterns. the main risk is partial failure (set succeeds, daemon notify fails), but this is recoverable via unlock.
