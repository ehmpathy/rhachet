# self-review: has-questioned-questions

## triaged questions from the vision

### question 1: should we emit a warn if daemon notification fails?

**can this be answered via logic?** yes

**analysis**:
- daemon notification failure is rare (daemon not active, socket unreachable)
- a warn would clutter output for edge case
- vault write succeeded, which is the important part
- user can always unlock to sync daemon
- if we warn, user might think set failed (unclear)

**answer**: no. silent best-effort is correct. if users report confusion, we can add `--verbose` output later. no wisher input needed.

---

### question 2: should set --verbose show "daemon notified" confirmation?

**can this be answered via logic?** yes

**analysis**:
- verbose mode is for debug purposes
- "daemon notified" is useful info for debug of keyrack state
- but this is a nice-to-have, not a blocker for the fix
- can be added later without any change to the core fix

**answer**: yes, eventually. but not needed for the initial fix. scope is minimal: fix the defect first, add observability later.

---

### question 3: should there be a --no-daemon-notify flag?

**can this be answered via logic?** yes

**analysis**:
- use case: user wants to set a key but not notify daemon (why?)
- possible scenario: user is about to unlock anyway, skip redundant notify
- but redundant notify is harmless (daemon just sees same key again)
- extra flag increases complexity for no clear benefit

**answer**: no. can't think of a use case where a skip of notify would be needed. if one emerges, add the flag later.
