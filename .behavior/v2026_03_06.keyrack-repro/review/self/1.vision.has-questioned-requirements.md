# self-review: has-questioned-requirements

## requirement 1: get works after set regardless of unlock order

**who said this?** the wisher, based on a consumer's reproduction command

**evidence?** the reproduction demonstrates: `init → unlock → set → get` fails with exit 2

**what if we didn't do this?** users would continue to hit this confusing behavior. the workaround (unlock after set) is unintuitive and undocumented.

**is scope right?** yes — minimal scope. we only change `set` to notify daemon. no api changes.

**simpler way?**
- alternative: fail-fast with helpful error message on `get` when key was recently set but daemon has stale state
- but this is harder to implement (requires tracking "recently set" state) and provides worse UX than just making it work
- the chosen approach is simpler: notify daemon, done

**verdict**: requirement is valid. belongs in scope.

---

## requirement 2: 9h expiry default

**who said this?** i assumed it based on `unlock` behavior using 9h

**evidence?** looked at how `unlock` constructs `KeyrackKeyGrant` — uses same pattern

**what if we used different expiry?**
- if too short: key expires before user expects, get fails
- if too long: security concern (key stays in daemon cache longer than intended)

**is this the right default?**
- should match what `unlock` would use for consistency
- 9h is a workday — reasonable for interactive use
- for ci/cd, shorter might be better but 9h is still reasonable

**verdict**: assumption is reasonable. could add `--expires` flag later if needed.

---

## requirement 3: daemon notification is best-effort (silent failure)

**who said this?** i designed it this way

**evidence?** follows defensive design — vault is source of truth, daemon is cache

**what if daemon notify fails and we error?**
- set would fail even though vault write succeeded
- user would need to retry set, which would re-write vault (idempotent but wasteful)
- or user would need to manually unlock

**what if daemon notify fails and we warn?**
- this is an option. user sees "warning: daemon not notified, run unlock to sync"
- pros: user understands why get might fail
- cons: clutters output for rare edge case

**verdict**: silent best-effort is correct for now. vault write is the important part. if this causes confusion, we can add --verbose output later.

---

## requirement 4: no api changes

**who said this?** inferred from "backwards compatible" goal in wish

**evidence?** wish says "fix it" not "redesign the api"

**what if we did change the api?**
- could add `set --notify-daemon` flag (default true)
- but this adds complexity for no benefit — users want it to just work

**verdict**: no api changes is correct.

## what i might have wrong

1. **the 9h expiry might not match user expectations**
   - if unlock was called with different duration, there's a mismatch
   - but unlock will overwrite daemon state, so it's recoverable

2. **partial failure could still confuse users**
   - set succeeds, daemon notify fails silently, get fails
   - user sees: set ✓, get ✗ — still confusing
   - mitigation: this is rare (daemon unreachable), and unlock will fix it

---

## conclusion

requirements are valid. scope is minimal. no obvious over-engineering. the main risk is partial failure confusion, but the fallback (unlock syncs daemon) makes it recoverable.
