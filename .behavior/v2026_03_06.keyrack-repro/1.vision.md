# vision: keyrack get-after-set

## the outcome world

### before

a developer sets a credential and immediately tries to use it:

```sh
keyrack init --owner myapp --pubkey ~/.ssh/id_ed25519.pub
keyrack unlock --owner myapp --prikey ~/.ssh/id_ed25519 --env all
echo "my-secret" | keyrack set --owner myapp --key API_KEY --env all --vault os.secure
keyrack get --owner myapp --key API_KEY --env all  # exit 2: locked!?
```

confusion. the key was just set. why is it "locked"?

the developer digs through docs, tries different orders, wastes time. eventually discovers the workaround: unlock must come *after* set. but this is unintuitive — why would setting a key require unlocking again?

### after

the same flow just works:

```sh
keyrack init --owner myapp --pubkey ~/.ssh/id_ed25519.pub
keyrack unlock --owner myapp --prikey ~/.ssh/id_ed25519 --env all
echo "my-secret" | keyrack set --owner myapp --key API_KEY --env all --vault os.secure
keyrack get --owner myapp --key API_KEY --env all  # → "my-secret" ✓
```

no surprises. set a key, get it back. the daemon is notified automatically.

### the "aha" moment

> "wait, i don't need to unlock again after set? it just works?"

the value clicks when developers realize they can trust the natural order of operations. set means set. get means get. no hidden state synchronization required.

## user experience

### usecases

1. **ci/cd credential injection** — set credentials during pipeline setup, immediately use them in subsequent steps
2. **interactive credential management** — set a key, verify it, use it in the same session
3. **scripted automation** — bash scripts that set and get without worrying about daemon state

### contract inputs & outputs

**before (broken)**:
```
input:  set key → get key
output: exit 2 (locked)
```

**after (fixed)**:
```
input:  set key → get key
output: the secret value
```

### timeline

1. `keyrack init` — initialize keyrack for an owner
2. `keyrack unlock` — authenticate and load extant keys into daemon
3. `keyrack set` — write new key to vault AND notify daemon
4. `keyrack get` — retrieve key from daemon (now works immediately)

## mental model

### how users would describe this

> "keyrack remembers keys i set, even in the same session"

### analogies

- **password manager autosave** — when you save a new password, you don't need to "refresh" to use it
- **filesystem cache** — when you write a file, you can read it back immediately
- **database transaction** — after commit, the data is visible

### terms

| user term | our term |
|-----------|----------|
| "remembers" | daemon cache updated |
| "session" | daemon lifetime |
| "locked" | key not in daemon cache |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| get works after set | ✓ daemon notified on set |
| no order dependency | ✓ unlock before or after set both work |
| no user action required | ✓ automatic daemon notification |
| backwards compatible | ✓ no api changes, just behavior improvement |

### pros

- zero friction — it just works
- no api changes — extant scripts continue to work
- defensive — if daemon unreachable, set still succeeds (vault written)
- pit of success — users can't accidentally get into the broken state

### cons

- slight latency increase — set now checks daemon reachability and sends notification
- network roundtrip — daemon notification is a socket call
- partial atomicity — vault write succeeds, daemon notify could fail (daemon will sync on next unlock)

### edgecases

| edgecase | behavior |
|----------|----------|
| daemon not running | set succeeds, daemon not notified (unlock required later) |
| daemon unreachable | set succeeds, daemon not notified (same as above) |
| daemon rejects key | set succeeds, error logged, unlock required later |
| no secret provided | no daemon notification (no value to cache) |

## open questions & assumptions

### assumptions

1. **9h expiry default is acceptable** — keys set via `set` use 9h expiry like unlock does
2. **grade inference is correct** — we infer key grade from vault+mech, same as unlock
3. **daemon socket path is computable** — we have owner, can derive socket path

### questions to validate

1. should we emit a warning if daemon notification fails?
2. should set --verbose show "daemon notified" confirmation?
3. should there be a --no-daemon-notify flag for special cases?

## what is awkward?

### design tensions

1. **partial failure** — vault write succeeds but daemon notify fails. the key is stored but not immediately usable. this is recoverable (unlock will sync) but may confuse users who see set succeed but get fail.

2. **implicit vs explicit** — we're making daemon notification implicit. some users may prefer explicit control. but the pit of success argues for implicit — most users want it to just work.

3. **expiry mismatch** — if user set a key with set, then unlocks with different duration, the daemon may have two different expiry times for the same key. the unlock would overwrite, so this is fine, but it's a subtle interaction.

### mitigations

- if daemon notify fails, set still succeeds (vault is source of truth)
- unlock will always sync daemon to vault state
- no api changes means no migration burden
