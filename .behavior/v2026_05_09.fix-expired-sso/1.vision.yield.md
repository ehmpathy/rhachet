# vision: fix expired sso validation gap

## the outcome world

### day-in-the-life: before

```bash
$ rhx keyrack unlock --owner ehmpath --env prep

🔓 keyrack unlock
   └─ ahbode.prep.AWS_PROFILE
      ├─ env: prep
      ├─ vault: aws.config
      └─ expires in: 540m  # looks good!

$ aws sts get-caller-identity

aws: [ERROR]: Credentials were refreshed, but the refreshed credentials are still expired.
```

the human is confused: keyrack said the credentials are valid for 540 minutes, but aws says they're expired? did keyrack lie?

### day-in-the-life: after

```bash
$ rhx keyrack unlock --owner ehmpath --env prep

🔓 keyrack unlock
   └─ ahbode.prep.AWS_PROFILE
      ├─ with sso prior?
      │  └─ ✗ sts credentials expired, will re-auth
      [browser opens for SSO login]
      ├─ env: prep
      ├─ vault: aws.config
      └─ expires in: 60m  # actual expiration from aws

$ aws sts get-caller-identity

{
  "UserId": "AROA...:user@example.com",
  "Account": "123456789012",
  "Arn": "arn:aws:sts::123456789012:assumed-role/Role/user@example.com"
}
```

keyrack detects the expired sts credentials and triggers re-auth automatically. no surprises.

### the "aha" moment

"keyrack doesn't just check if i'm logged into sso — it checks if my actual aws credentials work. no more false positives."

## user experience

### usecases

| usecase | goal | contract |
|---------|------|----------|
| unlock prep credentials | get valid aws credentials for prep env | `rhx keyrack unlock --owner ehmpath --env prep` |
| verify session is valid | confirm credentials work before long job | `aws sts get-caller-identity` |

### timeline

1. human runs `rhx keyrack unlock`
2. keyrack validates via `aws configure export-credentials`:
   - forces credential refresh (validates SSO token implicitly)
   - checks `AWS_CREDENTIAL_EXPIRATION` timestamp (validates STS)
3. if fails → trigger sso login flow
4. credentials are ready for use

## evaluation

### how well does it solve the goals?

| goal | before | after |
|------|--------|-------|
| no false positives | ❌ says valid when expired | ✓ detects expired sts |
| auto re-auth | ❌ manual intervention | ✓ triggers login flow |

### pros

- eliminates the "keyrack says valid but aws says expired" confusion
- no extra user action required — just works
- backward compatible (no contract changes)

### cons

- slightly slower validation (one extra aws cli call)
- adds ~200-500ms to unlock time

### edge cases

| edge case | pit of success |
|-----------|----------------|
| sso valid, sts expired | detect via export-credentials, trigger re-auth |
| sso expired, sts cached | detect via list-accounts, trigger re-auth |
| both expired | detect via list-accounts first, trigger re-auth |
| both valid | fast path, no re-auth needed |

## open questions & assumptions

### assumptions

1. `aws configure export-credentials` returns accurate expiration times
2. the error message "refreshed credentials are still expired" comes from aws cli, not sdk
3. sts credentials have a fixed ~1 hour expiration from aws sso

### questions answered

- **why doesn't list-accounts catch this?** — it validates the sso access token, not the sts credentials. they have separate expirations.
- **why does sts get-caller-identity lie?** — aws cli caches sts credentials in `~/.aws/cli/cache/`. the cache can contain valid-format but expired credentials.

### to validate with wisher

- none — root cause is clear from research

### to research externally

- none — aws cli behavior is documented in issue #9845

## groundwork

### external research

**aws cli issue #9845**: "sts get-caller-identity uses cached credentials"
- confirmed: aws cli caches sts credentials separately from sso access tokens
- the cache can lie about validity
- solution: use `aws configure export-credentials` which refreshes and returns actual expiration

### internal research

**current validation flow** (`vaultAdapterAwsConfig.ts:77-182`):
1. for sso profiles: validates via `aws sso list-accounts --access-token`
2. validates sts via `aws configure export-credentials --profile`
3. then calls `aws sts get-caller-identity --profile` for username
4. if all succeed, returns `{ valid: true, username }`

**the gap**: step 1 is redundant — `export-credentials` already validates SSO implicitly

**fix location**: `src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts`
- remove `list-accounts` call entirely
- use `export-credentials` as primary validator (validates both SSO and STS)
- keep `get-caller-identity` only for username extraction
- if credentials expire in < 5 minutes, return `{ valid: false }` to trigger re-auth

**vocab to reuse**:
- `validateSsoSession` — the function that does validation
- `isUnlocked` / `unlock` — vault adapter methods

**stdout patterns to match**:
```
   ├─ with sso prior?
   │  ├─ ✓ username, access confirmed
   │  └─ ✓ clear, will reuse
```

## what is awkward?

### time window

there's a small time window where:
1. validation passes (credentials have >5min left)
2. user runs long job
3. credentials expire mid-job

mitigated by the 5-minute buffer. most operations complete within 5 minutes.

### two validation layers

the fix simplifies to two calls:
1. sts credentials (export-credentials) — validates both SSO and STS
2. sts identity (get-caller-identity) — extracts username for display

export-credentials handles both failure modes:
- expired SSO session → command fails (can't refresh STS without valid SSO)
- expired STS credentials → command refreshes if SSO valid, fails if SSO invalid

### "540m" display

the current "expires in: 540m" comes from the requested duration, not actual expiration. after this fix, the display should show the actual `AWS_CREDENTIAL_EXPIRATION` from the credentials. (enhancement for later)
