# vision: keyrack output modes

## the outcome world

### day-in-the-life: before

```bash
# to get just the secret for pipe use, parse json manually
rhx keyrack get --key API_KEY --env test --json | jq -r '.grant.key.secret'

# to source keys in shell, call sdk from node
node -e "require('rhachet/keyrack').keyrack.source({ env: 'test', owner: 'ehmpath' })"

# to source a single key for shell, chain jq + export
export API_KEY=$(rhx keyrack get --key API_KEY --env test --json | jq -r '.grant.key.secret')
```

friction: jq dependency, verbose, error handle scattered.

### day-in-the-life: after

```bash
# get just the secret value (pipe friendly)
rhx keyrack get --key API_KEY --env test --value

# source all repo keys into current shell
eval "$(rhx keyrack source --env test --owner ehmpath)"

# source a single key
eval "$(rhx keyrack source --key API_KEY --env test --owner ehmpath)"

# strict mode: fail if any key absent (default)
eval "$(rhx keyrack source --env test --owner ehmpath --strict)"

# lenient mode: source what's available, skip absent
eval "$(rhx keyrack source --env test --owner ehmpath --lenient)"
```

single commands, no jq, consistent error behavior.

### the "aha" moment

"wait, i can just `--value` to get the secret? and `source` outputs export statements i can eval?" — shell users everywhere

## user experience

### usecases

| usecase | contract | output |
|---------|----------|--------|
| pipe secret to command | `keyrack get --key X --value` | raw secret, no newline |
| set env var in shell | `eval "$(keyrack source --key X)"` | `export X='secret'` |
| source all repo keys | `eval "$(keyrack source --env test)"` | multiple exports |
| ci/cd setup | `keyrack source --env test --strict` | exports or exit 2 |
| local dev (optional keys) | `keyrack source --env test --lenient` | exports available keys |

### contract inputs & outputs

#### `keyrack get --value` / `--output value`

```
input:
  --key <keyname>     required (single key only)
  --env <env>         required
  --value             shorthand flag for --output value
  --output <mode>     explicit output mode selection (value, json, vibes)

output (stdout):
  raw secret value (no final newline)

exit codes:
  0 = granted
  2 = not granted (blocked/locked/absent)

note:
  --value is an alias for --output value
  --json is an alias for --output json
  default output is vibes (turtle treestruct)
```

#### `keyrack source`

```
input:
  --key <keyname>     optional (single key) or omit for all repo keys
  --env <env>         required
  --owner <owner>     required
  --strict            default: fail if any key not granted
  --lenient           alternative: skip absent keys

output (stdout):
  export KEY1='secret1'
  export KEY2='secret2'

exit codes:
  0 = all requested keys exported (or lenient mode)
  2 = strict mode and some keys not granted

hint on exit 2:
  "some keys not granted. use --lenient if partial results are acceptable."
```

### timeline: single key

```
user runs: rhx keyrack get --key API_KEY --env test --value
         → daemon returns grant
         → stdout: sk-abc123...
         → exit 0
```

### timeline: source all

```
user runs: eval "$(rhx keyrack source --env test --owner ehmpath)"
         → cli invokes keyrack get --for repo --json
         → filters to env
         → emits export statements
         → shell eval sets env vars
         → exit 0
```

## mental model

### how users describe it

> "keyrack get with --value gives me the secret, source gives me shell exports"

> "it's like `pass show` for secrets and `source .env` for env setup"

### analogies

| concept | analogy |
|---------|---------|
| `--output value` / `--value` | `jq -r .value` built-in |
| `--output json` / `--json` | structured output for programmatic use |
| `source` | `source .env` but from vault |
| `--strict` | `set -e` for credentials |
| `--lenient` | `|| true` for optional secrets |

### user terms vs our terms

| user says | we call it |
|-----------|------------|
| "the secret" | `grant.key.secret` |
| "export it" | `keyrack source` |
| "fail if absent" | `--strict` (default) |
| "skip absent" | `--lenient` |

## evaluation

### how well does it solve the goals?

| goal | solution | fit |
|------|----------|-----|
| pipe secret to command | `--value` flag | exact |
| shell setup | `source` subcommand | exact |
| sdk parity | same --strict/--lenient | exact |

### pros

- zero dependencies (no jq required)
- consistent with sdk (same modes, same semantics)
- composable (pipe, eval, subshell)
- turtle vibes maintained for human output

### cons

- `source` is a new subcommand (not just a flag)
- `--value` only works with single key (by design)

### edgecases and pit of success

| edgecase | behavior |
|----------|----------|
| `--value` without `--key` | error: --value requires --key |
| `--value` with `--for repo` | error: --value requires --key |
| secret contains single quote | shell-escaped: `'sec'\''ret'` |
| secret contains newlines | preserved in output |
| key not granted + strict | exit 2, stderr shows status |
| key not granted + lenient | skip, no export for that key |

## open questions & assumptions

### assumptions

1. `--value` should not print a final newline (for pipe use)
2. `source` output uses single quotes (handles most special chars)
3. `--strict` is the default (fail-fast philosophy)
4. `source` without `--key` sources all repo manifest keys for env

### questions [answered]

1. **should `--value` be a flag or output mode like `--output value`?**
   [answered] support both. `--output <mode>` is the full format (value, json, human). `--value` and `--json` are shorthand aliases.

2. **should `source` support `--for repo` syntax or just default to repo?**
   [answered] no `--for repo`. omit `--key` = all repo keys. simpler, intuitive.

3. **should lenient mode emit notices to stderr for skipped keys?**
   [answered] no. SDK lenient mode is silent; CLI matches SDK. use `--json` to debug.

### sdk parity note

`source --key X` is CLI ergonomics extension beyond SDK. SDK's `sourceAllKeysIntoEnv` has no key filter — it sources ALL repo keys. CLI supports both:
- `source` (no --key): all repo keys (SDK parity)
- `source --key X`: single key (wish request, CLI extension)

### external research needed

none — shell escape patterns are well-understood.

## what is awkward?

### `source` as subcommand vs flag

alternative: `keyrack get --for repo --output shell`

but: `source` is clearer intent, matches mental model of `source .env`

### `--value` only for single key

could support `--value` for `--for repo` to emit newline-separated values.

but: ambiguous ordering, no key names — probably not useful.

### `--strict` as default

some may expect lenient by default.

but: fail-fast is safer for ci/cd; lenient is opt-in escape hatch.

### shell escape edge cases

extreme edge cases (null bytes, etc) may not escape perfectly.

mitigation: document limits, recommend `--json` for programmatic use.
