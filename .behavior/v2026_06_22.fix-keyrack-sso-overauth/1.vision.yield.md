# vision: keyrack SSO session reuse

## the outcome world

### before

```
[t0] keyrack unlock --env test
     → SSO browser auth (expected, first time)

[t1] keyrack unlock --env prod
     → SSO browser auth (unnecessary — same user, same SSO domain)

[t2] keyrack unlock --env test
     → SSO browser auth AGAIN (prod unlock cleared test's token)
```

every profile switch triggers re-auth, even when the same SSO user is valid for both.

### after

```
[t0] keyrack unlock --env test
     → SSO browser auth (expected, first time)

[t1] keyrack unlock --env prod
     → no re-auth (same awsSsoUsername, reuse session)

[t2] keyrack unlock --env test
     → no re-auth (token still valid)
```

re-auth only when the expected user differs from the cached user.

### the "aha" moment

"i can switch between test, prep, and prod without the browser popup every time — keyrack knows it's still me."

## user experience

### usecases

| goal | before | after |
|------|--------|-------|
| unlock test then prod | 2 browser auths | 1 browser auth |
| unlock prod then test | 2 browser auths (test token was cleared) | 1 browser auth |
| switch SSO users | manual logout, then re-auth | automatic: keyrack detects mismatch, clears, re-auths |

### contract inputs & outputs

**on `keyrack set --vault aws.config`:**

```
input:  keyrack set --key AWS_PROFILE --env prod --vault aws.config
output: 🔐 keyrack set (org: ehmpathy, env: prod)
           └─ ehmpathy.prod.AWS_PROFILE
              ├─ mech: EPHEMERAL_VIA_AWS_SSO
              └─ vault: aws.config
```

internally, `meta.awsSsoUsername` is captured from ARN session name after SSO auth.

**on `keyrack unlock` (same user):**

no browser popup — session reused internally. no stdout change.

**on `keyrack unlock` (different user):**

```
output: 🔓 keyrack unlock
           └─ ehmpathy.prod.AWS_PROFILE
              ├─ with sso prior?
              │  ├─ ✗ https://d-12345abcde.awsapps.com/start, expires 2099-01-01T00:00:00Z
              │  │  ├─ expected: bob
              │  │  └─ observed: alice
              │  └─ ✓ cleared, re-auth triggered
              ├─ env: prod
              ├─ org: ehmpathy
              ├─ vault: aws.config
              └─ expires in: 540m
```

clear visibility into why re-auth was needed.

### timeline

1. `keyrack set` — SSO auth, ARN captured to `meta.awsSsoUsername`
2. `keyrack unlock` — check cached session ARN vs stored ARN
3. match → reuse session
4. mismatch → clear session, re-auth, log clearly

## mental model

### how users would describe it

"keyrack remembers who i logged in as, so it doesn't make me re-auth unless i'm a different user."

### analogies

- like a hotel keycard that works across multiple room types (test, prep, prod) as long as it's the same guest
- like browser cookies that persist across tabs until you explicitly log out

### terms

| user term | our term |
|-----------|----------|
| "my SSO login" | `meta.awsSsoUsername` (username from ARN session) |
| "session" | cached SSO token in `~/.aws/sso/cache/` |
| "profile" | `exid` (aws profile name) |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| avoid unnecessary re-auth on profile switch | yes — same user = reuse session |
| clear session on user switch | yes — different ARN = clear + re-auth |
| visibility into logout reason | yes — log shows cached vs expected ARN |

### pros

- eliminates friction for same-user multi-env workflows
- maintains security for multi-user scenarios
- clear logs when sessions are cleared

### cons

- requires `keyrack set` to re-run for extant keys (to capture ARN)
- adds ~1 API call (`sts get-caller-identity`) on unlock to verify cached session

### pit of success

| edge case | behavior |
|-----------|----------|
| `meta.awsSsoUsername` absent | failfast: "re-set the key" |
| cached token expired | normal flow: fresh auth |
| cached token for wrong user | clear, re-auth, log clearly |
| no cached token | normal flow: fresh auth |

## open questions & assumptions

### assumptions

1. `sts get-caller-identity` returns consistent ARN format across SSO sessions
2. cached SSO token can be validated via `sts get-caller-identity` before full unlock
3. session clear for domain doesn't affect unrelated profiles

## groundwork

### external research

**AWS STS get-caller-identity:**
- returns `{ UserId, Account, Arn }` — ARN format is `arn:aws:sts::{account}:assumed-role/{role}/{session}`
- works with cached SSO credentials (no browser needed if token valid)
- fails with `ExpiredTokenException` if token expired

### internal research

**extant code paths:**

- `vaultAdapterAwsConfig.ts` — current unlock flow, calls `validateSsoSession()`
- `unlockKeyrackKeys.ts:200-234` — vault.get() always called, no daemon check
- `clearAwsSsoCacheForDomain()` — clears all tokens for SSO domain (the bug)

**contracts to conform to:**

- `KeyrackHostVaultAdapter.set()` returns `{ mech, exid?, meta? }` (add meta)
- `KeyrackKeyHost.meta: Record<string, unknown> | null` (add field)
- `isKeyrackKeyHostAttrsEqual()` — update to compare `meta` instead of `vaultRecipient`

**vocab:**

- `awsSsoUsername` — ARN from sts get-caller-identity
- `ageKeyRecipient` — pubkey for age encryption (os.secure)

**stdout patterns:**

- `🔐 keyrack set` / `🔓 keyrack unlock` headers
- tree structure with `├─`, `└─` for nested output

## what is awkward?

### re-set requirement

extant keys lack `meta.awsSsoUsername`. users must re-run `keyrack set` to capture it.

mitigation: failfast with clear message — "key lacks awsSsoUsername; re-set the key"

### username extraction from ARN

ARN format: `arn:aws:sts::{account}:assumed-role/{role}/{session}`

we extract `{session}` as the username.

**verified via live test:**
```
arn:aws:sts::805192865516:assumed-role/AWSReservedSSO_ehmpathy-demo-sso_18136d510d9aa5f5/demo-agent
                                                                                         └── session name
```

session name `demo-agent` is stable across logins — not timestamp-based.

this means same user with different roles still matches (only session name compared).

## critical: test and snapshot coverage

**test coverage and snapshot coverage is the most important part of this work.**

- exhaustive test coverage against every combinatorial boundary case (see wish)
- snapshot coverage for all stdout outputs — enables visual verification in PRs
- snapshots catch regressions and format drift
- every new output path must have a snapshot test

the implementation is secondary to the test coverage. without tests, the behavior is unverified.
