# blocker.1: race condition in host manifest write (read-modify-write without atomicity)

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/access/daos/daoKeyrackHostManifest/index.ts:130-150 (set function writeFileSync)
- src/domain.operations/keyrack/setKeyrackKeyHost.ts:132-140 (calls daoKeyrackHostManifest.set with full manifest)

The `daoKeyrackHostManifest.set` function performs a read-modify-write sequence on the encrypted host manifest file without any locking or transaction. When two instances of this code run concurrently (e.g., two keyrack set commands), each reads the current state, modifies it in memory, then writes the entire file back. The second write overwrites the first's changes, leading to data loss. This violates the rule against race conditions (read-modify-write without protection). The same pattern exists in `setKeyrackKeyHost` which calls `daoKeyrackHostManifest.set({upsert: manifestUpdated})` after modifying the in-memory manifest.

To fix: use an atomic file write pattern (write to temp file, rename) and consider file-level locking or a version check to detect concurrent modifications. Alternatively, redesign to avoid full-file overwrite in concurrent scenarios.

Location 1: `daoKeyrackHostManifest.set` does direct writeFileSync without checking if the file changed since read.
Location 2: `setKeyrackKeyHost` reads context.hostManifest (loaded earlier), modifies hosts, then passes the full manifest to set. The set then writes the entire file, ignoring any concurrent changes.

**snippet**:
```ts
// daoKeyrackHostManifest.set (index.ts):
  // write encrypted content with restricted permissions
  writeFileSync(path, ciphertext, 'utf8');
  chmodSync(path, 0o600);

// setKeyrackKeyHost.ts:
  const manifestUpdated = new KeyrackHostManifest({
    ...hostManifest,
    hosts: hostsUpdated,
  });
  await daoKeyrackHostManifest.set({ upsert: manifestUpdated });
```

# nitpick.1: order-dependence: hostManifest must be pre-loaded before set/unlock

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/keyrack/setKeyrackKeyHost.ts:76-80
- src/domain.operations/keyrack/session/unlockKeyrackKeys.ts:62-67

Several domain operations (setKeyrackKeyHost, unlockKeyrackKeys) require that context.hostManifest is already set (loaded via a prior call). They throw UnexpectedCodePathError if missing. While this is explicit, it creates an implicit ordering requirement that could surprise callers. The rule considers order-dependence a behavior hazard if not documented. Here it is documented in comments, but the enforcement is runtime error rather than compile-time guarantee. This is a minimal concern (nitpick) because the error message is clear.

**snippet**:
```ts
// setKeyrackKeyHost.ts:
  if (!context.hostManifest)
    throw new UnexpectedCodePathError(
      'hostManifest required for set; call daoKeyrackHostManifest.get() first',
      { slug: input.slug },
    );

// unlockKeyrackKeys.ts:
  if (!context.hostManifest)
    throw new UnexpectedCodePathError(
      'host manifest not found. run: rhx keyrack init',
      { owner: input.owner },
    );
```
