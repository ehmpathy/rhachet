# nitpick.1: identity extracted from ContextKeyrack and passed to vault adapter as string

**rule**: .agent/repo=.this/role=keyrack/briefs/rule.require.context-for-identity.md

**locations**:
- src/domain.operations/keyrack/session/unlockKeyrackKeys.ts:137-145

In `unlockKeyrackKeys.ts`, the identity is extracted from `context.identity.getOne({ for: 'manifest' })` and then passed as a plain string to `adapter.isUnlocked` and `adapter.unlock`. The rule states that operations needing identity access should receive `ContextKeyrack` directly, not extracted identities. While vault adapter interfaces currently accept `identity: string | null`, this pattern duplicates the logic that the context already encapsulates and loses the lazy‑caching behavior. Consider updating vault adapters to accept `ContextKeyrack` (or a lighter identity container) and let them call `context.identity` themselves.

**snippet**:
```ts
    const identity = await context.identity.getOne({ for: 'manifest' });

    // unlock vault if needed
    const isUnlocked = await adapter.isUnlocked({
      exid: hostConfig.exid,
      identity,
      meta: hostConfig.meta,
    });
    if (!isUnlocked) {
      await adapter.unlock({
        identity,
        exid: hostConfig.exid,
        silent: true,
        meta: hostConfig.meta,
      });
    }
```
