# blocker.7: write-only vault keys silently omitted during bulk unlock without clear error

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- src/domain.operations/keyrack/session/unlockKeyrackKeys.ts:~115-120

when unlocking keys in bulk, if a key's vault is write-only (e.g., github.secrets), it is silently added to the omitted list with reason 'remote' and the loop continues. the user receives no error and may not notice the omission in the output list. this is a friction hazard because a blocked state is not clearly communicated. (if a specific key is requested, an error is thrown, which is correct, but the bulk path silently skips.)

**snippet**:
```ts
if (adapter.get === null) {
  if (input.key) {
    throw new ConstraintError(`${vault} cannot be unlocked`, {
      slug: effectiveSlug,
      vault,
      hint: 'write-only vault; secrets cannot be retrieved via api',
    });
  }
  // bulk unlock → skip silently, add to omitted
  keysOmitted.push({ slug: effectiveSlug, reason: 'remote' });
  continue;
}
```

# blocker.8: lenient key sourcing may cause unclear later failures in aws.whoami.sh

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- .agent/repo=.this/role=any/skills/aws.whoami.sh:~20

the script uses `--lenient` when sourcing keyrack credentials, which silences errors for missing keys. if a required key is absent, the script does not fail immediately; later, `aws sts get-caller-identity` may fail with a cryptic error message (e.g., "config profile missing") that does not clearly indicate the root cause — a missing credential. this violates the principle that errors should be actionable and immediate.

**snippet**:
```bash

```
