# blocker.1: eval without exit code check hides command failure

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- .agent/repo=.this/role=any/skills/aws.whoami.sh:29

The script uses `eval "$(rhx keyrack source ...)"` with `--lenient`. If the inner command fails (e.g., network error, missing credentials), its stdout is empty and eval silently does nothing. The script continues as if success, hiding real errors. This is a failhide pattern. Must check exit code of the command substitution or use a guard that fails fast.

Fix: capture stdout separately and check exit code, or use `set -e` (already set but does not apply inside command substitution). Consider: `output=$(rhx keyrack source ...) && eval "$output" || exit 1`

**snippet**:
```sh
# source keyrack credentials (lenient to allow partial key sourcing)
eval "$(rhx keyrack source --owner ehmpath --env "$ENV" --lenient)"
```

# blocker.2: error logged via console.log + process.exit instead of throwing proper error class

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.require.failloud.md

**locations**:
- src/contract/cli/invokeKeyrack.ts:482-493

Multiple CLI actions in invokeKeyrack.ts handle errors by printing a message to stdout and calling process.exit(2). This violates rule.require.failloud: errors must use proper error classes (BadRequestError, ConstraintError) with full context. The pattern also prevents proper error propagation and testing.

Fix: replace each instance with `throw new ConstraintError(...)` or `throw new BadRequestError(...)` with contextual metadata.

Representative occurrence in `keyrack set` action (similar pattern in del, list, source actions).

**snippet**:
```ts
if (!repoManifest) {

    console.log('   └─ tip: for sudo credentials without keyrack.yml, use --org @all');
    console.log('');
    process.exit(2);
  }
  console.log('');
  console.log('✋ no keyrack.yml found');
  console.log("   └─ tip: run 'npx rhachet keyrack init --org <your-org>' to create one");
  console.log('');
  process.exit(2);
}
```

# nitpick.1: process.exit(2) after error output – prefer throw

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.require.failfast.md

**locations**:
- src/contract/cli/invokeKeyrack.ts:330-332
- src/contract/cli/invokeKeyrack.ts:605-608
- src/contract/cli/invokeKeyrack.ts:620-623

In `keyrack get` and `keyrack source` actions, after outputting error messages via stderr/stdout, the code calls `process.exit(2)`. While exit code is correct, it does not throw an error class. This prevents callers from catching the error and reduces testability. Consider throwing a `ConstraintError` after output, or returning a structured error.

Fix: throw `ConstraintError` after output, or refactor to let the error propagate naturally.

**snippet**:
```ts
console.log(formatKeyrackGetAllOutput({ attempts }));

// exit 2 if any key was not granted (blocked by constraints)
if (!isAllAttemptsGranted({ attempts })) {
  process.exit(2);
}
```
