# blocker.1: aws.config set does not verify roundtrip for exid case

**rule**: .agent/repo=.this/role=keyrack/briefs/rule.require.vault-roundtrip-verify.md

**locations**:
- src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts

In vaultAdapterAwsConfig.ts, the set method performs roundtrip verification only when exid is not provided (guided setup). When exid is provided (e.g., --exid profile-name), it merely checks that the profile exists in ~/.aws/config, but does not verify that the profile can be successfully read (get) after storage. The rule requires that all vault adapters storing secrets must verify roundtrip on set. The exception for 'inherently verified' vaults (os.direct, os.daemon) does not apply to aws.config. This leaves the key unverified for the common --exid path; user may only discover failure at unlock time.

**snippet**:
```ts
// exid case: only checks profile exists, no roundtrip verify
if (input.exid) {
  const profileExists = checkProfileExists(profileName);
  if (!profileExists) {
    throw new ConstraintError(
      `aws profile '${profileName}' not found in ~/.aws/config`,
      { slug: input.slug, profileName, hint: 'check the profile name' },
    );
  }
}

// guided setup (no exid) has roundtrip verify:
if (!input.exid) {
  console.log('   │');
  console.log('   └─ perfect, now lets verify...');
  // ... full unlock/get/relock roundtrip
}

```
