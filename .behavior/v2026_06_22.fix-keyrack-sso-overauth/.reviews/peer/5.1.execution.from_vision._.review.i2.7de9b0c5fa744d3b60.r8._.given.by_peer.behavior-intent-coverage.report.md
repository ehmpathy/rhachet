# blocker.3: Unlock CLI output does not display SSO prior status as specified in wish/vision

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.require.behavior-intent-coverage.md

**locations**:
- src/domain.operations/keyrack/session/unlockKeyrackKeys.ts:113-119
- src/domain.operations/keyrack/adapters/vaults/aws.config/vaultAdapterAwsConfig.ts:201-225

The wish and vision specify that `keyrack unlock` should display the 'with sso prior?' status (whether session was reused, cleared due to mismatch, or expired). This output is implemented in vaultAdapterAwsConfig.unlock but only when silent=false. In the orchestrator unlockKeyrackKeys, the vault adapter is called with silent:true, suppressing this output. The CLI output is then constructed from the returned grants (emitted via emitKeyrackKeyBranch) which does not include the 'with sso prior?' branch. Therefore users do not see the session reuse/clear information, violating the intended behavior.

Specifically:
- unlockKeyrackKeys.ts passes silent:true
- vaultAdapterAwsConfig.unlock only prints 'with sso prior?' when !input.silent
- The CLI output does not include this information

**snippet**:
```ts
// unlockKeyrackKeys.ts (orchestrator)
await adapter.unlock({
  identity,
  exid: hostConfig.exid,
  silent: true,  // ← suppresses the vault adapter's output
  meta: hostConfig.meta,
});

// vaultAdapterAwsConfig.unlock (vault adapter)
if (!input.silent) {
  console.log('   │');
  console.log('   ├─ with sso prior?');
  console.log(`   │  ├─ ✓ ${sessionResult.username}, access confirmed`);
  console.log('   │  └─ ✓ clear, will reuse');
}
// This output is never shown because silent:true in orchestrator
```
