# blocker.1: findsert silently falls through to upsert when context is missing

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- src/access/daos/daoKeyrackHostManifest/index.ts:207-212

In `daoKeyrackHostManifest.set`, when `input.findsert` is specified but `context` is undefined, the function skips reading the extant manifest. `manifestFound` remains `null`, so the findsert check is bypassed, and the function proceeds to write a new manifest, effectively performing an upsert. This silently hides the programming error (caller forgot context) and risks data loss. The rule forbids failhide: real errors must be surfaced, not hidden.

**snippet**:
```ts
    let manifestFound: KeyrackHostManifest | null = null;
    if (input.findsert && existsSync(path) && context) {
      // for findsert, we need to read the extant manifest
      const result = await daoKeyrackHostManifest.get({ owner }, context);
      manifestFound = result?.manifest ?? null;
    }
```

# nitpick.1: assertion failure uses console.log/process.exit(2) instead of throwing a proper error class

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.require.failloud.md

**locations**:
- src/contract/cli/invokeKeyrack.ts:1363-1365
- src/contract/cli/invokeKeyrack.ts:1380-1384
- src/contract/cli/invokeKeyrack.ts:1406-1411
- src/contract/cli/invokeKeyrack.ts:1432-1437

In the 'list' and other commands, when a required condition fails (e.g., no host manifest found), the code prints a message to stdout and calls `process.exit(2)`. The rule requires using dedicated error classes (e.g., `ConstraintError`, `BadRequestError`) with full context for every error path. This pattern lacks the visibility and introspectability of thrown error objects.

**snippet**:
```ts
if (!context.hostManifest) {
  console.log('');
  console.log('✋ no host manifest found');
  console.log('');
  process.exit(2);
}
```
