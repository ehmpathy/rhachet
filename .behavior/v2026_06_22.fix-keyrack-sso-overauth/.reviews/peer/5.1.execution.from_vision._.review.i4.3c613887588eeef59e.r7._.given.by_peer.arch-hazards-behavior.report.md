# blocker.1: read-modify-write race condition on host manifest

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/domain.operations/keyrack/setKeyrackKeyHost.ts

The setKeyrackKeyHost function reads the host manifest from memory (context.hostManifest), modifies its hosts map, and writes the entire manifest back via daoKeyrackHostManifest.set. This is a non-atomic read-modify-write pattern: between reading hostManifest.hosts and writing the updated manifest, a concurrent call could have modified the manifest, causing the later write to silently overwrite those changes. This violates the rule against race conditions: 'does this write to shared state that others read concurrently? are there read-modify-write sequences without locks or transactions?' The host manifest file is a shared resource that multiple processes or async operations may access. No locking or atomic file operations are used.

**snippet**:

# nitpick.1: hidden side effect: get mutates context argument

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/access/daos/daoKeyrackHostManifest/index.ts

The daoKeyrackHostManifest.get function has a hidden side effect: it mutates the context object by adding a hostManifest property. The function is named 'get', suggesting it is read-only, but it also writes to an input argument. This violates the rule against hidden side effects: 'does the function do more than its name suggests? are there mutations to inputs or global state?' While this side effect is documented in a .note comment, the behavior is still unexpected for a getter and could lead to subtle bugs if callers are not aware.

**snippet**:
```ts
  // set hostManifest on context for caller access
  context.hostManifest = manifest;

  return { manifest };
}
```
