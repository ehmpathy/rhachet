# blocker.1: Race condition in daoKeyrackHostManifest.set due to read-modify-write without file locking

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/access/daos/daoKeyrackHostManifest/index.ts:91-126

The set method performs a read-modify-write sequence: it checks file existence via existsSync, optionally reads the current manifest via get(), then writes the updated manifest. If two processes call set simultaneously (e.g., concurrent `rhx keyrack set` commands or parallel test runs), they may read the same initial state and overwrite each other's updates. This can cause lost updates, duplicate records on retry, and inconsistent host manifest state. The method lacks file-level locking or atomic write operations to prevent this.

**snippet**:
```ts
    // check if manifest already exists (need context to decrypt for findsert check)
    let manifestFound: KeyrackHostManifest | null = null;
    if (input.findsert && existsSync(path) && context) {
      // for findsert, we need to read the extant manifest
      const result = await daoKeyrackHostManifest.get({ owner }, context);
      manifestFound = result?.manifest ?? null;
    }

    // handle findsert: return found if exists with same uri
    if (input.findsert && manifestFound) {
      if (manifestFound.uri === input.findsert.uri) return manifestFound;
      throw new BadRequestError(
        'can not findsert; manifest already exists with different uri',
        { uriFound: manifestFound.uri, uriDesired: input.findsert.uri },
      );
    }
```

# nitpick.1: Hidden side effect: daoKeyrackHostManifest.get mutates context

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/access/daos/daoKeyrackHostManifest/index.ts:92-94

The `get` function modifies `context.hostManifest` as a side effect, despite its name suggesting read-only behavior. While documented, this can surprise callers who pass the same context to subsequent operations, especially in concurrent or retry scenarios. Consider returning the manifest in the result and letting the caller decide if mutation is needed.

**snippet**:
```ts
    // set hostManifest on context for caller access
    context.hostManifest = manifest;

    return { manifest };
```
