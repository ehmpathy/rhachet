# nitpick.1: Fragile meta comparison in isKeyrackKeyHostAttrsEqual

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.verification/rule.require.test-coverage-by-grain.md

**locations**:
- src/domain.operations/keyrack/isKeyrackKeyHostAttrsEqual.ts

The `isKeyrackKeyHostAttrsEqual` uses `JSON.stringify()` to compare the `meta` fields of two hosts. This is fragile because object key order may differ, causing false negatives. Although JSON.stringify respects insertion order in modern V8, it is an implementation detail and could fail if objects are serialized differently (e.g., from different sources). A more robust approach would be to use a known-key comparison or `lodash.isequal` / a helper that compares keys and values independent of order.

Recommend using a utility that performs deep equality without relying on sort order, or explicitly comparing each known meta key.

**snippet**:
```ts
return (
  input.hostFound.mech === input.attrs.mech &&
  input.hostFound.vault === input.attrs.vault &&
  input.hostFound.exid === input.attrs.exid &&
  input.hostFound.env === input.attrs.env &&
  input.hostFound.org === input.attrs.org &&
  JSON.stringify(input.hostFound.meta) === JSON.stringify(input.attrs.meta) // fragile
);
```
