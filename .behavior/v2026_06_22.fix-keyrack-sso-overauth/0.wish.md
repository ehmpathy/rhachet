wish = fix keyrack SSO overauth: same domain + same user = reuse session

## .core problem

**multiple AWS profiles share the same SSO domain but keyrack triggers re-auth every time you switch profiles.**

example: `ehmpathy.demo.test`, `ehmpathy.demo.prep`, `ehmpathy.demo.prod` all use:
- same SSO domain: `https://d-906617ed8b.awsapps.com/start`
- same user: `demo-agent`

when you unlock test, then unlock prep:
- **current behavior**: triggers browser SSO re-auth (annoying, slow)
- **desired behavior**: reuses extant session (same domain, same user)

## .core solution

compare username before deciding to re-auth:

| same domain? | same username? | action |
|--------------|----------------|--------|
| yes | yes | **reuse session** (no re-auth) |
| yes | no | clear session, trigger re-auth |
| no | n/a | trigger fresh auth |

store `awsSsoUsername` in `meta` on `keyrack set`. on `keyrack unlock`, compare cached session username against stored username.

---

## .implementation

replace `KeyrackKeyHost.vaultRecipient` with generic `meta` field for vault-specific metadata.

- `os.secure`: `meta.ageKeyRecipient` (pubkey for age encryption)
- `aws.config`: `meta.awsSsoUsername` (username from sts get-caller-identity ARN)

this enables `keyrack unlock` to:
- detect if cached session belongs to expected user
- skip re-auth when same user
- clear session only when different user

## .why

`vaultRecipient` was os.secure-specific but never used. generic `meta` enables per-vault metadata without field proliferation.

```ts
// meta types per vault
interface KeyrackKeyHostMetaOsSecure {
  ageKeyRecipient: string; // pubkey for age encryption
}

interface KeyrackKeyHostMetaAwsConfig {
  awsSsoUsername: string; // ARN from sts get-caller-identity
}

// lookup type (KeyrackKeyHost.vault determines which meta shape)
type KeyrackKeyHostMetaOf<V extends KeyrackHostVault> =
  V extends 'os.secure' ? KeyrackKeyHostMetaOsSecure :
  V extends 'aws.config' ? KeyrackKeyHostMetaAwsConfig :
  null;

// KeyrackKeyHost stores generic meta at runtime
interface KeyrackKeyHost {
  vault: KeyrackHostVault;
  meta: Record<string, unknown> | null; // replaces vaultRecipient
}
```

## .data flow

vault.set() returns `{ mech, exid?, meta? }`. meta is derived internally by each vault:

- `os.secure`: derives `ageKeyRecipient` from context on set
- `aws.config`: derives `awsSsoUsername` from `sts get-caller-identity` after SSO auth

no CLI flag needed — vaults derive meta internally.

## .removal

drop `vaultRecipient` entirely:

- remove `KeyrackKeyHost.vaultRecipient` field
- remove `--vaultRecipient` CLI flag from `keyrack set`
- remove all propagation through `setKeyrackKey`, `setKeyrackKeyHost`
- update schema, tests, snapshots

## .behaviors

### on `keyrack set` (for `os.secure`)

return the recipient used for encryption:

1. vault.set() returns `{ mech, meta: { ageKeyRecipient: context.hostManifest.recipients[0] } }`
2. `setKeyrackKeyHost` persists `meta` to `KeyrackKeyHost`

no new derivation — just return what it already uses.

### on `keyrack set` (for `aws.config`)

after SSO auth completes:
1. call `sts get-caller-identity` to get ARN
2. return `{ mech, exid, meta: { awsSsoUsername: arn } }` from `vault.set()`
3. `setKeyrackKeyHost` persists `meta` to `KeyrackKeyHost`

### on `keyrack unlock` (for `aws.config`)

1. **if `meta.awsSsoUsername` is absent** → failfast: "key lacks awsSsoUsername; re-set the key"

2. **if cached token exists for domain**:
   - call `sts get-caller-identity` with cached token to get current ARN
   - if ARN matches `meta.awsSsoUsername` → reuse session, skip re-auth
   - if ARN differs → **log clearly**, then clear and trigger fresh auth

3. **if no cached token** → trigger fresh auth (normal flow)

### logout visibility

when a session is cleared due to recipient mismatch, log clearly:

```
clear SSO session: cached recipient differs from expected
  cached:   arn:aws:iam::123456789012:user/alice
  expected: arn:aws:iam::123456789012:user/bob
```

this makes it obvious why re-auth was triggered.

## .test coverage

**this is the most important part of the work.**

heavy test coverage and snapshot coverage required against every combinatorial boundary case. the snapshots enable visual verification of output in PRs and catch regressions.

### phase 1: happy and pause paths

| case | scenario | expected |
|------|----------|----------|
| happy | set → unlock → unlock | second unlock skips re-auth |
| pause | set without SSO | failfast: SSO auth required |
| pause | unlock with absent `awsSsoUsername` | failfast: re-set key |
| pause | unlock with mismatched recipient | clear logged, re-auth triggered |

### phase 2: exhaustive combinatorial

| cached token | awsSsoUsername | cached ARN vs expected | expected behavior |
|--------------|----------------|------------------------|-------------------|
| absent | absent | n/a | failfast: re-set key |
| absent | present | n/a | trigger fresh auth |
| present | absent | n/a | failfast: re-set key |
| present | present | match | reuse session, skip re-auth |
| present | present | mismatch | log clear, trigger re-auth |
| present | absent | n/a | failfast: re-set key |
| expired | present | match | trigger fresh auth |
| expired | present | mismatch | log clear, trigger re-auth |

### phase 3: multi-env sequences

| sequence | expected |
|----------|----------|
| test → test | no re-auth (same profile, same recipient) |
| test → prep | no re-auth (same profile, same recipient) |
| test → prod (same user) | no re-auth (different profile, same recipient) |
| test → prod (different user) | re-auth (different profile, different recipient) |
| prod → test (same user) | no re-auth |
| prod → test (different user) | re-auth |
