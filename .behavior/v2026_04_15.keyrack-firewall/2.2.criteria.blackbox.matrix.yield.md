## matrix 1: credential process

the core credential handle logic.

| ind: declared in keyrack.yml | ind: present in secrets | ind: JSON format | ind: mech field | ind: blocked pattern | dep: action result | dep: env export | dep: output |
|------------------------------|-------------------------|------------------|-----------------|----------------------|--------------------|-----------------|--------------
| no | yes | - | - | - | skip | none | silent |
| yes | no | - | - | - | continue | none | reports absent |
| yes | yes | malformed | - | - | fail | none | parse error + hint |
| yes | yes | valid | unknown mech | - | fail | none | unknown mech + supported list |
| yes | yes | valid | valid mech | - | translate | token | granted + mech + expiry |
| yes | yes | valid | absent | yes (ghp_*, AKIA*) | fail | none | blocked + reason + fix |
| yes | yes | valid | absent | no | passthrough | original | granted |
| yes | yes | n/a (plain string) | absent | yes (ghp_*, AKIA*) | fail | none | blocked + reason + fix |
| yes | yes | n/a (plain string) | absent | no | passthrough | original | granted |

**legend**:
- `-` = not applicable / don't care
- `valid mech` = EPHEMERAL_VIA_GITHUB_APP, EPHEMERAL_VIA_AWS_SSO, etc
- `blocked pattern` = ghp_*, gho_*, ghu_*, ghr_*, AKIA*

**note**: when mech field is valid, translation happens regardless of blocked pattern (the translated token is short-lived by design).

## matrix 2: fail-fast behavior

when multiple secrets are processed.

| ind: secret 1 result | ind: secret 2 result | dep: action result | dep: secret 1 exported | dep: secret 2 exported |
|----------------------|----------------------|--------------------|-----------------------|-----------------------|
| success | success | success | yes | yes |
| success | fail | fail | no | no |
| fail | success | fail | no | no |
| fail | fail | fail | no | no |

**principle**: partial success is forbidden. if any secret fails, no credentials are exported.

## gap analysis

### no gaps found

all combinations are covered by the blackbox criteria:

1. ✅ declared + present + valid mech → translate (usecase.1)
2. ✅ declared + present + absent mech + blocked → fail (usecase.3)
3. ✅ declared + present + absent mech + not blocked → passthrough (usecase.4)
4. ✅ declared + present + malformed JSON → fail (usecase.6)
5. ✅ declared + present + unknown mech → fail (usecase.6)
6. ✅ declared + absent → report absent, continue (usecase.9)
7. ✅ not declared + present → skip (usecase.2)
8. ✅ multiple secrets with mixed results → fail fast (usecase.7)
9. ✅ keyrack.yml absent → fail (usecase.8)
10. ✅ per-job isolation → each job needs own step (usecase.10)

## decomposition analysis

### dimensions: 5 independent variables for matrix 1

this is at the edge of maintainability. however, the variables are tightly coupled:

- `declared in keyrack.yml` + `present in secrets` = filter logic
- `JSON format` + `mech field` + `blocked pattern` = translation logic

these two groups are naturally sequential (filter first, then translate), so the matrix remains reasonable.

### recommendation: no decomposition needed

the credential process matrix has 5 dimensions but only 9 meaningful rows (not 32 = 2^5) because many combinations are invalid:
- if not declared → skip (no further dimensions matter)
- if not present → report absent (no further dimensions matter)
- if malformed JSON → fail (mech/blocked don't matter)
- if valid mech → translate (blocked pattern doesn't apply)

the dimensions collapse in practice, the matrix stays tractable.
