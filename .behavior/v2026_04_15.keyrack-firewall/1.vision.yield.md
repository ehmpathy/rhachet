# 1.vision.yield — keyrack firewall

## the outcome world

### before: the gap

today, keyrack works beautifully on dev machines:
- \`keyrack set\` stores a github app JSON blob in os.secure
- \`keyrack unlock\` translates JSON blob → \`ghs_*\` token via mechAdapterGithubApp
- \`keyrack get\` returns the short-lived token from daemon
- firewall blocks long-lived tokens (ghp_*, AKIA*) at get time

but in CI, the flow breaks:
\`\`\`
github secret: {"appId":"123","privateKey":"...","installationId":"456","mech":"EPHEMERAL_VIA_GITHUB_APP"}
    ↓
workflow env var: EHMPATH_BEAVER_GITHUB_TOKEN=\${{ secrets.EHMPATH_BEAVER_GITHUB_TOKEN }}
    ↓
jest setup: keyrack.source({ env: 'test', owner: 'ehmpath' })
    ↓
keyrack get --for repo → os.envvar vault → PERMANENT_VIA_REPLICA (passthrough)
    ↓
test code: keyrack.get({ for: { key: 'EHMPATH_BEAVER_GITHUB_TOKEN' } })
    ↓
returns: raw JSON blob (not translated!)
    ↓
test expects ghs_* token → fails
\`\`\`

the problem: in CI, env vars are read via os.envvar vault with PERMANENT_VIA_REPLICA mechanism. this mechanism is passthrough — it doesn't read the \`mech\` field from self-descriptive blobs, so no translation happens.

### after: the unified flow

\`\`\`
github secrets (all of them): { EHMPATH_BEAVER_GITHUB_TOKEN: json_blob, OTHER_SECRET: ..., ... }
    ↓
workflow env var: SECRETS=\${{ toJSON(secrets) }}
    ↓
workflow step: npx rhachet keyrack firewall --env test --from 'json(env://SECRETS)' --into github.actions
    ↓
CLI reads keyrack.yml → filters to only declared keys
    ↓
CLI parses blob → reads mech: EPHEMERAL_VIA_GITHUB_APP
    ↓
CLI runs mechAdapterGithubApp.deliverForGet() → ghs_* token
    ↓
CLI outputs:
  - ::add-mask::\$token — masks value in logs (GitHub Actions workflow command)
  - echo "\$key=\$token" >> \$GITHUB_ENV — exports for subsequent steps
    ↓
subsequent steps in same job have translated keys in env
    ↓
keyrack.source() → os.envvar → passthrough → ghs_* token
    ↓
test code works!
\`\`\`

## design: CLI over action

### why CLI over GitHub Action

**testability**. a CLI command can be tested via normal integration tests with mocked env vars and temp files. a GitHub Action requires either \`act\` (flaky) or real workflow runs (slow). by pushing all logic into the CLI, we get fast, reliable tests.

| approach | test method | reliability |
|----------|-------------|-------------|
| GitHub Action | \`act\` (local) | flaky, env differences |
| GitHub Action | real workflow runs | slow, 30s+ per iteration |
| CLI command | jest with temp files | fast, deterministic |

### the CLI design

\`\`\`
npx rhachet keyrack firewall --env test --from 'json(env://SECRETS)' --into github.actions
\`\`\`

**input mechanism**: \`--from 'json(env://SECRETS)'\`
- read JSON from env var named \`SECRETS\`
- extensible: \`--from 'json(stdin://)'\` for stdin input
- CLI filters to keys declared in \`keyrack.yml\`

**output mechanism**: \`--into github.actions\`
- outputs \`::add-mask::\$secret\` for each granted key (GitHub Actions runtime masking)
- appends \`\$key=\$secret\` to \`\$GITHUB_ENV\` file
- outputs \`::notice::\` for expiry times
- extensible: \`--into json\` for structured output (tests, programmatic use)

**critical**: add firewall step to EACH job that needs credentials. GitHub blocks masked values in job outputs, so credentials cannot flow between jobs.

**pros**:
- testable: CLI can be tested without GitHub Actions infrastructure
- ergonomic: no manual secret enumeration
- keyrack.yml is single source of truth for which keys matter
- same mechanism adapters as dev machine
- no action.yml to maintain
- no ncc bundle required

**cons**:
- passes ALL secrets to CLI (even ones not in keyrack.yml) — but CLI only processes keys in keyrack.yml
- requires rhachet to be installed (vs self-contained action)

## research findings

### 1. does \$GITHUB_ENV work for subsequent steps? ✅ confirmed

**answer**: yes. per [GitHub docs](https://docs.github.com/en/actions/reference/workflows-and-actions/workflow-commands):

> "you can make an environment variable available to any subsequent steps in a workflow job by defining or updating the environment variable and writing this to the GITHUB_ENV environment file"

the syntax is:
\`\`\`bash
echo "{name}={value}" >> \$GITHUB_ENV
\`\`\`

**citation**: [Store information in variables - GitHub Docs](https://docs.github.com/en/actions/learn-github-actions/variables)

### 2. is there a better way to pass secrets? ❌ no native alternative

**answer**: \`toJSON(secrets)\` is the standard approach. there is no built-in way to iterate secrets.

**decision**: implement as CLI command. no external deps. reuse extant mechanism adapters + add secrets parser + translation + export.

### 3. how does ::add-mask:: work?

**answer**: \`::add-mask::\` is a GitHub Actions workflow command. when a step outputs \`::add-mask::VALUE\`, GitHub Actions masks that VALUE in all subsequent log output for the job.

the syntax is:
\`\`\`bash
echo "::add-mask::\$SECRET_VALUE"
\`\`\`

this is the runtime equivalent of \`core.setSecret()\` from @actions/core, but works from any shell command — no Node.js action required.

**citation**: [Masking a value in a log - GitHub Docs](https://docs.github.com/en/actions/using-workflows/workflow-commands-for-github-actions#masking-a-value-in-a-log)

---

**key insight**: the firewall CLI:
1. accepts ALL secrets via \`--from 'json(env://SECRETS)'\` — ergonomic, no manual input list
2. filters to only keys declared in \`keyrack.yml\` — no wasted translation
3. exports to \`\$GITHUB_ENV\` — subsequent steps in same job have translated keys
4. masks all translated values via \`::add-mask::\` — no log leaks

**important**: firewall must be added as a step to EACH job that needs credentials. credentials cannot be passed between jobs via outputs (GitHub blocks masked values in outputs for security).

### the "aha" moment

the keyrack firewall CLI is **not** a new credential store. it's a **translator** that sits between github secrets and your workflow:

1. github secrets store the **source credentials** (JSON blobs with \`mech\` field)
2. the firewall CLI **translates** source → usable secret via mechanism adapters
3. subsequent steps see **translated credentials** in their env

same mechanism adapters. same translation logic. just run at workflow start instead of daemon unlock.

## user experience

### usecase 1: repo maintainer sets up CI (single job)

**goal**: enable CI to use github app credentials for tests

**before (manual, error-prone)**:
\`\`\`yaml
# .github/workflows/test.yml
jobs:
  test:
    steps:
      - run: npm ci
      - run: |
          # manually generate token from github app creds
          TOKEN=\$(npm run --silent generate-github-token)
          echo "GITHUB_TOKEN=\$TOKEN" >> \$GITHUB_ENV
      - run: npm test
\`\`\`

**after (declarative, consistent)**:
\`\`\`yaml
# .github/workflows/test.yml
jobs:
  test:
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci

      # keyrack firewall: translate and validate secrets
      - run: npx rhachet keyrack firewall --env test --from 'json(env://SECRETS)' --into github.actions
        env:
          SECRETS: \${{ toJSON(secrets) }}

      - run: npm test
        # EHMPATH_BEAVER_GITHUB_TOKEN is already a usable ghs_* token
\`\`\`

### usecase 2: developer adds a new credential

**goal**: add AWS credentials to CI tests

**timeline**:
1. developer stores credentials locally: \`rhx keyrack set --key AWS_CREDS --vault os.secure --mech EPHEMERAL_VIA_AWS_SSO\`
2. developer creates github secret with JSON blob that includes \`mech\` field
3. developer adds \`AWS_CREDS\` to \`keyrack.yml\` — firewall will now filter for it
4. CI automatically translates JSON blob → AWS session credentials

**no workflow changes needed** — the \`toJSON(secrets)\` pattern passes all secrets, firewall filters via keyrack.yml.

### usecase 3: debug a credential failure

**goal**: understand why a credential fails in CI

**workflow**:
\`\`\`yaml
- run: npx rhachet keyrack firewall --env test --from 'json(env://SECRETS)' --into github.actions
  env:
    SECRETS: \${{ toJSON(secrets) }}
\`\`\`

**outputs (on success)**:
\`\`\`
🔐 keyrack firewall
   ├─ env: test
   └─ grants
      └─ GITHUB_TOKEN
         ├─ mech: EPHEMERAL_VIA_GITHUB_APP
         ├─ translated: ghs_*** (1 hour expiry)
         └─ status: granted 🔑
\`\`\`

**outputs (on failure)**:
\`\`\`
🔐 keyrack firewall
   └─ GITHUB_TOKEN
      ├─ status: blocked 🚫
      │  ├─ detected github classic pat (ghp_*)
      │  └─ its dangerous to use long lived tokens in CI
      └─ fix: use a github app instead of a classic PAT
\`\`\`

## evaluation

### how well does it solve the goals?

| goal | solved? | how |
|------|---------|-----|
| translate JSON blobs to tokens in CI | yes | CLI runs mechanism adapters |
| block dangerous credentials in CI | yes | firewall validation in CLI |
| same logic in dev and CI | yes | reuses mechanism adapters |
| no host manifest needed in CI | yes | blob self-describes via \`mech\` field |
| testable | yes | CLI can be tested without GitHub Actions |

### pros

1. **testable** — CLI can be tested via normal integration tests
2. **unified mechanism adapters** — same code runs locally and in CI
3. **self-descriptive blobs** — no host manifest lookup needed in CI
4. **firewall built-in** — blocks ghp_*, AKIA* before they reach workflow
5. **transparent** — CLI logs show what was translated
6. **minimal workflow changes** — one step, declarative inputs
7. **no action.yml** — no ncc bundle, no action maintenance
8. **extensible** — \`--from\` and \`--into\` flags support multiple formats

### cons

1. **CLI must be added to every workflow** — not automatic
2. **secrets must contain \`mech\` field** — requires migration for extant secrets
3. **token refresh not possible** — tokens expire, job must finish before expiry
4. **requires rhachet installation** — npm ci must complete before firewall step

### edge cases and pit of success

| edge case | behavior |
|-----------|----------|
| secret lacks \`mech\` field | CLI assumes PERMANENT_VIA_REPLICA (passthrough) + firewall validation |
| secret contains ghp_* token | firewall blocks with clear error message |
| secret JSON is malformed | CLI fails fast with parse error |
| secret mech is unknown | CLI fails fast: "unknown mechanism: X" |
| token generation fails (rate limit, etc) | CLI fails fast with github API error |
| multiple secrets, one fails | CLI fails fast, reports which one failed |
| secret not in manifest | skipped silently |
| multiline secret value (PEM key) | handled via null-delimited parse |

## open questions & assumptions

### assumptions we've made

1. **self-descriptive blobs are the path forward** — the \`.todo.self-descriptive-mech-blobs.md\` spec is the right approach
2. **CLI is the right delivery mechanism** — maximizes testability over action convenience
3. **firewall validation should run at translate time** — vs at usage time
4. **single CLI handles all mechanisms** — vs one command per mechanism type

### questions to validate with wisher

1. **backwards compatibility** [wisher]: should we support secrets without \`mech\` field? (assume REPLICA + firewall)

### questions answered via logic

1. **error behavior** [answered]: fail-fast on first error. rationale: security operations should fail early. if one secret is blocked, stop immediately rather than expose others.

2. **OIDC integration** [answered]: leave to aws-actions/configure-aws-credentials. rationale: OIDC is a distinct mechanism with well-supported extant actions. we should focus on self-descriptive blob translation, not reinvent OIDC.

3. **CLI vs action** [answered]: CLI. rationale: testability. a CLI command can be tested via normal integration tests. a GitHub Action requires \`act\` (flaky) or real workflow runs (slow).

4. **multi-secret input format** [answered]: use \`--from 'json(env://SECRETS)'\` with \`SECRETS: \${{ toJSON(secrets) }}\`. rationale: pass all secrets as a JSON object, CLI filters via keyrack.yml. handles multiline JSON because GitHub's \`toJSON()\` properly escapes.

### questions answered via additional analysis

5. **ghs_* token classification** [answered]: \`ghs_*\` tokens are GitHub App installation access tokens, which are inherently short-lived (1 hour max, enforced by GitHub's API). the firewall comment incorrectly labeled them as "long-lived". fix required: remove \`ghs_*\` from LONG_LIVED_PATTERNS in \`mechAdapterReplica.ts\`. this is safe because 1-hour tokens have limited blast radius and align with EPHEMERAL mechanism goals.

## what is awkward?

### 1. two places to configure credentials

dev machine has \`keyrack.host.yml\` with vault + mech per key. CI has secrets with \`mech\` embedded in the JSON blob. this is intentional (no host manifest in CI), but means:
- adding a new credential requires updating both host manifest (for dev) AND github secret (for CI)
- the \`mech\` in host manifest could theoretically differ from \`mech\` in blob

**mitigation**: validation that host manifest mech matches blob mech when both are present.

### 2. token expiry in long-running jobs

github app tokens expire in 1 hour. if a CI job runs longer than that, the token becomes invalid mid-job.

**mitigation options**:
- warn if job is likely to exceed 55 minutes
- support token refresh mid-job (requires more complex logic)
- document the limitation clearly

**current stance**: document limitation, most CI jobs are < 1 hour.

### 3. requires rhachet installation

unlike a self-contained GitHub Action, the CLI requires \`npm ci\` to complete before the firewall step can run. this means:
- node_modules must be available
- slightly more workflow boilerplate (setup-node + npm ci before firewall)

**mitigation**: this is already the common pattern. most workflows run \`npm ci\` early anyway.

### 4. the name "firewall" might confuse

"firewall" typically means blocking. this CLI also translates. some users might not understand why a "firewall" is needed when they just want credentials.

**decision**: product name is "firewall" (the keyrack firewall). the name emphasizes security — the CLI blocks dangerous credentials first, translates second.
