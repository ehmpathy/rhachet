# vision: keyrack --prikey on set operations

## the outcome world

### what does a day-in-the-life look like with this in place?

alice is setting up credentials for a shared robot account on a build server. the robot has its own ssh keypair stored at `/opt/robot/.ssh/id_ed25519`. she initializes a keyrack owner for the robot, then sets api keys to it — all in one flow:

```sh
# initialize the robot's keyrack owner with its custom ssh key
rhx keyrack init --owner robot --prikey /opt/robot/.ssh/id_ed25519

# set a secret to the robot's vault, specifying the private key for decryption
echo "sk-prod-abc123" | rhx keyrack set --key OPENAI_API_KEY --owner robot --prikey /opt/robot/.ssh/id_ed25519

# later, the robot unlocks and uses its secrets
rhx keyrack unlock --owner robot --prikey /opt/robot/.ssh/id_ed25519
```

she didn't need to manually unlock first. she didn't need to copy keys around. the `--prikey` flag let her directly decrypt and update the robot's host manifest.

### what's the before/after contrast?

**before (today's defect):**
```sh
# alice sets up robot's keyrack
rhx keyrack init --owner robot --prikey /opt/robot/.ssh/id_ed25519

# alice tries to set a key... but can't specify the private key
echo "sk-prod-abc123" | rhx keyrack set --key OPENAI_API_KEY --owner robot
# ERROR: could not decrypt host manifest — no matching ssh key found

# workaround: manually unlock first, hope session persists
rhx keyrack unlock --owner robot --prikey /opt/robot/.ssh/id_ed25519
# then quickly try set again... but unlock doesn't persist across commands
echo "sk-prod-abc123" | rhx keyrack set --key OPENAI_API_KEY --owner robot
# still fails — each command needs to decrypt independently
```

**after (with this feature):**
```sh
# works in a single atomic command
echo "sk-prod-abc123" | rhx keyrack set --key OPENAI_API_KEY --owner robot --prikey /opt/robot/.ssh/id_ed25519
# decrypted manifest with provided key, stored secret, re-encrypted manifest
```

### what's the "aha" moment where the value clicks?

> "oh — `--prikey` isn't just for unlock. it's for any operation that needs to read my encrypted manifest. of course set needs it too, it has to decrypt before it can update!"

the mental model crystallizes: keyrack operations that touch the encrypted host manifest need a way to decrypt it. currently only `unlock` lets you specify how. this feature makes that capability symmetric.

## user experience

### what usecases do folks fulfill? what goals?

| usecase | goal |
|---------|------|
| robot credentials | set api keys for service accounts with dedicated ssh keypairs |
| team member isolation | each developer uses `--owner $USER` with their own ssh key |
| ci/cd provisioning | commands set credentials using explicit key paths (no ssh-agent) |
| multi-tenant machines | multiple owners share a host, each with isolated vaults |
| recovery workflows | re-set a key after rotating secrets, using backup ssh key |

### what contract inputs & outputs do they leverage?

**inputs:**
```sh
rhx keyrack set \
  --key <KEY_NAME>           # required: the key identifier
  --owner <OWNER_ID>         # optional: owner namespace (default: null)
  --prikey <PATH>            # NEW: explicit ssh private key for decryption
  --vault <VAULT_TYPE>       # optional: os.direct, os.secure, os.daemon, etc
  --mech <MECHANISM>         # optional: mechanism constraint
  # ... other extant flags
```

**outputs:**
- key stored in vault at `~/.rhachet/keyrack/vault/{vault_type}/owner={owner}/{key}`
- host manifest updated and re-encrypted
- repo manifest (`.agent/keyrack.yml`) updated for non-sudo keys

### what would it look like to leverage them?

**scenario 1: robot account setup**
```sh
# generate dedicated keypair for robot
ssh-keygen -t ed25519 -f /opt/robot/.ssh/id_ed25519 -N ""

# initialize robot's keyrack
rhx keyrack init --owner robot --prikey /opt/robot/.ssh/id_ed25519

# set multiple secrets
echo "sk-prod-abc" | rhx keyrack set --key OPENAI_API_KEY --owner robot --prikey /opt/robot/.ssh/id_ed25519
echo "ghp_xyz123" | rhx keyrack set --key GITHUB_TOKEN --owner robot --prikey /opt/robot/.ssh/id_ed25519
```

**scenario 2: developer onboarding**
```sh
# developer uses their non-standard key location
rhx keyrack init --owner vlad --prikey ~/.ssh/vlad_work_key

# set their personal api keys
echo "my-secret" | rhx keyrack set --key MY_API_KEY --owner vlad --prikey ~/.ssh/vlad_work_key
```

**scenario 3: ci command without ssh-agent**
```sh
#!/bin/bash
# CI environment: ssh-agent not running, key at known path
PRIKEY="/var/secrets/deploy-key"

# set rotated secret
echo "$NEW_SECRET" | rhx keyrack set \
  --key DEPLOY_SECRET \
  --owner ci \
  --prikey "$PRIKEY"
```

### what timelines do they go through?

1. **setup (once)**: `keyrack init --owner X --prikey /path/to/key`
2. **populate (few times)**: `keyrack set --key Y --owner X --prikey /path/to/key`
3. **use (many times)**: `keyrack unlock --owner X --prikey /path/to/key` then `keyrack get --key Y`

note: `--prikey` is universal across `init`, `set`, and `unlock` — same flag, same path, every time. `get` reads from the daemon (already unlocked state) and doesn't need `--prikey`.

## mental model

### how would users describe this to a friend?

> "keyrack encrypts your secrets with your ssh key. when you want to unlock or add secrets, you tell it which private key to use for decryption. the `--prikey` flag works on `unlock` and `set`."

### what analogies or metaphors fit?

**the lockbox analogy:**
- your keyrack is a lockbox
- your ssh public key is the lock
- your ssh private key is the key to open it
- `--prikey` tells keyrack "here's my key, open the lockbox so i can add something"

**the gpg analogy:**
- similar to `gpg --decrypt --recipient` needing the private key
- `--prikey` is like specifying `--secret-keyring` in gpg

### what terms would they use vs what terms would we use?

| user term | our term |
|-----------|----------|
| "my key" | ssh private key, prikey |
| "unlock key" | age identity (derived from ssh key) |
| "my vault" | owner-namespaced keyrack |
| "encrypted config" | host manifest |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| set keys to custom owners | yes | `--prikey` enables decryption |
| robot/service accounts | yes | explicit key path, no ssh-agent needed |
| ci/cd environments | yes | no interactive prompts, command-based |
| symmetry with unlock | yes | same flag, same behavior |
| discoverability | yes | users who know `unlock --prikey` will expect it on `set` |

### what are the pros? the cons?

**pros:**
- unblocks the primary defect: can't set keys to custom owners
- consistent ux: `--prikey` works the same way across commands
- command-based: explicit paths work in non-interactive environments
- no new concepts: reuses extant `--prikey` pattern from unlock

**cons:**
- slightly longer command when needed (but only when discovery fails)
- no real cons: users use the same `--prikey` path everywhere, and the default case still works without `--prikey` via discovery

### what edgecases exist and how do our contracts keep users in a pit of success?

| edgecase | pit of success |
|----------|----------------|
| wrong prikey provided | fail-fast with clear error: "could not decrypt manifest with provided key" |
| prikey doesn't exist | fail-fast: "file not found: {path}" |
| prikey is pubkey by mistake | fail-fast: "expected private key, got public key" |
| prikey without owner | works fine: decrypts default (null) owner's manifest |
| prikey permissions wrong | fail-fast: "permission denied reading key" |

### anything awkward?

**potential confusion: when is `--prikey` needed?**
- answer: when ssh-agent doesn't have the key and standard paths don't match
- mitigation: error message should suggest `--prikey` when decryption fails

**`--pubkey` still works on init**
- `init --pubkey` remains supported (backwards compat)
- `init --prikey` is the new preferred way (derives pubkey automatically)
- both accept private key paths; `--prikey` is just clearer naming

---

## summary

the vision is simple: `--prikey` becomes a universal flag across `init`, `set`, and `unlock`. same flag, same path, every operation that touches the encrypted host manifest. users learn one pattern and it works everywhere.

**scope:**
- add `--prikey` to `keyrack init` (derives pubkey from private key)
- add `--prikey` to `keyrack set` (enables decryption for custom owners)

this unblocks the critical workflow: set credentials to custom-owner vaults encrypted with non-standard ssh keys.
