# blueprint: keyrack --prikey on init and set

## overview

add `--prikey` flag to `keyrack init` and `keyrack set` commands, enabling explicit ssh private key specification for manifest encryption/decryption. this mirrors the pattern already used in `keyrack unlock`.

## implementation details

### 1. invokeKeyrack.ts: init command

```typescript
// add option after --pubkey (line ~53)
.option('--prikey <path>', 'ssh private key path (derives pubkey automatically)')

// update opts type
async (opts: {
  owner?: string;
  for?: string;
  pubkey?: string;
  prikey?: string;  // add
  label?: string;
  // ...
}) => {
  // --prikey takes precedence over --pubkey (both accept private key paths)
  const keyPath = opts.prikey ?? opts.pubkey;

  const result = await initKeyrack({
    owner,
    pubkey: keyPath,  // initKeyrack already handles private key paths
    label: opts.label,
    // ...
  });
}
```

### test fixture requirements

the tests will use the `with-vault-os-direct` fixture, which includes:
- a test ssh keypair at a known location
- keyrack.yml with org configured

for the custom prikey path tests, the test will:
1. generate a fresh keypair in the temp repo
2. use that keypair path for all operations
3. verify the full flow works with non-default key locations

## risks and mitigations

| risk | mitigation |
|------|------------|
| breaking change to extant flows | `--prikey` is additive; all paths still work without it |
| inconsistent behavior init vs set | both now accept `--prikey` with identical semantics |
| test isolation | each test uses fresh temp repo with isolated keypair |
| backwards compat | `--pubkey` on init remains supported |

## scope boundaries

**in scope:**
- add `--prikey` to `init` command
- add `--prikey` to `set` command

- acceptance tests for full roundtrip

**out of scope:**
- changes to `unlock` (already has `--prikey`)
- changes to `get` (reads from daemon, doesn't need key)
- changes to DAO layer (already supports `prikey`)
