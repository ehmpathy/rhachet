# vision: fix keyrack sudo key get after unlock

## the outcome world

### day-in-the-life

user sets up a sudo credential:
```bash
rhx keyrack set --key GITHUB_TOKEN --env sudo --vault 1password
# enters 1password uri, key is stored in encrypted host manifest
```

user unlocks for a session:
```bash
rhx keyrack unlock --env sudo --key GITHUB_TOKEN
# daemon spawns, key is unlocked for 30 minutes
```

user retrieves the key:
```bash
rhx keyrack get --env sudo --key GITHUB_TOKEN
# shows "granted" with the secret value
```

all parts work. no surprises.

### before/after contrast

| before | after |
|--------|-------|
| unlock shows key but with `-270m` expiration | unlock shows key with correct `30m` expiration |
| status shows "no keys unlocked" | status shows the unlocked key |
| get returns "absent" | get returns "granted" |
| user is confused | user is confident |

### aha moment

when the user runs `keyrack get` after `keyrack unlock` and sees the key immediately available, the value clicks: the keyrack daemon does its job as a session cache.

## user experience

### usecases

1. **set + unlock + get roundtrip** — most common flow for developers who need sudo credentials
2. **unlock multiple sudo keys** — users with multiple sudo credentials unlock them one by one
3. **relock sudo keys** — user ends session early via `keyrack relock --env sudo`
4. **status check** — user verifies which sudo keys are currently unlocked

### contract inputs/outputs

**unlock:**
- input: `--env sudo --key <name>`
- output: shows key slug, env, org, vault, and expiration time

**get:**
- input: `--env sudo --key <name>`
- output: shows "granted" with key details, or "locked" with unlock hint, or "absent" with set hint

### timelines

```
t0: keyrack set --key TOKEN --env sudo --vault 1password
    → stores pointer in encrypted host manifest

t1: keyrack unlock --env sudo --key TOKEN
    → spawns daemon (if not active)
    → decrypts manifest, fetches secret from 1password
    → stores grant in daemon memory with 30m TTL

t2: keyrack get --env sudo --key TOKEN
    → queries daemon for the key
    → returns "granted" with secret

t3: (30m later)
    → TTL expires, daemon purges key

t4: keyrack get --env sudo --key TOKEN
    → returns "locked" (key exists but not unlocked)
```

## mental model

### how users would describe this

> "keyrack is like a secure clipboard for credentials. you unlock the keys you need for your session, and they're available for 30 minutes. after that, you unlock again."

### analogies

- **ssh-agent**: you unlock your SSH key once, then it's available for subsequent connections
- **macOS keychain**: unlock at login, credentials available for the session
- **1password app**: unlock with biometrics, items available until lock

### terms

| user term | system term |
|-----------|-------------|
| "my credentials" | sudo keys in host manifest |
| "unlock" | populate daemon with grant |
| "session" | daemon lifetime with TTL |
| "locked" | key exists but not in daemon |
| "available" | key granted from daemon |

## evaluation

### how well does it solve the goals

the goal is simple: after unlock, get should return the key.

current state: this **works** in the test suite with `os.direct` vault. the tests pass.

the user's issue appears to be either:
1. ~~**stale daemon bytecode**: daemon was active with old code that had a bug~~ [disproven] — daemon was freshly spawned
2. **1password-specific edge case**: the 1password adapter or TTL calculation has a defect
3. **environmental**: system clock offset, or daemon startup time issue
4. **IPC type mismatch** [new hypothesis]: expiresAt may lose type in daemon communication (string vs number)

### pros

- simple mental model: unlock → get → use
- explicit TTL enforcement (30m for sudo)
- daemon is ephemeral (memory-only, no persistence)

### cons

- daemon bytecode persistence: if user upgrades rhachet but daemon is still active, old bytecode is used
- no diagnostic output when defects occur
- `-270m` expiration is unclear (why negative?)

### edgecases and pit of success

| edgecase | current behavior | ideal behavior |
|----------|------------------|----------------|
| daemon has stale bytecode | silent failure | `daemon prune` kills and restarts |
| system clock is wrong | negative TTL | log warn, fail fast |
| 1password auth expires mid-unlock | silent failure | clear error message |
| key set with org A, get from repo with org B | mismatch | clear error about org mismatch |

## what is awkward

### feels off

1. **the -270m is displayed but unexplained**: user sees a negative number with no hint about what went wrong. [same as Q1] — trace TTL calculation.
2. **"absent" is inaccurate**: the key exists in host manifest but daemon reports it as absent because TTL expired immediately. [research] — why does status say "no keys unlocked" instead of "key expired"?
3. ~~**no daemon version check**: if bytecode is stale, unlock silently fails.~~ [moot] — daemon was fresh, so version check would not have helped here.

### design fights mental model

- user expects unlock to "just work"
- user doesn't know about daemon bytecode persistence
- user can't diagnose why get returns absent after unlock succeeded

### uncomfortable tradeoffs

- **daemon is in-memory only**: fast and secure, but loses keys on restart
- **no daemon restart on upgrade**: would be disruptive mid-session
- **sudo keys require explicit --key flag**: inconvenient but intentional (limit blast radius)

## proposed tests to add

based on the spec briefs for keyrack, we need to ensure:

### integration tests

1. **env=all fallback for sudo**: ensure `unlock --env sudo --key X` can find `org.all.X` if `org.sudo.X` doesn't exist
2. **TTL calculation sanity**: ensure expiresAt is always in the future
3. **1password roundtrip** (integration): set → unlock → get with mocked `op` CLI

### acceptance tests

1. **daemon restart mid-session**: kill daemon, run get, should show locked (not error)
2. **concurrent unlock**: two unlocks in parallel should both succeed
3. **negative TTL detection**: if expiresAt is in the past, fail fast with clear error
