# vision: boot.yml for repo=.this/role=any

## the outcome world

### before

every brief in `.agent/repo=.this/role=any/briefs/` gets booted as `.say` — full inline content.

current state:
```
root briefs:     ~39k chars (11 files)
domain.thought/: ~35k chars (6 files)
infra.composition/: ~5k chars (2 files)
────────────────────────────────
total:           ~79k chars (~20k tokens)
```

every session pays this token cost regardless of task relevance.

### after

a `boot.yml` controls which briefs are said vs reffed:

```yaml
briefs:
  say:
    - define.rhachet.v3.md          # core identity, always needed
    - define.agent-dir.md           # explains .agent/ structure
```

unmatched briefs become refs automatically — no need for explicit ref globs.

### the aha moment

session boot reports:
```
.agent/repo=.this/role=any
  ├─ briefs
  │  ├─ say = 2 (~10k chars)
  │  └─ ref = 17
  └─ skills = 6
```

tokens dropped from ~20k to ~3k. refs are there when needed but don't cost tokens until accessed.

## mental model

### how users describe this

> "boot.yml controls what goes into context by default vs what's just referenced"

> "say = inline content, ref = pointer. like the difference between inlining a module vs importing it"

### analogies

**table of contents vs full chapters**
- say = chapters printed inline
- ref = "see chapter 5" pointers

**http HEAD vs GET**
- ref = HEAD (know it exists, no body)
- say = GET (full content)

**git sparse checkout**
- say = checked out files
- ref = known paths, not materialized

### terms

| user term | internal term | meaning |
|-----------|---------------|---------|
| "full brief" | say | content inline in boot |
| "brief ref" | ref | pointer only, read on demand |
| "boot config" | boot.yml | curation manifest |

## evaluation

### pros

1. **token efficiency** — only load what's frequently needed
2. **discoverability preserved** — refs still appear, just not inline
3. **explicit curation** — author declares importance hierarchy
4. **no code change** — boot.yml is pure config

### cons

1. **maintenance burden** — must update boot.yml when adding briefs
2. **judgment calls** — which briefs deserve say vs ref?
3. **ref friction** — mechanic must explicitly read ref content

### edgecases

| case | handling |
|------|----------|
| new brief added | defaults to ref (unmatched by say globs) |
| glob mismatch | brief appears as ref |
| empty say array | ref all — minimal boot |
| no boot.yml | say all — current behavior preserved |

pit of success: if you don't configure, you get current behavior (say all).

## what is awkward

### judgment about say vs ref

who decides which briefs deserve full inline? current heuristic:
- identity briefs (define.rhachet, define.agent-dir) = say
- howto briefs for active tasks = say
- deep domain theory = ref
- philosophy/history = ref

but this is subjective. the wisher should review.

### maintenance coupling

adding a brief requires updating boot.yml if you want it said. if you forget, it defaults to ref (unmatched by say globs). this is intentional — new briefs don't inflate context until explicitly curated.

### no runtime validation

if boot.yml references a glob that matches no files, no error — just empty set. could be confusing if typo in glob.
