## user experience

### usecases

| goal | contract |
|------|----------|
| discover available skills | `rhx list` |
| search by substring | `rhx list radio` (contains "radio") |
| search by glob | `rhx list 'radio.*'` (glob pattern) |
| filter by repo | `rhx list --repo ehmpathy` |
| filter by role | `rhx list --role mechanic` |
| show all skills | `rhx list --all` |
| get quick help | `rhx --help`, `rhx -h`, or `rhx help` |
| get skill help | `rhx [skill] --help` (passthrough, already works) |

### search behavior

| input | has glob chars (`*?[]`) | behavior |
|-------|-------------------------|----------|
| `rhx list radio` | no | contains "radio" |
| `rhx list 'radio.*'` | yes | glob: matches pattern |
| `rhx list '*commit*'` | yes | glob: matches pattern |
| `rhx list git.release` | no | contains "git.release" |

**note**: glob patterns require quotes to prevent shell expansion.

### timeline

1. user installs rhachet
2. user runs `rhachet roles link` to link roles
3. user runs `rhx list` to see available skills
4. user runs `rhx [skill]` to execute

## evaluation

### pros

- **discoverability**: users find skills without browsing filesystems
- **consistency**: matches familiar cli patterns (--help, list)
- **simplicity**: no new concepts, just surfaces extant structure

### cons

- **output size**: many skills → long output. mitigated by repo/role filters and `... (N more)` truncation

### edgecases

| case | handling |
|------|----------|
| no roles linked | `rhx list` shows: "no skills found. run `rhachet roles link` first" |
| `rhx help [skill]` | not supported (use `rhx [skill] --help` instead) |
| `rhx --help [garbage]` | `--help` takes precedence, shows help |
| `rhx list --help` | shows list-specific help |

### test criteria

#### rhx --help

| given | when | then |
|-------|------|------|
| rhachet installed | `rhx --help` | shows commands treestruct with ref: $readme |
| rhachet installed | `rhx -h` | same as `rhx --help` |
| rhachet installed | `rhx help` | same as `rhx --help` |
| rhachet installed | `rhx --help garbage` | `--help` takes precedence, shows help |

#### rhx list

| given | when | then |
|-------|------|------|
| roles linked with 42 skills | `rhx list` | shows "42 skills found" + treestruct by repo > role > skill |
| no roles linked | `rhx list` | shows "0 skills found" + hint to run `rhachet roles link` |
| roles linked | `rhx list --help` | shows list-specific help with options |

#### rhx list [pattern]

| given | when | then |
|-------|------|------|
| 4 skills contain "radio" | `rhx list radio` | shows "4 skills found" + header `rhx list '*radio*'` |
| 4 skills match git.commit.* | `rhx list 'git.commit.*'` | shows "4 skills found" + header `rhx list git.commit.*` |
| no skills match | `rhx list nonexistent` | shows "0 skills found" |
| pattern has `*?[]` | `rhx list 'foo*'` | treated as glob |
| pattern has no `*?[]` | `rhx list foo` | treated as contains (rendered as `'*foo*'`) |

#### rhx list --repo / --role

| given | when | then |
|-------|------|------|
| skills in repo ehmpathy | `rhx list --repo ehmpathy` | shows only ehmpathy skills |
| skills in role mechanic | `rhx list --role mechanic` | shows only mechanic skills |
| combined filters | `rhx list radio --repo bhuild` | shows radio skills in bhuild only |

#### rhx list --all

| given | when | then |
|-------|------|------|
| 30 skills in one role | `rhx list` | shows 10 + "... (20 more, use --all to see all)" |
| 30 skills in one role | `rhx list --all` | shows all 30, no truncation |

## what is awkward?

### `help` vs `--help` vs `-h`

all three should work as synonyms:
- `rhx help` = subcommand form
- `rhx --help` = long flag form (Unix standard)
- `rhx -h` = short flag form (Unix standard)

the flag forms require intercept before skill dispatch. current `bin/rhx` does not check for flags — it passes all args to `run --skill`.

**resolution**: intercept `--help`, `-h`, and `help` in `bin/rhx` before dispatch.

### reserved command names

`list`, `help`, `keyrack`, `upgrade`, `enroll` are now reserved. if a skill has these names, it's shadowed.

**resolution**: document that these are reserved. skills should use namespaced names (e.g., `my.list` not `list`).

### truncation UX

"... (N more)" is simple but loses information. user must filter to see all.

**resolution**: acceptable tradeoff. filtering via `--repo`/`--role` is explicit.
