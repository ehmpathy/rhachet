│  │  ### 2. `getSkillCountLine` implicitly encodes structure knowledge — behavior hazard 🔴
│  │  
│  │  **file:** `src/domain.operations/invoke/getSkillListTreestruct/getSkillCountLine.ts`
│  │  
│  │  ```ts
│  │  export const getSkillCountLine = (input: { count: number }): string => {
│  │    if (input.count === 0) return '   └─ 0 skills found'; // last child
│  │    if (input.count === 1) return '   ├─ 1 skill found';  // has siblings
│  │    return `   ├─ ${input.count} skills found`;            // has siblings
│  │  };
│  │  ```
│  │  
│  │  the choice of `├─` vs `└─` depends on whether repos will follow in the output — that is a structural fact the orchestrator knows but the transformer must silently assume. when count > 0, repos follow, so `├─`. when count = 0, only the hint follows (a child, not sibling), so `└─`.
│  │  
│  │  this coupling is invisible in the function signature. if the orchestrator ever reorders sections (e.g. adds a header between count and repos, or removes the hint line), the prefix breaks silently.
│  │  
│  │  **fix:** the orchestrator (`getSkillListTreestruct`) should pass `isLast: boolean` to `getSkillCountLine`. the transformer formats; the orchestrator decides structure.

│  │  ### 3. `getRawArgsAfterRun` searches process.argv for literal `'run'` — behavior hazard 🔴
│  │  
│  │  **file:** `src/contract/cli/invokeRun.ts:21-26`
│  │  
│  │  ```ts
│  │  const getRawArgsAfterRun = (): string[] => {
│  │    const argv = process.argv;
│  │    const runIdx = argv.indexOf('run');
│  │    if (runIdx === -1) return [];
│  │    return argv.slice(runIdx + 1);

│  │  
│  │  **hazards:**
│  │  - if the cwd path or any earlier argv contains the word `'run'` (e.g. `/home/runner/run-scripts/...`), the slice point is wrong

│  │  
│  │  **fix:** extract raw passthrough args from commander's parsed `args` on the `run` command action (or use `process.argv.slice(process.argv.indexOf('--skill') + 2)` as a narrower heuristic, though injecting from commander is cleaner).
