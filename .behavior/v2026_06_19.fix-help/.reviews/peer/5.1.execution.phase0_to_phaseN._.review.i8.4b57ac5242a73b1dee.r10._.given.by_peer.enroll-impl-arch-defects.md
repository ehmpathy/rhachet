│  │  ## architectural review: `vlad/fix-help`
│  │  
│  │  ### 1. scope leak — `list` nested under `run` [blocker]
│  │  
│  │  `invokeRunList` is registered as a subcommand of `run`, meaning the CLI path is `rhachet run list`. But `list` is a peer-level discovery concern, not a sub-operation of execution. `run` = execute a skill. `list` = discover available skills. These are orthogonal.
│  │  
│  │  **consequences:**
│  │  - `invokeRunList` receives a `runCommand: Command` — it is bound to `run`, not to `program`. If it ever needs to be top-level, it requires refactoring its signature.
│  │  - `bin/rhx` must know the internal `run list` path and special-case it: `exec "$SCRIPT_DIR/run" run list "$@"`. This is a scope coupling — the dispatcher knows the internal routing of `run`'s subcommands.
│  │  - The `emitHelpOutput` inside `invokeRun` lists `list` as a sibling of skills, contradicting its structural placement as a child of `run`.
│  │  
│  │  **fix direction:** register `invokeRunList` against `program` (as a top-level command), the same way `invokeRun` is registered.
│  │  
│  │  ---
│  │  
│  │  ### 2. scope leak — `emitHelpOutput` describes the program, not `run` [blocker]
│  │  
│  │  `emitHelpOutput` lives in `invokeRun.ts` but emits top-level CLI structure:
│  │  
│  │  ```ts
│  │  console.log(`🪨 rhx [command] [...args]`);
│  │  console.log(`   ├─ commands`);
│  │  console.log(`   │  ├─ [skill]     run a skill (default)`);
│  │  console.log(`   │  ├─ list        show available skills`);
│  │  ```
│  │  
│  │  This is the program's help, not the `run` command's help. Two defects:
│  │  - The `run` command has leaked ownership of the entire CLI's self-description.
│  │  - If someone runs `rhachet run --help` directly (bypassing `bin/rhx`), they get `rhx`-branded output from the `run` command — misrepresenting the binary.
│  │  
│  │  **fix direction:** extract `invokeHelp` as a top-level command registered on `program`, and remove `emitHelpOutput` from `invokeRun`.
│  │  
│  │  ---
│  │  
│  │  ### 3. behavior hazard — `getRawArgsAfterRun` uses `process.argv.indexOf('run')` [blocker]
│  │  
│  │  ```ts
│  │  const getRawArgsAfterRun = (): string[] => {
│  │    const argv = process.argv;
│  │    const runIdx = argv.indexOf('run');
│  │    if (runIdx === -1) return [];
│  │    return argv.slice(runIdx + 1);
│  │  };
│  │  ```
│  │  
│  │  `indexOf('run')` finds the first occurrence of the literal string `'run'` in `process.argv`. If a skill slug or argument contains `'run'` earlier in the array (e.g., `rhx run-tests --arg run`), it slices from the wrong index. Also: this reads `process.argv` as an implicit global — it breaks the input-context pattern. Args should be passed explicitly.
│  │  
│  │  **fix direction:** Commander already has parsed args available; pass the raw remainder through Commander's `.args` or pass `process.argv` explicitly as an input so the function has no hidden dep.
│  │  
│  │  ---
│  │  
│  │  ### 4. maintenance + failhide hazard — `findUnique*` called twice in error handlers [blocker]
│  │  
│  │  In `invokeRun.ts`, when `InitExecutionError` is caught:
│  │  
│  │  ```ts
│  │  catch (error) {
│  │    if (error instanceof InitExecutionError) {
│  │      const init = findUniqueInitExecutable({ ... }); // 🔴 second call
│  │      console.error(`💪 init role repo=...`);
│  │      process.exit(error.exitCode);
│  │    }
│  │    throw error;
│  │  }
│  │  ```
│  │  
│  │  `findUniqueInitExecutable` was already called inside `performRunViaInitMode`. The error handler calls it again to produce a log line — but:
│  │  - If the second `findUniqueInitExecutable` throws (e.g., the skill file was deleted between calls), that exception escapes the `catch` block — the `InitExecutionError` is silently swallowed. This is a failhide hazard.
│  │  - If it succeeds but returns different results (file system changed), the log is inconsistent with what was actually executed.
│  │  
│  │  **fix direction:** `performRunViaInitMode` / `performRunViaCommandMode` should return the resolved executable so the catch block can use it without re-discovery.
│  │  
│  │  ---
│  │  
│  │  ### 5. behavior hazard — `getCommandHeader` omits `--repo`/`--role` context [nitpick → maintenance]
│  │  
│  │  When user runs `rhx list git --repo test`, the output header is:
│  │  
│  │  ```
│  │  🪨 rhx list '*git*'
│  │  ```
│  │  
│  │  `--repo test` is silently dropped from the self-description. The `getSkillListTreestruct` transformer only receives a pre-filtered `skills` array; it doesn't know what filters were applied upstream.
│  │  
│  │  **fix direction:** pass `{ repo, role }` into `getSkillListTreestruct` (and through to `getCommandHeader`) so the header can reflect the full invocation context. This also improves decomposability — `getSkillListTreestruct` becomes a complete description of what was queried.
│  │  
│  │  ---
│  │  
│  │  ### 6. arch smell — filter responsibility split without a unified spec [nitpick]
│  │  
│  │  Repo/role filtering happens in `discoverSkillExecutables`. Slug/pattern filtering happens in `getSkillListTreestruct`. The `invokeRunList` orchestrator composes these, but they share no unified "filter spec" type. If a third dimension of filtering is added later, it must be plumbed independently through both layers. A single `FilterSpec` object passed top-down would keep this decomposable.
