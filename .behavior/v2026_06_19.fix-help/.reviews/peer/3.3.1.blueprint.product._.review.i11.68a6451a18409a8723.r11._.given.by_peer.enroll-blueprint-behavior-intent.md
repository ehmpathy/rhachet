│  │  #### 🔴 blocker.4 — bare `rhx` (no args) is the original friction but remains unaddressed
│  │  
│  │  **the wish says:**
│  │  > `rhx` — user wonders: "what skills do i have available?" no way to know without .agent/ inspection
│  │  
│  │  **the vision's "before" block explicitly flags this.** but neither the criteria, the blackbox matrix, nor the blueprint specify any change to what bare `rhx` produces. the `bin/rhx` dispatch short-circuits for `--help`, `-h`, `help`, and `list`, then falls through to `run --skill $@` — which with no args will fail with the existing error path (likely "skill not found" or "no skill provided").
│  │  
│  │  the "aha moment" in the vision is: "new user links roles, runs `rhx list`, sees exactly what they can do." but a new user will likely type `rhx` first. if bare `rhx` still errors without a helpful hint, the original friction survives.
│  │  
│  │  **fix needed:** specify behavior for bare `rhx` (no args): either show help, show `rhx list` output, or show a hint. add it to the criteria matrix and acceptance tests.

│  │  #### 🔴 blocker.5 — `rhx list --help` is subject to the same commander.js interception bug as `rhx run --help`
│  │  
│  │  **location:** `invokeRunList.ts` command registration
│  │  
│  │  the blueprint registers `--help` as an option on the `list` subcommand:
│  │  ```ts
│  │  .option('--help', 'show list help')
│  │  .action(handleList)
│  │  ```
│  │  
│  │  same root cause as b2: commander intercepts `--help` on any subcommand before `.action()`. the `if (opts.help)` check in `handleList` will never fire. `rhx list --help` will render commander's default help — not the custom treestruct specified in usecase.6 and test case1 `[t1]`.
│  │  
│  │  this is a distinct instance from b2 (which affects `run --help`). **both need the fix** but the blueprint fix options proposed in b2 only address the `run --help` path.

│  │  #### 🟠 nitpick.6 — `rhx help [skill]` error path has no test, but vision explicitly calls it out as "not supported"
│  │  
│  │  **location:** vision edgecases table:
│  │  > `rhx help [skill]` | not supported (use `rhx [skill] --help` instead)
│  │  
│  │  with the `help` short-circuit added to `bin/rhx`, `rhx help git.commit.set` would be intercepted as `help` and show the global help (ignoring `git.commit.set`). the user gets no feedback that their intent wasn't served.
│  │  
│  │  the blueprint has no test for this path. a user following natural intuition (`rhx help git.commit.set`) gets silently misled.
│  │  
│  │  **fix needed:** either add an acceptance test asserting the current behavior (show help, ignore arg — document that `--help` takes precedence) or add a hint in the help output: `to get skill-specific help: rhx [skill] --help`.
