# vision: fix keyrack get AWS_PROFILE

## the outcome world

### before

```bash
$ rhx keyrack get --key AWS_PROFILE --env test
# returns: '{"AWS_ACCESS_KEY_ID":"ASIA3W6J4C3WCTG6MEL4","AWS_SECRET_ACCESS_KEY":"...","AWS_SESSION_TOKEN":"..."}'

$ export AWS_PROFILE=$(rhx keyrack get --key AWS_PROFILE --env test --silent)
# AWS_PROFILE='{"AWS_ACCESS_KEY_ID":"ASIA3W6J4C3WCTG6MEL4",...}'
# ❌ broken — AWS SDK doesn't understand JSON, expects profile name
```

### after

```bash
$ rhx keyrack get --key AWS_PROFILE --env test
# returns: ehmpathy.demo

$ export AWS_PROFILE=$(rhx keyrack get --key AWS_PROFILE --env test --silent)
# AWS_PROFILE=ehmpathy.demo
# ✅ works — AWS SDK resolves credentials from ~/.aws/config profile
```

### the "aha" moment

"keyrack just gives me the profile name. i set `AWS_PROFILE` and aws cli works. no json parse, no credential shuffle."

## user experience

### usecases

| goal | command | outcome |
|------|---------|---------|
| check configured profile | `rhx keyrack get --key AWS_PROFILE --env test` | shows profile name |
| use aws cli | `export AWS_PROFILE=$(rhx keyrack get ...)` | aws cli works |
| use aws sdk | `process.env.AWS_PROFILE = ...` | sdk resolves credentials |

### timeline

1. user runs `rhx keyrack set --key AWS_PROFILE --env test --vault aws.config`
2. guided setup: select sso portal → account → role → profile name stored
3. user runs `rhx keyrack unlock --env test`
4. sso login via browser if session expired
5. user runs `rhx keyrack get --key AWS_PROFILE --env test`
6. keyrack returns: `ehmpathy.demo` (the profile name, not credentials)
7. user exports `AWS_PROFILE=ehmpathy.demo`
8. aws cli/sdk resolves credentials from the sso session

## mental model

### how users describe this

> "keyrack manages which aws profile i use. i unlock it, it opens the sso login in my browser. then i just `export AWS_PROFILE=` the value and aws works."

### analogies

- keyrack is like a keychain — it holds the *name* of the profile, not the actual credentials
- the profile name is like a valet ticket — you give it to the valet (aws sdk), they fetch your car (credentials)
- keyrack unlocks the session, but aws sdk still does the credential resolution

### terms

| user says | keyrack means |
|-----------|---------------|
| "my aws profile" | profile name in ~/.aws/config |
| "unlock aws" | trigger sso login for the profile |
| "get my credentials" | return the profile name (sdk resolves rest) |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| AWS_PROFILE works with aws cli | ✅ | profile name works directly |
| AWS_PROFILE works with aws sdk | ✅ | sdk resolves from profile |
| no json parse required | ✅ | single string value |
| credentials stay secure | ✅ | never exposed; sso session handles |

### pros

- **simplicity**: one env var, one value, works everywhere
- **security**: credentials never leave sso session; keyrack only stores profile name
- **portability**: profile name works across terminals, scripts, ide

### cons

- requires sso session to be valid when aws cli/sdk runs
- if user wants explicit credentials (for tools that don't support profiles), they must run `aws configure export-credentials` directly

### edgecases

| edgecase | how we handle |
|----------|---------------|
| sso session expired | `keyrack get` still returns profile name; aws cli will prompt for re-login |
| profile doesn't exist | `keyrack set` validates profile exists in ~/.aws/config |
| user wants raw credentials | out of scope — use `aws configure export-credentials` directly |
| daemon has old cached JSON | user runs `keyrack relock` then `keyrack unlock` to refresh (keys expire in 9h anyway) |

## what is awkward?

### the current design tension

the `mechAdapterAwsSso.deliverForGet` was designed to "transform source → usable secret". for sso, it interpreted "usable secret" as the resolved credentials json. but for `AWS_PROFILE`, the "usable secret" IS the profile name — the aws sdk does the transformation.

### the fix

the mech IS needed — it validates the SSO session (prompts for browser login if expired). but the mech should return the profile name as the secret, not the resolved credentials JSON.

the transformation for aws.config:
- **input**: profile name (`ehmpathy.demo`)
- **mech work**: validate SSO session via `aws configure export-credentials`
- **output**: profile name (`ehmpathy.demo`) — not credentials JSON

the AWS SDK handles credential resolution from the profile name. keyrack's job is to validate the session and return the profile name.

### proposed fix location

`mechAdapterAwsSso.deliverForGet()` — change the return value from credentials JSON to profile name.

```ts
// current (buggy) — in mechAdapterAwsSso.ts
deliverForGet: async (input) => {
  const profile = input.source;

  // export credentials (validates SSO session)
  const { stdout } = await execAsync(
    `aws configure export-credentials --profile "${profile}" --format env-no-export`,
  );

  // parse credentials...
  // ...

  return { secret: JSON.stringify(credentials), expiresAt };  // ← bug: returns JSON
},

// proposed (fixed)
deliverForGet: async (input) => {
  const profile = input.source;

  // validate SSO session (prompts browser login if expired)
  const { stdout } = await execAsync(
    `aws configure export-credentials --profile "${profile}" --format env-no-export`,
  );

  // parse for expiration time only
  const lines = stdout.trim().split('\n');
  let expiresAt: string | undefined;
  for (const line of lines) {
    const match = line.match(/^AWS_CREDENTIAL_EXPIRATION=(.*)$/);
    if (match?.[1]) expiresAt = match[1];
  }
  if (!expiresAt) {
    expiresAt = addDuration(asIsoTimeStamp(new Date()), { minutes: 55 });
  }

  return { secret: profile, expiresAt };  // ← fix: return profile name
},
```

the vault adapter (`vaultAdapterAwsConfig.get()`) stays unchanged — it should continue to call `mech.deliverForGet()` to trigger SSO session validation.
