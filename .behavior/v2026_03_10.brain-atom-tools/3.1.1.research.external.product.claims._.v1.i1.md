# research: external product claims for BrainAtom tool support

> what facts, assumptions, questions, and opinions are relevant to this wish?

---

## claims

### [FACT] 1: LLMs are trained on tool use patterns

> [1] "Claude is capable of interacting with tools and functions, allowing you to extend Claude's capabilities to perform a wider variety of tasks." — anthropic docs [^1]

tool use is a native capability of modern LLMs, not a workaround or prompt hack.

### [FACT] 2: tool use produces outsized capability gains

> [2] "Tool access is one of the highest-leverage primitives you can give an agent. On benchmarks like LAB-Bench FigQA (scientific figure interpretation) and SWE-bench (real-world software engineering), adding even simple tools produces outsized capability gains, often surpassing human expert baselines." — anthropic docs [^1]

tool use measurably improves LLM performance on complex tasks.

### [FACT] 3: providers assign unique IDs to correlate tool calls with results

> [3] anthropic: `"id": "toolu_01A09q90qw90lq917835lq9"` in tool_use content blocks — anthropic docs [^1]

> [4] openai: `"id": "call_12345xyz"` in tool_calls array — openai docs [^2]

provider-assigned IDs are required for correlation in multi-turn tool flows.

### [FACT] 4: both providers use JSON Schema for tool input schemas

> [5] anthropic uses `input_schema` with JSON Schema format — anthropic docs [^1]

> [6] openai uses `parameters` with JSON Schema format — openai docs [^2]

JSON Schema is the industry standard for tool input definition.

### [FACT] 5: parallel tool calls are natively supported

> [7] openai: "The model may choose to call multiple functions in a single turn." — openai docs [^2]

brains can request multiple tool executions simultaneously.

---

### [SUMP] 1: tool definitions should be per-request, not global

the vision assumes tools are plugged per-ask:

> [8] "tool definitions at call time: tools are plugged per-ask, not registered globally" — vision.md

**rationale**: matches rhachet's stateless atom philosophy. each ask() call is independent.

**alternative**: global tool registry per BrainAtom instance (would add state).

### [SUMP] 2: tool results flow via episode continuation

the vision assumes tool results are injected via episode continuation:

> [9] "tool results via episode continuation: caller injects tool results into next exchange" — vision.md

**rationale**: keeps BrainAtom stateless; caller owns execution.

**alternative**: dedicated `atom.askWithToolResults()` method.

### [SUMP] 3: string serialization suffices for tool flow

the vision assumes tool calls serialize to BrainExchange.input/output:

> [10] "string serialization for tool flow: tool calls serialize to output, tool results serialize to input" — vision.md

**rationale**: natural fit — brain's output is the call, next input is the result.

### [SUMP] 4: validation is caller's responsibility

the vision assumes rhachet passes through tool invocations without validation:

> [11] "the invocation is included with the raw input [...] the caller can validate and handle the error" — 2.1.criteria.blackbox.md

**rationale**: keeps rhachet thin; callers may have their own validation logic.

---

### [KHUE] 1: should rhachet support strict mode?

strict mode guarantees schema conformance but has constraints:

> [12] openai: "structured outputs are not supported with parallel function calls—when using structured outputs, set `parallel_tool_calls` to `false`." — openai docs [^2]

**tradeoff**: strict mode vs parallel tool calls (cannot have both with openai).

**question**: should rhachet expose strict mode as an option, or leave it to adapters?

### [KHUE] 2: should tool input schema use zod or JSON Schema?

rhachet callers typically use zod for schema definition.

providers require JSON Schema.

**question**: should `ToolDefinition.schema.input` accept zod (and convert internally) or require JSON Schema?

### [KHUE] 3: how should episode hash include tool calls?

BrainEpisode has `hash: string` that's content-derived.

**question**: should the hash include tool call content? this affects deduplication and replay.

### [KHUE] 4: what happens when brain returns both output AND tool calls?

the criteria allows both:

> [13] "when the brain returns both output and tool calls, then output is TOutput, then call.tools is populated" — 2.1.criteria.blackbox.md

**question**: how should callers prioritize? process output first, then tools? or tools first?

---

### [OPIN] 1: tool use is fundamentally different from structured output

some argue tool use should be distinct from structured output:

> [14] "Each tool defines a contract: you specify what operations are available and what they return; Claude decides when and how to call them." — anthropic docs [^1]

tool use is about capability extension, not just data extraction.

### [OPIN] 2: provider-agnostic abstraction is worth the complexity

rhachet's philosophy is abstraction:

> [15] "provider-agnostic tool format: ToolDefinition abstracts anthropic/openai" — vision.md

**tradeoff**: abstraction adds complexity but enables portability.

### [OPIN] 3: BrainAtom should stay single-turn

the vision emphasizes:

> [16] "BrainAtom stays stateless. BrainAtom stays single-turn. but each turn can now request tool execution before the next turn." — vision.md

tool support shouldn't change BrainAtom's fundamental nature.
