# research: external product access for BrainAtom tool support

> what remote repositories (databases, apis, filesystems) do we need to access?

---

## summary

BrainAtom tool support requires integration with brain supplier APIs (Anthropic, OpenAI) that provide native tool use capabilities. both providers:

1. accept tool definitions as part of the inference request
2. return tool invocations as part of the response content
3. accept tool results via continuation messages
4. support parallel tool calls in a single response
5. provide optional strict mode for schema conformance

---

## remote access requirements

### 1. anthropic messages api

**endpoint**: `https://api.anthropic.com/v1/messages`

**interface**: REST API via HTTP, SDK packages available

**contract for tool definitions** (input):
```json
{
  "tools": [
    {
      "name": "get_weather",
      "description": "Get the current weather in a given location",
      "input_schema": {
        "type": "object",
        "properties": {
          "location": { "type": "string", "description": "The city and state" }
        },
        "required": ["location"]
      }
    }
  ]
}
```

**contract for tool invocations** (output):
```json
{
  "content": [
    { "type": "text", "text": "I'll check the weather." },
    {
      "type": "tool_use",
      "id": "toolu_01A09q90qw90lq917835lq9",
      "name": "get_weather",
      "input": { "location": "San Francisco, CA" }
    }
  ]
}
```

**contract for tool results** (continuation):
```json
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_01A09q90qw90lq917835lq9",
      "content": "15 degrees"
    }
  ]
}
```

### 2. openai chat completions api

**endpoint**: `https://api.openai.com/v1/chat/completions`

**interface**: REST API via HTTP, SDK packages available

**contract for tool definitions** (input):
```json
{
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "get_weather",
        "description": "Get the current weather",
        "parameters": {
          "type": "object",
          "properties": {
            "location": { "type": "string" }
          },
          "required": ["location"],
          "additionalProperties": false
        }
      }
    }
  ]
}
```

**contract for tool invocations** (output):
```json
{
  "tool_calls": [
    {
      "id": "call_12345xyz",
      "type": "function",
      "function": {
        "name": "get_weather",
        "arguments": "{\"location\":\"Paris, France\"}"
      }
    }
  ]
}
```

**contract for tool results** (continuation):
```json
{
  "role": "tool",
  "tool_call_id": "call_12345xyz",
  "content": "15 degrees"
}
```

## lessons

### lesson 1: tool definitions use JSON Schema for input validation

both providers use JSON Schema (not zod) for tool input schemas.

> [1] "Define client tools with names, descriptions, and input schemas in your API request." — anthropic docs [^1]

> [2] "Parameters: JSON schema that defines input arguments" — openai docs [^2]

**implication**: rhachet should accept JSON Schema at the provider boundary, even if callers use zod (convert via zod-to-json-schema).

### lesson 2: tool invocations include provider-assigned correlation IDs

both providers assign unique IDs to each tool call for correlation with results.

> [3] anthropic uses `id` in the `tool_use` content block:
> ```json
> { "type": "tool_use", "id": "toolu_01A09q90qw90lq917835lq9", "name": "get_weather", "input": {...} }
> ```
> — anthropic docs [^1]

> [4] openai uses `id` in the `tool_calls` array:
> ```json
> { "id": "call_12345xyz", "type": "function", "function": { "name": "get_weather", "arguments": "..." } }
> ```
> — openai docs [^2]

**implication**: rhachet's `ToolInvocation.exid` correctly maps to provider-assigned IDs.

### lesson 3: tool results travel as continuation messages

both providers expect tool results in subsequent messages, correlated by the tool call ID.

> [5] anthropic: "Return the results in a new `user` message with a `tool_result` content block" — anthropic docs [^1]

> [6] anthropic tool result format:
> ```json
> { "type": "tool_result", "tool_use_id": "toolu_01A09q90qw90lq917835lq9", "content": "15 degrees" }
> ```
> — anthropic docs [^1]

> [7] openai: "After you execute the function, send results back via a tool message with: role: 'tool', tool_call_id: matches the original call's id" — openai docs [^2]

**implication**: rhachet's episode continuation via BrainExchange.input/output naturally fits. tool results serialize into the next exchange input.

### lesson 4: parallel tool calls supported

both providers support multiple tool calls in a single response.

> [8] anthropic: response can include multiple `tool_use` content blocks in the same `content` array — anthropic docs [^1]

> [9] openai: "The model may choose to call multiple functions in a single turn. You can prevent this by set `parallel_tool_calls` to `false`." — openai docs [^2]

**implication**: rhachet's `call.tools: ToolInvocation[]` array correctly supports parallel calls.

### lesson 5: strict mode guarantees schema conformance (optional)

both providers offer strict mode for guaranteed schema adherence.

> [10] anthropic: "Add `strict: true` to your tool definitions to ensure Claude's tool calls always match your schema exactly, which eliminates type mismatches or absent fields." — anthropic docs [^1]

> [11] openai: "When enabled, it 'ensures function calls reliably adhere to the function schema.' Requirements include: additionalProperties must be false for each object." — openai docs [^2]

**implication**: strict mode is provider-specific and has constraints (e.g., openai requires `additionalProperties: false`, no parallel calls with strict). rhachet may expose this as optional, but validation responsibility remains with callers.

### lesson 6: input is parsed JSON, not raw string (anthropic)

anthropic provides parsed JSON in the `input` field.

> [12] anthropic tool_use block shows parsed JSON:
> ```json
> { "type": "tool_use", "id": "toolu_01A09q90qw90lq917835lq9", "name": "get_weather", "input": { "location": "San Francisco, CA", "unit": "celsius" } }
> ```
> — anthropic docs [^1]

### lesson 7: arguments are JSON string (openai)

openai provides arguments as a JSON-encoded string.

> [13] openai: "function.arguments: '{\"location\":\"Paris, France\"}'—JSON-encoded parameters" — openai docs [^2]

**implication**: rhachet adapters must handle both formats — anthropic provides parsed objects, openai provides JSON strings. the adapter normalizes to rhachet's `ToolInvocation.input: TInput`.

### lesson 8: stop_reason indicates tool use intent

both providers signal tool use via stop reason.

> [14] anthropic: "For client tools, the API response has a `stop_reason` of `tool_use`, which signals Claude's intent." — anthropic docs [^1]

**implication**: rhachet can use stop_reason to determine if the brain requested tools vs answered directly.

### lesson 9: both output and tool calls can coexist

anthropic explicitly shows text and tool_use in the same response.

> [15] anthropic response example:
> ```json
> { "content": [{ "type": "text", "text": "I'll check the current weather in San Francisco for you." }, { "type": "tool_use", "id": "toolu_01A09q90qw90lq917835lq9", "name": "get_weather", "input": {...} }] }
> ```
> — anthropic docs [^1]

**implication**: rhachet's design that allows both `output` and `call.tools` to be populated is correct.

## best practices

### 1. provider-agnostic abstraction

rhachet should define its own `ToolDefinition` and `ToolInvocation` types that abstract provider differences:

| rhachet | anthropic | openai |
|---------|-----------|--------|
| `slug` | `name` | `function.name` |
| `description` | `description` | `function.description` |
| `schema.input` | `input_schema` | `function.parameters` |
| `exid` | `id` | `id` |
| `input` | `input` (object) | `function.arguments` (string → parse) |

### 2. validation responsibility

> [16] anthropic: "the invocation is included with the raw input [...] the caller can validate and handle the error" — implied from behavior [^1]

callers should validate tool inputs against schemas. rhachet passes through what the brain returns; validation is caller's responsibility.

### 3. string serialization for episode flow

tool calls serialize naturally into BrainExchange:
- exchange N: output = tool call request (as JSON string or structured)
- exchange N+1: input = tool result (as string)

this matches the vision's assumption #3.

## map to rhachet contracts

### input: ToolDefinition

```ts
interface ToolDefinition {
  slug: string;           // maps to anthropic.name, openai.function.name
  description: string;    // maps to anthropic.description, openai.function.description
  schema: {
    input: JSONSchema;    // maps to anthropic.input_schema, openai.function.parameters
  };
}
```

### output: ToolInvocation

```ts
interface ToolInvocation<TInput = unknown> {
  exid: string;     // maps to anthropic.tool_use.id, openai.tool_calls[].id
  slug: string;     // maps to anthropic.tool_use.name, openai.tool_calls[].function.name
  input: TInput;    // maps to anthropic.tool_use.input, JSON.parse(openai.tool_calls[].function.arguments)
}
```

### flow

1. caller plugs tools via `plug.tools: ToolDefinition[]`
2. adapter converts to provider format (anthropic or openai)
3. brain responds with tool invocations
4. adapter converts to rhachet's `call.tools: ToolInvocation[]`
5. caller executes tools, injects results into next exchange
6. brain incorporates results, may request more tools or answer

## conclusion

the external APIs (anthropic, openai) align well with rhachet's proposed design:

| rhachet concept | provider support |
|-----------------|------------------|
| `plug.tools` | ✓ both accept tool definitions in request |
| `call.tools` | ✓ both return tool invocations in response |
| parallel calls | ✓ both support multiple tool calls per response |
| episode continuation | ✓ both accept tool results via messages |
| provider-assigned IDs | ✓ both assign unique IDs for correlation |

the adapter layer must handle:
- schema format differences (anthropic `input_schema` vs openai `parameters`)
- input format differences (anthropic object vs openai JSON string)
- field name differences (see map table above)
