# research: external product domain for BrainAtom tool support

> what domain objects are involved with this wish?

---

## domain objects

### entities

| entity | description | provider term |
|--------|-------------|---------------|
| Tool | a capability that can be invoked by the brain | anthropic: `tool`, openai: `tool` |
| ToolInvocation | a request from the brain to execute a tool | anthropic: `tool_use`, openai: `tool_call` |
| ToolResult | the output from tool execution | anthropic: `tool_result`, openai: tool message |

### literals

| literal | description | provider term |
|---------|-------------|---------------|
| ToolSchema | JSON Schema that defines valid tool input | anthropic: `input_schema`, openai: `parameters` |
| ToolName | unique identifier for the tool | anthropic: `name`, openai: `function.name` |
| ToolDescription | human-readable description of what the tool does | anthropic: `description`, openai: `function.description` |
| ToolCallId | provider-assigned correlation ID | anthropic: `id`, openai: `id` |
| ToolInput | the arguments to pass to the tool | anthropic: `input`, openai: `arguments` |

### events

| event | description | when |
|-------|-------------|------|
| ToolRequested | brain requests tool execution | brain returns tool_use/tool_call |
| ToolExecuted | caller executes the tool | between brain turns |
| ToolResultProvided | caller provides result back | next user message |

## relationships

### treestruct: tool use flow

```
BrainAtom.ask()
├── input
│   ├── prompt: string
│   ├── schema.output: z.Schema
│   └── plug.tools: ToolDefinition[]
│       └── ToolDefinition
│           ├── slug: string
│           ├── description: string
│           └── schema.input: JSONSchema
└── output
    ├── output: TOutput | null
    ├── call: { tools: ToolInvocation[] } | null
    │   └── ToolInvocation
    │       ├── exid: string
    │       ├── slug: string
    │       └── input: TInput
    ├── episode: BrainEpisode
    └── metrics: BrainOutputMetrics
```

### treestruct: provider API shapes

```
anthropic
├── request.tools[]
│   ├── name: string
│   ├── description: string
│   └── input_schema: JSONSchema
└── response.content[]
    ├── type: "text" | "tool_use"
    └── (tool_use) → id, name, input

openai
├── request.tools[]
│   ├── type: "function"
│   └── function
│       ├── name: string
│       ├── description: string
│       └── parameters: JSONSchema
└── response.tool_calls[]
    ├── id: string
    ├── type: "function"
    └── function
        ├── name: string
        └── arguments: string (JSON)
```

### dependencies

```
ToolInvocation depends on ToolDefinition
  → slug must match a plugged tool
  → input must conform to tool's schema

ToolResult depends on ToolInvocation
  → tool_use_id/tool_call_id must match invocation's exid

BrainEpisode depends on ToolInvocation
  → tool calls serialize into episode exchanges
```

## composition

### how domain objects compose to support wish

1. **caller defines tools** via `ToolDefinition[]`
   - each tool has slug, description, schema.input

2. **adapter converts to provider format**
   - anthropic: `name`, `description`, `input_schema`
   - openai: `function.name`, `function.description`, `function.parameters`

3. **brain returns invocations** via `ToolInvocation[]`
   - each invocation has exid, slug, input
   - adapter normalizes from provider format

4. **caller executes tools**
   - matches slug to tool implementation
   - validates input against schema (optional)
   - executes and captures result

5. **caller continues episode**
   - injects tool results into next exchange
   - brain incorporates results
   - may request more tools or answer
