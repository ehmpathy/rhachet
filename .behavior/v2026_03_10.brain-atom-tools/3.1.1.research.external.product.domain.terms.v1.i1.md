# research: domain terms for BrainAtom tool support

> what terms should we coin for new domain objects?

---

## terms to coin

### 1. ToolDefinition

**what**: a declaration of a tool that can be invoked by the brain

**options considered**:

| option | pros | cons |
|--------|------|------|
| `ToolDefinition` | clear, matches openai SDK | verbose |
| `ToolSpec` | short, implies specification | overloaded ("spec" = test) |
| `Tool` | simple | ambiguous (definition vs invocation?) |
| `Capability` | semantic richness | abstract, unfamiliar |
| `Implement` | bluecollar metaphor | implies execution, not definition |

**parallel concepts**:
- healthcare: `Procedure` (defined before performed)
- culinary: `Recipe` (defined before executed)
- bluecollar: `Blueprint` (defined before built)

**recommendation**: `ToolDefinition` — explicit, unambiguous, industry-aligned.

---

### 2. ToolInvocation

**what**: a request from the brain to execute a tool

**options considered**:

| option | pros | cons |
|--------|------|------|
| `ToolInvocation` | precise, implies "call" | verbose |
| `ToolCall` | short, matches openai | "call" is overloaded |
| `ToolUse` | matches anthropic | sounds like a noun, not a request |
| `ToolRequest` | clear intent | doesn't imply the brain initiated |
| `Dispatch` | action-oriented | too generic |

**parallel concepts**:
- healthcare: `Order` (doctor orders a procedure)
- culinary: `Ticket` (chef requests a dish)
- bluecollar: `WorkOrder` (requester orders work done)

**recommendation**: `ToolInvocation` — precise, implies the brain initiated the call.

---

### 3. exid (external ID)

**what**: provider-assigned correlation ID for a tool call

**options considered**:

| option | pros | cons |
|--------|------|------|
| `exid` | short, explicit "external" | unfamiliar term |
| `id` | simple | ambiguous (internal vs external) |
| `callId` | descriptive | ties to "call" terminology |
| `correlationId` | precise | verbose |
| `toolCallId` | explicit | redundant within ToolInvocation |

**parallel concepts**:
- healthcare: `CaseNumber` (external reference)
- culinary: `TicketNumber` (external tracker)
- bluecollar: `JobNumber` (external reference)

**recommendation**: `exid` — short, explicit about external origin.

---

### 4. slug (tool identifier)

**what**: unique identifier for the tool

**options considered**:

| option | pros | cons |
|--------|------|------|
| `slug` | familiar pattern, url-safe | less explicit than "name" |
| `name` | simple, matches providers | overloaded in code |
| `identifier` | precise | verbose |
| `key` | short | generic |
| `handle` | action-oriented | unfamiliar in this context |

**parallel concepts**:
- culinary: menu items have slugs (e.g., `soup-of-day`)
- recreation: routes have slugs (e.g., `trail-42`)

**recommendation**: `slug` — familiar pattern, implies unique identifier.

---

### 5. plug.tools (input field)

**what**: the field where tools are plugged into BrainAtom

**options considered**:

| option | pros | cons |
|--------|------|------|
| `plug.tools` | metaphor-rich, implies "pluggable" | less explicit |
| `tools` | simple | doesn't imply optionality |
| `extensions` | implies "extend capabilities" | generic |
| `capabilities` | semantic richness | verbose |
| `functions` | matches openai | ties to one provider |

**parallel concepts**:
- bluecollar: `Attachments` (tools attached to machine)
- recreation: `Gear` (equipment for activity)

**recommendation**: `plug.tools` — aligns with extant `BrainAtomPlug` type, implies pluggability.

---

### 6. call.tools (output field)

**what**: the field where tool invocations appear in output

**options considered**:

| option | pros | cons |
|--------|------|------|
| `call.tools` | reads naturally "brain's call for tools" | nested structure |
| `toolCalls` | flat, explicit | doesn't group with other "call" concepts |
| `invocations` | precise | verbose |
| `requests` | clear intent | doesn't imply brain initiated |

**parallel concepts**:
- healthcare: `Referral` (doctor refers to specialist)
- culinary: `Order` (chef orders ingredient)

**recommendation**: `call.tools` — reads naturally, groups potential future "call" fields.

## relationship treestruct

```
BrainAtomPlug (input config)
└── tools: ToolDefinition[]
    └── ToolDefinition
        ├── slug: string
        ├── description: string
        └── schema.input: JSONSchema

BrainOutput (output)
└── call: { tools: ToolInvocation[] } | null
    └── ToolInvocation
        ├── exid: string
        ├── slug: string
        └── input: TInput
```

## ubiquitous language summary

| term | definition |
|------|------------|
| `ToolDefinition` | declaration of a tool that can be invoked by the brain |
| `ToolInvocation` | a request from the brain to execute a tool |
| `exid` | provider-assigned external ID for correlation |
| `slug` | unique identifier for the tool |
| `plug.tools` | input field where tools are plugged in |
| `call.tools` | output field where tool invocations appear |
