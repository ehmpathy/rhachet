# handoff: BrainAtom tool support extension

> from bhrain to rhachet

## context

bhrain's BrainArch1 is an agentic brain that orchestrates:
1. LLM inference
2. tool execution
3. loop until done

bhrain CONFORMS to rhachet. no custom atom plugins.

rhachet's `genContextBrain({ choice: { repl: 'bhrain/arch1@claude/sonnet' } })` discovers bhrain's repl and composes it with the atom.

## the gap

| BrainArch1 needs | BrainAtom.ask() provides |
|------------------|--------------------------|
| `messages[]` with roles | `prompt` string |
| `tools[]` definitions | none (plug.tools absent) |
| `toolCalls[]` in response | structured output via schema |

**BrainAtom.ask() is designed for single-turn inference with structured output. BrainArch1 needs multi-turn tool execution.**

## what bhrain requests

### 1. tools in plug

add `tools?: ToolDefinition[]` to BrainAtomPlug:

```ts
interface BrainAtomPlug {
  tools?: ToolDefinition[];  // <-- ADD THIS
}
```

**why plug?** tools are plugins. you plug tools into the brain to use them.

### 2. call.tools in output

add `call?: { tools: ToolInvocation[] }` to BrainOutput:

```ts
interface BrainOutput {
  // ... extant fields
  call?: {
    tools: ToolInvocation[];  // <-- ADD THIS
  }
}
```

### 3. BrainExchange extension (maybe)

current BrainExchange:
```ts
{ input: string; output: string }
```

tool flow has more structure:
- tool call: `{ name: string; args: object }` in output
- tool result: `{ toolCallId: string; result: string }` in input
- multi-tool: single turn can have multiple tool calls

options:
- **A**: extend BrainExchange to support structured tool flow
- **B**: serialize tool flow into input/output strings (lossy but backward-compatible)

bhrain will conform to whatever rhachet decides.

## type definitions (proposed)

```ts
interface ToolDefinition {
  name: string;
  description: string;
  inputSchema: z.Schema<unknown>;
  strict?: boolean;
}

interface ToolInvocation {
  id: string;
  name: string;
  input: unknown;
}
```

these should align with anthropic and openai tool formats for cross-provider compatibility.

## the contract

```ts
ask<TOutput>(input: {
  prompt: string;
  schema: { output: z.Schema<TOutput> };
  plug?: {
    tools?: ToolDefinition[];  // plug tools in
  };
  on?: { episode: BrainEpisode };
  role: { briefs?: Artifact[] };
}) => Promise<BrainOutput<TOutput, 'atom'> & {
  call?: {
    tools: ToolInvocation[];  // tool calls out
  }
}>
```

## acceptance

bhrain can proceed when:

1. BrainAtomPlug accepts `tools?: ToolDefinition[]`
2. BrainOutput includes `call?: { tools: ToolInvocation[] }`
3. episode continuation handles tool flow (via exchange extension or serialization)

## timeline

bhrain is blocked until this extension ships. no workarounds - tools must be native LLM features, not prompt-serialized.
