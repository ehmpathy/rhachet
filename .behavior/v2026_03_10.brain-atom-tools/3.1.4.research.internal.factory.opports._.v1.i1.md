# research: internal factory opportunities for BrainAtom tool support

> what opportunities exist in the current codebase that we can leverage?

---

## opportunities identified

### 1. [OPPORTUNITY] conditional type pattern already exists

**file**: `src/domain.objects/BrainOutput.ts`

**pattern**: `AsBrainOutputSeriesFor<T>` narrows series based on BrainGrain

> [1] "export type AsBrainOutputSeriesFor<T extends BrainGrain> = T extends 'repl'
>   ? BrainSeries
>   : T extends 'atom'
>     ? null
>     : BrainSeries | null;" — BrainOutput.ts:12-16

**leverage for**: create parallel conditional types for call and output based on TPlug:
- `AsBrainOutputCallFor<TPlug>` — narrows to `{ tools } | null` vs `null`
- `AsBrainOutputOutputFor<TOutput, TPlug>` — narrows to `TOutput | null` vs `TOutput`

---

### 2. [OPPORTUNITY] BrainAtomPlugs is ready to extend

**file**: `src/domain.objects/BrainAtomPlugs.ts`

**pattern**: interface is a placeholder with explicit todo

> [2] "export interface BrainAtomPlugs {
>   ...
>   output?: never; // todo: allow configuration
> }" — BrainAtomPlugs.ts:10-16

**leverage for**: add `tools?: ToolDefinition[]` as a new optional field; no contract breaks

---

### 3. [OPPORTUNITY] exid pattern is established

**file**: `src/domain.objects/BrainEpisode.ts`

**pattern**: supplier-assigned external IDs use `exid: string | null`

> [3] "exid: string | null;
>   ...
>   .note = opaque to rhachet; null if not provided by supplier" — BrainEpisode.ts:27-28, 25-26

**leverage for**: `ToolInvocation.exid` follows same pattern; providers return call IDs that rhachet treats as opaque

---

### 4. [OPPORTUNITY] domain-objects library is used throughout

**file**: `src/domain.objects/BrainAtom.ts`

**pattern**: DomainEntity for identity, DomainLiteral for value objects

> [4] "export class BrainAtom extends DomainEntity<BrainAtom> implements BrainAtom {
>   public static unique = ['repo', 'slug'] as const;
> }" — BrainAtom.ts:61-63

**leverage for**: create ToolDefinition and ToolInvocation as DomainLiteral classes

---

### 5. [OPPORTUNITY] plugs pass-through already works

**file**: `src/domain.operations/brainAtom/askViaBrainAtom.ts`

**pattern**: askViaBrainAtom passes plugs to atom.ask

> [5] "return input.atom.ask(
>     {
>       plugs: input.plugs,
>       ...
>     },
>   );" — askViaBrainAtom.ts:26-33

**leverage for**: once BrainAtomPlugs.tools is defined, it will automatically pass through

### 6. [OPPORTUNITY] type tests pattern is established

**file**: `src/domain.objects/BrainAtom.types.test.ts`

**pattern**: compile-time type tests with @ts-expect-error

> [6] "// @ts-expect-error - 'wrong' property does not exist on { answer: string }
>   const _wrong = result.output.wrong;" — BrainAtom.types.test.ts:30-31

**leverage for**: add type tests that verify conditional type narrower works correctly

---

### 7. [OPPORTUNITY] BrainExchange string serialization fits tool flow

**file**: `src/domain.objects/BrainExchange.ts`

**pattern**: input and output are strings

> [7] "input: string;
>   ...
>   output: string;" — BrainExchange.ts:19, 23

**leverage for**: tool flow serializes naturally:
- tool call → `exchange.output` (JSON stringified invocation)
- tool result → next `exchange.input` (result string)

no structural changes needed to BrainExchange.

---

### 8. [OPPORTUNITY] nested domain objects pattern established

**file**: `src/domain.objects/BrainOutput.ts`

**pattern**: static nested declaration for hydration

> [8] "public static nested = {
>     metrics: BrainOutputMetrics,
>     episode: BrainEpisode,
>     series: BrainSeries,
>   };" — BrainOutput.ts:55-58

**leverage for**: if call is a nested object, add to nested declaration

## summary

| opportunity | benefit |
|-------------|---------|
| conditional type pattern | minimal new pattern; follow extant code |
| BrainAtomPlugs placeholder | clean extension point |
| exid pattern | consistent ID handler |
| domain-objects library | standard class structure |
| plugs pass-through | no operation changes needed |
| type tests pattern | verification strategy ready |
| string serialization | no BrainExchange changes needed |
| nested domain objects | hydration pattern ready |

these opportunities enable implementation with minimal deviation from extant patterns.
