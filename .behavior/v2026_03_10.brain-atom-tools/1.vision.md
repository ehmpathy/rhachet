# vision: BrainAtom tool support

> what does the world look like when BrainAtom supports tool calls?

---

## the outcome world

### before: the gap

today, BrainAtom is a single-turn inference primitive. you give it a prompt, you get structured output. clean, simple, deterministic-ish.

```ts
const { output } = await atom.ask({
  prompt: 'summarize this document',
  schema: { output: z.object({ summary: z.string() }) },
  role: { briefs },
});
// output.summary = "the document discusses..."
```

but what if the brain needs to *do* something before it can answer? what if summarizing requires fetching the document first?

today's workaround: the caller orchestrates everything manually.

```ts
// caller must orchestrate
const document = await fetchDocument(url);  // caller fetches
const { output } = await atom.ask({
  prompt: `summarize this: ${document}`,    // caller injects
  schema: { output: z.object({ summary: z.string() }) },
  role: { briefs },
});
```

this breaks down when:
- the brain should decide *which* tool to call
- the brain should decide *how many* calls to make
- the brain's judgment matters more than the caller's

bhrain's BrainArch1 hit this wall. it needs to orchestrate:
1. LLM inference → "i need to call tool X with args Y"
2. tool execution → actual result
3. LLM inference → "based on that, here's my answer"

but BrainAtom can't express step 1. there's no way to say "the brain wants to call a tool."

### after: the bridge

with tool support, BrainAtom becomes a proper inference primitive that can *request* tool calls:

```ts
const answer = await atom.ask({
  prompt: 'what files are in the src/ directory?',
  schema: { output: z.object({ files: z.array(z.string()) }) },
  plug: {
    tools: [
      {
        slug: 'os.fileops.ls',
        description: 'list files in a directory',
        schema: { input: z.object({ path: z.string() }) },
      },
    ],
  },
  role: { briefs },
});

// the brain might respond with a tool call request
if (answer.call?.tools.length) {
  // answer.call.tools = [{ exid: 'call_1', slug: 'os.fileops.ls', input: { path: 'src/' } }]
  // caller executes tool, continues episode with result
}
// or it might just answer directly
// answer.output = { files: ['index.ts', 'utils.ts'] }
```

the brain decides. the caller executes. rhachet stays in the middle as the honest broker.

### the "aha" moment

the value clicks when you realize: **tool support isn't about making BrainAtom do more. it's about making BrainAtom *communicate* more.**

the brain already *thinks* about tools. modern LLMs are trained on tool use. they naturally reason about when to call functions. BrainAtom just couldn't *express* that reasoning.

now it can. the brain says "i want to call this tool." the caller (e.g., BrainArch1) executes it. the conversation continues.

BrainAtom stays stateless. BrainAtom stays single-turn. but each turn can now request tool execution before the next turn.

## user experience

### usecase 1: agentic brain orchestrator (bhrain's BrainArch1)

**goal**: build an agentic brain that loops until done

**today's pain**:
```ts
// BrainArch1 can't use BrainAtom because it can't express tool calls
// forced to use raw provider SDKs, losing rhachet's abstractions
```

**tomorrow's flow**:
```ts
// BrainArch1 uses BrainAtom, gets tool calls in response
const answer = await atom.ask({
  prompt: userMessage,
  plug: { tools: availableTools },
  schema: { output: responseSchema },
  on: { episode: priorEpisode },  // continuation
  role: { briefs },
});

if (answer.call?.tools.length) {
  // execute tools
  const toolResults = await Promise.all(
    answer.call.tools.map(call => executeToolCall(call))
  );
  // continue episode with tool results
  // (inject tool results into next exchange)
}
```

**timeline**:
1. user sends message
2. BrainArch1 calls BrainAtom.ask() with tools plugged in
3. brain responds with tool calls (or direct answer)
4. BrainArch1 executes tool calls
5. BrainArch1 continues episode with tool results
6. repeat until brain responds with final answer

### usecase 2: structured extraction with fallback tools

**goal**: extract structured data, with tool access for missing info

```ts
const answer = await atom.ask({
  prompt: 'extract the customer info from this email',
  schema: {
    output: z.object({
      name: z.string(),
      email: z.string(),
      phone: z.string().nullable(),
    }),
  },
  plug: {
    tools: [
      {
        slug: 'stripe.customer.lookup',
        description: 'lookup customer by email in stripe',
        schema: { input: z.object({ email: z.string() }) },
      },
    ],
  },
  role: { briefs },
});

// brain might: extract from email directly, OR request stripe lookup
```

### contract inputs & outputs

**input additions**:
```ts
interface BrainAtomPlug {
  tools?: ToolDefinition[];
}

interface ToolDefinition {
  slug: string;
  description: string;
  schema: { input: z.Schema<unknown> };
}
```

**output additions**:
```ts
// add TPlug generic, conditional outcome types
type BrainOutput<TOutput, TBrainGrain, TPlug extends BrainAtomPlug = {}> = {
  // ... extant fields, plus:
  output: TPlug extends { tools: ToolDefinition[] } ? TOutput | null : TOutput;
  call: TPlug extends { tools: ToolDefinition[] } ? { tools: ToolInvocation[] } | null : null;
}

interface ToolInvocation<TInput = unknown> {
  exid: string;      // provider-assigned call ID (for result correlation)
  slug: string;      // which tool
  input: TInput;     // parsed arguments
}
```

**full output shape**:
```ts
type BrainOutput<TOutput, TBrainGrain, TPlug extends BrainAtomPlug = {}> = {
  // the outcome (direct outputs) — type narrows based on TPlug
  output: TPlug extends { tools: ToolDefinition[] } ? TOutput | null : TOutput;
  call: TPlug extends { tools: ToolDefinition[] } ? { tools: ToolInvocation[] } | null : null;

  // metadata (about how the brain worked)
  metrics: BrainOutputMetrics;
  episode: BrainEpisode;
  series: BrainSeries | null;   // null for BrainAtom (stateless)
}
```

**progressive complexity via generic**:
- no tools plugged → `output: TOutput`, `call: null` (brain must answer)
- tools plugged → `output: TOutput | null`, `call: { tools } | null` (brain may defer)

callers opt into tooluse complexity. without tools, no null checks needed — backwards compatible.

**why `output` and `call`?**

both `output` and `call` are the **outcome** of the brain — direct outputs, not metadata:

| field | what | why this name |
|-------|------|---------------|
| `output` | structured content the brain produced | the brain's direct output |
| `call` | tools the brain wants invoked | the brain's call for tools |

contrast with metadata fields (`metrics`, `episode`, `series`) which describe *how* the brain worked, not *what* it produced.

`call.tools` reads naturally: "the brain's call for tools" — clear, no made-up words.

## mental model

### how users describe it

> "BrainAtom can now ask for help. you give it tools, it might ask you to call one. you call it, tell it the result, and it continues."

> "it's like asking someone a question, and they say 'let me check something first' — you help them check, then they answer."

### analogies that fit

**the waiter analogy**:
you order food (prompt). the waiter (brain) might say "let me check if we have that" (tool call). you wait. they come back with info (tool result). then they confirm your order (final output).

**the assistant analogy**:
you ask your assistant to book a flight. they might say "i need to check availability" (tool call). you give them access to the booking system. they check and report back.

### terms: user vs rhachet

| user says | rhachet says |
|-----------|--------------|
| "tools" | `plug.tools` |
| "function call" | `call.tools` |
| "the brain wants to call X" | `answer.call.tools` contains invocation |
| "tool result" | injected into next exchange input |
| "the brain answered" | `answer.output` is populated, `answer.call` is null |

## evaluation

### how well does it solve the goals?

**bhrain's BrainArch1**: ✅ fully unblocked
- can use BrainAtom with tools
- gets tool calls in structured response
- can orchestrate tool execution
- can continue episode with results

**general agentic patterns**: ✅ enabled
- any caller can plug tools into BrainAtom
- tool calls are first-class in the response
- episode continuation handles multi-turn flows

### pros

1. **minimal change**: only two additions (plug.tools, call.tools)
2. **backward compatible**: no breaking changes to extant contracts
3. **provider agnostic**: abstracts anthropic/openai tool formats
4. **composable**: works with extant episode continuation
5. **stateless preservation**: BrainAtom stays single-turn

### cons

1. **complexity increase**: callers must handle tool call flow
2. **schema divergence**: tool results are untyped (injected as strings)
3. **episode serialization**: tool flow must serialize into exchanges

### edgecases & pit of success

| edgecase | pit of success design |
|----------|----------------------|
| brain returns both output AND tool calls | response type allows both; caller decides precedence |
| tool call with invalid schema | schema.input validation at plug time |
| no tools plugged but brain wants to call | brain can't request tools not in plug |
| no tool calls | `call` is null when brain answers directly |
| parallel tool calls | `call.tools` is array; all calls in one response |

## open questions & assumptions

### assumptions we've made

1. **tool definitions at call time**: tools are plugged per-ask, not registered globally
   - rationale: matches rhachet's stateless atom philosophy
   - alternative: global tool registry per BrainAtom instance

2. **tool results via episode continuation**: caller injects tool results into next exchange
   - rationale: keeps BrainAtom stateless; caller owns execution
   - alternative: BrainAtom.askWithToolResults() method

3. **string serialization for tool flow**: tool calls serialize to output, tool results serialize to input
   - rationale: natural fit — brain's output is the call, next input is the result
   - no extension needed — BrainExchange.input/output already fits

4. **provider-agnostic tool format**: ToolDefinition abstracts anthropic/openai
   - rationale: rhachet's role is abstraction
   - risk: some provider features may not map cleanly

## what is awkward?

### tool definition schema vs output schema

BrainAtom.ask() already has `schema.output`. now we add `plug.tools[].schema.input`.

```ts
atom.ask({
  schema: { output: z.object({ answer: z.string() }) },  // output schema
  plug: {
    tools: [
      { schema: { input: z.object({ path: z.string() }) } },   // tool input schema
    ],
  },
});
```

two different schema concepts in one call. not confusing per se, but adds cognitive load.

### naming: plug.tools vs something else?

the wish specified `plug.tools`. rationale: "tools are plugins you plug into the brain."

but `tools` also appears in output (`call.tools`). does `plug.tools` → `answer.call.tools` read clearly?

alternative considered: `extensions.tools` or `capabilities.tools`. but `plug` aligns with extant `BrainAtomPlug` type.

---

## summary

BrainAtom tool support bridges the gap between single-turn inference and agentic orchestration. by adding `plug.tools` (input) and `call.tools` (output), we enable:

- bhrain's BrainArch1 to use rhachet's BrainAtom
- any agentic caller to leverage tool use via atoms
- provider-agnostic tool definitions

the design is minimal, backward-compatible, and stateless.
