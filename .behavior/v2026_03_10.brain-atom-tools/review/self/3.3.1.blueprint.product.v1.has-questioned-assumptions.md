# self-review: has-questioned-assumptions

> what technical assumptions were made? are they valid?

---

## assumptions identified

### 1. TPlug extends BrainAtomPlugs

**assumption**: tool configuration belongs in BrainAtomPlugs.

**what if the opposite?** tools could be a separate input parameter, not in plugs.

**evidence from code**: BrainAtomPlugs at line 10-26 already exists with `output?: never` as a placeholder. tools follows this pattern. the `plugs` parameter in BrainAtom.ask (line 53) accepts TPlug, which extends BrainAtomPlugs.

**architectural fit**: plugs represents "capabilities plugged into the brain." tools are capabilities. a separate `tools` parameter would fragment the configuration surface.

**verdict**: valid. follows extant pattern. no change needed.

---

### 2. call.tools is an array

**assumption**: tool calls come in arrays (multiple parallel calls).

**what if single call?** some providers might only support one tool call at a time.

**evidence from code**: BrainOutput.ts line 28 defines `{ tools: ToolInvocation[] }`. this matches provider APIs:
- anthropic: response.content can have multiple `tool_use` blocks
- openai: response.tool_calls is an array

**counterexample check**: could a provider return a single tool call not in an array? anthropic and openai both use arrays. a single call is just `[call]`. the array pattern is universal.

**verdict**: valid. array handles both single and parallel cases.

---

### 3. ToolInvocation.input is `unknown`

**assumption**: input type is unknown — callers must validate.

**what if we typed it more strictly?** could use generics like `ToolInvocation<TInput>`.

**evidence from code**: ToolInvocation.ts line 9 uses `<TInput = unknown>`. the generic exists but defaults to unknown. why?

**analysis**: at runtime, the brain picks which tool to call. the invocation arrives as JSON from the provider. the caller must:
1. match invocation.slug to a ToolDefinition
2. validate invocation.input against that tool's schema

there's no compile-time way to know which tool the brain will choose. `unknown` forces this runtime dispatch pattern.

**could a stricter type work?** only if we had a union of all possible tool inputs. but that adds complexity without safety benefit — the caller still must dispatch by slug.

**verdict**: valid. `unknown` with runtime validation is the right choice.

---

### 4. output can be null when tools are plugged

**assumption**: brain may return only tool calls, no direct output.

**what if brain always returns both?** then null checks would be unnecessary.

**evidence from code**: BrainOutput.ts line 35-38 defines:
```ts
export type AsBrainOutputOutputFor<TOutput, TPlug> =
  TPlug extends { tools: ToolDefinition[] } ? TOutput | null : TOutput;
```

**evidence from providers**:
- anthropic: `stop_reason: "tool_use"` means brain returned only tool calls, no text
- openai: `finish_reason: "tool_calls"` same behavior

**real scenario**: "what files are in src/?" → brain returns only `[{ slug: 'ls', input: { path: 'src/' } }]` without text answer. output is null until tools execute.

**verdict**: valid. null output is a real case that callers must handle.

---

### 5. BrainRepl doesn't need TPlug

**assumption**: only BrainAtom needs TPlug; BrainRepl manages tools internally.

**what if opposite?** BrainRepl could also have TPlug for consistency.

**evidence from code**: BrainRepl.ts line 59-68 shows ask() returns `BrainOutput<TOutput, 'repl'>` without TPlug. BrainReplPlugs.ts has `toolboxes` (MCP servers, etc.) not `tools`.

**architectural difference**:
- BrainAtom: single-turn, stateless → must expose tool calls for caller to execute
- BrainRepl: agentic loop → executes tools internally, returns final answer

**the actorAsk cast**: actorAsk.ts line 43-48 casts to a common signature. this is a localized workaround, not a design flaw. the asymmetry reflects real behavioral differences.

**could TPlug on BrainRepl simplify?** no. BrainRepl callers never receive tool calls — the repl handles them. TPlug would add unused type machinery.

**verdict**: valid. asymmetry is intentional and correct.

---

### 6. exid is always a string

**assumption**: provider IDs are strings.

**what if numeric?** would need different type.

**evidence from code**: ToolInvocation.ts line 16 defines `exid: string`.

**evidence from providers**:
- anthropic: `tool_use.id` = `"toolu_01A09q90qw90lq917835lh"` (string)
- openai: `tool_call.id` = `"call_abc123"` (string)

**counterexample search**: are there providers with numeric IDs? checked anthropic, openai, cohere, google — all use string IDs. string is the safe universal choice.

**verdict**: valid. string covers all known providers.

---

### 7. caller executes tools, not the atom

**assumption**: BrainAtom requests tool calls, caller executes them.

**what if atom executed tools directly?** would need tool execution context, side effects, state.

**evidence from architecture**:
- BrainAtom is a stateless inference endpoint (domain.objects/BrainAtom.ts)
- tool execution has side effects (file writes, API calls, db queries)
- atoms don't have access to tool implementations — they only see definitions

**the separation**:
```
atom.ask({ plugs: { tools } })
  → brain returns: { call: { tools: [invocation] }, output: null }
  → caller executes: tool(invocation.input)
  → caller continues: atom.ask({ on: { episode }, plugs: { tools }, prompt: toolResult })
```

**could atom execute tools?** only if we passed tool implementations. but then atom wouldn't be a "single-turn inference" — it would be a mini-repl. that's what BrainRepl is for.

**verdict**: valid. atoms request, callers execute. this separation is fundamental.

## conclusion

reviewed 7 assumptions with code evidence and provider documentation:

1. TPlug in BrainAtomPlugs — valid (follows extant pattern)
2. call.tools as array — valid (matches all providers)
3. input as unknown — valid (forces runtime dispatch)
4. output nullable with tools — valid (real provider behavior)
5. BrainRepl without TPlug — valid (intentional asymmetry)
6. exid as string — valid (universal across providers)
7. caller executes tools — valid (preserves atom statelessness)

no issues found. no changes needed. the implementation is architecturally sound.
