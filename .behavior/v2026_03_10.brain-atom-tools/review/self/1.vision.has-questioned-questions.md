## question 1: BrainExchange extension vs string serialization

**from vision**: should we extend BrainExchange to handle structured tool flow, or is string serialization acceptable?

### can this be answered via research?

**yes, partially.** provider APIs give insight into how tool flow is structured:

**anthropic** ([tool use docs](https://platform.claude.com/docs/en/agents-and-tools/tool-use/overview)):
- tool calls are `tool_use` content blocks with `id`, `name`, `input`
- tool results are `tool_result` content blocks with `tool_use_id`, `content`
- both are structured, not strings

**openai** ([function call docs](https://developers.openai.com/api/docs/guides/function-calling/)):
- tool calls have `id`, `function.name`, `function.arguments` (JSON string)
- tool results reference `call_id` to pair with calls
- structured pair is essential for parallel calls

### can this be answered via logic?

**yes.** the logic favors structured:

1. **parallel tool calls require correlation**: both providers use IDs to correlate calls with results. string serialization loses this.

2. **replay/observability**: structured tool flow enables proper weave observation. strings obscure what happened.

3. **provider translation**: rhachet must translate to/from provider formats. structured internal format makes this cleaner.

### self-answer

**recommend option A: extend BrainExchange with structured tool flow.**

rationale:
- providers use structured formats
- parallel calls require ID correlation
- observability benefits from structure
- backward compatible if fields are optional

proposed extension:
```ts
interface BrainExchange {
  hash: string;
  input: string;
  output: string;
  exid: string | null;
  // new optional fields
  toolCalls?: { id: string; name: string; input: unknown }[];
  toolResults?: { id: string; result: unknown }[];
}
```

**verdict**: ✅ self-answered. recommend structured extension.

## question 2: parallel vs sequential tool calls

**from vision**: should we support parallel tool calls in a single response?

### can this be answered via research?

**yes.**

**anthropic**: supports multiple `tool_use` blocks in one response

**openai** ([parallel tool calls](https://community.openai.com/t/parallel-function-calling/626868)):
- `tool_calls` is an array
- can set `parallel_tool_calls: false` to limit to one
- all modern models support parallel calls

### can this be answered via logic?

**yes.** the contract already supports parallel:

```ts
interface BrainOutput {
  call?: {
    tools: ToolInvocation[];  // array = parallel
  };
}
```

the question is really: does rhachet need to do extra work?

**no.** the array handles parallel. orchestration is caller's concern.

### self-answer

**parallel is already supported by design.**

- `calls` is an array — multiple calls in one response
- caller decides: execute in parallel or sequential
- no rhachet changes needed

**verdict**: ✅ self-answered. already supported.

## question 3: strict mode for schema enforcement

**from vision**: should we expose openai's `strict` parameter?

### can this be answered via research?

**yes.**

**openai** ([function call docs](https://developers.openai.com/api/docs/guides/function-calling/)):
> `strict: true` ensures function calls reliably adhere to the function schema, instead of best effort. recommended to always enable.

**anthropic**: no equivalent `strict` parameter. schema adherence is default behavior.

### can this be answered via logic?

**yes.**

- `strict` is openai-specific
- rhachet abstracts providers
- optional field doesn't force provider couple
- can be ignored for anthropic, applied for openai

### self-answer

**include as optional field.**

```ts
interface ToolDefinition {
  name: string;
  description: string;
  inputSchema: z.Schema<unknown>;
  strict?: boolean;  // openai-specific, ignored for other providers
}
```

**verdict**: ✅ self-answered. include as optional.

## question 4 (from assumptions): output type when tool calls present

**from assumptions review**: what is `output: TOutput` when brain requests tool calls?

### can this be answered via research?

**yes.**

**anthropic**: response can have both `text` blocks AND `tool_use` blocks. content and tool calls can coexist.

**openai**: `content` may be null when only tool calls, or populated with reason alongside tool calls.

### can this be answered via logic?

**partially.** options:

A. `output: TOutput | null` — null when only tool calls
B. `output: TOutput` always, possibly empty object
C. union type: `{ output: TOutput; tool: null } | { output: null; tool: { calls } }`

**A is cleanest**:
- explicit: null = "brain didn't give final answer"
- callers can check: `if (result.output === null) { /* handle tools */ }`
- matches provider semantics (content can be null/empty with tool calls)

### self-answer

**recommend `output: TOutput | null`.**

when brain requests tool calls without final answer, output is null. when brain answers directly (or answers alongside tool calls), output is populated.

```ts
interface BrainOutput<TOutput, TBrainGrain> {
  output: TOutput | null;  // null when pending tool results
  call?: { tools: ToolInvocation[] };
  // ...
}
```

**verdict**: ⚠️ partially answered. recommend null, but wisher should confirm.

## question 5 (from assumptions): zod vs JSON Schema for tool definitions

**from assumptions review**: should tool inputSchema use zod or JSON Schema?

### can this be answered via research?

**yes.**

**providers use JSON Schema**:
- anthropic: `input_schema` is JSON Schema
- openai: `parameters` is JSON Schema

**rhachet currently uses zod** for output schemas — but zod can convert to JSON Schema via `zod-to-json-schema`.

### can this be answered via logic?

**yes.** options:

A. **zod only**: consistent with rhachet output schemas
B. **JSON Schema only**: aligned with providers
C. **both**: accept either, convert as needed

**A is preferred** for consistency:
- rhachet already uses zod for output schemas
- `zod-to-json-schema` converts at provider boundary
- callers get zod benefits (inference, validation)
- provider alignment handled internally

### self-answer

**recommend zod for consistency.**

rhachet converts to JSON Schema when it calls providers. callers use zod throughout.

```ts
interface ToolDefinition {
  inputSchema: z.Schema<unknown>;  // zod, converted to JSON Schema for providers
}
```

**verdict**: ✅ self-answered. zod for consistency.

## second pass: deeper reflection

after re-reading, i found my answers were too quick. let me stress-test them.

### issue 1: output: TOutput | null is a break change

my recommendation of `output: TOutput | null` would break every caller.

**today**: callers know output is `TOutput`, no null check needed:
```ts
const { output } = await atom.ask({ ... });
console.log(output.summary);  // safe
```

**with my change**: all callers must check null even if they don't use tools:
```ts
const { output } = await atom.ask({ ... });
if (output === null) throw new Error('unexpected');  // forced
console.log(output.summary);
```

**better approach**: conditional type based on whether tools are plugged:

```ts
type BrainOutput<TOutput, TPlugs> = TPlugs extends { tools: ToolDefinition[] }
  ? { output: TOutput | null; call?: { tools: ToolInvocation[] } }
  : { output: TOutput; tool?: never };
```

this preserves backward compatibility:
- no tools plugged → output is `TOutput`, no null
- tools plugged → output is `TOutput | null`, caller must handle

**revised verdict**: ⚠️ this is complex type engineering. wisher should confirm approach.

### issue 2: BrainExchange hash with tool fields

i said "backward compatible if fields are optional" but didn't think through hash:

```ts
interface BrainExchange {
  hash: string;  // content-derived
  // ...
  toolCalls?: [...];
  toolResults?: [...];
}
```

if hash is content-derived:
- does hash include toolCalls/toolResults?
- if yes: same input/output with different tool calls → different hash
- if no: two exchanges that differ only by tool calls → same hash (collision)

**resolution**: hash MUST include tool fields when present. this is implementation detail, not contract question.

### issue 3: zod vs JSON Schema round-trip

i said "zod for consistency" but consider:
- caller has JSON Schema from OpenAPI spec
- must convert to zod to pass to rhachet
- rhachet converts back to JSON Schema for provider

this is a wasteful round-trip.

**better approach**: accept both, prefer zod for new tools:

```ts
interface ToolDefinition {
  inputSchema: z.Schema<unknown> | JSONSchema;  // accept both
}
```

rhachet detects type and converts as needed.

**revised verdict**: ⚠️ "zod only" was too quick. consider accept both.
