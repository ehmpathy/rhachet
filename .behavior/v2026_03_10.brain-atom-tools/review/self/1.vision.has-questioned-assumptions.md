this review surfaces hidden assumptions in the vision for BrainAtom tool support.

## critical assumption discovered: output is always typed, even with tool calls

### what do we assume here without evidence?

the vision shows:

```ts
const answer = await atom.ask<{ files: string[] }>({
  schema: { output: z.object({ files: z.array(z.string()) }) },
  plug: { tools: [...] },
  ...
});

if (result.call?.tools.length) {
  // tool calls requested
}
// result.output = { files: ['index.ts'] }  // always typed?
```

we assume `result.output: TOutput` is always typed. but when the brain requests tool calls instead of direct answer, what is `output`?

### what evidence supports this assumption?

none. the vision marks this as "awkward" (lines 336-350) but doesn't resolve it.

### what if the opposite were true?

option A: output is `TOutput | null` when tool calls present
- cleaner semantics: "brain didn't answer, it asked for tools"
- caller knows definitively: `if (result.output === null) { /* handle tools */ }`

option B: output is always `TOutput`, possibly empty/default
- backward compatible with current behavior
- but what's the "default" for arbitrary schemas?

option C: output and call.tools are mutually exclusive via union type
- `type BrainResult = { output: TOutput; tool: null } | { output: null; tool: { calls: [...] } }`
- most precise, but most complex

### did the wisher actually say this, or did we infer it?

the wisher didn't address this. they focused on adding call.tools, not on how output behaves when tools are called.

**verdict**: ⚠️ critical gap. must resolve before implementation.

## hidden assumption: zod schemas for tool definitions

### what do we assume here without evidence?

the vision uses zod schemas for tool input:

```ts
interface ToolDefinition {
  inputSchema: z.Schema<unknown>;
}
```

### what evidence supports this assumption?

rhachet uses zod for output schemas, so consistency suggests zod for input schemas.

### what if the opposite were true?

providers use JSON Schema, not zod:
- anthropic: `input_schema: { type: 'object', properties: {...} }`
- openai: `parameters: { type: 'object', properties: {...} }`

if we require zod, callers must:
- convert JSON Schema → zod (non-trivial)
- maintain two schema formats
- lose runtime JSON Schema tooling

alternative: accept JSON Schema directly, or zod-to-json-schema

### did the wisher actually say this, or did we infer it?

the wisher used zod in their example but didn't mandate it. they noted:
> these should align with anthropic and openai tool formats

which suggests JSON Schema alignment might be preferred.

**verdict**: ⚠️ assumption worth questioning. JSON Schema might be more appropriate.

### what do we assume here without evidence?

```ts
interface ToolInvocation {
```

we assume rhachet parses the tool arguments. but:
- who validates against inputSchema?
- what happens if validation fails?

### what evidence supports this assumption?

providers return parsed JSON in their responses. rhachet would pass this through.

### what if the opposite were true?

if rhachet doesn't validate:
- caller must validate call.tools[].input against their schema
- invalid inputs reach caller code

if rhachet does validate:
- invalid tool calls become errors before caller sees them
- but this adds rhachet complexity

### did the wisher actually say this, or did we infer it?

the wisher didn't specify. this is implementation detail.

**verdict**: ⚠️ needs clarification. who validates tool inputs?

## assumption 1: tools are plugged per-ask, not registered globally

### what do we assume here without evidence?

the vision assumes tools are passed per-call via `plug.tools`:

```ts
atom.ask({
  plug: { tools: [...] },  // per-call
  ...
})
```

not registered once and reused:

```ts
const atom = genBrainAtom({ tools: [...] });  // global
atom.ask({ ... });  // tools already registered
```

### what evidence supports this assumption?

rhachet's philosophy: BrainAtom is stateless. each call is independent. no mutable state between calls.

from `define.rhachet.v3.md`:
> BrainAtom: single inference call, returns output directly, no series awareness — stateless

### what if the opposite were true?

if tools were global:
- atoms would have state
- callers couldn't change tools mid-conversation
- some usecases become awkward (e.g., progressive tool disclosure)

but global registration could:
- reduce call-site verbosity
- enable tool cache at provider level

### did the wisher actually say this, or did we infer it?

the wisher's contract shape implies per-call:

```ts
ask<TOutput>(input: {
  plug?: {
    tools?: ToolDefinition[];  // inside input
  };
  ...
})
```

but they didn't explicitly state "tools must be per-call, not global."

### what exceptions or counterexamples exist?

some usecases might prefer global:
- long-lived agents with stable toolsets
- performance optimization (avoid re-send of tool schemas)

but these are orchestration concerns, not atom concerns. the caller can cache and re-pass tools.

**verdict**: ⚠️ assumption is reasonable but worth confirm. question for wisher: should atoms support global tool registration as an alternative?

## assumption 2: tool results are injected via episode continuation

### what do we assume here without evidence?

the vision assumes tool results flow back via the episode:

```ts
// step 1: get tool call
const answer = await atom.ask({ ... });

// step 2: execute tool
const toolResult = await executeTool(result.call.tools[0]);

// step 3: continue with result (inject into prompt)
const continued = await atom.ask({
  prompt: `tool result: ${toolResult}`,
  on: { episode: result.episode },
  ...
});
```

not via a dedicated API:

```ts
const continued = await atom.askWithToolResults({
  episode: result.episode,
  toolResults: [{ id: 'call_123', result: toolResult }],
});
```

### what evidence supports this assumption?

rhachet's statelessness: BrainAtom doesn't "remember" tool calls. the caller manages state.

from the handoff:
> bhrain will conform to whatever rhachet decides

the wisher is flexible on this.

### what if the opposite were true?

if atoms had dedicated tool result handle:
- cleaner API for tool flow
- explicit correlation between call ID and result
- provider-level optimization possible

but:
- atoms would need to "know" about prior tool calls
- breaks stateless philosophy
- duplicates orchestration logic

### did the wisher actually say this, or did we infer it?

the wisher posed this as a question:

> options:
> - **A**: extend BrainExchange to support structured tool flow
> - **B**: serialize tool flow into input/output strings (lossy but backward-compatible)

they explicitly asked rhachet to decide.

### what exceptions or counterexamples exist?

provider SDKs often have explicit tool result APIs:
- anthropic: tool results as separate content blocks
- openai: tool messages with `tool_call_id`

the "inject into prompt" pattern is a simplification that may lose fidelity.

**verdict**: ✅ assumption is sound as default. but the vision correctly identifies this as awkward and poses it as an open question.

## assumption 3: string serialization for tool results in BrainExchange

### what do we assume here without evidence?

the vision assumes tool results serialize to strings in BrainExchange:

```ts
interface BrainExchange {
  hash: string;
  input: string;   // tool result becomes a string here
  output: string;
  exid: string | null;
}
```

not structured:

```ts
interface BrainExchange {
  hash: string;
  input: string;
  output: string;
  exid: string | null;
  toolCall?: { name: string; input: unknown };      // structured
  toolResult?: { id: string; result: unknown };     // structured
}
```

### what evidence supports this assumption?

the extant BrainExchange contract uses strings. backward compatibility matters.

### what if the opposite were true?

structured tool flow would:
- enable better tools (replay, debug, analytics)
- preserve type information
- match provider SDK semantics

but:
- breaks backward compatibility
- increases complexity
- may be over-engineered for initial release

### did the wisher actually say this, or did we infer it?

the wisher explicitly posed both options:

> - **A**: extend BrainExchange to support structured tool flow
> - **B**: serialize tool flow into input/output strings

this is a design choice, not an assumption.

### what exceptions or counterexamples exist?

sophisticated use cases (observability, audit, replay) benefit from structured data. simple use cases don't need it.

**verdict**: ✅ correctly identified as an open question in the vision. not an assumption.

## assumption 4: provider-agnostic tool format

### what do we assume here without evidence?

the vision assumes rhachet defines its own ToolDefinition:

```ts
interface ToolDefinition {
  name: string;
  description: string;
  inputSchema: z.Schema<unknown>;
  strict?: boolean;
}
```

not pass-through provider formats:

```ts
// anthropic format
{ name, description, input_schema: { type: 'object', properties: {...} } }

// openai format
{ type: 'function', function: { name, description, parameters: {...} } }
```

### what evidence supports this assumption?

rhachet's core value is abstraction. from briefs:
> rhachet's role is abstraction

provider-specific formats would leak implementation details.

### what if the opposite were true?

pass-through would:
- support provider-specific features
- avoid translation overhead
- risk future format changes

but:
- callers must know provider details
- no portability across providers
- defeats rhachet's purpose

### did the wisher actually say this, or did we infer it?

the wisher's proposed types are provider-agnostic:

```ts
interface ToolDefinition {
  name: string;
  description: string;
  inputSchema: z.Schema<unknown>;
  strict?: boolean;
}
```

they also noted:
> these should align with anthropic and openai tool formats for cross-provider compatibility

### what exceptions or counterexamples exist?

some provider features may not map cleanly:
- anthropic's `cache_control` on tools
- openai's `strict` mode (included as optional)
- future provider-specific extensions

**verdict**: ✅ assumption is sound. acknowledged risk in vision ("some provider features may not map cleanly").

### critical findings

1. **output type with tool calls**: the vision doesn't specify what `output: TOutput` contains when the brain requests tool calls instead of answering. this is a critical contract question.

2. **schema format**: zod vs JSON Schema is more than a preference — it affects developer experience and provider alignment. the wisher's "align with anthropic and openai" note suggests JSON Schema might be more appropriate.

3. **validation responsibility**: who validates tool call inputs against their schemas? rhachet or caller? this affects error surfaces and caller complexity.

### recommendations

before implementation, resolve:
1. output type behavior when call.tools is populated
2. schema format (zod vs JSON Schema vs both)
3. validation responsibility for tool inputs
