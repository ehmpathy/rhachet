### 2. schema pattern

**search**: `grep -n "schema:\s*\{" src/domain.objects/`

**found**: consistent `schema: { x }` nested structure

```
src/domain.objects/BrainAtom.ts:57:      schema: { output: z.Schema<TOutput> };
```

### 3. exid convention

**search**: `grep -n "exid" src/domain.objects/`

**found**: extant convention for provider-assigned external IDs

```
src/domain.objects/BrainExchange.ts:34:  exid: string | null;
src/domain.objects/BrainEpisode.ts:28:  exid: string | null;
src/domain.objects/BrainSeries.ts:28:  exid: string | null;
src/domain.objects/keyrack/KeyrackKeyHost.ts:22:  exid: string | null;
```

**our choice**: ToolInvocation.exid ✓ follows extant convention

note: extant usages have `exid: string | null` because provider may not assign. ToolInvocation uses `exid: string` (required) because providers always assign tool call IDs.

| wish proposed | implemented | why |

these are refinements for consistency, not arbitrary divergences.

## conclusion

all names follow extant conventions:
- file names: {Concept}.ts
- conditional types: As{X}For<T>
- generic parameters: T{Concept}
- fields: camelCase, descriptive
- schema: schema.{x} pattern

deviations from wish were for alignment with extant codebase conventions.
