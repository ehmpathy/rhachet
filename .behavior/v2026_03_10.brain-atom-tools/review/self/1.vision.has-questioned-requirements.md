# self-review: has-questioned-requirements

this review covers requirements from the wish for BrainAtom tool support.

---

## requirement 1: research how/why brain suppliers support tool calls

### who said this was needed? when? why?

the wisher stated this in the handoff:

> 1. research how and why brain suppliers like openai and anthropic support tool calls via their direct api calls against direct inference

this was part of the original wish, stated at the start.

### what evidence supports this requirement?

the evidence is implicit: to design a good abstraction over provider tool APIs, we need to comprehend how they work under the hood. without this research:
- we might design contracts that don't map to provider semantics
- we might miss important provider-specific behaviors
- we might create leaky abstractions

### what if we didn't do this?

we'd design blind. the contract shapes would be guesses rather than informed decisions. we might:
- miss parallel tool call support (both providers support it)
- miss the strict schema enforcement feature (openai-specific)
- create awkward serialization patterns that fight provider formats

### is the scope too large, too small, or misdirected?

scope is appropriate. the research should produce a brief that captures:
- why tool calls are special (vs just JSON output)
- how providers handle them differently
- what abstractions make sense

### could we achieve the goal in a simpler way?

no — comprehension of the problem space is prerequisite to solution design. we could skip to implementation, but that risks rework.

**verdict**: ✅ requirement justified. research first, design second.

## requirement 2: add tools to BrainAtomPlug

### who said this was needed? when? why?

the wisher explicitly requested:

> add `tools?: ToolDefinition[]` to BrainAtomPlug

rationale from handoff:

> **why plug?** tools are plugins. you plug tools into the brain to use them.

### what evidence supports this requirement?

bhrain's BrainArch1 is blocked without this:

> BrainArch1 needs `tools[]` definitions in the request

the handoff shows a clear gap:

| BrainArch1 needs | BrainAtom.ask() provides |
|------------------|--------------------------|
| `tools[]` definitions | none (plug.tools absent) |

### what if we didn't do this?

bhrain can't use rhachet's BrainAtom. they'd be forced to:
- use raw provider SDKs directly
- lose rhachet's abstractions (briefs, episodes, metrics)
- duplicate tool format translation per provider

### is the scope too large, too small, or misdirected?

scope is minimal and precise. one optional field in one interface. no API-break changes.

### could we achieve the goal in a simpler way?

this IS the simplest way. alternatives considered:
- separate method `atom.askWithTools()` — adds API surface unnecessarily
- global tool registry — adds state, violates atom's stateless philosophy

**verdict**: ✅ requirement justified. minimal addition, clear value.

## requirement 3: add call.tools to BrainOutput

### who said this was needed? when? why?

the wisher explicitly requested:

> add `call?: { tools: ToolInvocation[] }` to BrainOutput

rationale from handoff:

> `toolCalls[]` in response — needed for brain to express tool call requests

### what evidence supports this requirement?

without tool calls in output, the brain can't communicate:
- "i want to call this tool"
- "with these arguments"
- "before i can answer"

the handoff shows this gap:

| BrainArch1 needs | BrainAtom.ask() provides |
|------------------|--------------------------|
| `toolCalls[]` in response | structured output via schema |

### what if we didn't do this?

the brain would have to encode tool calls in the structured output schema. but:
- callers would need custom schemas per-use
- tool call semantics would be ad-hoc
- no standard way to correlate calls with results

### is the scope too large, too small, or misdirected?

scope is minimal. one optional field in the response. backward compatible — extant callers see `tool: undefined`.

### could we achieve the goal in a simpler way?

alternatives considered:
- encode tool calls in output schema — shifts complexity to callers
- return tool calls in a separate channel — overcomplicates API

the proposed `call?: { calls: [...] }` is the cleanest approach.

**verdict**: ✅ requirement justified. minimal addition, standard semantics.

## requirement 4: possibly extend BrainExchange

### who said this was needed? when? why?

the wisher posed this as a question, not a requirement:

> options:
> - **A**: extend BrainExchange to support structured tool flow
> - **B**: serialize tool flow into input/output strings (lossy but backward-compatible)
>
> bhrain will conform to whatever rhachet decides.

### what evidence supports this requirement?

tool flow has more structure than simple string input/output:
- tool call: `{ name, args }` in brain output
- tool result: `{ toolCallId, result }` in next input
- multi-tool: single turn can have multiple calls

string serialization works but loses type information.

### what if we didn't do this?

string serialization is viable. bhrain can serialize tool results into prompts:

```ts
prompt: `Tool result for call_123: {"files": ["a.ts", "b.ts"]}`
```

lossy, but functional.

### is the scope too large, too small, or misdirected?

this is appropriately scoped as a question, not a requirement. the vision document captures both options and defers to wisher input.

### could we achieve the goal in a simpler way?

yes — option B (string serialization) is simpler. option A (structured) is cleaner but more work.

**verdict**: ⚠️ this is a question, not a requirement. correctly categorized. decision deferred to wisher.
