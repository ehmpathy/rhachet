# self-review: has-questioned-deletables

> can any components be removed or simplified?

---

## components reviewed

### 1. ToolDefinition.ts

**can this be removed?** no.

this is the core contract for how callers define tools. without it, there's no type-safe way to plug tools into BrainAtom.

**simplest version?** current version is minimal: slug, description, schema.input. each field is necessary:
- slug: identifies the tool
- description: helps brain understand when to use it
- schema.input: validates invocation arguments

---

### 2. ToolInvocation.ts

**can this be removed?** no.

this is the core contract for tool call responses. the brain needs to express "call this tool with these args."

**simplest version?** current version is minimal: exid, slug, input. each field is necessary:
- exid: provider correlation ID (required for continuation)
- slug: which tool
- input: parsed arguments

---

### 3. BrainAtomPlugs.tools

**can this be removed?** no.

this is how tools are plugged into the brain. the alternative would be putting tools in input directly, but plugs is the established pattern for brain configuration.

---

### 4. conditional types (AsBrainOutputCallFor, AsBrainOutputOutputFor)

**can these be removed?** no.

these enable progressive complexity — the key design goal. without them:
- callers without tools would need null checks
- backward compatibility would break

**can they be simplified?** they're already minimal — single conditional expressions.

---

### 5. TPlug generic

**can this be removed?** no.

this is what enables the conditional types to narrow based on whether tools are plugged. without the generic, we'd either:
- always require null checks (bad DX for simple cases)
- never narrow types (lose type safety for tool cases)

---

### 6. actorAsk cast

**can this be removed?** only if we also add TPlug to BrainRepl.ask.

the cast exists because BrainAtom.ask and BrainRepl.ask now have different generic signatures. TypeScript can't call a union of functions with incompatible generics.

**alternatives considered:**
1. add TPlug to BrainRepl — bigger change, but more consistent
2. extract common "Askable" interface — adds abstraction layer
3. keep the cast — minimal, localized, explains itself

current choice (cast) is the smallest change. the cast is documented and limited to one file.

---

## conclusion

**no deletables found.** each component exists for a specific reason:
- domain objects define the contract
- conditional types enable progressive complexity
- TPlug generic enables type narrowing
- actorAsk cast is a localized workaround for union type limitations

the implementation is minimal. no unused code, no over-engineering.
