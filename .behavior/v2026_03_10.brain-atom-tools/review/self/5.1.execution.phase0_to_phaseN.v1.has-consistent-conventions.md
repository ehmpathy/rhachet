## name conventions analyzed

### Brain* prefix

**extant pattern**: all brain-related interfaces use `Brain` prefix:
- BrainAtom, BrainRepl, BrainSpec, BrainOutput, BrainEpisode, BrainExchange, BrainSeries, BrainPlug, BrainReplPlugs, BrainHook, BrainHookFilter, BrainHooksAdapter

**our names**:
- BrainPlug (renamed from BrainAtomPlugs)
- BrainPlugToolDefinition
- BrainPlugToolInvocation

**verdict**: consistent. `Brain` prefix preserved. `BrainPlug` is the parent, `BrainPlugTool*` are children.

---

### AsBrainOutput*For conditional types

**extant pattern**: `AsBrainOutputSeriesFor<T>` in BrainOutput.ts

**our names**:
- AsBrainOutputCallFor
- AsBrainOutputOutputFor

**verdict**: consistent. same prefix `AsBrainOutput`, same suffix `For`, same file location.

### slug vs name

**extant usage**: searched for `slug` vs `name` in domain objects:
- BrainAtom: `slug: string` (unique identifier)
- BrainSupplierSlug: uses `slug`
- RoleSkill: uses `slug`
- BrainRepl: `slug: string`

**our usage**:
- BrainPlugToolDefinition: `slug: string` (unique identifier)
- BrainPlugToolInvocation: `slug: string` (which tool)

**verdict**: consistent. `slug` is the extant term for unique identifiers in this codebase.

---

### exid (external ID)

**extant usage**: searched for `exid` in domain.objects/:
- BrainExchange: `exid: string | null`
- BrainEpisode: `exid: string | null`
- BrainSeries: `exid: string | null`

**our usage**:
- BrainPlugToolInvocation: `exid: string` (not nullable)

**difference noted**: extant `exid` fields are nullable, ours is required.

**analysis**: this difference is intentional, not a convention divergence:
- BrainExchange.exid is nullable because suppliers may not provide it
- BrainPlugToolInvocation.exid is required because providers always provide tool call IDs
  - anthropic: tool_use blocks always have `id`
  - openai: tool_calls always have `id`

**verdict**: consistent in term usage (`exid`), deliberately different in nullability (domain-appropriate).

---

### schema.input vs input_schema

**provider APIs use**: `input_schema` (anthropic), `parameters` (openai)

**our usage**: `schema: { input: z.Schema }` — follows extant rhachet pattern of `schema: { output }` in BrainAtom.ask

**verdict**: consistent with rhachet's extant `schema` object pattern, not provider APIs.

## reflection

i questioned each name choice against extant conventions:

1. **Brain* prefix**: grepped for `^export interface Brain` — found 15+ matches. we follow the same prefix.

2. **AsBrainOutput*For**: grepped for `^export type As` — found only our types and one extant (`AsBrainOutputSeriesFor`). we match it exactly.

3. **slug**: grepped for `slug:` — found 40+ usages across domain objects. `slug` is the established term. we use it.

4. **exid**: grepped for `exid:` — found 20+ usages. noticed extant uses `string | null`, ours is `string`. investigated: provider APIs always supply tool call IDs, so non-nullable is correct. term consistent, nullability domain-appropriate.

5. **schema.input**: mirrors extant `schema: { output }` in BrainAtom.ask. structural consistency.

i found one deliberate difference (exid nullability) and articulated why it holds.

---

## verdict

**pass** — all names follow extant conventions. no new terms introduced where extant terms exist.
