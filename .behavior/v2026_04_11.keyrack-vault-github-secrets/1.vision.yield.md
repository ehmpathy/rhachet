# vision: keyrack vault github.secrets

## the outcome world

### before: manual json formatting pain

setting up github app credentials for github actions cicd is a multi-step pain:

1. create github app in org
2. download private key (.pem)
3. find app id and installation id via api or ui
4. manually construct json blob with exact field names
5. base64 encode or escape newlines in the pem
6. paste into github secrets ui
7. pray you got the format right

if the format is wrong, you find out when the workflow fails at 2am.

### after: guided setup flows to github

```bash
$ rhx keyrack set --key GITHUB_APP_CREDS --vault github.secrets --mech EPHEMERAL_VIA_GITHUB_APP

🔐 keyrack set ehmpathy.all.GITHUB_APP_CREDS via EPHEMERAL_VIA_GITHUB_APP
   │
   ├─ which github org?
   │  ├─ options
   │  │  ├─ 1. ehmpathy
   │  │  ├─ 2. otherorg
   │  └─ choice: 1
   │     └─ ehmpathy ✓
   │
   ├─ which github app?
   │  ├─ options
   │  │  ├─ 1. seaturtle-bot (id: 123456)
   │  │  ├─ 2. other-app (id: 654321)
   │  └─ choice: 1
   │     └─ seaturtle-bot ✓
   │
   ├─ which github app secret?
   │  └─ private key path (.pem): ~/Downloads/seaturtle-bot.pem
   │
   └─ ✓ pushed to github.secrets (no roundtrip — write-only vault)

🔐 keyrack set (org: ehmpathy, env: all)
   └─ ehmpathy.all.GITHUB_APP_CREDS
      ├─ mech: EPHEMERAL_VIA_GITHUB_APP
      └─ vault: github.secrets
```

the mech adapter handles the json construction. keyrack handles the github api call. no manual formatting. no base64 encoding. no prayer.

### the "aha" moment

when the developer realizes they can run `rhx keyrack set --key GITHUB_APP_CREDS --vault github.secrets` and have the interactive prompts guide them through setting up github app credentials that flow directly into github secrets — no clipboard, no manual json, no second-guessing the format.

## user experience

### usecases

| goal | contract | outcome |
|------|----------|---------|
| set github app to gh secrets | `rhx keyrack set --key X --vault github.secrets --mech EPHEMERAL_VIA_GITHUB_APP` | guided setup → gh api → secret stored |
| set any key to gh secrets | `rhx keyrack set --key X --vault github.secrets` | prompt → gh api → secret stored |
| delete key from gh secrets | `rhx keyrack del --key X --vault github.secrets` | gh api DELETE → secret removed |
| check if key was set | `rhx keyrack status --key X` | shows `locked` (was set but not retrievable) |
| get key value | `rhx keyrack get --key X` | failfast: "github secrets cannot be retrieved" |

### contract: `--vault github.secrets`

**vault adapter declaration:**
```typescript
{
  name: 'github.secrets',
  set: async (input, context) => { ... },  // gh api PUT
  get: null,                                // write-only vault
  del: async (input, context) => { ... },  // gh api DELETE
}
```

`get: null` makes write-only explicit at the interface level — failfast happens at dispatch, not deep in the call stack.

**supported mechs:**
- `PERMANENT_VIA_REPLICA` — plain text secrets
- `EPHEMERAL_VIA_GITHUB_APP` — json blob with appId, installationId, privateKey

**inputs:**
- `--key`: the key name (becomes the github secret name)
- `--env`: environment (for keyrack slug; v1 writes to repo-level secrets only)
- `--mech`: mechanism (optional, inferred if unambiguous)

**outputs (set):**
```json
{
  "slug": "ehmpathy.all.GITHUB_APP_CREDS",
  "vault": "github.secrets",
  "mech": "EPHEMERAL_VIA_GITHUB_APP",
  "exid": "ehmpathy/rhachet.GITHUB_APP_CREDS"
}
```

**outputs (get):**
```json
{
  "status": "locked",
  "slug": "ehmpathy.all.GITHUB_APP_CREDS",
  "message": "github secrets cannot be retrieved via api",
  "fix": null
}
```

### timeline

1. **set**: mech guided setup → value acquired → gh api writes secret
2. **list**: shows key in host manifest with `vault: github.secrets`
3. **status**: shows `locked` (we know it was set)
4. **get**: failfast (retrieval impossible by design)
5. **unlock**: no-op (no local cache for github secrets)

## mental model

### how users describe it

> "keyrack can push secrets directly to github secrets so i don't have to copy-paste json blobs"

> "it's like the os.secure vault but writes to github instead of my local machine"

> "write-only vault — keyrack remembers what you set, but can't read it back"

### analogies

| concept | analogy |
|---------|---------|
| github.secrets vault | dropbox (upload works, download doesn't) |
| status = locked | "i know i mailed the letter, but i can't read it anymore" |
| exid | the "receipt" — proof you sent it, not the content itself |

### terms

| user term | keyrack term |
|-----------|--------------|
| "github secret" | vault entry with `vault: github.secrets` |
| "can't retrieve" | `status: locked` with `fix: null` |
| "secret reference" | `exid: owner/repo.SECRET_NAME` |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| use keyrack guided setup for github secrets | ✓ | mech prompts work unchanged |
| avoid manual json formatting | ✓ | mech constructs json internally |
| support EPHEMERAL_VIA_GITHUB_APP | ✓ | same adapter, different destination |
| prevent accidental exfiltration | ✓ | no get possible |
| audit trail | ✓ | host manifest records what was set |

### pros

- **reuses mech adapters**: no new acquisition logic needed
- **pit of success**: format is always correct (mech constructs it)
- **audit trail**: host manifest shows vault=github.secrets
- **consistent ux**: same `keyrack set` command, different destination
- **no exfiltration risk**: keyrack can't read github secrets back

### cons

- **no roundtrip verification**: can't confirm the value was written correctly
- **no rotation detection**: if someone changes the secret in github ui, keyrack doesn't know
- **get always fails**: users must remember this vault is write-only
- **daemon is useless**: no value to cache for github secrets

### edgecases and pit of success

| edgecase | behavior |
|----------|----------|
| user runs `get --key X` on github.secrets key | failfast: "github secrets cannot be retrieved" |
| user runs `unlock --key X` on github.secrets key | failfast: "github.secrets cannot be unlocked" |
| user runs `unlock --for repo` with github.secrets keys | skip silently (don't break workflow) |
| key already set in github | upsert semantics via gh api |
| user runs `del --key X` on github.secrets key | gh api DELETE → removes from github and host manifest |
| gh cli not authenticated | failfast: "gh auth required" |
| repo doesn't exist | failfast: "repo not found" |
| user lacks write access | failfast: "permission denied" |

## what is awkward?

### `status: locked` is semantically overloaded

for os.secure, `locked` means "was set but not unlocked for this session". for github.secrets, `locked` means "was set but can never be retrieved". these are different states masquerading as the same.

**tradeoff**: introducing a new status like `remote` adds complexity. using `locked` reuses extant semantics but loses precision.

**proposed resolution**: use `locked` but include `fix: null` to signal no unlock is possible.

### no roundtrip verification

every other vault can verify the set worked. github.secrets cannot. the user must trust the gh api response.

**tradeoff**: acceptable because gh api is reliable and returns errors on failure.

### unlock is meaningless

for github.secrets, `unlock` does not apply. the key is never "locked" in the daemon sense — it simply isn't stored locally.

**proposed resolution**:
- `unlock --key X` specifically → failfast: "github.secrets cannot be unlocked"
- `unlock --for repo` bulk unlock → skip silently (don't break workflow)

### daemon provides no value

the daemon caches unlocked values for fast access. for github.secrets, there is no local value to cache. the daemon sees `vault: github.secrets` and knows `get` will always fail.

**tradeoff**: acceptable. the daemon correctly reports `status: locked` because the value isn't available.
