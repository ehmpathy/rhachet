# vision: add onTalk hook to Role.build

## the outcome world

### day-in-the-life: before

a developer building the achiever role wants to accumulate user asks as they arrive — to track what the human requested vs what got done.

they write the hook command (`userpromptsubmit.ontalk.sh`), then realize rhachet doesn't support `onTalk`. so they:

1. create an init that uses `jq` to inject the hook into `.claude/settings.json`
2. document that users must run `rhachet roles init --role achiever` manually
3. accept that unlink won't clean up the hook (orphaned hooks accumulate)
4. accept that the hook is invisible to rhachet's hook inspection tools

the workaround works but feels wrong — they're bypassing the abstraction they should be using.

### day-in-the-life: after

the developer declares the hook directly in their role:

```typescript
export const ROLE_ACHIEVER: Role = Role.build({
  slug: 'achiever',
  hooks: {
    onBrain: {
      onTalk: [{
        command: './.agent/repo=bhrain/role=achiever/inits/claude.hooks/userpromptsubmit.ontalk.sh',
        timeout: 'PT5S',
      }],
    },
  },
});
```

`rhachet roles link --role achiever` syncs the hook to `.claude/settings.json` automatically. unlink removes it. the hook is visible in rhachet's hook tooling.

### the "aha" moment

"wait, i can just declare `onTalk` like i declare `onBoot`? and it handles the settings.json sync and cleanup? i don't need the init workaround?"

## user experience

### usecases

| usecase | goal | hook |
|---------|------|------|
| accumulate asks | track what human requested during session | `onTalk` |
| audit trail | log every prompt for compliance | `onTalk` |
| context capture | save prompt to context window before processing | `onTalk` |
| intent detection | classify prompt intent before brain processes | `onTalk` |

### contract inputs

```typescript
// role declaration
hooks: {
  onBrain: {
    onTalk: [{
      command: string;       // executable to run when user submits prompt
      timeout: IsoDuration;  // e.g., 'PT5S' for 5 seconds
    }];
  };
}
```

### contract outputs

the hook command receives the prompt via stdin (claude code's behavior). it can:
- read the prompt
- emit output to influence behavior
- fail to block the prompt (if exit code != 0)

### timeline

```
1. developer declares onTalk in Role.build
2. `rhachet roles link --role myrole` syncs hook to settings.json
3. user starts claude code session
4. user submits prompt → hook fires → command runs
5. `rhachet roles unlink --role myrole` removes hook from settings.json
```

## mental model

### how users describe it

"rhachet lets me run a command every time the user sends a message. i just add `onTalk` to my role's hooks."

### analogies

| rhachet term | claude code term | analogy |
|--------------|------------------|---------|
| `onBoot` | `SessionStart` | server startup |
| `onTool` | `PreToolUse` | middleware |
| `onStop` | `Stop` | shutdown hook |
| `onTalk` | `UserPromptSubmit` | request interceptor |

`onTalk` is like express middleware that fires on every HTTP request — except the "request" is a user prompt.

### term mapping

| rhachet | claude code | when fires |
|---------|-------------|------------|
| onTalk | UserPromptSubmit | user submits a prompt (before brain processes) |

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| declare onTalk in Role.build | yes | symmetric with onBoot/onTool/onStop |
| map to UserPromptSubmit | yes | add entry to EVENT_MAP in translateHook |
| auto-sync on link | yes | add onTalk extraction to syncOneRoleHooksIntoOneBrainRepl |
| auto-cleanup on unlink | yes | extant pruneOrphanedRoleHooks handles it (no change needed) |
| remove init workaround | yes | achiever role can delete init.claude.hooks.sh |

### implementation notes

the sync machinery uses hardcoded event extraction, not dynamic iteration:

```typescript
// syncOneRoleHooksIntoOneBrainRepl.ts - extractDeclaredHooks()
for (const h of onBrain.onBoot ?? []) { ... }
for (const h of onBrain.onTool ?? []) { ... }
for (const h of onBrain.onStop ?? []) { ... }
// must add: for (const h of onBrain.onTalk ?? []) { ... }
```

this means implementation requires changes to:
1. `BrainHookEvent` type — add `'onTalk'`
2. `RoleHooksOnBrain` interface — add `onTalk` property
3. `extractDeclaredHooks` — add onTalk extraction block
4. `EVENT_MAP` — add `onTalk: 'UserPromptSubmit'`
5. `ClaudeCodeSettings` schema — add `UserPromptSubmit` event type

### pros

- **symmetry**: `onTalk` fits naturally alongside `onBoot`, `onTool`, `onStop`
- **discoverability**: developers see all hook options in one interface
- **lifecycle management**: link/unlink handles sync automatically
- **observability**: rhachet can inspect all hooks uniformly

### cons

- **name divergence**: rhachet uses `onTalk`, claude code uses `UserPromptSubmit`
  - mitigation: this divergence already extant (onBoot → SessionStart, etc.) and is documented

### pit of success

- developers can't forget to clean up hooks — unlink handles it
- developers can't accidentally orphan hooks — sync is idempotent
- developers can't misconfigure — type system enforces valid hook shape

## open questions & assumptions

### assumptions [all resolved]

1. **UserPromptSubmit behavior** [answered]: fires before brain acts, receives prompt via stdin as JSON with `prompt` field. can block via `decision: "block"` in output. confirmed via claude code docs.
2. **timeout semantics** [answered]: timeout works the same as other hooks — verified via translateHook.ts which applies timeout uniformly
3. **matcher semantics** [answered]: matcher field is silently ignored for UserPromptSubmit — hook fires on every prompt. use `*` wildcard.

### wisher decisions [all resolved]

1. **filter support** [answered]: no — UserPromptSubmit is a single event with no sub-events. filter.what not needed.
2. **PostUserPromptSubmit** [answered]: no — mirror onTool which only maps to PreToolUse. no onTalkAfter.
3. **opencode scope** [answered]: yes — support onTalk for opencode via `chat.message` event.

### implementation scope

onTalk maps to:
- claude code: `UserPromptSubmit`
- opencode: `chat.message`

changes needed:
1. `BrainHookEvent` type — add `'onTalk'`
2. `RoleHooksOnBrain` interface — add `onTalk` property
3. `extractDeclaredHooks` — add onTalk extraction block
4. claude code `EVENT_MAP` — add `onTalk: 'UserPromptSubmit'`
5. opencode adapter — add `onTalk: 'chat.message'` map
6. `ClaudeCodeSettings` schema — add `UserPromptSubmit` event type

## what is awkward?

### name: onTalk

`onTalk` is chosen for brevity and intuition ("when the user talks"). alternatives:

| name | pros | cons |
|------|------|------|
| `onTalk` | short, intuitive | "talk" might imply voice? |
| `onPrompt` | maps to "prompt" concept | could confuse with system prompt |
| `onAsk` | rhachet uses "ask" already | might confuse with ask command |
| `onMessage` | clear | verbose |
| `onUserPrompt` | explicit | verbose |

recommendation: stick with `onTalk` — it's short, the context makes it clear, and it fits the seaturtle "let's talk" vibe.

### asymmetry with PostToolUse [resolved]

claude code supports both `PreToolUse` and `PostToolUse`, but rhachet's `onTool` only maps to `PreToolUse`.

**decision:** mirror onTool — add `onTalk` only (maps to Pre variant). no onTalkAfter. post-hooks deferred to future work.

### filter semantics [resolved]

`onBoot` uses `filter.what` to distinguish SessionStart vs PreCompact vs PostCompact.

**decision:** no filter support for onTalk. UserPromptSubmit is a single event with no sub-events. matcher field is silently ignored by claude code.
