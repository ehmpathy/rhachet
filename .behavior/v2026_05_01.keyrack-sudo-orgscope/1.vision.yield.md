# vision: keyrack sudo org-scoped unlock

## the outcome world

### before

vlad is in `svc-ahction` repo, which declares `org: ahction` in its keyrack manifest.

```bash
$ rhx keyrack unlock --env sudo --key AWS_PROFILE

🔓 keyrack unlock
   ├─ ahbode.sudo.AWS_PROFILE    # wrong! i'm in ahction repo
   │  ├─ env: sudo
   │  ├─ org: ahbode
   │  └─ expires in: 30m
   └─ ahction.sudo.AWS_PROFILE    # correct
      ├─ env: sudo
      ├─ org: ahction
      └─ expires in: 30m
```

ahbode credentials unlocked when working in ahction repo = security violation. keys from org A should never be accessible when working in org B's context.

### after

```bash
$ rhx keyrack unlock --env sudo --key AWS_PROFILE

🔓 keyrack unlock
   └─ ahction.sudo.AWS_PROFILE
      ├─ env: sudo
      ├─ org: ahction
      ├─ vault: aws.config
      └─ expires in: 30m
```

only keys for the current repo's org are unlocked. the firewall respects context.

### the "aha" moment

keyrack is a firewall, not a vault. firewalls respect boundaries. if i'm standing in ahction's territory, i should only access ahction's resources — even for sudo credentials.

## user experience

### usecase.1 = unlock sudo key in repo context

**goal:** unlock aws sso credentials for deployment

**contract:**
```bash
$ rhx keyrack unlock --env sudo --key AWS_PROFILE
```

**input:** `--env sudo`, `--key AWS_PROFILE`, implicit org from repo manifest

**output:** only keys matching `{repo.org}.sudo.AWS_PROFILE` pattern

### usecase.2 = unlock cross-org sudo key

**goal:** unlock a credential that spans all orgs (e.g., shared infrastructure)

**contract:** use full slug syntax (extant behavior):
```bash
$ rhx keyrack unlock --env sudo --key @all.sudo.INFRA_TOKEN
```

**output:** `@all.sudo.INFRA_TOKEN` — explicitly cross-org

### usecase.3 = unlock without repo context

**goal:** unlock sudo credentials outside any repo (no keyrack.yml)

**constraint:** no repo manifest means no org context

**expected behavior:** fail-fast with clear error

```bash
$ rhx keyrack unlock --env sudo --key AWS_PROFILE

🔐 keyrack unlock
   └─ ✗ cannot determine org for sudo unlock
      ├─ no keyrack.yml found in repo
      └─ hint: run from a repo with keyrack.yml, or specify full slug:
         rhx keyrack unlock --env sudo --key ahction.sudo.AWS_PROFILE
```

### timeline

1. user runs `unlock --env sudo --key AWS_PROFILE`
2. keyrack reads repo manifest to get org (`ahction`)
3. keyrack searches host manifest for `ahction.sudo.AWS_PROFILE`
4. keyrack unlocks ONLY that key
5. tools can now access the credential via `keyrack get`

## mental model

### how users would describe this

> "sudo unlock respects which org i'm working in. if i'm in my ahction repo, it only unlocks ahction credentials."

### analogies

**building keycard analogy:** when you badge into building A, you only get access to building A's resources — not building B's elevators.

**git remote analogy:** when you're in a repo, `git push` goes to that repo's remote, not some other repo you also have access to.

### terms

| user term | keyrack term | meaning |
|-----------|--------------|---------|
| "which org am i in" | repo manifest org | the org declared in keyrack.yml |
| "my sudo keys" | org-scoped sudo keys | keys matching `{repo.org}.sudo.*` |
| "shared keys" | cross-org keys | keys with `org: @all` |

## evaluation

### how well does it solve the goals?

| goal | status |
|------|--------|
| prevent cross-org key leakage | solved — org filter enforced |
| maintain sudo key isolation | preserved — still requires `--key` |
| support cross-org keys when needed | supported — full slug syntax |
| fail-fast on ambiguous context | solved — no repo manifest = error |

### pros

- **security:** prevents accidental access to wrong org's credentials
- **consistency:** sudo keys now behave like regular keys (org-scoped)
- **explicit:** full slug always works as escape hatch
- **pit of success:** default behavior is the secure behavior

### cons

- **breaking change:** users who relied on unlocking multiple orgs at once need to adjust
- **extra typing:** users with cross-org keys need explicit `@all` or full slug

### edge cases & pit of success

| edge case | handling |
|-----------|----------|
| no repo manifest | fail-fast with hint to use full slug |
| full slug provided | exact match, bypasses org inference |
| `@all` org key | matches regardless of repo org |
| key name matches but wrong org | NOT unlocked (correct behavior) |

## open questions & assumptions

### assumptions

1. **repo manifest is authoritative for org.** if user is in a repo, that repo's org should scope their sudo access.

2. **full slug is the escape hatch.** users can always specify `ahbode.sudo.AWS_PROFILE` directly if they really need cross-org access.

3. **@all keys should be accessible from any repo.** cross-org keys are explicitly declared as such.

### questions for wisher

1. **[answered] what about repos that don't declare an org?** should they fail-fast, or fall back to some default?
   - **answer:** fail-fast is the secure default. if no repo manifest, user must specify full slug. this follows pit-of-success: ambiguous context = explicit input required.

2. **[answered] should @all keys be unlocked automatically when unlock org-specific keys?** e.g., if user has both `ahction.sudo.X` and `@all.sudo.X`, should both unlock?
   - **answer:** no. this follows extant `env=all` pattern — fallback happens at lookup-time (`keyrack get`), not at unlock-time. explicit unlock is safer for sudo credentials.

3. **[answered] should full slug allow cross-org unlock?** e.g., user in ahction repo runs `--key ahbode.sudo.AWS_PROFILE`. should this unlock ahbode's key?
   - **answer:** yes. full slug is the explicit escape hatch. the security issue is *implicit* cross-org unlock (key name matches multiple orgs). when user specifies full slug, their intent is explicit.
   - **evidence:** the wish says "absolutely not. not allowed" about ahbode key unlocked when user asked for `--key AWS_PROFILE` (key name only). user didn't ask for ahbode's key — keyrack gave it anyway. that's the violation.

4. **[answered] what if repo has keyrack.yml but no org declared?** different from "no keyrack.yml at all".
   - **answer:** same behavior — fail-fast. org is required for sudo unlock. the root manifest in the extends chain must declare an org.
   - **evidence:** define.root-org-inheritance.md — root manifest org wins in extends chain. if no root declares org, there's no org to scope by.

## what is awkward?

### the full slug escape hatch is verbose

```bash
rhx keyrack unlock --env sudo --key ahction.sudo.AWS_PROFILE
```

users must type the full `org.env.key` pattern when they need cross-org access. this is verbose but explicit. most users will be in the correct repo context so this shouldn't be common.

### @all keys might surprise users

if user has both `ahction.sudo.X` and `@all.sudo.X`, should both unlock? current design says no — only exact match or `@all` fallback at get-time. but this means `@all` keys need explicit unlock.

this mirrors the extant `env=all` fallback behavior (lookup-time, not unlock-time), so it's consistent.

### no repo = no sudo unlock (without full slug)

users who run `unlock --env sudo` from outside a repo will get an error. this is correct (fail-fast), but might frustrate users who "just want to unlock a key."

the hint provides the path forward: use full slug.
