# research: internal prod code patterns

## pattern.1 = enrollment mechanism

### what

enrollment spawns the claude cli with a custom config for enrolled role hooks, but auth credentials load independently from `~/.claude/.credentials.json`.

### citations

**[1]** `src/domain.operations/enroll/enrollBrainCli.ts:8-12`
```typescript
/**
 * .what = spawns brain CLI with --setting-sources local --settings <configPath>
 * .why = skips user/project settings, uses ONLY the specified config, auth loads from credentials file
 *
 * .note = --setting-sources local skips user (~/.claude/settings.json) and project (.claude/settings.json)
 * .note = auth credentials load independently from ~/.claude/.credentials.json (not affected by --setting-sources)
 * .note = --settings <path> loads settings from specified file only
 */
```

**[2]** `src/contract/cli/invokeEnroll.ts:131-136`
```typescript
enrollBrainCli({
  brain: enrollment.brain,
  configPath,
  args: passthroughArgs,
  cwd: gitroot,
});
```

### relation to wish

auth credentials load independently from enrollment. to implement auth rotation, enrollment must inject auth via a different mechanism — either env vars or a custom apiKeyHelper.

### verdict

[EXTEND] — extend enrollment to optionally accept `--creds` URI and configure auth rotation via apiKeyHelper mechanism.

## pattern.2 = keyrack vault adapters

### what

keyrack uses vault adapters to abstract credential storage. credentials are fetched from different backends (envvar, daemon, direct, secure, 1password) with a unified interface.

### citations

**[3]** `src/domain.operations/keyrack/adapters/vaults/os.envvar/vaultAdapterOsEnvvar.ts:38-41`
```typescript
// reads from process.env directly using raw key names
// always unlocked, no authentication needed
process.env['ANTHROPIC_AUTH_TOKEN']
```

**[4]** `src/domain.operations/keyrack/genContextKeyrackGrantGet.ts:66-68`
```typescript
// vault types: os.envvar, os.daemon, os.direct, os.secure, 1password
```

### relation to wish

the vault adapter pattern enables credentials to be sourced from multiple locations. auth pool tokens would be stored in os.secure (encrypted) and unlocked into os.daemon for rotation.

### verdict

[REUSE] — reuse vault adapters for token storage; os.secure for persistent, os.daemon for session cache.

## pattern.3 = keyrack mechanism adapters

### what

mechanism adapters transform credentials from stored format to usable secret. replica mechanism is passthrough; others (github_app, aws_sso) transform source credentials into ephemeral tokens.

### citations

**[5]** `src/domain.operations/keyrack/adapters/mechanisms/mechAdapterReplica.ts:110-112`
```typescript
deliverForGet: async (input) => {
  return { secret: input.source };  // Identity passthrough
}
```

**[6]** `src/domain.operations/keyrack/genContextKeyrackGrantGet.ts:54-61`
```typescript
// PERMANENT_VIA_REPLICA: passthrough, no transformation
// PERMANENT_VIA_REFERENCE: 1password passthrough with exid lookup
// EPHEMERAL_VIA_SESSION: daemon-stored credentials
// EPHEMERAL_VIA_GITHUB_APP: github app JSON → installation token
// EPHEMERAL_VIA_AWS_SSO: aws sso profile → temporary session credentials
```

### relation to wish

auth pool rotation is NOT a transformation mechanism (input token = output token). however, the rotation logic (which token to use) could be implemented as a new mechanism adapter or as a separate auth pool service.

### verdict

[EXTEND] — consider an auth pool mechanism or service that selects the best token from the pool and returns it via the replica passthrough pattern.

## pattern.4 = keyrack key resolution

### what

keyrack resolves credentials in priority order: envvar first, then daemon, then reports locked/absent. slugs use format `org.env.KEY_NAME`.

### citations

**[7]** `src/domain.operations/keyrack/getKeyrackKeyGrant.ts:74-133`
```typescript
// resolution order:
// 1. os.envvar check (line 77): process.env[keyName]
// 2. os.daemon check (line 105): in-memory daemon cache
// 3. locked or absent (line 131-155): determine status
```

**[8]** `src/domain.operations/keyrack/asKeyrackKeyName.ts`
```typescript
// extracts raw key name from slug
// slug.split('.').slice(2).join('.') — strips org.env prefix
```

### relation to wish

auth pool keys would follow the same slug format (`ehmpath.prep.CLAUDE_OAUTH_TOKEN_1`). the resolution order means pool tokens should be unlocked into daemon for quick access.

### verdict

[REUSE] — reuse slug format and resolution order; pool tokens stored under their individual slugs.

## pattern.5 = keyrack source into env

### what

keyrack can source granted credentials into `process.env` for child processes.

### citations

**[9]** `src/domain.operations/keyrack/sourceAllKeysIntoEnv.ts:123-131`
```typescript
// inject granted secrets into process.env
for (const key of keysForEnv) {
  if (key.status !== 'granted') continue;
  // extract env var name from slug (e.g., "ehmpathy.test.OPENAI_API_KEY" → "OPENAI_API_KEY")
  const envVarName = key.grant.slug.split('.').pop();
  if (envVarName && !process.env[envVarName]) {
    process.env[envVarName] = key.grant.key.secret;
  }
}
```

### relation to wish

if apiKeyHelper is supported, we can implement a helper operation that sources the best token from the pool and outputs it. the operation would use keyrack to fetch all pool tokens and select the best one.

### verdict

[REUSE] — reuse source pattern for injecting auth into enrolled brain processes.

## pattern.6 = repo manifest declaration

### what

keyrack uses a repo manifest (`.agent/keyrack.yml`) to declare required credentials per environment.

### citations

**[10]** `src/domain.objects/keyrack/KeyrackRepoManifest.ts`
```yaml
# example format

  - ANTHROPIC_AUTH_TOKEN
env.test:
  - OPENAI_API_KEY
```

### relation to wish

auth pool tokens would NOT be declared in repo manifest. instead, they are user-specific (stored in host manifest) and referenced via `--creds` URI at enrollment time.

### verdict

[REUSE] — repo manifest pattern stays unchanged; pool tokens live in host manifest only.

## summary

| pattern | verdict | rationale |
|---------|---------|-----------|
| enrollment mechanism | [EXTEND] | inject auth via apiKeyHelper, not credentials file |
| vault adapters | [REUSE] | os.secure + os.daemon cover storage needs |
| mechanism adapters | [EXTEND] | add rotation selection logic |
| key resolution | [REUSE] | slug format and resolution order work as-is |
| source into env | [REUSE] | for apiKeyHelper output |
| repo manifest | [REUSE] | unchanged; pool tokens in host manifest |
| keyrack sdk | [EXTEND] | add auth-pool namespace |

## key insights

1. **auth independence**: claude cli loads auth from `~/.claude/.credentials.json` independently of enrollment config. we must intercept via apiKeyHelper or env var injection.

2. **apiKeyHelper path**: if claude cli supports apiKeyHelper (research question), we can implement rotation as a helper operation that keyrack sources.

3. **state file location**: rotation state should live at `~/.rhachet/auth-pool/state.json` (per vision), separate from keyrack daemon.

4. **slug pattern**: pool tokens use `owner.env.CLAUDE_OAUTH_TOKEN_*` format, stored in os.secure vault.
