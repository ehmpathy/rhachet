# q9: keyrack glob support — yield

## answer

**no, glob patterns NOT supported today**

## source

`src/domain.operations/keyrack/session/unlockKeyrackKeys.ts`

current behavior:

```ts
// filter by key: match full slug or key name suffix
slugsForEnv = keyInput
  ? allSlugsForEnv.filter(
      (slug) => slug === keyInput || slug.endsWith(`.${keyInput}`),
    )
  : allSlugsForEnv;
```

supports:
- full slug: `org.env.KEY`
- key name suffix: `KEY` (matches any slug that ends with `.KEY`)
- no key: unlock all for env

does NOT support:
- `KEY_*` glob patterns
- `*_TOKEN` prefix patterns

## implications for auth-pool

auth-pool does NOT need keyrack glob support:

1. creds URI explicitly names tokens: `keyrack://owner/env/TOKEN_1,TOKEN_2,TOKEN_3`
2. auth-pool parses URI and iterates over explicit token names
3. calls `keyrack get --key $slug` for each token

no glob required — explicit token list in URI is sufficient for spike.

## future enhancement

if glob support desired later:
- add glob match to `unlockKeyrackKeys`
- use `minimatch` or similar for pattern expansion
