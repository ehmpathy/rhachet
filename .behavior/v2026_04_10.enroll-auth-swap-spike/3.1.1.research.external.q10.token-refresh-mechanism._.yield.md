# q10: token refresh mechanism — yield

## answer

**apiKeyHelper = called periodically during active session**
**env vars = read once at spawn, NOT re-read**

## source

[Claude Code Authentication](https://code.claude.com/docs/en/authentication)

> **Refresh intervals**: by default, `apiKeyHelper` is called after 5 minutes or on HTTP 401 response. Set `CLAUDE_CODE_API_KEY_HELPER_TTL_MS` environment variable for custom refresh intervals.

> `apiKeyHelper` executable output. Use this for **dynamic or rotated credentials**, such as short-lived tokens fetched from a vault.

## behavior

| credential source | when read | mid-session rotation? |
|-------------------|-----------|----------------------|
| `ANTHROPIC_API_KEY` | process spawn | NO — cached at startup |
| `ANTHROPIC_AUTH_TOKEN` | process spawn | NO — cached at startup |
| `CLAUDE_CODE_OAUTH_TOKEN` | process spawn | NO — cached at startup |
| `apiKeyHelper` | every 5 min OR on 401 | **YES** — re-called periodically |

## critical insight

for mid-session rotation to work, we MUST use `apiKeyHelper`. env var changes after Claude Code spawns will NOT be picked up.

## flow

```
claude spawns
    ↓
apiKeyHelper called → returns token A
    ↓
... 5 minutes pass (or 429 → then 401 on retry) ...
    ↓
apiKeyHelper called again → returns token B (rotated!)
    ↓
Claude uses token B for subsequent requests
```

## implications

- env var approach won't work for active sessions
- apiKeyHelper IS the integration point (confirmed)
- TTL configurable via `CLAUDE_CODE_API_KEY_HELPER_TTL_MS`
- rotation latency = up to TTL (5 min default)
