## usecase.2 = automatic rotation on rate limit

### matrix: rotation decision

| ind: active token state | ind: next candidate state | ind: candidate rate-limit age | dep: rotation outcome | dep: state file update |
|------------------------|--------------------------|------------------------------|----------------------|------------------------|
| rate-limited | healthy | n/a | rotates to next | marks active as rate-limited |
| rate-limited | rate-limited | < cooldown | skips, tries next-next | marks active as rate-limited |
| rate-limited | rate-limited | > cooldown | attempts recovery | clears if success |
| rate-limited | all others rate-limited (< cooldown) | n/a | exhaustion error | marks active as rate-limited |

### dimensions

- **active token state**: the token that just hit a rate limit
- **next candidate state**: the state of the next token in round-robin order
- **candidate rate-limit age**: how long ago the candidate was marked rate-limited
- **rotation outcome**: what the system does
- **state file update**: how the persistent state changes

### gaps

none — the blackbox criteria covers all four combinations.

### decomposition notes

3 independent variables is manageable. no decomposition needed.

## usecase.3 = per-brain isolation

### matrix: isolation boundaries

| ind: brain enrollment | ind: pool overlap | ind: rotation action | dep: cross-brain effect | dep: global auth state |
|----------------------|------------------|---------------------|------------------------|----------------------|
| X with [A,B,C], Y with [D,E,F] | none | X rotates A→B | Y unaffected | unchanged |
| X with [A,B,C], same for terminal 2,3 | full (same brain) | T1 rotates A→B | T2,T3 see B as active | unchanged |

### dimensions

- **brain enrollment**: which brains exist with which pools
- **pool overlap**: whether brains share tokens
- **rotation action**: which brain/terminal triggers rotation
- **cross-brain effect**: impact on other brains/terminals
- **global auth state**: impact on non-enrolled processes

### gaps

**gap identified**: the criteria does not specify behavior when two brains share some tokens (partial overlap). this may be out of scope for spike, but worth a callout.

### decomposition notes

2 independent variables. no decomposition needed.

## usecase.4 = error states

### matrix: error conditions

| ind: pool health | ind: keyrack state | ind: token validity | dep: error type | dep: exit code | dep: user guidance |
|-----------------|-------------------|--------------------|-----------------|--------------|--------------------|
| all rate-limited | unlocked | all valid | exhaustion | 2 | wait or add tokens |
| some healthy | unlocked | one expired | auth failure (then rotate) | 0 (if others work) | - |
| all expired | unlocked | all expired | exhaustion | 2 | regenerate tokens |
| any | locked | any | keyrack-locked | 2 | unlock command hint |

### dimensions

- **pool health**: rate-limit status across all tokens
- **keyrack state**: whether keyrack session is active
- **token validity**: whether tokens are expired (auth failure vs rate limit)
- **error type**: the kind of error surfaced
- **exit code**: process exit status
- **user guidance**: hint provided to user

### gaps

**gap identified**: what happens when expired token is detected but all other tokens are rate-limited? the criteria implies rotation continues, but does not specify the combined case explicitly.

### decomposition notes

3 independent variables is manageable. no decomposition needed.

### gaps to address

1. **partial pool overlap** (usecase.3): if brain X and Y share some tokens, how do rotations interact? recommend: declare out of scope for spike (each enrolled brain must use disjoint pools).

2. **expired + rate-limited combined** (usecase.4): if an expired token is detected and rotation finds all other tokens rate-limited, the outcome should be exhaustion error with combined guidance ("some tokens expired, others rate-limited").
