# vision: enroll-auth-swap

## the architecture

rhachet defines the BrainAuthAdapter contract. brain suppliers implement it. rhachet orchestrates.

## the key insight (claude-specific)

for claude, **apiKeyHelper is THE integration point for mid-session rotation.**

env vars (`ANTHROPIC_API_KEY`, `ANTHROPIC_AUTH_TOKEN`) are read ONCE at process spawn and cached. you cannot rotate them mid-session.

apiKeyHelper is an executable configured in `settings.json` that Claude Code calls:
- on startup
- every 5 minutes (configurable via `CLAUDE_CODE_API_KEY_HELPER_TTL_MS`)
- on HTTP 401 (auth failure)

```
┌─────────────────────────────────────────────────────────────────┐
│  Claude Code (when BrainAuthAdapter configures pool strategy)   │
│       │                                                         │
│       ▼  (every 5 min or on 401)                                │
│  ┌──────────────┐                                               │
│  │ apiKeyHelper │ → runs: npx rhachet brains auth get --from ...  │
│  └──────┬───────┘                                               │
│         │                                                       │
│         ▼                                                       │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │ rhachet brains auth get                                     │ │
│  │ ────────────────────────────────────────────────────────── │ │
│  │ 1. parse --from strategy (pool or solo)                    │ │
│  │ 2. if pool: read state, find best available token          │ │
│  │ 3. print token to stdout                                   │ │
│  └────────────────────────────────────────────────────────────┘ │
│         │                                                       │
│         ▼                                                       │
│  Claude uses returned token for API calls                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**this is why it enables rotation:**

- token A gets rate-limited (429)
- next apiKeyHelper call (within 5 min), `brains auth get` returns token B
- Claude now uses token B
- seamless rotation, no restart needed

**vs env vars:**

| approach | mid-session rotation |
|----------|---------------------|
| `ANTHROPIC_API_KEY=xxx claude` | NO — read once at spawn |
| `apiKeyHelper → returns xxx` | YES — re-called periodically |

## outcome world

### before (current pain)

developer has 3 claude subscriptions. in heavy work sessions, hits 5hr usage limit on subscription A. must:
1. notice the rate limit error
2. open browser, log into subscription B account
3. run `/login` in each terminal
4. repeat across all active terminals
5. lose flow state and context

after 5hrs on B, repeat for C. after C, back to A (hopefully refreshed). constant manual rotation.

### after (desired state)

developer has 3 claude subscriptions configured as a pool. hits usage limit on subscription A:
1. next apiKeyHelper call (within 5 min) detects token A is limited
2. `brains auth get` returns token B instead
3. Claude uses token B for subsequent requests
4. developer never notices the swap

the auth rotation is invisible. developer stays in flow.

## user experience

### setup: one-time configuration

**step 1: generate tokens for each subscription**

for each subscription account, log in and generate a long-lived OAuth token:
```sh
# log into subscription 1 account via browser
claude setup-token  # outputs 1-year OAuth token → store this
```

repeat for each subscription (2, 3, etc).

**step 2: store tokens in keyrack**
```sh
rhx keyrack set CLAUDE_TOKEN_1 --owner ehmpath --vault os.secure --env all
rhx keyrack set CLAUDE_TOKEN_2 --owner ehmpath --vault os.secure --env all
rhx keyrack set CLAUDE_TOKEN_3 --owner ehmpath --vault os.secure --env all
```

**step 3: configure auth pool**

auth can be configured at init time (repo-wide default) or enroll time (role-specific override):

```sh
# at init time — sets default for all enrollments in this repo
npx rhachet init --auth 'pool(keyrack://ehmpath/all/CLAUDE_TOKEN_1,CLAUDE_TOKEN_2,CLAUDE_TOKEN_3)'

# at enroll time — overrides default for this specific enrollment
rhx enroll --brain claude --roles mechanic --auth 'pool(keyrack://ehmpath/all/CLAUDE_TOKEN_1,CLAUDE_TOKEN_2,CLAUDE_TOKEN_3)'
```

**init vs enroll:**

| command | scope | persistence |
|---------|-------|-------------|
| `npx rhachet init --auth ...` | repo-wide default | stored in `.agent/` config |
| `rhx enroll --auth ...` | this enrollment only | runtime override |

this mirrors how `--roles` works: init sets defaults, enroll overrides.

**what these commands do:**
- validate tokens exist in keyrack
- register the BrainAuthAdapter for the brain
- configure apiKeyHelper in `~/.claude/settings.json`
- wire up the `brains auth get` command as apiKeyHelper

**auth strategies:**

| strategy | syntax | behavior |
|----------|--------|----------|
| default | `default` | brain's native auth (env var, /login, etc) |
| solo | `solo(source://...)` | explicit single token from source |
| pool | `pool(source://...)` | round-robin pool via apiKeyHelper |

**sources:**

| source | syntax | example |
|--------|--------|---------|
| env | `env://VAR_NAME` | `solo(env://ANTHROPIC_API_KEY)` |
| keyrack | `keyrack://owner/env/KEY` | `solo(keyrack://ehmpath/all/CLAUDE_TOKEN)` |

- `--auth default` — restores native brain auth behavior
- `--auth solo(...)` — explicit single token from source
- `--auth pool(...)` — round-robin rotation from source

**step 4: unlock keyrack at session start**
```sh
rhx keyrack unlock --owner ehmpath --env all
```

done. Claude Code now uses the auth pool for automatic token rotation.

### daily flow

1. start terminal, unlock keyrack once
2. run `claude` as normal
3. hit rate limit on token 1 → apiKeyHelper returns token 2
4. continue work without interruption
5. hit rate limit on token 2 → apiKeyHelper returns token 3
6. continue work
7. by the time token 3 is limited, token 1 may have refreshed

no manual intervention in the work session.

## mental model

### analogy: load balancer with failover

like a load balancer that routes traffic to healthy backends:
- each subscription = backend server
- rate limit = server unhealthy
- rotation = failover to next healthy server
- apiKeyHelper = the load balancer

### terminology

| term | definition |
|------|------------|
| auth pool | set of subscription tokens available for rotation |
| apiKeyHelper | Claude Code mechanism that calls an executable to get auth token |
| rotation | swap from limited token to available one |
| rate limit | 429 response — subscription quota exceeded |
| creds URI | `keyrack://$owner/$env/$key1,$key2,...` — explicit list of token keys |

## contract: BrainAuthAdapter

brain suppliers implement BrainAuthAdapter — symmetric to BrainHooksAdapter:

```ts
/**
 * .what = adapter contract for brain auth configuration
 * .why = brain suppliers implement this to support auth sync
 *
 * .note = mirrors BrainHooksAdapter pattern (inversion of control)
 */
interface BrainAuthAdapter {
  slug: string;
  dao: BrainAuthAdapterDao;
  capacity: BrainAuthCapacityDao;
}

/**
 * .what = DAO interface contract for brain auth CRUD operations
 * .why = enables declastruct pattern with idempotent get/set/del semantics
 *
 * .note = uses method syntax for bivariance (see define.bivariance-for-generics)
 * .note = singleton per brain — no `by` input needed (unlike BrainHooksAdapterDao)
 */
interface BrainAuthAdapterDao {
  get: {
    one(): Promise<BrainAuthSpecShape | null>;
  };
  set: {
    findsert(input: { spec: BrainAuthSpecShape }): Promise<BrainAuthSpecShape>;
    upsert(input: { spec: BrainAuthSpecShape }): Promise<BrainAuthSpecShape>;
  };
  del(): Promise<void>;
}

/**
 * .what = capacity status for a single auth token
 * .why = enables selection of token with most capacity left
 */
interface BrainAuthCapacity {
  authSlug: string;
  used: number;
  limit: number;
  left: number;
  refreshAt: IsoTimeStamp | null;
}

/**
 * .what = readonly DAO for auth capacity queries
 * .why = brain suppliers implement capacity queries; rhachet queries to select best token
 *
 * .note = readonly — no set/del methods
 */
interface BrainAuthCapacityDao {
  get: {
    one(input: { by: { slug: string } }): Promise<BrainAuthCapacity | null>;
    all(): Promise<BrainAuthCapacity[]>;
  };
}

/**
 * .what = auth spec as CLI string
 * .why = human-readable format for --auth and --from flags
 *
 * .examples:
 *   - 'default'
 *   - 'solo(keyrack://ehmpath/all/TOKEN_1)'
 *   - 'pool(keyrack://ehmpath/all/TOKEN_1,TOKEN_2,TOKEN_3)'
 */
type BrainAuthSpecWords = string;

/**
 * .what = auth spec as structured object
 * .why = machine-readable format for internal logic
 */
interface BrainAuthSpecShape {
  strategy: 'default' | 'solo' | 'pool';
  source: string | null; // e.g., 'keyrack://ehmpath/all/TOKEN_1,TOKEN_2'
}

/**
 * .what = union of both representations
 */
type BrainAuthSpec = BrainAuthSpecWords | BrainAuthSpecShape;

/**
 * .what = transforms between representations
 */
const asBrainAuthSpecShape = (input: { from: BrainAuthSpecWords }): BrainAuthSpecShape;
const asBrainAuthSpecWords = (input: { from: BrainAuthSpecShape }): BrainAuthSpecWords;
```

**where each is used:**

| representation | where |
|----------------|-------|
| `BrainAuthSpecWords` | CLI flags (`--auth`, `--from`), settings.json |
| `BrainAuthSpecShape` | internal logic, DAO methods, enrollment manifest |

**inversion of control:**

rhachet doesn't know how to configure auth for each brain. the brain supplier provides the adapter:

```ts
// in rhachet-brains-anthropic
export const genBrainAuthAdapterForClaudeCode = (input: {
  repoPath: string;
}): BrainAuthAdapter => ({
  slug: 'claude-code',
  dao: {
    get: { async one() { /* read from settings.json */ } },
    set: {
      async findsert({ spec }) { /* if exists return it, otherwise write */ },
      async upsert({ spec }) { /* write apiKeyHelper to settings.json */ },
    },
    del: async () { /* remove apiKeyHelper from settings.json */ },
  },
  capacity: {
    get: {
      async one({ by }) { /* query usage for one token via anthropic api */ },
      async all() { /* query usage for all tokens in pool */ },
    },
  },
});
```

**implementation per brain:**

| brain | default | solo | pool |
|-------|---------|------|------|
| claude | native (env var, /login) | apiKeyHelper | apiKeyHelper |
| openai | native (env var) | env var | — |
| gemini | native (env var) | env var | — |

for claude, solo and pool both use `apiKeyHelper` in `settings.json` — the difference is whether the helper returns one token or rotates through a pool.

## integration: enrollment manifest

auth is part of the enrollment manifest — different auth configs produce different enrollment hashes:

```ts
interface BrainCliEnrollmentManifest {
  brain: BrainSlug;
  roles: RoleSlug[];
  auth: BrainAuthSpecShape | null;  // ← new: auth spec affects hash
}
```

**why auth affects the hash:**

```sh
# same brain, same roles, different auth → different config files
rhx enroll --brain claude --roles mechanic                           # hash: a1b2c3d4
rhx enroll --brain claude --roles mechanic --auth default            # hash: a1b2c3d4 (same)
rhx enroll --brain claude --roles mechanic --auth 'pool(keyrack://...)' # hash: e5f6g7h8 (different)
```

this ensures:
- enrollments with pool auth get apiKeyHelper configured
- enrollments with default auth use native brain auth
- no collision between different auth configurations

**config file pattern:**

```
.claude/settings.enroll.$hash.local.json
```

each unique `{ brain, roles, auth }` combination gets its own config file.

## contract: brains auth get

### interface: brains auth get

```sh
npx rhachet brains auth get --from $spec --into $format
```

**arguments:**

| arg | description | example |
|-----|-------------|---------|
| `--from` | auth spec (strategy + source) | `pool(keyrack://ehmpath/all/TOKEN_1,TOKEN_2)` |
| `--into` | output format | `claude.apiKeyHelper`, `env.ANTHROPIC_API_KEY` |

**examples:**

```sh
# for Claude Code apiKeyHelper (prints raw token)
npx rhachet brains auth get --from 'pool(keyrack://ehmpath/all/TOKEN_1,TOKEN_2,TOKEN_3)' --into claude.apiKeyHelper

# for solo token
npx rhachet brains auth get --from 'solo(keyrack://ehmpath/all/TOKEN_1)' --into claude.apiKeyHelper
```

**behavior for `pool(...)`:**
1. read pool state from `~/.rhachet/brain/auth/state.json`
2. if current token is NOT marked as limited, return it
3. if current token IS limited, try next token in pool
4. mark tokens as limited when they fail (tracked via state file or external signal)
5. round-robin until one works or all exhausted

**behavior for `solo(...)`:**
1. fetch token from source
2. return it

**output formats:**

| format | output |
|--------|--------|
| `claude.apiKeyHelper` | raw token to stdout |
| `env.$VAR` | `export $VAR=token` |

### interface: settings.json

```json
{
  "apiKeyHelper": "npx rhachet brains auth get --from 'pool(keyrack://ehmpath/all/TOKEN_1,TOKEN_2,TOKEN_3)' --into claude.apiKeyHelper"
}
```

Claude Code calls this executable:
- every 5 min (default TTL)
- on 401 response

### interface: shared state

```json
// ~/.rhachet/brain/auth/state.json
{
  "activeTokenKey": "CLAUDE_TOKEN_2",
  "lastRotation": "2026-04-11T14:30:00Z",
  "limited": {
    "CLAUDE_TOKEN_1": {
      "since": "2026-04-11T14:25:00Z",
      "until": "2026-04-11T19:25:00Z"
    }
  }
}
```

all terminals on same machine read from same state file. atomic reads, no coordination needed.

## evaluation

### what we confirmed via research

| question | answer |
|----------|--------|
| mid-session rotation? | YES — via apiKeyHelper only, NOT env vars |
| env vars rotate? | NO — cached at spawn |
| apiKeyHelper TTL? | 5 min default, configurable via `CLAUDE_CODE_API_KEY_HELPER_TTL_MS` |
| apiKeyHelper trigger? | on startup, every TTL, on 401 |
| token validity? | 1 year via `claude setup-token` |
| rate limit scope? | per-organization/subscription, NOT per-IP |
| per-terminal isolation? | YES — each process independent |
| keyrack glob support? | NO — use explicit comma-separated list in URI |
| proactive rotation? | headers available, but reactive (on 429) is OK for spike |

### pros

- invisible to user after setup
- no workflow changes needed
- leverages Claude Code's apiKeyHelper mechanism (no custom patch)
- works with keyrack security model
- each terminal independent (no cross-process interference)
- scales to any pool size

### cons

- requires N separate logins to generate N tokens (one-time)
- token refresh (yearly) requires manual re-generation
- rotation latency up to TTL (5 min default) — user may see brief error
- 429 (rate limit) does NOT trigger apiKeyHelper — only 401 does

### pit of success

- default behavior is single token (no pool)
- pool opt-in via explicit configuration
- rotation only triggers on actual failure
- clear error if all tokens exhausted

### potential issue: 429 vs 401

apiKeyHelper triggers on **401 (auth failure)**, not 429 (rate limit).

if Claude Code receives 429, it may retry with same token. the token won't rotate until:
- the retry eventually fails with 401, OR
- the next TTL refresh (5 min)

**mitigation**: keep TTL low (5 min default is reasonable). worst case, user waits 5 min for rotation.

## scope: spike

### in scope

1. `npx rhachet brains auth get` — returns token based on spec and format
2. shared state file for pool state
3. keyrack integration for secure token storage
4. settings.json configuration for apiKeyHelper

### out of scope

- proactive rotation (rotate before limit) — reactive is OK
- team pools (shared across machines) — single machine for spike
- usage dashboard (which subscription, quota left) — not needed
- automatic token refresh (yearly expiry) — manual re-gen is fine

## awkward aspects

### 5-minute rotation latency

if token 1 hits 429 at minute 0:
- apiKeyHelper was just called
- next call at minute 5
- user may see errors for up to 5 minutes

**mitigation**:
- can reduce TTL via `CLAUDE_CODE_API_KEY_HELPER_TTL_MS`
- or accept 5 min latency as tolerable for spike

### token generation requires browser

`claude setup-token` requires browser OAuth flow. each subscription needs:
1. log into anthropic.com as that subscription's account
2. run `claude setup-token`
3. copy token to keyrack

one-time per year. acceptable for spike.

### no automatic 429 → rotation

apiKeyHelper triggers on 401, not 429. if Claude handles 429 with built-in retry:
- retries with same (limited) token
- may fail repeatedly until 401 or TTL refresh

**observation**: in practice, Claude may convert 429 → 401 after retries exhaust. needs empirical validation.
