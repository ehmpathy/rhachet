## q1: rate limit http code

**answer:** `429 rate_limit_error`

**source:** [Anthropic API Errors](https://platform.claude.com/docs/en/api/errors)

> 429 - `rate_limit_error`: Your account has hit a rate limit.

**implications:**
- `isAuthPoolTokenRateLimited` should detect 429 status code
- 401 is authentication error, NOT rate limit
- implementation matches original assumption

## q2: apiKeyHelper trigger scope

**answer:** called after 5 minutes OR on HTTP 401 response

**source:** [Claude Code Authentication](https://code.claude.com/docs/en/authentication)

> Refresh intervals: by default, `apiKeyHelper` is called after 5 minutes or on HTTP 401 response. Set `CLAUDE_CODE_API_KEY_HELPER_TTL_MS` environment variable for custom refresh intervals.

> `apiKeyHelper` output. Use this for dynamic or rotating credentials, such as short-lived tokens fetched from a vault.

**implications:**
- **critical finding:** `apiKeyHelper` is designed exactly for our use case
- no need for manual env var swaps
- auth-pool can implement an apiKeyHelper executable that returns best available token
- `CLAUDE_CODE_API_KEY_HELPER_TTL_MS` controls check frequency
- 401 triggers immediate refresh (rotation on exhaustion)

## q3: env var for subscription tokens

**answer:** multiple options in precedence order

**source:** [Claude Code Authentication](https://code.claude.com/docs/en/authentication)

| precedence | env var | header | use case |
|------------|---------|--------|----------|
| 1 | cloud provider vars | varies | bedrock, vertex, foundry |
| 2 | `ANTHROPIC_AUTH_TOKEN` | `Authorization: Bearer` | LLM gateway/proxy |
| 3 | `ANTHROPIC_API_KEY` | `X-Api-Key` | direct API access |
| 4 | `apiKeyHelper` output | depends on format | **dynamic/rotating credentials** |
| 5 | `CLAUDE_CODE_OAUTH_TOKEN` | OAuth | CI/pipelines |
| 6 | subscription OAuth | OAuth | default `/login` |

**implications:**
- `apiKeyHelper` sits at precedence 4
- can override subscription OAuth (precedence 6)
- returns key that gets sent appropriately based on format
- this is the integration point for auth-pool

## q4: token format and validity

**answer:** depends on token type

**source:** [Claude Code Authentication](https://code.claude.com/docs/en/authentication)

| token type | validity | format |
|------------|----------|--------|
| `CLAUDE_CODE_OAUTH_TOKEN` | 1 year | OAuth token from `claude setup-token` |
| subscription OAuth | session-based | stored in macOS Keychain or `~/.claude/.credentials.json` |
| `ANTHROPIC_API_KEY` | unlimited | API key from Console |

> Generate a long-lived token: For CI pipelines or other environments where interactive browser login isn't available, generate a one-year OAuth token with `claude setup-token`

**implications:**
- `claude setup-token` generates 1-year OAuth token
- can store these in keyrack
- each subscription can have its own long-lived token
- rotation is about selecting which token, not refreshing tokens

## revised implementation approach

### before research
assumed: swap env vars directly when rate limited

### after research
discovered: use `apiKeyHelper` mechanism

```
┌─────────────────────────────────────────────────────────────────┐
│                  revised auth-pool flow                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  claude cli                                                     │
│       │                                                         │
│       ▼  (every 5 min or on 401)                                │
│  ┌──────────────┐                                               │
│  │ apiKeyHelper │ → runs executable                             │
│  └──────┬───────┘                                               │
│         │                                                       │
│         ▼                                                       │
│  ┌────────────────────────────────────┐                         │
│  │ auth-pool executable               │                         │
│  │ ──────────────────────────────     │                         │
│  │ 1. read AuthPoolState              │                         │
│  │ 2. check rate limit status         │                         │
│  │ 3. return best available token     │                         │
│  └────────────────────────────────────┘                         │
│         │                                                       │
│         ▼                                                       │
│  ┌──────────────────────────────────┐                           │
│  │ keyrack get --key AUTH_TOKEN_*  │                            │
│  │ (returns token from pool)        │                           │
│  └──────────────────────────────────┘                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### key changes to blueprint

1. **no env var manipulation** — use `apiKeyHelper` setting instead
2. **shell executable integration** — auth-pool provides executable at known path
3. **simpler orchestration** — claude cli handles refresh timing
4. **keyrack stores tokens** — from `claude setup-token` per subscription

### settings.json integration

```json
{
  "apiKeyHelper": "rhx authpool.get-best-token"
}
```

or:

```json
{
  "apiKeyHelper": "$HOME/.claude/auth-pool/get-best-token.sh"
}
```
