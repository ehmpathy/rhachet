# reflection: premortem

reflection for mech adapters own their prompts.

---

## premortem: imagine we shipped and it failed

project forward: we shipped this, and it failed badly. why?

### most plausible failure scenarios

1. **aws sso flow regressed**: users who relied on aws.iam.sso vault find their workflows broken. the restructure moved prompts from vault to mech, but subtle differences in behavior caused failures. trust in keyrack damaged.

2. **github app guided setup fails silently**: the gh cli dependency is not available or not authenticated. instead of graceful fallback to manual json, the error is cryptic. users abandon the feature.

3. **pem file format corrupted**: the newline conversion from .pem file to json escapes incorrectly. credentials appear to store successfully but fail on unlock with "invalid private key" errors. hard to diagnose.

4. **vault/mech compatibility matrix incomplete**: certain vault × mech combinations fail at runtime with unclear errors. users discover incompatibilities via trial and error instead of clear fail-fast guidance.

5. **test coverage missed edge cases**: unit tests pass but integration tests were not comprehensive. real-world scenarios (single org, zero apps, expired sessions) fail in production.

---

## what obvious signs did we ignore?

- **aws sso browser auth is hard to test**: we knew the browser flow could not be automated, but we assumed unit tests with mocks were sufficient
- **github api shapes vary**: we assumed consistent response structure, but pagination, rate limits, and permission errors were not handled
- **pem files have multiple formats**: we assumed RSA keys, but ECDSA and Ed25519 keys exist and may require different format
- **vault rename requires migration**: aws.iam.sso → aws.config requires users to update their commands and configs

---

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| gh cli is installed and authenticated | guided setup fails, fallback to manual json needed | medium |
| github api returns consistent structure | parse errors on unexpected shapes | low |
| .pem files are always RSA format | other key types fail to validate | low |
| aws cli export-credentials works uniformly | platform-specific issues (windows, macos) | low |
| single org/app auto-select is desired | users want explicit choice even with one option | low |
| vault/mech matrix is complete | new mechs or vaults break assumptions | medium |
| zero backwards compat is acceptable | users relied on undocumented behavior | low |
| roundtrip verification catches issues | some failures only appear on real unlock | medium |

---

## what is the nightmare scenario?

### worst plausible outcome

aws sso users upgrade to new version and their credentials stop work. the vault rename (aws.iam.sso → aws.config) is not communicated clearly. users run `keyrack unlock` and get "vault not found" errors. they cannot access their aws resources. production deployments fail.

### what would make us deeply regret this?

- extant workflows break without clear migration path
- users lose trust in keyrack reliability
- security-related failures (credentials exposed, stored incorrectly)

### what would cause the most damage?

- aws sso regression: users cannot authenticate to aws
- credentials stored incorrectly: secrets not encrypted or accessible by wrong parties

### what would be hardest to recover from?

- trust damage: users switch to alternative tools
- security incident: credentials compromised

---

## what mitigations to consider?

| risk | mitigation | cost of mitigation |
|------|------------|--------------------|
| aws sso regression | comprehensive integration tests against real aws cli (mocked browser) | medium — test complexity |
| gh cli not available | explicit fallback with clear instructions | low — error message |
| pem format issues | validate pem content before store, support multiple key types | low — validation |
| vault rename confusion | clear error message if aws.iam.sso used, suggest aws.config | low — error message |
| incomplete compat matrix | fail-fast with alternatives on incompatible combo | low — validation |
| roundtrip verification misses issues | test unlock → get cycle, not just set | low — test coverage |
| migration breaks | document migration path in release notes | low — documentation |

---

## additional risks

### technical risks

- **daemon state mismatch**: daemon caches secrets by slug. if slug format changes, stale entries may cause issues.
- **manifest version compatibility**: host manifests may need migration if structure changes.
- **context injection changes**: if mech adapters need different context, callers must be updated.

### process risks

- **scope creep**: "while we're here" additions delay ship
- **test suite time**: comprehensive integration tests slow down CI
- **documentation debt**: internal restructure not reflected in docs

### user risks

- **confusion about mech vs vault**: users may not understand the distinction
- **prompt fatigue**: too many prompts in guided setup
- **fallback quality**: manual json fallback must be usable, not just present
