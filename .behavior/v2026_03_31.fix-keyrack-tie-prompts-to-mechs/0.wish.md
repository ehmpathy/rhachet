wish =

keyrack vaults should use mech adapters, to enable each mechanism to prompt for exactly what it needs to be set into that vault

for example, MECH: EPHEMERAL_VIA_GITHUB_APP

needs to prompt for each part of the json structure it persists, to ensure its usable on unlock

---

and probably for EPHEMERAL_VIA_AWS_SSO

the vault is actually aws.config

and the mech is the one that should do all that stdout emission

-----

the mech adapter should be looked up by the vault, via a common get mech interface, just like the vaults are looked up by the domain operations

---

and, mech inference should still work as expected, since we often use the vault as the biggest signal of which mech to infer

and some vaults will want to failfast for incompatible mechs

---

e.g., os.secure should failfast on the EPHEMERAL_VIA_AWS_SSO mech probably, since that one is so tied to aws config vault patterns

but would be fine with the EPHEMERAL_VIA_GITHUB_APP

---

we want to generically refactor everything, with zero backwards compat, and full domain.operation architectural decomposition, reuse, and recomposition


----

also, specifically for the EPHEMERAL_VIA_GITHUB_APP

hopefully the prompt will start with asking 
1. which of the orgs to target? which has the installation already
    - give a list of options from the api, ask to choose a number (1-n) ; just like we have with aws sso
2. which of the apps installed for this org?
    - give a list of options from the api, ask to choose a number (1-n) ; just like we have with aws sso
3. the private key associated with this installation
    - ask app admin to mint a fresh one, if you dont have access to this already
    - pass in a relative path to the .pem, and we'll format it as needed

since bothi nstallation id and app id come from the api per org

---

the best verification becomes

1. aws sso works as it always did via stdout, only the internals change (i.e., prompt comes from mech); (except maybe the vault is changed, to just 'aws.credentials')
    - and lets make it clear that only 'aws.credentials' vault supports AWS_PROFILE keys; also, there should be some inference operation which infers the mechanism and vault based on key name for this already. lets ensure to maintain that.

2. ephemeral via github now prompts for the inputs it needs, like aws.iam.sso mechanism does, and can be used with any vault that supports it! (os.secure, 1password, etc)
    - explicitly forbid os.direct from ephemeral support, because it cant secure the source key -> save callers from themselves


---

also, when a vault can infer multiple different options for the same key based on name

e.g., 
- AWS_PROFILE -> always infer as --vault aws.credentials; then, vault aws.credentials upon sight of AWS_PROFILE would not know whether the choice is EPHEMERAL_VIA_AWS_SSO or! PERMANENT_VIA_AWS_KEY ; and so, if --mech was not a supplied input, it must ask
- GITHUB_PROFILE -> cant infer --vault; then, cant infer mech choice either, since it could be EPHEMERAL_VIA_GITHUB_APP or PERMANENT_VIA_REPLICA (the replica = the single github pat)

so, probably, we need inference adapters too - that the vaults can invoke to get an stdin response of _which_ mech to use if not specified on input
