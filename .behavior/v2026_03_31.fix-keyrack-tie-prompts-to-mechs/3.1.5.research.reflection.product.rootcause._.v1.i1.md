# reflection: root cause

reflection for mech adapters own their prompts.

---

## what is the symptom?

### what is broken or absent?

EPHEMERAL_VIA_GITHUB_APP has no guided setup. users must manually construct a json blob:

```json
{
  "appId": "...",
  "privateKey": "-----BEGIN RSA PRIVATE KEY-----\n...",
  "installationId": "..."
}
```

meanwhile, EPHEMERAL_VIA_AWS_SSO has beautiful guided setup — but only when used with aws.iam.sso vault.

### how does it manifest?

- github app users spend time on manual json construction
- typos and format errors cause cryptic unlock failures
- discoverability is zero — users must read docs to know what fields are needed
- github app feels like a second-class mechanism compared to aws sso

### when does it occur?

at `keyrack set` time when a user wants to store github app credentials. the vault prompts for "the secret" — but for github app, the secret is a structured blob that requires multiple pieces.

---

## 5 whys tree

### branch 1: github app has no guided setup

1. **why?** → vault adapter prompts for "the secret" but does not know what pieces compose a github app credential
2. **why?** → mech-specific knowledge (appId, installationId, privateKey) is not accessible to the vault adapter
3. **why?** → the mech adapter only has validate and translate methods, not promptForSet
4. **why?** → prompts were added to vaults when they were the only entry point for set
5. **why?** → **architecture conflates storage (vault) with credential acquisition (mech)**

### branch 2: aws sso guided setup works, but only with one vault

1. **why?** → guided setup for aws sso lives in aws.iam.sso vault adapter
2. **why?** → the setup produces an aws profile, which only aws.iam.sso vault can consume
3. **why?** → no separation between "how to acquire credentials" and "where to store them"
4. **why?** → vault/mech coupled state was acceptable when each vault had one mech
5. **why?** → **architecture conflates storage (vault) with credential acquisition (mech)**

### branch 3: mech inference does not exist

1. **why?** → users must explicitly specify --mech or the vault picks a default
2. **why?** → vaults do not have a standard way to list supported mechs
3. **why?** → mech compatibility is implicit, not declared
4. **why?** → no vault adapter interface for supportedMechs or checkMechCompat
5. **why?** → **interfaces evolved organically without mech portability as a goal**

## root cause summary

| layer | issue | is root cause? |
|-------|-------|----------------|
| symptom | github app has no guided setup | no |
| surface | vault prompts for "the secret" generically | no |
| middle | mech adapter lacks promptForSet method | partial |
| middle | vault/mech coupled state prevents portability | partial |
| root | **architecture conflates storage with credential acquisition** | **yes** |

---

## recommendation

### address root cause

separate storage (vault) from credential acquisition (mech):

1. **extend mech adapter interface**: add `promptForSet` method that runs guided setup and produces source credential
2. **extend vault adapter interface**: add `supportedMechs` list and `checkMechCompat` validation
3. **refactor set flow**: vault inference → vault lookup → mech inference → mech.promptForSet → vault.set(secret)
4. **move extant logic**: setupAwsSsoProfile moves from vault adapter to mech adapter

### why root cause fix is appropriate

- fixes github app guided setup (primary symptom)
- makes mechs portable across vaults (secondary benefit)
- enables mech inference when multiple mechs valid (secondary benefit)
- cleans up architecture for future extensibility (long-term benefit)
- zero backwards compat requirement per memory — clean refactor

### what follow-up is needed

- update vault adapter tests to verify mech receives secret from promptForSet
- update mech adapter tests to cover promptForSet flows
- document vault/mech compatibility matrix
- rename aws.iam.sso → aws.config per vision
