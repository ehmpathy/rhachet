# reflection: audience

reflection for mech adapters own their prompts.

---

## who is the audience?

### primary: developers who use keyrack with ephemeral credentials

developers who need short-lived tokens from long-lived source credentials:

- **github app users**: developers who use github app installations to authenticate CI/CD, bots, or automation. currently must craft json blobs manually.
- **aws sso users**: developers who authenticate via aws sso browser flow. currently works but setup is tied to aws.iam.sso vault.

these developers run `keyrack set` and expect guided setup — not manual json construction.

### secondary: devops and infrastructure engineers

engineers who configure credential management across teams:

- set up github apps for org-wide automation
- configure aws sso profiles for team members
- maintain vault configurations (os.secure, 1password, aws.config)

they care about consistency and portability of credential patterns.

### tertiary: keyrack maintainers

developers who extend keyrack:

- add new vaults (e.g., hashicorp vault, azure keyvault)
- add new mechs (e.g., EPHEMERAL_VIA_GOOGLE_OIDC)
- maintain extant adapters

they care about clean separation of concerns and testable contracts.

---

## why do they care?

### github app users

**pain**: manual json construction is error-prone
- must look up appId and installationId manually
- must format privateKey with correct newlines
- typos cause cryptic failures on unlock

**goal**: self-service credential setup
- discover orgs and apps via api
- point to .pem file, keyrack handles format
- verify setup works via roundtrip test

**frustration**: no guided path for ephemeral github credentials
- aws sso has guided setup, github app does not
- feels like a second-class citizen

### aws sso users

**pain**: none with current flow — guided setup works
- but internally coupled to aws.iam.sso vault

**goal**: maintain current experience
- same prompts, same flow, same output

**frustration**: cannot use aws sso with other vaults (if desired)
- not a common request, but limits flexibility

### devops engineers

**pain**: inconsistent setup experiences across mechs
- github app: manual json
- aws sso: guided prompts
- replica: simple secret prompt

**goal**: uniform patterns across credential types
- all mechs have guided setup
- all mechs produce well-formed source credentials

**frustration**: vault/mech coupler makes patterns hard to standardize
- setup logic scattered across vault adapters

### keyrack maintainers

**pain**: new mechs require vault modifications
- promptForSet logic baked into vaults
- each vault must understand each mech

**goal**: clean adapter interfaces
- mech adapter owns prompts and transformation
- vault adapter owns storage
- composition via dependency injection

**frustration**: test coverage requires vault × mech combinations
- hard to test mech logic in isolation

| aws sso users | low | low | no change to experience |

## what do they care about?

### github app users

**success criteria**:
- guided prompts discover orgs and apps
- single .pem path input, not manual json
- roundtrip verification on set
- clear error messages on failure

**acceptable tradeoffs**:
- extra prompts ok if they guide to success
- gh cli dependency ok (common tool)

**delight vs disappointment**:
- delight: auto-select when single org/app
- disappointment: fall back to manual json without clear guidance

### aws sso users

**success criteria**:
- same prompts as before
- same output as before
- no regressions

**acceptable tradeoffs**:
- internal restructure ok if invisible to user

**delight vs disappointment**:
- delight: identical experience
- disappointment: any change in flow or error messages

### devops engineers

**success criteria**:
- consistent pattern across all mechs
- clear vault/mech compatibility matrix
- fail-fast on incompatible combos

**acceptable tradeoffs**:
- more explicit --mech flag ok if auto-inference covers common cases

**delight vs disappointment**:
- delight: documentation of patterns, predictable behavior
- disappointment: inconsistent experiences between mechs

### keyrack maintainers

**success criteria**:
- mech adapter interface is testable in isolation
- vault adapter interface is pure storage
- composition is explicit via (input, context) pattern

**acceptable tradeoffs**:
- more files ok if separation is clean

**delight vs disappointment**:
- delight: add new mech without touch vault code
- disappointment: mech/vault coupled code remains

## how will they experience it?

### github app user journey

1. **discover**: "i need github app tokens in keyrack"
2. **invoke**: `rhx keyrack set --key GITHUB_TOKEN --vault os.secure --mech EPHEMERAL_VIA_GITHUB_APP`
3. **select org**: prompted with list from `gh api /user/orgs`, pick by number
4. **select app**: prompted with list from `gh api /orgs/{org}/installations`, pick by number
5. **provide pem**: prompted for path to .pem file
6. **verify**: keyrack runs roundtrip (unlock → get → relock)
7. **success**: see confirmation with slug, mech, vault

**friction points**:
- gh cli not authenticated → fallback to manual json (clear error)
- .pem path invalid → fail-fast with path shown
- no apps in org → clear error with guidance

### aws sso user journey

1. **discover**: "i need aws sso credentials in keyrack"
2. **invoke**: `rhx keyrack set --key AWS_PROFILE --vault aws.config`
3. **select mech**: prompted if vault supports multiple (sso vs key)
4. **guided setup**: same prompts as before (region, account, role)
5. **verify**: keyrack runs roundtrip
6. **success**: see confirmation

**friction points**:
- none expected — same as before
- vault rename (aws.iam.sso → aws.config) requires update to docs

### devops engineer journey

1. **audit**: "which mechs work with which vaults?"
2. **consult**: check vault/mech compatibility matrix
3. **configure**: set up credentials for team
4. **verify**: roundtrip works
5. **document**: record patterns for team

**friction points**:
- compatibility matrix must be clear and complete
- fail-fast errors must suggest alternatives

### maintainer journey

1. **extend**: "i need to add a new mech"
2. **implement**: create mech adapter with validate, translate, promptForSet
3. **register**: add to mech adapter registry
4. **test**: unit test mech in isolation, integration test with vaults
5. **document**: update vault/mech matrix

**friction points**:
- interface must be clear and documented
- test patterns must be reusable
