## claims: credential architecture

### [FACT] vault encryption requires multi-layer key separation

> "Vault encrypts the data using an encryption key (in the keyring)" and stores it in the storage backend. The encryption key itself is protected through an additional layer: the system encrypts it using a root key, which is then encrypted using an unseal key.

source: [HashiCorp Vault Seal Concepts](https://developer.hashicorp.com/vault/docs/concepts/seal) [official] 2026

**implication:** separation of storage from access is a proven pattern. vaults store encrypted data; access requires separate unlock mechanism.

### [FACT] sealed state separates storage access from decryption

> When a Vault server starts, it remains in a sealed state despite having access to physical storage. This means the system can "access the physical storage, but it cannot decrypt any of the data on it" without completing an unseal process.

source: [HashiCorp Vault Seal Concepts](https://developer.hashicorp.com/vault/docs/concepts/seal) [official] 2026

**implication:** keyrack daemon parallels this — vaults hold encrypted secrets, unlock transforms them to usable form.

### [FACT] 1Password separates storage from usage via secret references

> "you can use secret references to securely load information saved in 1Password into environment variables, configuration files, and scripts without exposing any secrets in plaintext."

source: [1Password Secret References](https://developer.1password.com/docs/cli/secret-references) [official] 2026

**implication:** op:// URI pattern validates separation of storage (vault) from usage (runtime injection).

### [FACT] 1Password provides three resolution methods for secrets

> `op read`: prints secrets to stdout. `op run`: scans environment variables for secret references, loads values, runs subprocess. `op inject`: replaces secret references in files with actual secrets.

source: [1Password Secret References](https://developer.1password.com/docs/cli/secret-references) [official] 2026

**implication:** multiple transformation methods for same source credential — supports mech adapter pattern where transform varies by context.

## claims: ephemeral vs permanent credentials

### [FACT] github app JWTs expire in 10 minutes maximum

> The `iat` claim should be set 60 seconds in the past to account for clock drift. Tokens expire within a strict 10-minute maximum window.

source: [GitHub JWT Generation](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/generating-a-json-web-token-jwt-for-a-github-app) [official] 2026

**implication:** JWT is intermediate credential; must transform to installation access token for actual use.

### [FACT] github installation access tokens expire in 1 hour

> "The installation access token will expire after 1 hour," requiring regeneration for continued access.

source: [GitHub Installation Access Tokens](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/generating-an-installation-access-token-for-a-github-app) [official] 2026

**implication:** EPHEMERAL_VIA_GITHUB_APP mech must transform source (private key + app id + installation id) to short-lived ghs_ token on unlock.

### [FACT] installation token generation requires JWT exchange

> The transformation from source credentials to usable tokens follows these steps: JWT Creation, Installation ID Retrieval, Token Exchange via POST to `/app/installations/INSTALLATION_ID/access_tokens`.

source: [GitHub Installation Access Tokens](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/generating-an-installation-access-token-for-a-github-app) [official] 2026

**implication:** mech adapter transform method must implement this multi-step exchange.

### [SUMP] dynamic secrets are safer than static secrets

> "When an application starts, it could request its database credentials, which, when dynamically generated, will be provided with new credentials for that session." Static credentials require rotation; dynamic credentials expire naturally upon service restart, reducing compromise window.

source: [OWASP Secrets Management Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Secrets_Management_Cheat_Sheet.html) [official] 2026

**implication:** ephemeral mechs (github app, aws sso) that produce short-lived tokens follow security best practices.

### [OPIN] secrets should expire after defined time where possible

> "secrets to expire after a defined time where possible"

source: [OWASP Secrets Management Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Secrets_Management_Cheat_Sheet.html) [official] 2026

**implication:** ephemeral mechs align with security recommendations; permanent mechs (REPLICA) are acceptable fallback.

## claims: aws sso credential flow

### [FACT] aws sso uses sso-session sections for token refresh

> Uses automated token refresh with `sso-session` sections in the `~/.aws/config` file. Legacy direct profile-level SSO configuration produces non-refreshable tokens.

source: [AWS CLI SSO Configuration](https://docs.aws.amazon.com/cli/latest/userguide/cli-configure-sso.html) [official] 2026

**implication:** aws.config vault must write sso-session format for proper token lifecycle.

### [FACT] aws sso guided setup prompts for session, url, region, scopes

> Guided setup prompts: SSO session name, SSO Start URL or Issuer URL, SSO region, Registration scopes (`sso:account:access`).

source: [AWS CLI SSO Configuration](https://docs.aws.amazon.com/cli/latest/userguide/cli-configure-sso.html) [official] 2026

**implication:** EPHEMERAL_VIA_AWS_SSO mech adapter must prompt for these fields.

### [FACT] aws sso tokens cache to ~/.aws/sso/cache/

> Authentication tokens: cached to disk under `~/.aws/sso/cache/` with filename based on `sso_start_url`. Credentials auto-renewed if SSO token valid.

source: [AWS CLI SSO Configuration](https://docs.aws.amazon.com/cli/latest/userguide/cli-configure-sso.html) [official] 2026

**implication:** aws.config vault writes to ~/.aws/config; sso cache is separate concern handled by aws cli.

### [FACT] aws sso supports PKCE and device authorization flows

> PKCE Authorization (Default, CLI v2.22.0+) requires browser on same device. Device Authorization (older versions or `--use-device-code`) can use different device.

source: [AWS CLI SSO Configuration](https://docs.aws.amazon.com/cli/latest/userguide/cli-configure-sso.html) [official] 2026

**implication:** unlock flow triggers browser auth; this is aws cli's responsibility, not keyrack's.

## claims: github api discovery

### [FACT] user organizations listed via GET /user/orgs

> The `GET /user/orgs` endpoint shows organizations where the authenticated user has membership. Fine-grained tokens may receive empty list due to scope limitations.

source: [GitHub Organizations API](https://docs.github.com/en/rest/orgs/orgs) [official] 2026

**implication:** guided setup for github app mech can enumerate user's orgs via gh cli.

### [FACT] org installations listed via GET /orgs/{org}/installations

> `GET /orgs/{org}/installations` requires authenticated user to be organization owner. Response includes app metadata: permissions, account details, repository selection scope.

source: [GitHub Organizations API](https://docs.github.com/en/rest/orgs/orgs) [official] 2026

**implication:** guided setup can enumerate apps installed for selected org; auto-select if single app.

### [FACT] app installations listed via GET /app/installations

> The endpoint `GET /app/installations` lists all installations for authenticated GitHub App. Requires JWT authentication.

source: [GitHub Apps API](https://docs.github.com/en/rest/apps/apps) [official] 2026

**implication:** alternative discovery path if we authenticate as the app itself.

## claims: cli ux patterns

### [FACT] prompts should only appear on interactive terminals

> "Only use prompts or interactive elements if `stdin` is an interactive terminal (a TTY)."

source: [Command Line Interface Guidelines](https://clig.dev/) [community] 2026

**implication:** mech guided setup must detect TTY; fallback to manual input if non-interactive.

### [FACT] prompts should never be mandatory

> "Never _require_ a prompt. Always provide a way of passing input with flags or arguments."

source: [Command Line Interface Guidelines](https://clig.dev/) [community] 2026

**implication:** all mech prompts must have flag equivalents (--app-id, --installation-id, --private-key-path).

### [SUMP] interactive setup is acceptable for initialization commands

> "You can ignore this guideline if your program is interactive by default (e.g. `npm init`)."

source: [Command Line Interface Guidelines](https://clig.dev/) [community] 2026

**implication:** keyrack set is an initialization command; guided prompts are appropriate.

### [FACT] password prompts should disable terminal echo

> "If you're prompting for a password, don't print it as the user types." This requires disabling terminal echo during input.

source: [Command Line Interface Guidelines](https://clig.dev/) [community] 2026

**implication:** if prompting for raw secrets, must hide input.

### [OPIN] suggest next commands to guide users

> "Suggest commands the user should run. When several commands form a workflow, suggesting to the user commands they can run next helps them learn how to use your program."

source: [Command Line Interface Guidelines](https://clig.dev/) [community] 2026

**implication:** after set success, hint at unlock command.

## claims: progressive disclosure

### [FACT] progressive disclosure shows essential options first

> "Initially, show users only a few of the most important options" with "a larger set of specialized options upon request."

source: [Nielsen Norman Group Progressive Disclosure](https://www.nngroup.com/articles/progressive-disclosure/) [research] 2026

**implication:** guided setup should prompt for required fields first; advanced options later.

### [FACT] staged disclosure guides users through linear sequences

> A related variant called staged disclosure guides users through linear task sequences. Wizards exemplify this approach, revealing task-relevant features one step at a time.

source: [Nielsen Norman Group Progressive Disclosure](https://www.nngroup.com/articles/progressive-disclosure/) [research] 2026

**implication:** mech guided setup is staged disclosure: org → app → pem path.

### [FACT] progressive disclosure reduces cognitive load

> For novice users, "hiding the advanced settings...helps novice users avoid mistakes and saves them the time they would have spent contemplating features that they don't need."

source: [Nielsen Norman Group Progressive Disclosure](https://www.nngroup.com/articles/progressive-disclosure/) [research] 2026

**implication:** auto-select single org/app reduces choices; skip prompts when unambiguous.

## claims: separation of concerns

### [FACT] twelve-factor requires strict config/code separation

> "strict separation of config from code" is mandatory. Configuration encompasses anything varying across deployments—database credentials, API keys, service endpoints.

source: [The Twelve-Factor App: Config](https://12factor.net/config) [methodology] 2026

**implication:** mech adapters separate prompt logic (code) from stored credentials (config).

### [FACT] environment groupings don't scale

> "Environment groupings" (development/prep/production) create "a combinatorial explosion of config which makes managing deploys very brittle."

source: [The Twelve-Factor App: Config](https://12factor.net/config) [methodology] 2026

**implication:** keyrack env.all pattern with fallback handles this gracefully.

### [SUMP] sidecars should handle secrets authentication

> "A sidecar container authenticates with the secrets manager (e.g., using a Kubernetes Service Account)" and retrieves secrets for the main application, preventing direct coupling.

source: [OWASP Secrets Management Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Secrets_Management_Cheat_Sheet.html) [official] 2026

**implication:** keyrack daemon acts as sidecar pattern — handles auth/retrieval, app just calls get.

### [FACT] two-phase commit decomposes operations into prepare and commit

> "Each node participating in the transaction acquires whatever it needs to assure that it will be able to do the commit in the second phase—for example, any locks that are required."

source: [Martin Fowler: Two Phase Commit](https://www.martinfowler.com/articles/patterns-of-distributed-systems/two-phase-commit.html) [methodology] 2026

**implication:** keyrack set flow parallels: prepare (prompt + validate) then commit (vault storage).

## claims: compatibility and fail-fast

### [SUMP] least privilege restricts access to required vaults only

> "using 1Password Service Accounts to follow the principle of least privilege," restricting processes to access only required vaults and items.

source: [1Password Secret References](https://developer.1password.com/docs/cli/secret-references) [official] 2026

**implication:** vault/mech compatibility validation enforces proper boundaries.

### [KHUE] how should incompatible vault/mech combos fail?

multiple sources emphasize fail-fast with clear guidance. "Catch errors and rewrite them for humans...the program is guiding them in the right direction."

source: [Command Line Interface Guidelines](https://clig.dev/) [community] 2026

**question:** should error messages include specific alternative suggestions, or just list valid options?

### [SUMP] engineers should not access all secrets

> "engineers should not have access to all secrets in the secrets management system."

source: [OWASP Secrets Management Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Secrets_Management_Cheat_Sheet.html) [official] 2026

**implication:** os.direct failing for ephemeral mechs protects source keys from exposure.

## conflict detection

### no major conflicts detected

sources converge on:
1. separation of storage from transformation is best practice
2. ephemeral credentials reduce compromise window
3. guided setup with progressive disclosure improves ux
4. fail-fast with clear errors guides users to success
5. token lifetimes: jwt (10min), installation token (1hr), sso token (refreshable)

### minor variance

| topic | variance |
|-------|----------|
| prompt necessity | clig says "never require prompts" but also accepts interactive setup commands |
| token strategy | some orgs prefer permanent tokens for simplicity despite security tradeoffs |

strong consensus across 12+ sources:
- **mech-vault separation** validated by hashicorp, 1password, aws, owasp patterns
- **guided setup** validated by clig, nngroup progressive disclosure research
- **ephemeral preference** validated by github (1hr tokens), aws (refreshable sso), owasp recommendations

## anti-patterns identified

| anti-pattern | source | why avoid |
|--------------|--------|-----------|
| hard-coded credentials | twelve-factor | violates separation, security risk |
| environment groupings | twelve-factor | combinatorial explosion |
| mandatory prompts without flag escape | clig | breaks automation |
| static secrets without rotation | owasp | compromise window |
| direct app-to-secrets coupling | owasp | sidecar pattern preferred |

## implementation implications

### for mech adapters

1. **promptForSet must detect TTY** — fallback to flag-based input
2. **transform method implements token exchange** — jwt → installation token for github app
3. **single option auto-select** — skip prompt when only one org/app available

### for vault adapters

1. **supportedMechs list** — explicit compatibility declaration
2. **checkMechCompat fail-fast** — clear error with alternatives
3. **aws.config writes sso-session format** — not legacy profile format

### for mech inference

1. **prompt only when multiple valid mechs** — auto-select if single option
2. **show descriptions with mech names** — "aws sso (short-lived tokens via browser)"

### for guided setup

1. **staged disclosure** — org → app → pem path
2. **progressive validation** — validate each step before next
3. **suggest next command** — hint unlock after set success
