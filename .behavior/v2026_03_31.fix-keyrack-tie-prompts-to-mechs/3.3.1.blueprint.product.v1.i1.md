# blueprint: mech adapters own their prompts

blueprint for mech adapters own their prompts.

---

## summary

refactor keyrack so that mech adapters own their guided setup prompts. this makes mechs portable across compatible vaults.

**changes:**
1. extend mech adapter interface with `acquireForSet` and `deliverForGet` methods
2. extend vault adapter interface with `mechs.supported`
3. add guided setup to github app mech (org → app → pem path)
4. move aws sso guided setup from vault adapter to mech adapter
5. rename `aws.iam.sso` vault to `aws.config`
6. add vault inference from key name (AWS_PROFILE → aws.config)
7. add mech inference when vault supports multiple mechs
8. remove prompts from vault adapters (vaults become pure storage)

**outcome:** EPHEMERAL_VIA_GITHUB_APP works with os.secure, 1password, or any compatible vault with the same guided setup experience.

## codepath tree

### set flow (target)

```
keyrack set --key GITHUB_TOKEN --vault os.secure --mech EPHEMERAL_VIA_GITHUB_APP

setKeyrackKey
├─ [+] inferVault (if --vault absent)
│  └─ AWS_PROFILE → aws.config, else null
│
├─ [○] vault adapter lookup
│  └─ context.vaultAdapters[input.vault]
│
├─ [~] vault.set({ slug, mech?, exid? })
│  │
│  ├─ [+] inferKeyrackMechForSet (if mech absent)
│  │  ├─ filter to vault.mechs.supported only
│  │  ├─ single compatible mech → auto-select
│  │  └─ multiple compatible mechs → prompt via stdin
│  │
│  ├─ [+] check mech compat (if mech explicit)
│  │  └─ fail-fast if mech not in vault.mechs.supported
│  │
│  ├─ [+] mech.acquireForSet({ keySlug })
│  │  ├─ EPHEMERAL_VIA_GITHUB_APP
│  │  │  ├─ gh api /user/orgs → list github orgs
│  │  │  ├─ prompt github org selection
│  │  │  ├─ gh api /orgs/{githubOrg}/installations → list
│  │  │  ├─ prompt app selection
│  │  │  ├─ prompt pem path
│  │  │  ├─ read pem file, escape newlines
│  │  │  └─ return json blob
│  │  ├─ EPHEMERAL_VIA_AWS_SSO
│  │  │  ├─ [←] reuse setupAwsSsoWithGuide logic
│  │  │  └─ return profile name
│  │  └─ PERMANENT_VIA_REPLICA
│  │     ├─ prompt secret via stdin
│  │     └─ return secret
│  │
│  ├─ [~] store secret (never leaves vault scope)
│  └─ [○] return { exid? }
│
├─ [○] daoKeyrackHostManifest.set
│
└─ [○] roundtrip validation
   ├─ unlock → get → relock
   └─ return success
```

### unlock flow (vault encapsulates delivery)

```
keyrack unlock --key GITHUB_TOKEN --env all

unlockKeyrackKey
├─ [○] vault adapter lookup
├─ [○] vault.unlock
├─ [~] vault.get({ slug }) → usable secret
│  │
│  ├─ [○] retrieve source from storage
│  │
│  ├─ [~] mech.deliverForGet({ source })
│  │  ├─ EPHEMERAL_VIA_GITHUB_APP
│  │  │  ├─ parse json blob
│  │  │  ├─ createAppAuth → token
│  │  │  └─ return { secret: ghs_..., expiresAt }
│  │  └─ PERMANENT_VIA_REPLICA
│  │     └─ return { secret: source } (identity)
│  │
│  └─ [○] return translated secret
│
└─ [○] daemon.set → store translated secret
```

### mech adapter interface (target)

```
KeyrackGrantMechanismAdapter
├─ [○] validate
│  └─ { source?, cached? } → { valid, reasons? }
├─ [~] acquireForSet
│  └─ { keySlug } → { source: string }
│  └─ note: keySlug is fully qualified (org.env.name) for display
│  └─ note: mech discovers external resources independently (e.g., gh api /user/orgs)
└─ [~] deliverForGet
   └─ { source } → { secret, expiresAt? }
   └─ note: replaces old translate method
```

### vault adapter interface (target)

```
KeyrackHostVaultAdapter
├─ [○] unlock
├─ [○] isUnlocked
├─ [○] get
├─ [~] set
│  └─ { slug, mech?, exid? } → void | { exid }
│     note: vault calls mech.promptForSet internally; secret never exposed
├─ [○] del
├─ [○] relock
└─ [+] mechs
   └─ supported: KeyrackGrantMechanism[]
      note: compat check is inline (mech in vault.mechs.supported)
```

## risks and mitigations

| risk | mitigation |
|------|------------|
| break extant aws sso flow | move logic, not rewrite; same prompts, new location |
| gh cli auth state detection | clear fallback path with per-field prompts |
| pem file format edge cases | validate format before store, fail fast with guidance |
| vault/mech matrix grows complex | explicit mechs.supported list, fail-fast with alternatives |

---

## out of scope

- new mechanisms (e.g., PERMANENT_VIA_AWS_KEY) — scaffold only
- keyrack daemon changes
- unlock flow structural changes (only mech.translate used, already correct)

## in scope (explicit)

- **usecase.2 (github app + 1password)** — 1password vault adapter supports EPHEMERAL_VIA_GITHUB_APP via mechs.supported, same guided setup as os.secure
