# evaluation: mech adapters own their prompts

evaluation of implementation relative to blueprint.

---

## summary (as implemented)

refactored keyrack so that mech adapters own their guided setup prompts. mechs are now portable across compatible vaults.

**changes implemented:**
1. extended mech adapter interface with `acquireForSet` and `deliverForGet` methods
2. extended vault adapter interface with `mechs.supported`
3. added guided setup to github app mech (org → app → pem path)
4. moved aws sso guided setup from vault adapter to mech adapter
5. renamed `aws.iam.sso` vault to `aws.config`
6. added vault inference from key name (AWS_PROFILE → aws.config)
7. added mech inference when vault supports multiple mechs
8. removed prompts from vault adapters (vaults became pure storage)

**outcome:** EPHEMERAL_VIA_GITHUB_APP works with os.secure, 1password, or any compatible vault with the same guided setup experience.

## codepath tree (as implemented)

### set flow

```
keyrack set --key GITHUB_TOKEN --vault os.secure --mech EPHEMERAL_VIA_GITHUB_APP

setKeyrackKey
├─ inferVault (if --vault absent)
│  └─ AWS_PROFILE → aws.config, else null
│
├─ vault adapter lookup
│  └─ context.vaultAdapters[input.vault]
│
├─ vault.set({ slug, mech?, exid? })
│  │
│  ├─ inferKeyrackMechForSet (if mech absent)
│  │  ├─ filter to vault.mechs.supported only
│  │  ├─ single compatible mech → auto-select
│  │  └─ multiple compatible mechs → prompt via stdin
│  │
│  ├─ check mech compat (if mech explicit)
│  │  └─ fail-fast if mech not in vault.mechs.supported
│  │
│  ├─ mech.acquireForSet({ keySlug })
│  │  ├─ EPHEMERAL_VIA_GITHUB_APP
│  │  │  ├─ gh api /user/orgs → list github orgs
│  │  │  ├─ prompt github org selection
│  │  │  ├─ gh api /orgs/{githubOrg}/installations → list
│  │  │  ├─ prompt app selection
│  │  │  ├─ prompt pem path
│  │  │  ├─ read pem file, escape newlines
│  │  │  └─ return json blob
│  │  ├─ EPHEMERAL_VIA_AWS_SSO
│  │  │  ├─ setupAwsSsoWithGuide logic
│  │  │  └─ return profile name
│  │  └─ PERMANENT_VIA_REPLICA
│  │     ├─ prompt secret via stdin
│  │     └─ return secret
│  │
│  ├─ store secret (never leaves vault scope)
│  └─ return { mech, exid? }
│
├─ daoKeyrackHostManifest.set
│
└─ roundtrip validation
   ├─ unlock → get → relock
   └─ return success
```

### unlock flow (vault encapsulates delivery)

```
keyrack unlock --key GITHUB_TOKEN --env all

unlockKeyrackKey
├─ vault adapter lookup
├─ vault.unlock
├─ vault.get({ slug, mech }) → usable secret
│  │
│  ├─ retrieve source from storage
│  │
│  ├─ mech.deliverForGet({ source })
│  │  ├─ EPHEMERAL_VIA_GITHUB_APP
│  │  │  ├─ parse json blob
│  │  │  ├─ createAppAuth → token
│  │  │  └─ return { secret: ghs_..., expiresAt }
│  │  └─ PERMANENT_VIA_REPLICA
│  │     └─ return { secret: source } (identity)
│  │
│  └─ return translated secret
│
└─ daemon.set → store translated secret
```

## mech adapter interface (as implemented)

```ts
interface KeyrackGrantMechanismAdapter {
  validate: (input: { source?: string; cached?: string }) => {
    valid: boolean;
    reasons?: string[];
  };

  acquireForSet: (input: { keySlug: string }) => Promise<{ source: string }>;

  deliverForGet: (input: {
    source: string;
  }) => Promise<{ secret: string; expiresAt?: IsoTimeStamp }>;
}
```

```ts
interface KeyrackHostVaultAdapter {
```

### minor divergences

1. **setupAwsSsoWithGuide.ts retained**: blueprint declared `[-] delete (content moved to mech)`. implementation retained the file and `mechAdapterAwsSso.acquireForSet` calls it. this is a composition pattern — the logic lives in the helper, mech adapter wraps it. outcome equivalent.

2. **additional vaults**: `os.daemon` and `os.envvar` also have `mechs.supported` added. blueprint marked these out of scope but they were completed for consistency.

3. **PERMANENT_VIA_AWS_KEY absent**: blueprint matrix shows this mech for aws.config, but implementation only has EPHEMERAL_VIA_AWS_SSO. blueprint explicitly noted "new mechanisms (e.g., PERMANENT_VIA_AWS_KEY) — scaffold only" in out of scope section.
