# vision: mech adapters own their prompts

## the outcome world

### before: prompts scattered, mechs limited

today, each vault adapter handles its own prompts — but mechs are second-class citizens:

```
keyrack set --key GITHUB_TOKEN --vault os.secure

enter secret for GITHUB_TOKEN: ********

🔐 keyrack set (org: ehmpathy, env: all)
   └─ ehmpathy.all.GITHUB_TOKEN
      ├─ mech: PERMANENT_VIA_REPLICA
      └─ vault: os.secure
```

the problem surfaces when you try to use `EPHEMERAL_VIA_GITHUB_APP`:

1. **no guided setup** — you must manually craft a json blob `{"appId": "...", "privateKey": "...", "installationId": "..."}`
2. **error-prone** — typos in json, wrong newlines in pem, missing fields
3. **discoverability is zero** — how do you know what installationId is? where do you find it?

meanwhile, `aws.iam.sso` has beautiful guided setup — but it's baked into the vault adapter, not the mechanism. this means `EPHEMERAL_VIA_AWS_SSO` only works with `aws.iam.sso` vault, never with `os.secure` or `1password`.

### after: mechs drive their own setup

each mechanism owns its prompts. vaults become pure storage backends.

```
keyrack set --key GITHUB_TOKEN --vault os.secure --mech EPHEMERAL_VIA_GITHUB_APP

🔐 keyrack set GITHUB_TOKEN via EPHEMERAL_VIA_GITHUB_APP
   │
   ├─ which org?
   │  ├─ options
   │  │  ├─ 1. ehmpathy
   │  │  └─ 2. bhuild
   │  └─ choice
   │     └─ 1 ✓
   │
   ├─ which app?
   │  ├─ options
   │  │  ├─ 1. beaver-by-bhuild (id: 3234162)
   │  │  └─ 2. seaturtle-ci (id: 8234521)
   │  └─ choice
   │     └─ 1 ✓
   │
   ├─ where's the private key?
   │  └─ path
   │     └─ ./beaver.2026-04-01.pem ✓
   │
   └─ perfect, now lets verify...
      ├─ ✓ unlock
      ├─ ✓ get
      └─ ✓ relock

🔐 keyrack set (org: ehmpathy, env: all)
   └─ ehmpathy.all.GITHUB_TOKEN
      ├─ mech: EPHEMERAL_VIA_GITHUB_APP
      └─ vault: os.secure
```

### the "aha" moment

"wait — i can use github app credentials with 1password now? and i get the same guided setup?"

yes. the mech adapter drives the prompts. the vault just stores whatever the mech produces.

## user experience

### usecase 1: github app token with os.secure

**goal:** store github app credentials securely, get short-lived tokens on unlock

**contract:**
```bash
rhx keyrack set --key GITHUB_TOKEN --env all --mech EPHEMERAL_VIA_GITHUB_APP --vault os.secure
```

**timeline:**
1. user selects org (from `gh api /user/orgs` or manual entry)
2. mech fetches apps via `gh api /orgs/{org}/installations`
3. user selects app → mech prompts for .pem path
4. mech formats credentials as json blob
5. vault adapter encrypts and stores the blob
6. on unlock: vault decrypts → mech translates json → short-lived ghs_ token

### usecase 2: github app token with 1password

**goal:** same as above, but store in 1password instead of local age-encrypted file

**contract:**
```bash
rhx keyrack set --key GITHUB_TOKEN --env all --mech EPHEMERAL_VIA_GITHUB_APP --vault 1password --exid op://vault/github-app-creds
```

**timeline:**
1. mech adapter runs same guided setup as above
2. mech produces json blob
3. 1password adapter stores blob at exid location
4. on unlock: 1password fetches → mech translates → ghs_ token

### usecase 3: aws sso (unchanged externally, restructured internally)

**goal:** maintain extant ux while moving prompts to mech

**contract:**
```bash
rhx keyrack set --key AWS_PROFILE --env test --vault aws.config
```

**timeline:**
1. vault supports multiple mechs → mech inference prompts via stdin
2. user selects mech (e.g., EPHEMERAL_VIA_AWS_SSO)
3. mech adapter runs guided setup (same prompts as today)
4. mech produces profile config
5. vault adapter writes to `~/.aws/config`
6. on unlock: vault retrieves → mech translates to sso session

## mental model

### how users describe it

> "the mechanism knows what it needs, the vault just stores it"

### analogies

| old model | new model |
|-----------|-----------|
| vault = bank + teller | vault = safe deposit box |
| vault handles prompts | mech handles prompts |
| vault + mech tightly coupled | mech portable across vaults |

**the factory analogy:**
- mech = the machine that manufactures credentials
- vault = the warehouse that stores them
- the machine knows what raw materials it needs (prompts)
- the warehouse doesn't care what's inside the boxes

### terms: user vs internal

| user says | we call it |
|-----------|------------|
| "storage backend" | vault |
| "credential type" | mech |
| "interactive setup" | guided setup |
| "the json blob thing" | source credential |
| "the actual token" | translated secret |

## evaluation

### how well does it solve the goals?

| goal | before | after |
|------|--------|-------|
| EPHEMERAL_VIA_GITHUB_APP has guided setup | ✗ manual json | ✓ org/app/pem prompts |
| mechs work with any compatible vault | ✗ coupled | ✓ portable |
| aws sso experience unchanged | ✓ | ✓ (restructured internally) |
| vault/mech compat validation | ✗ implicit | ✓ explicit fail-fast |

### pros

1. **discoverability** — github app setup becomes self-service
2. **portability** — EPHEMERAL_VIA_GITHUB_APP works with os.secure, 1password, etc
3. **separation of concerns** — vaults store, mechs transform
4. **consistency** — all guided setups follow the same pattern
5. **extensibility** — new mechs only need to implement adapter interface

### cons

1. **migration complexity** — internal restructure touches many files
2. **interface growth** — mech adapters gain `set` method with prompts
3. **testing surface** — more combinations to test (vault × mech)

### pit of success

| edgecase | protection |
|----------|------------|
| os.direct + ephemeral mechs | fail-fast: os.direct cannot secure source keys |
| os.secure + EPHEMERAL_VIA_AWS_SSO | fail-fast: aws sso only works with aws.config vault |
| missing --mech for multi-mech vaults | mech inference adapter prompts via stdin |
| incompatible vault/mech combo | fail-fast with clear error + suggested alternatives |
| single org/app available | auto-select, skip choice prompt |

## open questions & assumptions

### assumptions

1. **github api access** — we assume `gh` cli is authenticated and can fetch installations. **fallback:** if gh unavailable, allow manual json input as escape hatch.
2. **pem file format** — we assume user can provide a .pem file path (not paste content)
3. **aws.iam.sso → aws.config rename** — wisher confirmed this name
4. **zero backwards compat** — per memory, we delete old patterns, never shim
5. **vaults own storage format** — vaults still own how they store (aws.config writes to ~/.aws/config). mechs own prompts and transformation, not storage format.

### questions for wisher

1. **vault rename: [answered]** — `aws.iam.sso` becomes `aws.config`
   - named after `~/.aws/config` where profile configuration is stored
   - wisher confirmed in wish: "lets make it clear that only 'aws.config' vault supports AWS_PROFILE keys"

2. **mech inference when ambiguous: [answered]** — wisher confirmed mech inference adapters as the approach
   - AWS_PROFILE → infer --vault aws.config, but vault doesn't know if user wants EPHEMERAL_VIA_AWS_SSO or PERMANENT_VIA_AWS_KEY
   - GITHUB_TOKEN → can't infer vault, can't infer mech (EPHEMERAL_VIA_GITHUB_APP or PERMANENT_VIA_REPLICA)
   - **solution:** mech inference adapters that vaults invoke to prompt via stdin when --mech not supplied
   - guided prompts handle both mech selection and credential input

### external research needed

1. **github api shape** — verify `/orgs/{org}/installations` returns what we need
2. **pem newline handling** — verify .pem file read → json escape works correctly
3. **1password op cli** — verify we can store json blobs to arbitrary exid paths

## what is awkward?

### the --mech flag

users rarely think in terms of "mechanisms". they think:
- "i want github app tokens" not "i want EPHEMERAL_VIA_GITHUB_APP"
- "i want aws sso" not "i want EPHEMERAL_VIA_AWS_SSO"

**mitigation: mech inference adapters**

when --mech is not supplied and multiple mechs are valid for the vault/key combo, a mech inference adapter prompts via stdin:

```
keyrack set --key AWS_PROFILE --vault aws.config

🔐 keyrack set AWS_PROFILE
   │
   ├─ which mechanism?
   │  ├─ options
   │  │  ├─ 1. aws sso (EPHEMERAL_VIA_AWS_SSO) — short-lived tokens via browser login
   │  │  └─ 2. aws key (PERMANENT_VIA_AWS_KEY) — long-lived access key + secret
   │  └─ choice
   │     └─ 1 ✓
   │
   └─ ... (mech-specific prompts follow)
```

this means the guided flow handles both:
1. **mech selection** — when ambiguous, ask which type of credential
2. **credential input** — mech adapter prompts for what it needs

### vault/mech compatibility matrix

users must know which combos work:

| vault | PERMANENT_VIA_REPLICA | EPHEMERAL_VIA_GITHUB_APP | EPHEMERAL_VIA_AWS_SSO |
|-------|----------------------|--------------------------|----------------------|
| os.secure | ✓ | ✓ | ✗ |
| os.direct | ✓ | ✗ | ✗ |
| 1password | ✓ | ✓ | ✗ |
| aws.config | ✗ | ✗ | ✓ |

**requirement:** clear error messages with alternatives. "os.direct doesn't support EPHEMERAL_VIA_GITHUB_APP because it can't secure source keys. try os.secure or 1password." always guide callers to success.

### two-step setup awareness

for ephemeral mechs, users need to understand:
1. **set** stores source credentials (json blob, profile name)
2. **unlock** translates to usable tokens (ghs_ token, sso session)

this is fine for power users but may confuse newcomers.

**chosen approach:** guided setup makes this seamless — users don't need to understand the two-step process because the guided flow handles it transparently.
