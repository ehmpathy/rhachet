# blackbox criteria: mech adapters own their prompts

## episode experience

### usecase.1 = github app set with os.secure

```
given(user runs `keyrack set --key GITHUB_TOKEN --vault os.secure --mech EPHEMERAL_VIA_GITHUB_APP`)
  when(prompted for org)
    then(shows list of orgs from github api)
    then(user selects org by number)
      sothat(user discovers available orgs without manual lookup)

  when(prompted for app)
    then(shows list of apps installed for selected org)
    then(user selects app by number)
      sothat(user discovers app id and installation id without manual lookup)

  when(prompted for private key)
    then(accepts relative path to .pem file)
    then(reads and formats pem content)
      sothat(user provides one input instead of manual json construction)

  when(all prompts complete)
    then(mech produces json blob with appId, installationId, privateKey)
    then(vault encrypts and stores the blob)
    then(verifies via unlock → get → relock cycle)
    then(shows success with slug, mech, and vault)
```

### usecase.2 = github app set with 1password

```
given(user runs `keyrack set --key GITHUB_TOKEN --vault 1password --mech EPHEMERAL_VIA_GITHUB_APP --exid op://vault/creds`)
  when(prompted for org, app, pem)
    then(same guided setup as os.secure)
      sothat(mech prompts are portable across vaults)

  when(all prompts complete)
    then(mech produces json blob)
    then(1password adapter stores blob at exid location)
    then(shows success with slug, mech, and vault)
```

### usecase.3 = aws sso set with aws.config

```
given(user runs `keyrack set --key AWS_PROFILE --vault aws.config` without --mech)
  when(vault supports multiple mechs for this key)
    then(mech inference prompts via stdin)
    then(shows mech options: aws sso vs aws key)
    then(user selects mech by number)
      sothat(user discovers mech options without docs lookup)

  when(user selects EPHEMERAL_VIA_AWS_SSO)
    then(mech adapter runs guided setup for sso)
    then(prompts for region, account, role)
    then(mech produces profile config)
    then(vault writes to ~/.aws/config)
      sothat(aws sso experience unchanged from before)
```

### usecase.4 = github app unlock transforms to token

```
given(GITHUB_TOKEN set via EPHEMERAL_VIA_GITHUB_APP)
  when(user runs `keyrack unlock --key GITHUB_TOKEN --env all`)
    then(vault retrieves encrypted json blob)
    then(mech transforms json blob to short-lived ghs_ token)
    then(daemon stores translated token)
      sothat(ephemeral mechs produce usable tokens, not source credentials)

  when(user runs `keyrack get --key GITHUB_TOKEN --env all`)
    then(returns ghs_ token, not json blob)
      sothat(downstream tools receive usable credential)
```

## exchange experience

### usecase.5 = incompatible vault/mech fails fast

```
given(user runs `keyrack set --key GITHUB_TOKEN --vault os.direct --mech EPHEMERAL_VIA_GITHUB_APP`)
  when(vault checks mech compatibility)
    then(fails fast with clear error)
    then(error explains os.direct cannot secure source keys)
    then(error suggests alternatives: os.secure, 1password)
      sothat(user guided to success, not left confused)
```

### usecase.6 = aws sso with os.secure fails fast

```
given(user runs `keyrack set --key AWS_PROFILE --vault os.secure --mech EPHEMERAL_VIA_AWS_SSO`)
  when(vault checks mech compatibility)
    then(fails fast with clear error)
    then(error explains aws sso only works with aws.config vault)
    then(error suggests: use --vault aws.config)
      sothat(user learns vault/mech constraints)
```

### usecase.7 = single org auto-selects

```
given(user has access to exactly one org)
  when(prompted for org)
    then(auto-selects the single org)
    then(skips choice prompt)
    then(shows which org was selected)
      sothat(user not forced to choose when no choice exists)
```

### usecase.8 = single app auto-selects

```
given(selected org has exactly one app installed)
  when(prompted for app)
    then(auto-selects the single app)
    then(skips choice prompt)
    then(shows which app was selected)
      sothat(user not forced to choose when no choice exists)
```

### usecase.9 = gh cli unavailable fallback

```
given(gh cli not authenticated or unavailable)
  when(mech tries to fetch orgs or apps)
    then(falls back to manual json input)
    then(prompts for raw json blob)
      sothat(guided setup degrades gracefully)
```

### usecase.10 = explicit --mech skips inference

```
given(user runs `keyrack set --key AWS_PROFILE --vault aws.config --mech EPHEMERAL_VIA_AWS_SSO`)
  when(--mech is supplied)
    then(skips mech inference prompt)
    then(proceeds directly to mech guided setup)
      sothat(power users can skip selection step)
```

## boundary conditions

### usecase.11 = invalid pem path

```
given(user provides path to nonexistent .pem file)
  when(mech tries to read file)
    then(fails fast with clear error)
    then(error shows which path was tried)
      sothat(user knows exactly what to fix)
```

### usecase.12 = malformed pem content

```
given(user provides path to file that is not valid pem)
  when(mech tries to parse content)
    then(fails fast with clear error)
    then(error explains expected format)
      sothat(user knows pem vs other formats)
```

### usecase.13 = vault infers from key name

```
given(user runs `keyrack set --key AWS_PROFILE` without --vault)
  when(key name matches vault inference pattern)
    then(infers --vault aws.config)
      sothat(common keys require fewer flags)
```

### usecase.14 = vault inference impossible

```
given(user runs `keyrack set --key GITHUB_TOKEN` without --vault)
  when(key name has no inference pattern)
    then(fails fast with clear error)
    then(error lists available vaults)
      sothat(user learns --vault is required for this key)
```
