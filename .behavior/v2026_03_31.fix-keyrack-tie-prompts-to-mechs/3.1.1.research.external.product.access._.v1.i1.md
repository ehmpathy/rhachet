# research: external product access contracts

research for mech adapters own their prompts — remote access requirements

---

## summary

this document catalogs the external APIs, CLIs, and file formats that keyrack mech adapters must interact with:

1. **GitHub Apps API** — for EPHEMERAL_VIA_GITHUB_APP mech (organizations, installations, tokens)
2. **1Password CLI** — for 1password vault adapter (op:// URIs, JSON storage)
3. **AWS config files** — for aws.config vault and EPHEMERAL_VIA_AWS_SSO mech

## 1. GitHub Apps API

### 1.1 list user organizations

**endpoint:** `GET /user/orgs`

**source [1]** — [GitHub REST API: Organizations](https://docs.github.com/en/rest/orgs/orgs) [official]
> "List organizations for the authenticated user."

**source [2]** — [GitHub REST API: Organizations](https://docs.github.com/en/rest/orgs/orgs) [official]
> "For OAuth app tokens and personal access tokens (classic), this endpoint only lists organizations that your authorization allows you to operate on in some way." minimum required scopes are **user** or **read:org**.

**pagination:** "The number of results per page (max 100)" with a default of 30 items. uses standard page-based pagination.

**gh cli alternative:**

**source [3]** — [gh org list](https://www.mankier.com/1/gh-org-list) [official]
> "gh org list" lists organizations for the authenticated user. can also query with `gh api /user/orgs`.

### 1.2 list organization installations

**endpoint:** `GET /orgs/{org}/installations`

**source [4]** — [GitHub REST API: App Installations](https://docs.github.com/en/rest/apps/installations) [official]
> "List app installations accessible to the user access token." requires authentication with `Authorization: Bearer <YOUR-TOKEN>` and `Accept: application/vnd.github+json` headers.

**response format:** returns `total_count` (integer) and `installations` (array) with objects that contain `id`, `account`, `app_slug`, `permissions`, `created_at`, `updated_at`.

**source [5]** — [GitHub REST API: App Installations](https://docs.github.com/en/rest/apps/installations) [official]
> for installations accessible to app, endpoint is `GET /user/installations` with response schema that includes `total_count` and `installations` array with `app_slug` field.

### 1.3 JWT generation for GitHub Apps

**source [6]** — [Generate a JWT for a GitHub App](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/generating-a-json-web-token-jwt-for-a-github-app) [official]
> "your JWT must be signed using the `RS256` algorithm"

**required claims:**

**source [7]** — [Generate a JWT for a GitHub App](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/generating-a-json-web-token-jwt-for-a-github-app) [official]
> - **`iss`** (Issuer): "The client ID or application ID of your GitHub App"
> - **`iat`** (Issued At): "The time that the JWT was created"
> - **`exp`** (Expires At): "The expiration time of the JWT, after which it can't be used"

**clock drift protection:**

**source [8]** — [Generate a JWT for a GitHub App](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/generating-a-json-web-token-jwt-for-a-github-app) [official]
> "set this 60 seconds in the past and ensure that your server's date and time is set accurately"

**expiration constraint:**

**source [9]** — [Generate a JWT for a GitHub App](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/generating-a-json-web-token-jwt-for-a-github-app) [official]
> "The time must be no more than 10 minutes into the future"

### 1.4 installation access token generation

**endpoint:** `POST /app/installations/{installation_id}/access_tokens`

**source [10]** — [Generate an Installation Access Token](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/generating-an-installation-access-token-for-a-github-app) [official]
> requires headers: `Accept: application/vnd.github+json`, `Authorization: Bearer JWT`, `X-GitHub-Api-Version: 2026-03-10`. "if you are passing a JSON web token (JWT), you must use `Authorization: Bearer`"

**token lifetime:**

**source [11]** — [Generate an Installation Access Token](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/generating-an-installation-access-token-for-a-github-app) [official]
> "The installation access token will expire after 1 hour."

**best practice:**

**source [12]** — [Generate an Installation Access Token](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/generating-an-installation-access-token-for-a-github-app) [official]
> "You should keep your installation access token secure" — reference the guidance in "Best practices for GitHub App creation"

### 1.5 private key management

**key format:**

**source [13]** — [Manage Private Keys for GitHub Apps](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/managing-private-keys-for-github-apps) [official]
> "If you're using a library that requires a specific file format, the PEM file you download will be in `PKCS#1 RSAPrivateKey` format."

**storage recommendation:**

**source [14]** — [Manage Private Keys for GitHub Apps](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/managing-private-keys-for-github-apps) [official]
> "Consider storing the key in a key vault, such as Azure Key Vault, and making it sign-only."

**anti-pattern:**

**source [15]** — [Manage Private Keys for GitHub Apps](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/managing-private-keys-for-github-apps) [official]
> "You should not hard-code your private key in your app, even if your code is stored in a private repository."

**key rotation:**

**source [16]** — [Manage Private Keys for GitHub Apps](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/managing-private-keys-for-github-apps) [official]
> "should use multiple keys in order to rotate keys without downtime in the event of a key compromise." maximum 25 keys per app.

### 1.6 authentication methods overview

**source [17]** — [About GitHub App Authentication](https://docs.github.com/en/apps/creating-github-apps/authenticating-with-a-github-app/about-authentication-with-a-github-app) [official]

three authentication methods:
1. **as a GitHub App** — uses JWT
2. **as an app installation** — uses installation access token
3. **on behalf of a user** — uses user access token

> "Your app should authenticate as an app installation when you want to attribute app activity to the app"
> "Authenticate as an app installation is ideal for automation workflows that don't involve user input"

## 2. 1Password CLI

### 2.1 secret reference URI syntax

**source [18]** — [1Password Secret Reference Syntax](https://developer.1password.com/docs/cli/secret-reference-syntax/) [official]

**format:**
- without section: `op://<vault>/<item>/<field>`
- with section: `op://<vault>/<item>/<section>/<field>`

> "Secret references are case-insensitive and support these characters: alphanumeric characters (a-z, A-Z, 0-9), -, _, . and the whitespace character"

**escape rules:**
> "If a secret reference includes a whitespace, enclose the secret reference in quotation marks."
> "Any part of a secret reference that includes an unsupported character must be referred to by its unique identifier (ID) instead of its name."

### 2.2 secret reference usage

**source [19]** — [Use Secret References with 1Password CLI](https://developer.1password.com/docs/cli/secret-references/) [official]

three ways to use secret references:
> - use `op read` to write secrets to stdout or to a file
> - use `op run` to pass secrets as environment variables to a process
> - use `op inject` to inject secrets into configuration files or scripts

### 2.3 load secrets into environment

**source [20]** — [Load Secrets into Environment Variables](https://developer.1password.com/docs/cli/secrets-environment-variables/) [official]

> "1Password CLI loads the specified secrets, then runs the provided command in a subprocess with the secrets made available as environment variables only for the duration of the process."

**security benefit:**
> "This allows you to avoid hardcode any plaintext secrets and quickly switch between different sets of secrets for different development contexts."

**principle of least privilege:**
> "Authenticate with a 1Password Service Account to follow the principle of least privilege. You can scope service account access to specific vaults and 1Password Environments so that processes in your authorized terminal session can only access secrets required for a given purpose."

### 2.4 create items with JSON templates

**source [21]** — [Item JSON Template](https://developer.1password.com/docs/cli/item-template-json/) [official]
> "Item templates are formatted similarly to the JSON output for `op item get`."

**template structure:**
- item level: `title` and `category`
- section level: `id` (unique identifier) and `label` (display name)
- field level: `id`, `section id`, `type`, `label`, and `value`

**add custom fields:**
> "To add a custom field to the template, insert a new field JSON object into the `fields` array."

### 2.5 item creation commands

**source [22]** — [Create Items with 1Password CLI](https://developer.1password.com/docs/cli/item-create/) [official]

**syntax:** `op item create [flags] [assignment statements]`

**assignment statement format:**
> "section (Optional) The name of the section where you want to create the field. field The name of the field you want to create. fieldType The type of field you want to create."

**security alert:**
> "Command arguments can be visible to other processes on your machine." recommends JSON templates with stdin for sensitive data.

**stdin usage:**
```bash
op item template get Login | op item create --template -
```

## 3. AWS config files

### 3.1 file format specification

**source [23]** — [AWS Shared Config and Credentials Files](https://docs.aws.amazon.com/sdkref/latest/guide/file-format.html) [official]

> "Both the shared `config` and `credentials` files are plaintext files that contain only ASCII characters (UTF-8 encoded). They take the form of what are generally referred to as INI files."

**format rules:**
> - all entries take the form of `setting-name=value`
> - lines can be commented out with `#` prefix
> - section definitions use square brackets: `[section-name]`

### 3.2 config vs credentials file

**source [24]** — [Configuration and Credential File Settings](https://docs.aws.amazon.com/cli/latest/userguide/cli-configure-files.html) [official]

> "The AWS CLI stores sensitive credential information that you specify with `aws configure` in a local file named `credentials`, in a folder named `.aws` in your home directory. The less sensitive configuration options that you specify with `aws configure` are stored in a local file named `config`, also stored in the `.aws` folder in your home directory."

**credentials file restriction:**
> "Only these settings considered 'secrets' or sensitive can be stored in the `credentials` file: `aws_access_key_id`,`aws_secret_access_key`, and `aws_session_token`."

**profile name rules:**
> "When you name the profile in a `config` file, include the prefix word "`profile`", but do not include it in the `credentials` file."
> "Do **_not_** use the word `profile` when you create an entry in the `credentials` file."

### 3.3 SSO profile configuration

**source [25]** — [Configure IAM Identity Center with AWS CLI](https://docs.aws.amazon.com/cli/latest/userguide/cli-configure-sso.html) [official]

**profile section format:**
```ini
[profile my-dev-profile]
sso_session = my-sso
sso_account_id = 123456789011
sso_role_name = readOnly
region = us-west-2
output = json
```

**sso-session section format:**
```ini
[sso-session my-sso]
sso_region = us-east-1
sso_start_url = https://my-sso-portal.awsapps.com/start
sso_registration_scopes = sso:account:access
```

**required settings for sso-session:**
- `sso_start_url` (Required)
- `sso_region` (Required)
- `sso_registration_scopes` (Optional)

### 3.4 SSO login flow

**source [26]** — [Configure IAM Identity Center with AWS CLI](https://docs.aws.amazon.com/cli/latest/userguide/cli-configure-sso.html) [official]

> "To retrieve and cache a set of IAM Identity Center credentials, run this command for the AWS CLI to open your default browser and verify your IAM Identity Center log in."

```bash
aws sso login --profile my-dev-profile
```

**token cache:**
> "The authentication token is cached to disk under the `~/.aws/sso/cache` directory with a filename based on the `sso_start_url`."

**credential renewal:**
> "As long as you are signed in to IAM Identity Center and those cached credentials are not expired, the AWS CLI automatically renews expired AWS credentials when needed."

### 3.5 security best practices

**source [27]** — [SEC02-BP03: Store and Use Secrets Securely](https://docs.aws.amazon.com/wellarchitected/latest/framework/sec_identities_secrets.html) [official]

**core principle:**
> "The most secure credential is one that you do not have to store, manage, or handle."

**anti-patterns:**
> - "Not rotate credentials"
> - "Store long-term credentials in source code or configuration files"
> - "Store credentials at rest unencrypted"

**strategy — remove, replace, and rotate:**

| credential type | strategy |
|----------------|----------|
| IAM access keys | replace with IAM roles on EC2/Lambda, or use temporary credentials |
| SSH keys | replace with AWS Systems Manager Session Manager |
| database credentials | replace with Secrets Manager integration |
| API/OAuth tokens | rotate via AWS Secrets Manager |

## conflict detection

no conflicts detected between sources. all official documentation sources converge on:

1. **JWT expiration:** 10 minute maximum (sources [8], [9])
2. **installation token lifetime:** 1 hour (source [11])
3. **PEM format:** PKCS#1 RSAPrivateKey (source [13])
4. **1Password URI format:** op://vault/item/field (source [18])
5. **AWS config vs credentials:** sensitive values in credentials file (sources [23], [24])

---

## convergence signals

**strong convergence (3+ sources agree):**

1. **use key vaults for sensitive storage** — sources [14], [20], [27] all recommend vault storage over environment variables or config files
2. **short-lived tokens preferred** — sources [11], [17], [27] emphasize ephemeral credentials
3. **avoid hardcode secrets** — sources [15], [22], [27] warn against plaintext secrets

---

## anti-patterns

**sources warn against:**

1. **hardcode private keys** — source [15]: "You should not hard-code your private key in your app, even if your code is stored in a private repository."

2. **command-line secret exposure** — source [22]: "Command arguments can be visible to other processes on your machine."

3. **long-term credentials in files** — source [27]: anti-pattern is "Store long-term credentials in source code or configuration files"

4. **not rotate credentials** — sources [16], [27] emphasize rotation

## implementation implications for mech adapters

### EPHEMERAL_VIA_GITHUB_APP mech

**guided setup needs:**
1. list orgs via `gh api /user/orgs` or `GET /user/orgs`
2. list installations via `GET /orgs/{org}/installations`
3. prompt for .pem file path
4. store JSON blob: `{ appId, installationId, privateKey }`

**transform on unlock:**
1. generate JWT from appId + privateKey (RS256, 10min max, iat -60s)
2. call `POST /app/installations/{installation_id}/access_tokens`
3. return `ghs_` token (1 hour lifetime)

### 1password vault adapter

**storage via:**
1. `op item create --category Secure\ Note --title <key> --vault <vault>`
2. store JSON content via `--template` with stdin pipe

**retrieval via:**
1. `op read op://<vault>/<item>/<field>`

### aws.config vault adapter

**storage:**
1. write profile section to `~/.aws/config`
2. write sso-session section if needed
3. never write to `~/.aws/credentials` for SSO profiles

**SSO unlock:**
1. `aws sso login --profile <profile>`
2. credentials cached in `~/.aws/sso/cache`
