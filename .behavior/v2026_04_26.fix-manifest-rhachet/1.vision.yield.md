# vision: forbid npx in hook commands

## the outcome world

### before

a role author declares a hook like this:

```yaml
hooks:
  onBrain:
    onBoot:
      - command: npx rhachet roles boot --repo ehmpathy --role mechanic
        timeout: PT120S
```

when claude code starts a session:
1. hook runs via `npx rhachet ...`
2. npx resolves the package (network lookup, cache check)
3. adds 500ms-2s latency per hook
4. sometimes fails with cryptic npm errors
5. debugging is painful ("why is my hook slow/broken?")

### after

role author declares:

```yaml
hooks:
  onBrain:
    onBoot:
      - command: ./node_modules/.bin/rhachet roles boot --repo ehmpathy --role mechanic
        timeout: PT60S  # can be shorter now
```

or uses global:

```yaml
hooks:
  onBrain:
    onBoot:
      - command: rhachet roles boot --repo ehmpathy --role mechanic
        timeout: PT60S
```

when claude code starts:
1. hook runs directly (no npx resolution)
2. ~35-70ms startup vs 500ms-2s
3. no mysterious npm failures
4. diagnosis is straightforward

### the "aha" moment

"oh, my hooks are instant now — and they never fail randomly."

## user experience

### usecase: role author runs `repo introspect`

**goal**: generate manifest from role registry

**timeline**:
1. author runs `npx rhachet repo introspect`
2. rhachet loads registry, validates hooks
3. if hook uses `npx rhachet` or `npx rhx`:
   - failfast with clear error
   - shows forbidden command
   - suggests fix pattern
4. author fixes, reruns, succeeds

**contract**:
- input: registry with hooks containing `command: string`
- output: validation error or manifest yaml

### usecase: hook validation error

**what it looks like**:

follows treestruct pattern, no mascot, no capitals:

```
🔭 load getRoleRegistry from rhachet-roles-test...

✋ hooks with forbidden npx patterns
   │
   ├─ these hooks use npx which adds 500ms-2s latency per invocation:
   │
   ├─ mechanic
   │  ├─ hook: onBrain.onBoot[0]
   │  ├─ command: npx rhachet roles boot --role mechanic
   │  └─ hint: use './node_modules/.bin/rhachet roles boot --role mechanic'
   │
   └─ fix: replace 'npx rhachet' with './node_modules/.bin/rhachet' or 'rhachet' (global)
```

**also update extant assertion**: `assertRegistryBootHooksDeclared.ts` must be updated to:
- use ✋ instead of 🐢
- remove turtle mascot ("bummer dude...")
- use lowercase
- use same treestruct pattern

## mental model

### how users describe it

"rhachet won't let you use npx in hooks — it's too slow and flaky. use the direct path or global command."

### analogies

- like eslint forbidding `eval()` — not that it can't work, but the footgun risk outweighs the convenience
- like typescript strict null checks — catches bugs before runtime

### terms

| user says | we say |
|-----------|--------|
| "npx hook" | forbidden npx pattern |
| "direct path" | `./node_modules/.bin/...` |
| "global" | `rhachet` or `rhx` without path |

## evaluation

### how well does it solve the goals?

| goal | addressed? |
|------|-----------|
| prevent latency | yes — forbids npx at manifest time |
| prevent flaky hooks | yes — no npm resolution failures |
| clear diagnosis | yes — error message shows exact command and fix |
| easy fix | yes — hint shows exact replacement |

### pros

- catches problem at build time, not runtime
- clear error with actionable fix
- consistent with other manifest validations (boot hooks, executable skills)
- prevents support burden ("why is my hook slow?")

### cons

- breaking change for extant hooks using npx
- requires role authors to update
- global commands assume rhachet is installed globally

### pit of success

- failfast at `repo introspect`, not at session runtime
- error message shows exact command and fix
- both direct path and global patterns are valid alternatives

## open questions & assumptions

### assumptions

1. `./node_modules/.bin/rhachet` is reliable in installed packages
2. global `rhachet` / `rhx` commands are common enough to be a valid alternative
3. npx latency (500ms-2s) is unacceptable for hooks
4. hooks run frequently enough that latency compounds

### questions to validate

1. should we also catch `pnpm exec`, `yarn exec`, and `bunx`?
   - **[answered]**: only `npx` and `bunx` — they try to auto-install packages
   - `pnpm exec` and `yarn exec` are fine — they only run what's already installed
2. should we warn or error? (wish says failfast, so error)
   - **[answered]**: error — the wish explicitly says failfast
3. should we provide a migration command for extant roles?
   - **[answered]**: no — failfast with hint is sufficient

### external research needed

none — no external dependencies, purely internal validation

- "hint" — suggested fix pattern

## what is awkward?

### forced migration

extant role packages will fail on next `repo introspect`. this is intentional (failfast > silent performance regression) but requires migration work.

### global command assumption

suggesting `rhachet` (global) assumes it's installed. for ci/cd, `./node_modules/.bin/rhachet` is safer.

could address by:
- always suggest direct path first
- add "(if globally installed)" caveat for global suggestion

### pnpm/yarn edge cases

the wish only mentions `npx`, but `pnpm exec` and `yarn exec` have the same problem. should we include them now or defer?

**recommendation**: include them now — same root cause, same fix, prevents future "why didn't you catch pnpm exec?" questions.
