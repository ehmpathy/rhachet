# vision: fix keyrack cross-org fill

## the outcome world

### before

```
ahbode/svc-verifications on vlad/feat-patenter
> rhx keyrack fill --env prod

🔐 keyrack fill (env: prod, keys: 2, owners: 1)

🔑 key 1/2, USPTO_ODP_API_KEY, for 1 owner
   └─ for owner default
      ├─ set the key
      │  └─ enter secret for ahbode.prod.USPTO_ODP_API_KEY: ********
      └─ get after set, to verify
         ├─ ✓ rhx keyrack unlock --key USPTO_ODP_API_KEY --env prod
         └─ ✗ rhx keyrack get --key USPTO_ODP_API_KEY --env prod

⛈️ roundtrip verification failed: key set under ahbode but looked up under rhight
```

developer is confused. they entered the secret, but fill says it's absent. the slug on "enter secret for" shows `ahbode` (root org) while the key is declared in the `rhight` keyrack (extended).

### after

```
ahbode/svc-verifications on vlad/feat-patenter
> rhx keyrack fill --env prod

🔐 keyrack fill (env: prod, keys: 2, owners: 1)

🔑 key 1/2, USPTO_ODP_API_KEY, for 1 owner
   └─ for owner default
      ├─ set the key
      │  └─ enter secret for rhight.prod.USPTO_ODP_API_KEY: ********
      └─ get after set, to verify
         ├─ ✓ rhx keyrack unlock --key USPTO_ODP_API_KEY --env prod
         └─ ✓ rhx keyrack get --key USPTO_ODP_API_KEY --env prod

🔐 keyrack fill complete (2/2 keys verified)
```

fill respects the org embedded in each key's slug. keys from extended manifests are stored under their original org, not the root org.

### the "aha" moment

the slug **is** the source of truth for org. when the manifest says `rhight.prod.USPTO_ODP_API_KEY`, that's where it should be stored and retrieved. the root manifest's org (`ahbode`) is for keys declared in the root — not for keys inherited via `extends:`.

## user experience

### usecase: nested keyrack via extends

```yaml
# ahbode/svc-verifications/.agent/keyrack.yml (root)
org: ahbode
extends:
  - .agent/repo=rhight/role=patenter/keyrack.yml
env.prep:
  - AWS_PROFILE
env.prod:
  - AWS_PROFILE
```

```yaml
# .agent/repo=rhight/role=patenter/keyrack.yml (extended)
org: rhight
env.prod:
  - USPTO_ODP_API_KEY
```

when user runs `rhx keyrack fill --env prod` in ahbode/svc-verifications:

| key | declared in | stored under | looked up under |
|-----|-------------|--------------|-----------------|
| AWS_PROFILE | ahbode (root) | ahbode.prod.AWS_PROFILE | ahbode.prod.AWS_PROFILE |
| USPTO_ODP_API_KEY | rhight (extended) | rhight.prod.USPTO_ODP_API_KEY | rhight.prod.USPTO_ODP_API_KEY |

when user runs `rhx keyrack fill --env prep`:

| key | declared in | stored under | looked up under |
|-----|-------------|--------------|-----------------|
| AWS_PROFILE | ahbode (root) | ahbode.prep.AWS_PROFILE | ahbode.prep.AWS_PROFILE |

each key preserves its origin org through the full set → unlock → get cycle.

### timeline

1. user runs `rhx keyrack fill --env prod`
2. fill loads manifest (with extends resolution)
3. for each key slug:
   - extracts org from slug (not from `repoManifest.org`)
   - prompts with correct org in slug
   - stores under slug's org
   - verifies roundtrip under same slug
4. all keys verified, fill completes

## mental model

### how users describe it

> "fill puts each key where it belongs — under the org that declared it"

> "extended keys keep their original org, they don't get absorbed into the parent"

### analogy

like email forwarding: if alice@corp.com forwards bob@vendor.com's message to you, when you reply you send to bob@vendor.com, not alice@corp.com. the origin identity travels with the content.

### terms

| user term | internal term |
|-----------|---------------|
| "the key's org" | org segment of slug (`$org.$env.$key`) |
| "parent keyrack" | root manifest that uses `extends:` |
| "extended keyrack" | manifest referenced via `extends:` |

## evaluation

### pros

- **no surprises**: slug shown at prompt matches slug used for storage and lookup
- **org fidelity**: each key preserves its declaring manifest's org
- **simple mental model**: org comes from slug, period
- **backwards compatible**: keys declared in root still use root org

### cons

- none significant — this aligns behavior with user expectation

### edgecases

| case | behavior |
|------|----------|
| key declared in both root and extended | root wins (extant behavior via merge) |
| extended manifest has no org | fail-fast at hydration (extant validation) |
| slug explicitly includes org | already correct, no change needed |

## root cause summary

in `fillKeyrackKeys.ts`:

```ts
// line 257 (before)
await setKeyrackKey({
  key: keyName,
  env: input.env,
  org: repoManifest.org,  // ← uses root org, ignores slug's org
  ...
});
```

the slug `rhight.prod.USPTO_ODP_API_KEY` already encodes org=`rhight`, but the code passes org=`ahbode` (the root manifest's org).

### fix

extract org from slug:

```ts
// line 257 (after)
const orgFromSlug = slug.split('.')[0]!;
await setKeyrackKey({
  key: keyName,
  env: input.env,
  org: orgFromSlug,  // ← respects slug's org
  ...
});
```

or use a helper:

```ts
import { asKeyrackKeyOrg } from './asKeyrackKeyOrg';

const org = asKeyrackKeyOrg({ slug });
```
