# vision: github app credentials fix

## the outcome world

### before

```bash
# user pipes valid json with RSA private key
jq -n --rawfile key ~/Downloads/my-app.private-key.pem \
    '{appId: "123", privateKey: $key, installationId: "456"}' | \
  rhx keyrack set --key MY_GITHUB_TOKEN --env prep --vault os.secure --mech EPHEMERAL_VIA_GITHUB_APP

# looks successful...
🔐 keyrack set (org: ehmpathy, env: prep)
   └─ ehmpathy.prep.MY_GITHUB_TOKEN ✓

# but on get...
rhx keyrack get --key MY_GITHUB_TOKEN --env prep --allow-dangerous --json
{
  "grant": {
    "key": {
      "secret": "{"  # <-- MANGLED! only first char survived
    }
  }
}
```

frustration: valid input silently corrupted. user discovers hours later when github api calls fail.

### after

```bash
# same command, now works correctly
jq -n --rawfile key ~/Downloads/my-app.private-key.pem \
    '{appId: "123", privateKey: $key, installationId: "456"}' | \
  rhx keyrack set --key MY_GITHUB_TOKEN --env prep --vault os.secure --mech EPHEMERAL_VIA_GITHUB_APP

🔐 keyrack set (org: ehmpathy, env: prep)
   └─ ehmpathy.prep.MY_GITHUB_TOKEN ✓

# get returns exactly what was set
rhx keyrack get --key MY_GITHUB_TOKEN --env prep --allow-dangerous --json
{
  "grant": {
    "key": {
      "secret": "{\"appId\":\"123\",\"privateKey\":\"-----BEGIN RSA PRIVATE KEY-----\\nMIIE...\\n-----END RSA PRIVATE KEY-----\\n\",\"installationId\":\"456\"}"
    }
  }
}
```

the "aha" moment: stdin json with embedded newlines (like RSA keys) round-trips correctly.

## user experience

### usecase: store github app credentials

**goal**: configure ephemeral github tokens via app credentials

**contract**:
- input: json blob with `appId`, `privateKey`, `installationId` piped to stdin
- output: secret stored, retrievable exactly as provided

**timeline**:
1. user downloads private key from github app settings
2. user constructs json with jq (preserving newlines in key)
3. user pipes to `keyrack set`
4. keyrack stores the secret verbatim
5. later, `keyrack get` returns the exact json
6. github app adapter parses json, generates ephemeral token

### usecase: store any multiline secret

**goal**: store secrets containing newlines, special chars, nested json

**examples**:
- RSA/PEM keys
- multiline certificates
- json configs with embedded json strings
- base64-encoded blobs

## mental model

### how users describe it

> "i pipe json to keyrack, it stores it, i get it back unchanged."

### analogy

stdin → keyrack should be like `cat > file` — bytes in, same bytes out. no interpretation, no escaping, no truncation.

### terms

| user says | we say |
|-----------|--------|
| "my key got mangled" | stdin truncation bug |
| "only the first char" | premature stream close / buffer issue |
| "works without the real key" | newline handling defect |

## evaluation

### how well does it solve the goal?

**fully** — once fixed, any valid stdin content (including json with embedded newlines) stores and retrieves correctly.

### pros

- no new api surface — extant contract, just fixed
- backwards compatible — fix only affects write path
- users don't need to change their commands

### cons

- none identified — this is a pure bugfix

### edgecases & pit of success

| edge case | expected behavior |
|-----------|-------------------|
| json with `\n` in string values | preserved verbatim |
| binary content (non-utf8) | error or store as-is (tbd) |
| extremely large secrets (>1mb) | either work or fail-fast with size limit |
| empty stdin | fail-fast: "no input provided" |

## what is awkward?

### no awkwardness in the fix itself

this is a straightforward correctness bug:
- input: valid json
- expected: same json stored
- actual: truncated

### the awkward part is the silent failure

- `keyrack set` reports success even when it stored garbage
- user only discovers the bug at `keyrack get` time (or worse, at token generation time)

**potential improvement** (out of scope for this fix):
- validate stored content matches input before confirming success
- for `EPHEMERAL_VIA_GITHUB_APP`, validate json has required fields before storing

## summary

**root cause hypothesis**: stdin handling truncates at first newline or fails to read full stream.

**fix**: ensure stdin is read completely before processing, preserving all content including embedded newlines.

**tests needed**:
- unit: multiline json input → stored verbatim
- acceptance: full roundtrip with RSA-like content
