# blackbox criteria: github app credentials fix

## usecase.1 = multiline json via stdin

the primary fix — json with embedded newlines (like RSA keys) must round-trip correctly.

```
given(user has valid json with embedded newlines)
  # e.g., jq outputs pretty-printed json:
  # {
  #   "appId": "123",
  #   "privateKey": "-----BEGIN RSA...\n...\n-----END RSA-----",
  #   "installationId": "456"
  # }

  when(user pipes json to keyrack set)
    then(keyrack reports success)
      sothat(user knows the operation completed)

  when(user unlocks the key)
    then(keyrack reports unlock success)

  when(user gets the key with --allow-dangerous --json)
    then(secret field contains the EXACT json that was piped)
      sothat(round-trip preserves all content)
    then(json is parseable and contains all original fields)
      sothat(downstream consumers can use it)
```

## usecase.2 = single-line json via stdin (regression)

ensure the fix doesn't break compact json.

```
given(user has valid single-line json)
  # e.g., {"appId":"123","privateKey":"...","installationId":"456"}

  when(user pipes json to keyrack set)
    then(keyrack reports success)

  when(user gets the key)
    then(secret field contains the EXACT json that was piped)
      sothat(compact json still works)
```

## usecase.3 = interactive prompt (regression)

ensure fix doesn't break interactive use.

```
given(user runs keyrack set without piped stdin)
  given(terminal is interactive TTY)
    when(prompt appears for secret)
      then(user can type secret and press enter)
      then(keyrack stores the typed secret)
        sothat(interactive flow still works)
```

## usecase.4 = empty stdin

edge case: pipe is connected but empty.

```
given(user pipes empty content to keyrack set)
  # e.g., echo -n "" | keyrack set ...

  when(keyrack set runs)
    then(keyrack stores empty string OR fails fast with clear error)
      sothat(user knows what happened)
```

## usecase.5 = large content via stdin

edge case: very large secrets.

```
given(user has large json content)
  # e.g., >100KB of base64-encoded data

  when(user pipes content to keyrack set)
    then(keyrack stores the full content)
      sothat(large secrets are supported)

  when(user gets the key)
    then(secret field contains the EXACT content that was piped)
      sothat(no truncation for large values)
```

## usecase.6 = special characters in content

edge case: content with shell-special characters.

```
given(user has json with special characters)
  # e.g., $, `, \, ", ', newlines, tabs, unicode

  when(user pipes content to keyrack set)
    then(keyrack stores the content verbatim)
      sothat(special chars are not interpreted or escaped)

  when(user gets the key)
    then(secret field contains the EXACT content)
      sothat(byte-for-byte fidelity)
```

## exchange experience summary

| input | output |
|-------|--------|
| multi-line json via pipe | stored verbatim, retrievable exactly |
| single-line json via pipe | stored verbatim, retrievable exactly |
| interactive TTY input | prompted, stored as typed |
| empty pipe | empty string or clear error |
| large content (>100KB) | stored fully, no truncation |
| special characters | preserved byte-for-byte |

## error cases

| scenario | expected error |
|----------|----------------|
| stdin timeout (no input, no TTY) | clear message about no input |
| stdin read failure | clear message with cause |
