# vision: passwordless os.secure unlock

## the outcome world

### what does a day-in-the-life look like?

**before**: vlad sets a secret with a custom owner and prikey:

```sh
rhachet keyrack set EHMPATHY_SEATURTLE_PROD_GITHUB_TOKEN \
  --owner ehmpath \
  --prikey ~/.ssh/ehmpath \
  --env all \
  --value "ghp_xxx"
```

later, vlad tries to unlock:

```sh
rhachet keyrack unlock --owner ehmpath --prikey ~/.ssh/ehmpath --env all
# ⛈️ UnexpectedCodePathError: os.secure unlock requires identity or passphrase
```

vlad is confused. the prikey decrypted the manifest. the vault was encrypted with that same identity. why can't it unlock?

**after**: the same commands just work:

```sh
rhachet keyrack set EHMPATHY_SEATURTLE_PROD_GITHUB_TOKEN \
  --owner ehmpath \
  --prikey ~/.ssh/ehmpath \
  --env all \
  --value "ghp_xxx"

# later...
rhachet keyrack unlock --owner ehmpath --prikey ~/.ssh/ehmpath --env all
# 🔓 unlocked 1 key for ehmpath.all
```

### what's the before/after contrast?

| aspect | before | after |
|--------|--------|-------|
| unlock with prikey | fails with cryptic error | works seamlessly |
| env=all display | duplicated across all, prod, test | shows once under $org.all |
| tree output | broken structure with misaligned leaves | clean, readable tree |

### what's the "aha" moment?

the prikey unlocks the manifest → the manifest identity unlocks the vault → the vault releases the keys.

it's the same identity that flows through. the unlock should "just work" because set already proved the identity chain is valid.

## user experience

### usecases

1. **team-shared keyrack** — multiple team members share an org's secrets via a shared ssh key
2. **passwordless ci unlock** — ci/cd uses a machine key to unlock secrets without passphrase prompts
3. **multi-env secrets** — set once with `--env all`, access from any environment

### contract inputs & outputs

**set with custom prikey**:
```sh
rhachet keyrack set $KEY \
  --owner $ORG \
  --prikey $PATH_TO_KEY \
  --env all \
  --value $VALUE
```

output:
```
🔐 keyrack set (org: $ORG, env: all)
   └─ $ORG.all.$KEY
      ├─ mech: PERMANENT_VIA_REPLICA
      └─ vault: os.secure
   └─ done
```

**unlock with same prikey**:
```sh
rhachet keyrack unlock --owner $ORG --prikey $PATH_TO_KEY --env all
```

output:
```
🔓 keyrack unlock (org: $ORG, env: all)
   └─ $ORG.all.$KEY → unlocked
   └─ done
```

**list shows correct structure**:
```sh
rhachet keyrack list --owner $ORG --prikey $PATH_TO_KEY
```

output:
```
🔑 keyrack list (org: $ORG)
   └─ $ORG.all.$KEY
   └─ $ORG.prod.$OTHER_KEY
   └─ $ORG.test.$ANOTHER_KEY
```

keys with `--env all` appear once, not triplicated.

### timelines

1. **t0**: user runs `keyrack init --owner myorg --prikey ~/.ssh/myorg`
2. **t1**: user runs `keyrack set MY_SECRET --env all --value xxx`
3. **t2**: in a new shell, user runs `keyrack unlock --owner myorg --prikey ~/.ssh/myorg --env all`
4. **t3**: user runs `echo $MY_SECRET` → sees the value

the prikey flows through: init → set → unlock → get

## mental model

### how would users describe this to a friend?

> "i have an ssh key for my org. i use it to encrypt my secrets. later i use the same key to decrypt them. it just works — no passwords to remember."

### analogies

- **ssh key as a master key** — one key unlocks all in the vault
- **prikey = identity proof** — the manifest confirms "you are who you claim to be"
- **unlock = session auth** — once authenticated, all secrets in scope are accessible

### terms users use vs terms we use

| user term | our term |
|-----------|----------|
| "my ssh key" | prikey / identity |
| "org secrets" | keyrack / host manifest |
| "unlock my secrets" | unlock session via manifest identity |
| "set for all envs" | env=all with slug expansion |

## evaluation

### how well does it solve the goals?

| goal | solution quality |
|------|------------------|
| prikey unlock works | ✅ direct — identity flows through |
| env=all not duplicated | ✅ direct — store once, display once |
| tree structure clean | ✅ direct — proper indentation and alignment |

### pros

- **passwordless** — no passphrase prompts in ci/cd
- **consistent** — same prikey for init, set, unlock
- **simple** — one identity, one flow
- **secure** — age encryption with ssh keys

### cons

- **key management** — users must protect their prikey
- **migration required** — legacy passphrase-encrypted credentials must be re-encrypted with identity

### edgecases and pit of success

| edgecase | pit of success |
|----------|----------------|
| wrong prikey | manifest decryption fails early with clear error |
| absent prikey | fail fast with clear error — no passphrase fallback |
| prikey without read permissions | file access error before crypto attempt |
| env=all + env=prod same key | dedupe by slug — all takes precedence |
| corrupted vault | age decryption fails with diagnostic |

## hard requirements

### R1. identity must flow through to unlock

the manifest is decrypted with the prikey → session identity is set → but `unlock()` is called without the identity passed explicitly. the vault adapter doesn't know an identity is available.

**requirement**: pass identity from session to vault unlock explicitly.

### R2. env=all must not expand to duplicates

when `--env all` is used with a repo manifest, `getAllKeyrackSlugsForEnv()` expands to all envs that declare the key. this creates `$org.all.$key`, `$org.prod.$key`, `$org.test.$key`.

**requirement**: `--env all` stores under `$org.all.$key` only. get/unlock recognizes `$org.all.*` matches all envs.

### R3. tree output must render correctly

the current output has structural issues:
```
   └─ ehmpathy.all.TOKEN
      ├─ mech: ...
      └─ vault: ...
   └─ ehmpathy.prod.TOKEN  ← misaligned, should be peer
```

**requirement**: proper tree render with peer detection.

### R4. passphrase must be fully deprecated

passphrase was the original unlock mechanism before identity-based decryption. now that identity is the preferred path, passphrase must be fully removed from the unlock flow.

**rationale**:
- vaults that need interactive auth invoke their own prompts
- passphrase in the unlock signature is a leaky abstraction
- identity is more secure (ssh key vs plaintext)
- identity is more composable (no prompts in CI)

**requirement**:
- remove `--passphrase` flag from CLI
- remove `passphrase` from `unlock()` signature
- remove `promptHiddenInput` fallback in vault adapter
- vault adapters own their auth — if a vault needs a prompt, it invokes it

**migration**: credentials encrypted with passphrase before identity system will fail to decrypt. users must re-encrypt with identity via `keyrack set` with `--prikey`.

### R5. types must enforce identity flow

the bug was possible because `identity?: string` is optional in the unlock signature. optional attributes allow callers to forget.

**requirement**:
- change `identity?: string` to `identity: string | null`
- callers must explicitly pass `null` if no identity (forces conscious decision)
- aligns with `rule.forbid.undefined-inputs`
