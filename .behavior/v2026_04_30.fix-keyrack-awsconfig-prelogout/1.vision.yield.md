the fix:
1. **keyrack set**: pre-logout before browser auth via `clearAwsSsoCacheForDomain()` тАФ clears only the selected domain
2. **keyrack unlock**: verify identity matches тАФ if session exists but can't access the role, clear cache for that domain then login

**note**: targeted cache deletion is local-only (server-side session remains valid). other domains' sessions are preserved entirely тАФ no impact on unrelated workflows.

## user experience

### usecases

1. **clean setup** тАФ user sets up a new aws profile; no prior session; pre-logout is a no-op
2. **same-user refresh** тАФ user re-authenticates to same account; pre-logout clears stale token, re-auth works
3. **different-user switch (set)** тАФ user switches from alice@acme to bob@acme; pre-logout prevents "already signed in" collision
4. **multi-org workflows** тАФ user has profiles for ehmpathy.test, ahbode.prod, widget.prep; org labels help them pick the right domain
5. **wrong-user unlock** тАФ user has admin session cached, tries to unlock demo profile; keyrack detects admin can't access demo role, logs out admin, triggers fresh login for demo user
6. **same-user different-role** тАФ user has alice session, unlocks a different role alice can access; no logout needed, proceeds directly

### contract inputs & outputs

**pre-logout on set** (internal, automatic):
- input: `ssoStartUrl` (the domain to clear)
- behavior: call `clearAwsSsoCacheForDomain({ ssoStartUrl })` тАФ deletes cache files for this domain only
- output: only the selected domain's session cleared; other domains preserved

**identity check on unlock** (internal, automatic):
- input: profile name from host manifest (encodes sso_domain + sso_account_id + sso_role_name)
- behavior:
  1. run `aws sts get-caller-identity --profile <profile>`
  2. if success тЖТ current SSO user can access this role/account, proceed
  3. if fail (access denied or no session) тЖТ call `clearAwsSsoCacheForDomain({ ssoStartUrl })` then login
- unique key for access check: [sso_domain, sso_account_id, sso_role_name]
- output: valid session for the requested role/account

**org label display** (visual enhancement):
- input: list of `{ ssoStartUrl, ssoRegion }` from `listAwsSsoStartUrls()`
- behavior: extract subdomain from URL (e.g., `https://acme.awsapps.com/start` тЖТ `acme`)
- output: display `https://acme.awsapps.com/start, acme, us-east-1` instead of just the url

### timeline (keyrack set)

1. user runs `rhx keyrack set --slug ehmpathy.test.AWS_PROFILE`
2. guided setup shows sso domains with org labels
3. user picks domain #1 (acme)
4. keyrack calls `clearAwsSsoCacheForDomain({ ssoStartUrl })` (clears only this domain's session)
5. browser auth flow begins fresh
6. user completes auth, picks account, picks role
7. profile saved

### timeline (keyrack unlock with wrong user)

1. user runs `rhx keyrack unlock`
2. keyrack checks `aws sts get-caller-identity --profile ehmpathy.demo`
3. fails тАФ alice session can't access demo role in demo account
4. keyrack detects wrong session via access denied error
5. keyrack calls `clearAwsSsoCacheForDomain({ ssoStartUrl })` (clears only this domain's session)
6. keyrack triggers `aws sso login --profile ehmpathy.demo`
7. browser auth flow begins fresh
8. user completes auth as bob (demo user)
9. unlock succeeds

### analogies

**pre-logout (set)**: like a hotel that deactivates all room cards on checkout. when you check out (logout), all your cards stop working тАФ but you're about to check in again anyway, so you get fresh ones. keyrack does the checkout automatically before check-in.

**identity check (unlock)**: like a bouncer at a club. the bouncer checks if your wristband (session) grants access to this room (role/account). if not, you go back to the entrance (logout) and get the right wristband (re-auth).

**org labels**: like labeled mailboxes in an apartment building. instead of just "Box 1, Box 2", you see "Box 1 (Smith), Box 2 (Jones)". you know which is which.

## evaluation

### pros

- **eliminates manual sign-out** тАФ no more browser gymnastics when switching users
- **automatic** тАФ handles logout automatically so user doesn't have to think about it
- **org visibility** тАФ clearer which domain belongs to which org
- **smart identity check** тАФ detects wrong-user sessions on unlock and auto-corrects
- **no unnecessary logouts** тАФ same SSO user accessing different roles doesn't trigger logout
- **backwards compatible** тАФ same flow, just smarter

### cons

- **extra operation on set** тАФ adds ~10ms for cache file scan + deletion (much faster than `aws sso logout` call)
- **no-op on first use** тАФ pre-logout does zero work when there's no prior session (harmless, but visible output)
- **extra check on unlock** тАФ adds `get-caller-identity` call to verify access
- **local-only deletion** тАФ server-side session remains valid (user can still use other machines). this is fine: user re-auths immediately on this machine.

### pit of success

| edge case | handling |
|-----------|----------|
| no prior session | cache scan finds no files to delete тАФ silent no-op |
| multiple profiles same domain | all profiles cleared together (they share same cache file) |
| multiple domains | only target domain cleared; other domains preserved |
| cache dir not found | returns early with empty deleted list |
| malformed cache file | skipped silently (try/catch) |
| same user, different role | no clear needed тАФ user can access multiple roles |
| wrong user, can't access | detect via get-caller-identity fail, clear cache + re-auth |
| session expired (not wrong user) | normal login flow, no clear needed |

### litmus tests

these sequences validate the identity check logic:

| sequence | expect logout? | why |
|----------|----------------|-----|
| `use.ahbode.test` тЖТ `use.ahbode.prep` | no | same SSO user can access both roles |
| тЖТ `use.ehmpathy.demo --owner ehmpath` | yes | different SSO domain (ehmpathy vs ahbode) |
| `use.ehmpathy.demo --owner admin` | yes | admin user can't access demo role |
| тЖТ `use.ehmpathy.root --owner admin` | no | admin user can access root role |

## open questions & assumptions

### assumptions

1. **awsapps.com URL format** тАФ assumes sso start URLs follow `https://<subdomain>.awsapps.com/start` format. subdomain becomes the org label.
2. **targeted cache deletion** тАФ instead of `aws sso logout` (which is global), we delete specific cache files in `~/.aws/sso/cache/` by match on `startUrl` field. this is local-only (server-side session remains valid) and preserves other domains' sessions entirely. PoC verified: `clearAwsSsoCacheForDomain.ts` + `previewAwsSsoCacheForDomain.ts`.
3. **logout is idempotent** тАФ assumes logout when already logged out is harmless (returns exit 0 or error we can ignore).
4. **get-caller-identity fails for wrong user** тАФ assumes `aws sts get-caller-identity --profile X` fails when the cached SSO user can't access the profile's role/account.
5. **access check is sufficient** тАФ level 1: we check "can current user access this role?" not "is current user the expected user?". same user with access to multiple roles won't trigger unnecessary logout.

### questions to validate

1. **[answered] logout scope** тАФ does `aws sso logout --profile X` clear the session for all profiles that use the same `sso_start_url`? yes, and more тАФ `aws sso logout` clears ALL sessions across ALL domains globally. there is no way to limit logout to a single domain via the CLI. cache files are per-domain in `~/.aws/sso/cache/`, but the CLI command is global.

### research needed

- **[answered]** verify `aws sso logout` behavior with multiple profiles on same domain тАФ confirmed via aws docs (session is per-domain) and extant `relock` code
- **[answered]** verify error handle when no active session exists тАФ verified via `vaultAdapterAwsConfig.ts:340-350` which catches and ignores errors

## groundwork

### external research

**aws sso logout behavior**:
- per [AWS CLI docs](https://docs.aws.amazon.com/cli/latest/reference/sso/logout.html): `aws sso logout` invalidates ALL cached sso sessions
- **logout is global**: removes all cached AWS IAM Identity Center access tokens across ALL profiles and domains
- there is no built-in `--profile` or `--start-url` flag to limit logout scope
- cache files live in `~/.aws/sso/cache/` as JSON files per session, each with `initiateLoginUri` field that holds the domain
- **targeted logout via cache deletion**: delete specific cache files in `~/.aws/sso/cache/` by match on `startUrl` field. verified via PoC: access token files have `startUrl` at top level (easy to match). this is local-only тАФ server-side session remains valid, user re-auths on next use.
- **our approach**: targeted cache deletion. other domains' sessions preserved entirely.

**verified via code inspection**: `vaultAdapterAwsConfig.ts:335-352` already implements `relock` using `aws sso logout --profile <name>`. the `--profile` flag is a global AWS CLI option that specifies which profile's config to use, but **does NOT scope the logout**. logout is still global. we can simplify to just `aws sso logout` without a profile flag.

## what is awkward?

1. **pre-logout visibility** тАФ the message "clear stale session..." appears even when there's no session. this might confuse users. alternative: do it silently and only show output if there was actually a session to clear. but then the behavior is invisible.

2. **non-awsapps URLs** тАФ if a start URL doesn't follow `https://<subdomain>.awsapps.com/start` format (e.g., custom domains), subdomain extraction won't work. edge case тАФ most orgs use the default awsapps.com domain.

3. **[resolved] targeted logout** тАФ the wish says "ensure its targeted to that domain first though, so that unrelated domains aren't impacted". **solution**: `clearAwsSsoCacheForDomain()` deletes only cache files that match the target `ssoStartUrl`. other domains' sessions are preserved. verified via PoC test: 1 file matched for target domain, 16 files kept for other domains.
