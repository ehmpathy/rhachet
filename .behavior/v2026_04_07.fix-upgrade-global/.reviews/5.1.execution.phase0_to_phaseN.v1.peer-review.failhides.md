# blocker.1: Failhide in global upgrade try/catch

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.forbid.failhide.md.pt1.md

**locations**:
- src/domain.operations/upgrade/execUpgrade.ts

In execUpgrade.ts, the try/catch block around execNpmInstallGlobal catches all errors from the global install operation, logs them, and continues execution without rethrowing. This violates the failfast principle by hiding real errors (failhide), as the rule prohibits try/catch unless it allowlists specific errors and rethrows the rest. Although intentional for allowing local upgrade to proceed, it suppresses errors instead of propagating them, leading to potential silent failures that could mask issues. To fix, either rethrow the error after handling or use a more restricted catch that only handles expected errors.

**snippet**:
```ts
try {
        upgradedGlobal = execNpmInstallGlobal({ packages: ['rhachet'] });
      } catch (error) {
        // warn and continue (criteria usecase.3: exits with success sothat local not blocked)
        const message = error instanceof Error ? error.message : String(error);
        const isPermissionError =
          message.includes('EACCES') || message.includes('EPERM');
        console.log('');
        console.log('❌ rhachet upgrade globally failed');
        console.log(`   └── ${isPermissionError ? 'permission denied' : message}`);
        console.log('');
        upgradedGlobal = { upgraded: false, error: message };
      }
```

# blocker.2: Missing semantic exit codes in CLI

**rule**: .agent/repo=ehmpathy/role=mechanic/briefs/practices/code.prod/pitofsuccess.errors/rule.require.exit-code-semantics.md

**locations**:
- src/contract/cli/invokeUpgrade.ts

The CLI command in invokeUpgrade.ts does not set explicit exit codes based on operation results. For malfunction errors like global install failures (which are transient and allow retry), the process should exit with code 1. Currently, when global upgrade fails but local succeeds, the command exits successfully (implicit 0), violating the semantic exit code rule. This prevents callers from distinguishing between full success and partial failure for retry logic. To fix, inspect the UpgradeResult and call process.exit(1) if any upgrades failed due to malfunctions.

**snippet**:
```ts
        const result = await execUpgrade(
          {
            self: options.self,
            roleSpecs: options.roles,
            brainSpecs: options.brains,
            which: options.which,
          },
          context,
        );

        // summary
        console.log('');
        if (result.upgradedSelf) {
          console.log('✨ rhachet upgraded locally');
        }
        if (result.upgradedRoles.length > 0) {
          console.log(
            `✨ ${result.upgradedRoles.length} role(s) upgraded: ${result.upgradedRoles.join(', ')}`,
          );
        }
        if (result.upgradedBrains.length > 0) {
          console.log(
            `✨ ${result.upgradedBrains.length} brain(s) upgraded: ${result.upgradedBrains.join(', ')}`,
          );
        }
        if (result.upgradedGlobal?.upgraded) {
          console.log('✨ rhachet upgraded globally');
        }
        console.log('');
```
