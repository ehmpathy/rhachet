# vision: upgrade global rhachet by default

## the outcome world

### before

```
$ rhx upgrade
✨ rhachet upgraded (in project)
✨ 3 role(s) upgraded: ehmpathy/mechanic, ehmpathy/architect, ehmpathy/ergonomist

$ rhx --version
1.38.0  # still old — global install unchanged

$ cd ~/other-project && rhx --version
1.38.0  # same old global version
```

the user upgraded, but their muscle-memory `rhx` command is still on an old version. they discover this days later when a new feature doesn't work.

### after

```
$ rhx upgrade
✨ rhachet upgraded (global: 1.38.0 → 1.39.11)
✨ rhachet upgraded (local: 1.38.0 → 1.39.11)
✨ 3 role(s) upgraded: ehmpathy/mechanic, ehmpathy/architect, ehmpathy/ergonomist

$ rhx --version
1.39.11  # global is current

$ cd ~/other-project && rhx --version
1.39.11  # same everywhere
```

global and local stay in sync. no version drift. no surprises.

### the "aha" moment

"wait, `rhx upgrade` just keeps everything current everywhere? i don't have to think about which install to upgrade?"

## user experience

### usecase 1: default upgrade (most common)

**goal:** "i want the latest rhachet everywhere"

```bash
rhx upgrade
```

**behavior:**
1. upgrades local install (extant behavior)
2. upgrades roles and brains (extant behavior)
3. detects if rhachet is installed globally (npm only)
4. if yes, upgrades global install

**output:**
```
✨ rhachet upgraded (local)
✨ 3 role(s) upgraded: ehmpathy/mechanic, ehmpathy/architect, ehmpathy/ergonomist
✨ rhachet upgraded (global)
```

### usecase 2: explicit control

**goal:** "i only want to upgrade local, not global"

```bash
rhx upgrade --which local
```

**goal:** "i only want to upgrade global, not local"

```bash
rhx upgrade --which global
```

the default is `--which both` when no flag is specified

### usecase 3: npx invocation

**goal:** "i use npx, no global install"

```bash
npx rhachet upgrade
```

**behavior:**
- detects npx invocation → implies `--which local`
- no attempt to upgrade global
- no error, no noise

**output:**
```
✨ rhachet upgraded (local)
✨ 3 role(s) upgraded: ehmpathy/mechanic, ehmpathy/architect, ehmpathy/ergonomist
```

**the default rule:**
- `npx rhx upgrade` → local only by default (npx = no global install)
- `rhx upgrade` → global + local by default (rhx = global install detected)

## mental model

### how users would describe this to a friend

> "just run `rhx upgrade` — it upgrades everything. your global install, your project install, all the roles. you don't have to think about it."

### analogies

- like `brew upgrade` — upgrades both the formula and all installed packages
- like `rustup update` — keeps the toolchain current everywhere
- like `apt upgrade` — system-wide, not per-directory

### terms

| user term | internal term |
|-----------|---------------|
| "global rhx" | global npm install (`npm i -g rhachet`) |
| "project rhachet" | local devDependency or dependency |
| "everything" | global + local + roles + brains |

## evaluation

### how well does it solve the goals?

| goal | before | after |
|------|--------|-------|
| keep global current | manual `npm i -g rhachet` | automatic |
| avoid version drift | user must remember | handled |
| single command upgrade | partial (local only) | complete |

### pros

- **no cognitive load** — one command, everything upgraded
- **no version drift** — global and local stay in sync
- **backwards compatible** — default behavior extends, doesn't break
- **explicit control** — `--which local|global|both` for users who want isolation or specificity

### cons

- **longer upgrade time** — two installs instead of one
- **sudo may be needed** — global npm installs sometimes require elevated permissions
- **network twice** — fetches package twice (global + local)

### edge cases and pit of success

| edge case | behavior |
|-----------|----------|
| invoked via npx | implies `--which local` by default |
| invoked via rhx (global) | implies `--which both` by default |
| global install fails (permissions) | warn, proceed with local, surface hint |
| global and local at same version | report "already current", no-op |
| global newer than local | upgrade local to match (no downgrade global) |

## open questions & assumptions

### assumptions

1. **global installs are common** — users who type `rhx` typically have it installed globally
2. **npm global detection is reliable** — `npm list -g rhachet` or PATH check (npm only; volta/homebrew out of scope for MVP)
3. **permissions are usually not an issue** — most users have write access to their npm global prefix

### questions (triaged)

1. **[answered] should global upgrade be default or opt-in?**
   - answer: default (per wish: "by default, if installed globally")

2. **[answered] what if global upgrade fails?**
   - answer: warn and continue — local upgrade is core value, global is additive

3. **[answered] should we support non-npm global installs?**
   - answer: npm only for MVP — volta/homebrew are future scope if requested

4. **[answered] permission strategy?**
   - answer: attempt → catch EACCES → surface hint (no sudo prompt)

### research needed

1. **[research]** how to reliably detect global npm install location cross-platform
2. **[research]** how to detect if run via npx vs global install

## what is awkward?

### potential friction points

1. **two npm installs** — global and local are separate operations, could be slow
2. **permission errors** — global npm prefix may need sudo on some systems
3. **version mismatch messaging** — when global != local, what do we say?

### where design fights mental model

- users may expect `rhx upgrade` in a project to ONLY affect that project
- global upgrade as default may surprise users who want isolation

### uncomfortable tradeoffs

| tradeoff | decision |
|----------|----------|
| speed vs convenience | accept slower upgrade for simpler mental model |
| isolation vs consistency | prefer consistency (global + local match) |
| fail-fast vs resilience | warn and continue if global fails, so local still upgrades |

### mitigation

- clear output shows what was upgraded where
- `--which local` for users who want isolation
- helpful error messages when permissions fail
