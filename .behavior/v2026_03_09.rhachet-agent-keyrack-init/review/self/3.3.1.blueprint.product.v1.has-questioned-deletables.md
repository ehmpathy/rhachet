# self-review: has-questioned-deletables

## components questioned

### 1. getOrgFromPackageJson.ts

**can it be removed?** no.

- org detection is core to the wish
- fallback chain (organization → scoped name → repository) follows the vision
- each fallback serves a real use case:
  - organization field: explicit declaration
  - scoped name: npm convention (@org/pkg)
  - repository field: standard package.json field

**simplification?** the three fallbacks are necessary. if we remove any, we reduce compatibility.

---

### 2. discoverRoleKeyracks.ts

**can it be removed?** no.

- extends array must be auto-populated from role keyracks
- this is a single glob operation — already minimal

**simplification?** already as simple as possible: one glob, one sort.

---

### 3. --for host flag

**can it be removed?** **yes, potentially.**

- the wish says `.agent/keyrack` — repo keyrack only
- `--for host` was added in vision as scope extension
- host keyrack is a separate concern

**conclusion:** defer `--for host` to a future behavior. the wish asks for repo keyrack.

---

### 4. env.prep section

**can it be removed?** no.

- the wish explicitly shows `env.prep:` in the example
- follows the pattern: env.prod, env.prep, env.test

---

### 5. flag composition (--keys --hooks --roles)

**can it be removed?** no.

- follows extant invokeInit pattern
- composition is handled by extant code — no new work

---

### 6. output format tree

**can it be removed?** yes, simplify.

- the wish doesn't specify output format
- verbose tree is nice-to-have, not essential
- simplify to basic output: "created .agent/keyrack.yml" or "updated ..."
