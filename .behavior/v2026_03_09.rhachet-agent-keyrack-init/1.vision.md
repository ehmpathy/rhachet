# vision: rhachet init --keys

## the outcome world

### what does a day-in-the-life look like?

**before**: a developer clones a repo that uses rhachet roles. the roles need api keys (openai, anthropic, aws). they have to:
1. figure out which keys are needed (hunt through docs, ask teammates)
2. manually create `.agent/keyrack.yml`
3. guess the org name
4. manually find which roles have their own keyracks
5. manually compose the extends array
6. repeat this dance every time a new role is added

**after**: developer runs one command:
```sh
npx rhachet init --keys --roles mechanic dispatcher
```

done. the keyrack manifest is created with:
- org auto-detected from package.json (`@ehmpathy/rhachet` → `org: ehmpathy`)
- extends array populated from specified role keyracks (explicit control)
- env sections ready for key declarations

### before/after contrast

| aspect | before | after |
|--------|--------|-------|
| org discovery | manual lookup | auto from package.json |
| role keyracks | manual hunt | explicit via `--roles` |
| extends array | manual compose | auto-populated from specified roles |
| re-runs | manual update | idempotent upsert |
| new roles | manual add extends | specify in `--roles` on next init |

### the "aha" moment

you run `rhachet init --keys --roles mechanic dispatcher` in a fresh clone of a repo that uses `rhachet-roles-ehmpathy`. it outputs:

```
🔑 keyrack init
   ├─ org: ehmpathy (from package.json @ehmpathy/my-project)
   ├─ extends:
   │  ├─ .agent/repo=bhuild/role=dispatcher/keyrack.yml
   │  └─ .agent/repo=ehmpathy/role=mechanic/keyrack.yml
   └─ created .agent/keyrack.yml
```

you explicitly specified which roles need keys. the keyrack manifest is created with exactly those roles.

## user experience

### usecases

| usecase | command | outcome |
|---------|---------|---------|
| fresh repo setup | `rhachet init --keys --roles mechanic dispatcher` | creates .agent/keyrack.yml with org + extends for specified roles |
| new roles added | `rhachet init --keys --roles mechanic dispatcher newrole` | appends newrole keyrack to extends array |
| verify repo keyrack | `rhachet init --keys --roles mechanic` | shows current state (idempotent, no changes if up-to-date) |

**note:** host keyrack setup uses `rhachet keyrack init --for host`, not `init --keys`.

### contract inputs & outputs

**inputs:**
```sh
npx rhachet init --keys --roles mechanic           # specify which roles to extend (required)
npx rhachet init --keys --roles mechanic dispatcher # multiple roles
npx rhachet init --keys --roles mechanic --hooks   # composes with other init flags
```

**note:** `--keys` requires `--roles`. org is auto-detected from package.json (no `--org` flag). for custom org or host keyrack, use `rhachet keyrack init` instead.

**outputs:**
```yaml
# .agent/keyrack.yml
org: ehmpathy

extends:  # sorted alphabetically for deterministic diffs
  - .agent/repo=bhuild/role=dispatcher/keyrack.yml
  - .agent/repo=ehmpathy/role=mechanic/keyrack.yml

env.prod:
env.prep:
env.test:
```

### what would it look like to leverage them?

```sh
# 1. clone repo
git clone git@github.com:ehmpathy/cool-project.git
cd cool-project

# 2. install deps (links roles)
npm ci

# 3. init all (flags compose)
npx rhachet init --keys --roles mechanic dispatcher --hooks

# 4. unlock with your key
npx rhachet keyrack unlock --prikey ~/.ssh/id_ed25519

# 5. ready to work
```

### timelines

| phase | duration | what happens |
|-------|----------|--------------|
| validation | instant | validates specified roles have keyracks (fail-fast if absent) |
| discovery | instant | reads package.json repository for org |
| generation | instant | writes or upserts keyrack.yml with specified role keyracks |
| idempotent re-run | instant | detects no changes needed |

## mental model

### how would users describe this to a friend?

> "when i clone a repo, i run `init --keys --roles mechanic dispatcher` and it sets up my keyrack config. i tell it which roles i want, it figures out the org from package.json, and creates the manifest with those role keyracks."

### analogies

| analogy | maps to |
|---------|---------|
| `npm init` creates package.json | `rhachet init --keys --roles` creates keyrack.yml |
| `.gitignore` templates | keyrack.yml template with env sections |

| `git add file1 file2` specifies files | `--roles mechanic dispatcher` specifies roles |

## evaluation

### how well does it solve the goals?

| goal | solved? | how |
|------|---------|-----|
| reduce manual setup | ✅ | auto-detects org, validates specified roles |
| stay up to date | ✅ | append semantics add specified role keyracks to extends |
| pit of success | ✅ | explicit roles, fail-fast validation, clear output |
| composition | ✅ | role keyracks compose via extends |

### pros

- **explicit control** — user specifies which roles to extend via `--roles`
- **fail-fast validation** — errors immediately if specified role lacks keyrack
- **idempotent** — safe to run repeatedly
- **composable** — role keyracks compose via extends
- **transparent** — output shows exactly what was specified and created

### cons

- **package.json required** — must have org-detectable fields (organization, scoped name, or repository); for custom org use `keyrack init` instead
- **explicit roles required** — must specify `--roles` (no auto-discovery of all linked roles)

### edgecases

| edgecase | pit of success behavior |
|----------|------------------------|
| org detection | fallback chain: package.json#organization → scoped name (@org/) → repository field → error (use `keyrack init` for custom org) |
| `--roles` absent | fail-fast error: must specify which roles to extend |
| specified role lacks keyrack | fail-fast error: indicates which role lacks keyrack |
| manifest extant, extends outdated | append: add specified entries, preserve extant |

## open questions & assumptions

### assumptions

1. **org from package.json** — fallback chain: `organization` field → scoped name (`@org/`) → `repository` field owner
2. **role keyracks follow convention** — `.agent/repo=$repo/role=$role/keyrack.yml`
3. **extends is append-only** — we add specified entries, never remove
4. **explicit roles** — `--roles` flag is required; only specified roles are extended

### resolved via review

all open questions were self-answered via the wish:

1. **stale extends removal:** append-only. the wish says "upsert" — for extends, this means append (add absent, never remove).

2. **env sections:** use `env.prod`, `env.prep`, `env.test` as shown in the wish. if env.prep requires new system support, that's implementation work.

3. **org detection:** use package.json fallback chain (organization → scoped name → repository field). if detection fails, error and redirect to `keyrack init` for custom org.

4. **git-provider agnostic:** repository field works regardless of provider (github, gitlab, bitbucket).

5. **extends sorted:** yes, alphabetically by path. deterministic output for clean diffs.

## what is awkward?

### where does the design fight the user's mental model?

1. **"init" does two things** — host manifest vs repo manifest. users may confuse them.
   - mitigation: `init --keys` for repo keyrack, `keyrack init --for host` for host keyrack

2. **extends vs inline** — users might expect to declare keys inline, not via extends
   - mitigation: repo manifest has its own env sections for repo-specific keys

### tradeoffs that feel uncomfortable

1. **auto-detect only** — org auto-detection is convenient but magical. some users prefer explicit.
   - chosen: auto-detect only for `init --keys`. users who need custom org use `keyrack init` instead. transparency via output.

2. **append vs overwrite** — append preserves manual additions to extends but may leave stale entries
   - chosen: append semantics for extends (safe default). users can manually remove stale entries.

3. **silent vs verbose** — should success be quiet or show what happened?
   - chosen: verbose output shows org source and specified extends. transparency wins.

### command separation

`init --keys` only operates on repo keyrack (`.agent/keyrack.yml`) with auto-detected org. for more control, use `rhachet keyrack init`:
- `keyrack init` — both host and repo (default for backcompat)
- `keyrack init --for host` — host keyrack only (`~/.config/rhachet/keyrack.yml`)
- `keyrack init --for repo` — repo keyrack only, with `--org` option for custom org

this keeps `init --keys` simple while `keyrack init` provides full control.

### flag composition

`init` flags are composable — they do not early-return. when multiple flags are present, all execute in order:
1. `--roles` → init roles from packages (always runs when present)
2. `--keys` → init keyrack manifest (runs after roles)
3. `--hooks` → apply hooks
4. `--prep` → persist to package.json

example: `init --keys --roles mechanic --hooks` does all three.
