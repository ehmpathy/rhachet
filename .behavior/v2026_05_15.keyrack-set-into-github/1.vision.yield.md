# vision: keyrack set embeds mech into github.secrets

## the outcome world

### before

a developer sets up a github app credential for CI:

```bash
rhx keyrack set --key EHMPATH_BEAVER_GITHUB_TOKEN --env test --vault github.secrets
# selects: EPHEMERAL_VIA_GITHUB_APP
# selects: ehmpathy org
# selects: beaver-by-bhuild app
# provides: private key path
# output shows success with mech: EPHEMERAL_VIA_GITHUB_APP
```

CI runs. the keyrack get call returns raw JSON instead of a `ghs_*` token:

```
token prefix: {"appId":"3234162","
```

test fails. hours of debug. eventually discover the stored JSON lacks the `mech` field, so `inferKeyrackMechForGet` defaults to `PERMANENT_VIA_REPLICA` (passthrough).

manual workaround required:

```bash
jq -c -n --rawfile key ~/path/to/key.pem \
  '{appId: "3234162", privateKey: $key, installationId: "123", mech: "EPHEMERAL_VIA_GITHUB_APP"}' | \
gh secret set EHMPATH_BEAVER_GITHUB_TOKEN --repo ehmpathy/my-repo
```

### after

same setup flow:

```bash
rhx keyrack set --key EHMPATH_BEAVER_GITHUB_TOKEN --env test --vault github.secrets
# selects: EPHEMERAL_VIA_GITHUB_APP
# selects: ehmpathy org
# selects: beaver-by-bhuild app
# provides: private key path
# output shows success with mech: EPHEMERAL_VIA_GITHUB_APP
```

CI runs. keyrack get sees the embedded `mech` field, invokes the github app adapter, returns a `ghs_*` token:

```
token prefix: ghs_
```

test passes. no manual intervention required.

### the "aha" moment

> "it just works — the mechanism I selected at set time is remembered at get time, even across vaults, even in CI"

the contract is: **what you set is what you get**. the mechanism travels with the credential.

## user experience

### usecases

| usecase | goal |
|---------|------|
| CI github automation | store github app creds, get ephemeral tokens in CI workflows |
| cross-repo secrets | share github app across repos via github.secrets vault |
| token rotation | ephemeral tokens rotate automatically, no manual refresh |

### contract inputs & outputs

**set flow**:
```
input:  keyrack set --key NAME --env ENV --vault github.secrets
        (interactive: select mech, org, app, provide key)
output: secret stored with embedded mech
        { appId, privateKey, installationId, mech: "EPHEMERAL_VIA_GITHUB_APP" }
```

**get flow**:
```
input:  keyrack get --key NAME --env ENV
output: ghs_XXXX (short-lived installation token, 1hr expiry)
```

### timeline

1. **setup (once)**: `keyrack set` — interactive, 2-3 minutes
2. **usage (every CI run)**: `keyrack get` — automatic, sub-second
3. **maintenance (never)**: tokens rotate on each get, no manual refresh

## mental model

### how users describe this

> "keyrack stores my github app credentials and gives me fresh tokens whenever I need them. I set it up once and forget about it."

### analogies

- **valet key**: you give the valet your master key (github app), they give you back a temporary token. the master key stays in the vault.
- **badge printer**: your ID badge (github app) is stored securely. each time you need access, the printer makes a fresh visitor pass (installation token).

### terms

| user term | internal term |
|-----------|---------------|
| "github app" | EPHEMERAL_VIA_GITHUB_APP mechanism |
| "token" | installation access token (ghs_*) |
| "credentials" | source (appId + privateKey + installationId) |
| "vault" | github.secrets adapter |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| CI gets ephemeral tokens | yes — mech embedded, adapter invoked |
| no manual intervention | yes — set once, get works forever |
| secure credential storage | yes — private key never leaves github secrets |

### pros

- **invisible**: fix is internal, no user-faced changes
- **backwards compatible**: extant credentials can be updated manually
- **pit of success**: interactive flow always embeds mech correctly

### cons

- **migration**: extant credentials stored without mech need manual re-set or patch
- **bloat**: JSON payload grows by ~35 chars (mech field)

### edge cases

| edge case | handle |
|-----------|--------|
| user provides JSON with mech already | mech from adapter overwrites (source of truth is the interactive selection) |
| mech field corrupted | `inferKeyrackMechForGet` falls back to PERMANENT_VIA_REPLICA |
| different mech types in future | pattern extends — any mech can embed its type |

## open questions & assumptions

### assumptions

1. **mech is source of truth at set time** — the interactive selection or `--mech` flag determines the mechanism, not any embedded value in user-provided JSON
2. **JSON structure is mutable** — we can add fields to the stored JSON without break for consumers
3. **inferKeyrackMechForGet is the standard pattern** — all vaults use this for mechanism detection on get

### questions for wisher

none — all questions answered via logic.

### questions answered

1. **[answered]** should we provide a migration path for extant credentials?
   - **migration is impossible** for github.secrets vault
   - github secrets API does not support retrieval — write-only
   - we cannot read → parse → add mech → re-write
   - workarounds exist: re-run `keyrack set` or manual jq patch
   - no migration tool needed or possible

2. **[answered]** should the fix apply only to github app, or all mechanisms that store JSON?
   - only github app stores JSON to github.secrets currently
   - `PERMANENT_VIA_REPLICA` stores plain strings
   - future mechs that store JSON should follow the same pattern (embed mech)
   - no action needed beyond the github app fix

### external research needed

none — this is internal keyrack behavior, no external APIs affected

## groundwork

### external research

none — no external dependencies. the github secrets API and gh CLI behavior are unchanged.

### internal research

verified the below:

| what | where | result |
|------|-------|--------|
| JSON construction location | `src/domain.operations/keyrack/adapters/mechanisms/mechAdapterGithubApp.ts:220-225` | JSON lacks mech field |
| inferKeyrackMechForGet pattern | `src/domain.operations/keyrack/inferKeyrackMechForGet.ts` | looks for `parsed.mech`, falls back to PERMANENT_VIA_REPLICA |
| vault adapter interface | `src/domain.objects/keyrack/KeyrackHostVaultAdapter.ts` | vault.set returns `{ mech }` so orchestrator knows mechanism |
| mech adapter interface | `src/domain.objects/keyrack/KeyrackGrantMechanismAdapter.ts` | acquireForSet returns `{ source }` only — no mech |
| mechanism types | `src/domain.objects/keyrack/KeyrackGrantMechanism.ts` | EPHEMERAL_VIA_GITHUB_APP is valid type |

**the bug**: `mechAdapterGithubApp.acquireForSet()` constructs:
```typescript
const source = JSON.stringify({
  appId,
  installationId,
  privateKey,
});
```

**the fix**: include mech in JSON:
```typescript
const source = JSON.stringify({
  appId,
  installationId,
  privateKey,
  mech: 'EPHEMERAL_VIA_GITHUB_APP',
});
```

### contracts to conform to

- `KeyrackGrantMechanismAdapter.acquireForSet` returns `{ source: string }`
- `inferKeyrackMechForGet` expects `{ mech: KeyrackGrantMechanism }` in parsed JSON

### vocab to reuse

- `mech` — mechanism identifier
- `source` — the stored credential value
- `deliverForGet` — transform source to usable secret

### stdout patterns

no changes to CLI output — fix is internal to JSON payload.

## what is awkward?

### mech knowledge in mech adapter

the mech adapter now needs to know its own mech type to embed it. this is slightly circular — the adapter is named for the mech but doesn't explicitly declare "I am EPHEMERAL_VIA_GITHUB_APP".

**alternative**: vault adapter could inject mech into JSON after it receives source from mech adapter. but this means vault adapter needs to understand JSON structure.

**recommendation**: keep it simple — mech adapter knows its own type, embeds it directly. the slight redundancy is worth the simplicity.

### migration burden

extant credentials lack mech. users must either:
1. re-run `keyrack set` to overwrite
2. manually patch the secret via gh CLI
3. wait for a migration tool (if built)

this is a one-time cost. new credentials will just work.

### JSON bloat

mech field increases payload by ~35 chars. trivial for secrets API limits.

## verification

### roundtrip acceptance test

reproduce the reported scenario end-to-end:

```typescript
given('[case1] github app credentials set to github.secrets', () => {
  // setup: simulate what keyrack set produces for github.secrets vault
  const sourceJson = JSON.stringify({
    appId: '3234162',
    installationId: '123456',
    privateKey: TEST_PRIVATE_KEY,
    mech: 'EPHEMERAL_VIA_GITHUB_APP', // <-- the fix
  });

  when('[t0] CI injects secret as env var and keyrack get is called', () => {
    // simulate github actions: secret placed into env
    process.env.EHMPATHY_TEST_EHMPATH_BEAVER_GITHUB_TOKEN = sourceJson;

    const result = useThen('keyrack get returns token', async () =>
      keyrackGet({
        key: 'EHMPATH_BEAVER_GITHUB_TOKEN',
        env: 'test',
        owner: 'ehmpathy',
      }),
    );

    then('token is ghs_* format, not raw JSON', () => {
      expect(result.token.startsWith('ghs_')).toBe(true);
      expect(result.token).not.toContain('appId');
    });

    then('mech was detected from embedded field', () => {
      expect(result.mech).toBe('EPHEMERAL_VIA_GITHUB_APP');
    });
  });
});
```

**key assertions**:
1. token starts with `ghs_` (not `{"appId":`)
2. mech detected as `EPHEMERAL_VIA_GITHUB_APP` (not `PERMANENT_VIA_REPLICA`)

this test would have caught the bug — without `mech` in JSON, `inferKeyrackMechForGet` returns `PERMANENT_VIA_REPLICA` and passes through raw JSON.
