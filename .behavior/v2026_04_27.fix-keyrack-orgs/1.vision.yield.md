# vision: keyrack org inheritance consistency

## the outcome world

### a day-in-the-life with this in place

vlad is setting up a new repo `svc-jobs` for the `ahbode` org. the repo extends keyrack from `repo=ehmpathy/role=mechanic` for shared secrets. the root `.agent/keyrack.yml` declares `org: ahbode` to override the extended keyrack's `org: ehmpathy`.

```yaml
# .agent/keyrack.yml
org: ahbode
extends:
  - .agent/repo=ehmpathy/role=mechanic/keyrack.yml
```

vlad runs set:
```sh
rhx keyrack set --owner ehmpath --env prep --key EHMPATHY_SEATURTLE_GITHUB_TOKEN --vault os.secure
# 🔐 keyrack set (org: ahbode, env: prep)
#    └─ ahbode.prep.EHMPATHY_SEATURTLE_GITHUB_TOKEN
```

vlad runs unlock:
```sh
rhx keyrack unlock --owner ehmpath --env prep
# 🔓 keyrack unlock
#    └─ ahbode.prep.EHMPATHY_SEATURTLE_GITHUB_TOKEN
#       ├─ env: prep
#       ├─ org: ahbode
#       └─ expires in: 540m
```

vlad runs get:
```sh
rhx keyrack get --owner ehmpath --env prep --key EHMPATHY_SEATURTLE_GITHUB_TOKEN
# 🔐 keyrack
#    └─ ahbode.prep.EHMPATHY_SEATURTLE_GITHUB_TOKEN
#       └─ status: unlocked ✅
```

**the aha moment**: every command respects the root org override. keys defined in extended keyracks are re-slugged to match the consuming repo's org. no discontinuity.

### before/after contrast

| scenario | before (broken) | after (fixed) |
|----------|-----------------|---------------|
| set | writes to `ahbode.prep.*` | writes to `ahbode.prep.*` |
| unlock | reads from `ehmpathy.prep.*` | reads from `ahbode.prep.*` |
| get | can't find `ahbode.prep.*` | finds `ahbode.prep.*` |
| mental model | confusing, keys "disappear" | consistent, predictable |

## user experience

### usecases

1. **org-scoped repos extending shared keyrack**: a repo belonging to `ahbode` extends `ehmpathy` mechanic keyrack for common secrets
2. **consistent key lifecycle**: set → unlock → get works without surprises
3. **multi-org access patterns**: mechanics unlocking keys across orgs see consistent slugs

### contract inputs & outputs

**input**: repo manifest with org override + extends
```yaml
# root: .agent/keyrack.yml
org: ahbode  # ← override
extends:
  - .agent/repo=ehmpathy/role=mechanic/keyrack.yml  # ← has org: ehmpathy
keys:
  prep:
    - REPO_SPECIFIC_KEY  # defined here, uses ahbode
```

**hydration output**: all keys re-slugged to root org
```ts
{
  org: 'ahbode',
  keys: {
    'ahbode.prep.REPO_SPECIFIC_KEY': { ... },      // root key
    'ahbode.prep.EHMPATHY_SEATURTLE_GITHUB_TOKEN': { ... },  // extended key, RE-SLUGGED
  }
}
```

### timeline

1. `keyrack set` → writes `ahbode.prep.KEY` to host manifest + vault
2. `keyrack unlock` → reads repo manifest, finds `ahbode.prep.KEY`, unlocks from host
3. `keyrack get` → retrieves `ahbode.prep.KEY` from daemon

## mental model

### how users describe this to a friend

> "my repo's keyrack can extend another keyrack to inherit secrets. if my repo belongs to a different org, i override the org in my root manifest. all inherited keys get re-slugged to my org — they act like they were defined in my repo."

### analogies

- **domain override in DNS**: like a subdomain inheriting records from a parent but in its own namespace
- **class inheritance with override**: child class inherits methods but can rebind `this` to its own context
- **fork in git**: you fork a repo, but commits belong to your fork, not the original

### terminology

| user says | system uses |
|-----------|-------------|
| "my org" | `manifest.org` (root) |
| "the extended keyrack's org" | `extendedManifest.org` |
| "inherited keys" | keys merged from extended manifests |
| "re-slug" | reconstruct key slug with root org |

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| set/unlock/get consistency | yes — all commands use same slugs |
| cross-org extension | yes — extended keys re-slugged to root org |
| no surprises | yes — behavior matches mental model |

### pros

- **pit of success**: once org is declared at root, all commands "just work"
- **simplicity**: no special handling needed per-command
- **backward compatible**: repos with same org as extended see no change (slug preserved)

### cons

- **host manifest coupling**: the host manifest must also use the new slug, so the secret must be set with the correct org
- **migration**: if users have extant keys under the old org, they need re-set

### edgecases and pit-of-success

| edgecase | pit-of-success handling |
|----------|-------------------------|
| nested extends (A extends B extends C) | org resolved from outermost root, propagates down |
| same org as extended | no re-slug needed (already matches) |
| conflicting keys | root keys win (already implemented) |
| key already exists in host under wrong org | set command writes to correct org; old entry orphaned |

## open questions & assumptions

### assumptions

1. **org override propagates fully**: when root declares org, ALL extended keys adopt it
2. **host manifest keyed by slug**: the host manifest stores keys by their full slug, so changing the slug means the secret must be re-set
3. **backward compatibility**: extant repos with same org as extended see no behavior change

### questions for wisher

1. should we support partial org override (some keys keep original org)?
   - **assumption**: no, full override is simpler
2. what happens to orphaned keys in host manifest (old org)?
   - **assumption**: they're ignored; user must re-set
3. should `unlock` warn if extended keys were re-slugged?
   - **assumption**: no, consistent behavior is self-evident

## what is awkward?

### what feels off

1. **re-slugging is implicit**: users might not realize extended keys are re-slugged until they see the output
   - **mitigation**: consistent output shows the actual slug used

2. **host manifest must match**: if you set a key under the wrong org, it won't be found after re-slug
   - **mitigation**: set command always uses resolved org, so this only happens if user manually edits host manifest

3. **no migration path**: extant keys under old org require manual re-set
   - **mitigation**: this is a regression fix; users who hit the bug already need to re-set anyway

### tradeoffs

- **full override vs selective**: we chose full override for simplicity; partial override adds complexity
- **re-slug on hydration vs on-demand**: re-slugging at hydration time is simpler; on-demand would require tracking original org

## acceptance journeys to add

the wish explicitly requires thorough acceptance journey tests.

### test-first order (test-covered repairs)

1. **write acceptance tests that fail** — reproduce the defect via tests FIRST
2. **fix the code** — implement re-slug at hydration
3. **tests pass** — proof the fix works

this order ensures the defect cannot recur (test catches it) and documents the edge case.

### journeys to cover

1. **cross-org set/unlock/get**: set key in repo with org override, unlock, get — all consistent
2. **nested extends**: A(org:X) extends B(org:Y) extends C(org:Z) — all keys use org X
3. **same org**: repo extends keyrack with same org — no re-slug needed
4. **duplicate keys**: root and extended define same key name — root wins, both use root org
