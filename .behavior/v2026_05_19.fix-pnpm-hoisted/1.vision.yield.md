# vision: fix pnpm node-linker=hoisted compatibility

## the outcome world

### before
```bash
# user has node-linker=hoisted in .npmrc (common for react-native, metro, etc)
$ ./node_modules/.bin/rhx git.repo.test --what unit
./node_modules/.bin/rhx: 28: exec: /path/to/node_modules/.bin/run: not found

$ npx rhx git.repo.test --what unit
# same error — npx delegates to ./node_modules/.bin/rhx
```

user is stuck. claude code hooks fail. catch-22: can't use rhx to diagnose rhx.

### after
```bash
# same setup — node-linker=hoisted in .npmrc
$ ./node_modules/.bin/rhx git.repo.test --what unit
# works

$ npx rhx git.repo.test --what unit
# works
```

rhachet works regardless of pnpm linker mode. users don't need to know or care.

### the "aha" moment
"it just works" — no special configuration, no .npmrc edits, no workarounds.

## user experience

### usecases
| who | what | how |
|-----|------|-----|
| react-native dev | may use hoisted mode for metro compat | `npx rhx` works out of box |
| monorepo maintainer | uses hoisted for workspace compat | `rhx` commands work |
| new user | doesn't know about linker modes | never encounters the issue |

### contract (unchanged)
the public contract doesn't change. users still invoke:
- `npx rhx <skill>` or `./node_modules/.bin/rhx <skill>`
- `npx rhachet run --skill <skill>`

the fix is internal — symlink resolution in `bin/rhx`.

### timeline
1. user installs rhachet: `pnpm add -D rhachet`
2. user runs skill: `npx rhx git.repo.test --what unit`
3. works regardless of node-linker mode

## evaluation

### how well does it solve the goals?
| goal | solved? |
|------|---------|
| `rhx` works with hoisted | yes |
| `npx rhx` works with hoisted | yes |
| backward compatible | yes — extant installs unaffected |
| no user action required | yes — fix is in rhachet itself |

### pros
- zero user configuration
- fixes both `rhx` and `npx rhx`
- small, surgical change to `bin/rhx`
- follows pattern already in `bin/run`

### cons
- adds ~5 lines to `bin/rhx`
- uses bash instead of sh (matches `bin/run`)

### edge cases
| edge case | handling |
|-----------|----------|
| symlink chain (multiple hops) | while loop follows all hops |
| relative symlinks | resolved relative to link location |
| absolute symlinks | used directly |
| non-symlink invocation | loop exits immediately, no change |

## open questions & assumptions

### assumptions
1. pnpm hoisted mode creates symlinks (not shell stubs) on POSIX — verified via pnpm docs
2. the symlink resolution pattern in `bin/run` is correct — it's battle-tested
3. changing `rhx` from `#!/bin/sh` to `#!/usr/bin/env bash` is acceptable — `bin/run` already uses bash

### questions
- [answered] is bash dependency acceptable? — yes, `bin/run` already uses bash. implicit acceptance.
- [answered] any other entry points affected? — no, only `rhx` and `rhachet`. `rhachet` uses `run` which already has symlink logic.
- [answered] what about npm/yarn? — unaffected. they don't create the same symlink structure, or already work.
- [answered] does npx use same codepath? — yes, `npx rhx` = `./node_modules/.bin/rhx`. same binary.
- [research] how do we verify the fix works? — acceptance test via symlink simulation:

### verification strategy

acceptance test that simulates pnpm hoisted mode:

```typescript
// accept.blackbox/cli/rhx-symlink.acceptance.test.ts
given('[case1] rhx invoked via symlink', () => {
  const scene = useBeforeAll(async () => {
    const tempDir = await genTempDir();
    // create symlink: tempDir/rhx -> repo/bin/rhx
    await symlink({ at: `${tempDir}/rhx`, to: `${repoRoot}/bin/rhx`, mode: 'absolute' });
    return { symlinkPath: `${tempDir}/rhx` };
  });

  when('[t0] symlink is invoked with --help', () => {
    const result = useThen('invocation succeeds', async () =>
      execSync(`${scene.symlinkPath} --help`, { encoding: 'utf8' }),
    );

    then('help output is shown', () => {
      expect(result).toContain('rhachet');
    });
  });
});
```

this test:
1. creates a symlink to `bin/rhx` (simulates `node_modules/.bin/rhx` in hoisted mode)
2. invokes via that symlink
3. asserts it works (finds peer `run`, executes correctly)

## groundwork

### external research
checked pnpm documentation and issues:
- [pnpm issue #4782](https://github.com/pnpm/pnpm/issues/4782): confirms hoisted mode symlink behavior
- [pnpm settings docs](https://pnpm.io/settings): documents `node-linker` options
- [pnpm symlinked structure](https://pnpm.io/symlinked-node-modules-structure): explains how symlinks work

key finding: with `node-linker=hoisted`, pnpm creates symlinks in `node_modules/.bin/` that point to the hoisted package location, not into `.pnpm/` store.

### internal research
checked extant behavior:

**bin/rhx:7** — current (broken) resolution:
```sh
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
```
does NOT follow symlinks. when `$0` is a symlink, `SCRIPT_DIR` is the symlink's directory.

**bin/run:14-20** — working resolution:
```bash
EXECUTABLE="${BASH_SOURCE[0]}"
while [ -L "$EXECUTABLE" ]; do
  SCRIPT_DIR="$(cd "$(dirname "$EXECUTABLE")" && pwd)"
  EXECUTABLE="$(readlink "$EXECUTABLE")"
  [[ "$EXECUTABLE" != /* ]] && EXECUTABLE="$SCRIPT_DIR/$EXECUTABLE"
done
SCRIPT_DIR="$(cd "$(dirname "$EXECUTABLE")" && pwd)"
```
follows symlinks to find actual executable location.

**package.json:50-53** — bin entries:
```json
"bin": {
    "rhachet": "./bin/run",
    "rhx": "./bin/rhx"
}
```
`run` is NOT a bin entry — it's internal. so `node_modules/.bin/run` doesn't exist; only `rhachet` and `rhx` do.

## what is awkward?

### the sh → bash change
`rhx` currently uses `#!/bin/sh` while `run` uses `#!/usr/bin/env bash`. the symlink resolution uses bash-specific `[[ ]]` syntax and `BASH_SOURCE`. we'd need to either:
1. change `rhx` to bash (matches `run`, simple)
2. rewrite resolution for POSIX sh (more complex, diverges from `run`)

option 1 feels right — consistency with `run`, and bash is ubiquitous.

### the duplication
the symlink resolution logic would be duplicated between `rhx` and `run`. could extract to a shared file, but:
- adds complexity for ~5 lines
- sourcing another file has its own symlink issues
- `run` and `rhx` are the only two affected files

duplication is acceptable here — it's small, stable, and self-contained.
