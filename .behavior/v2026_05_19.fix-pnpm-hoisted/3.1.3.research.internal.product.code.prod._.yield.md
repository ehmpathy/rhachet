# internal product code research: prod codepaths

## pattern 1: symlink resolution in bin/run [REUSE]

**location**: `bin/run:13-20`

```bash
# resolve script directory (handles symlinks from node_modules/.bin/)
SCRIPT="${BASH_SOURCE[0]}"
while [ -L "$SCRIPT" ]; do
  SCRIPT_DIR="$(cd "$(dirname "$SCRIPT")" && pwd)"
  SCRIPT="$(readlink "$SCRIPT")"
  [[ "$SCRIPT" != /* ]] && SCRIPT="$SCRIPT_DIR/$SCRIPT"
done
SCRIPT_DIR="$(cd "$(dirname "$SCRIPT")" && pwd)"
```

**what it does**:
1. starts with `BASH_SOURCE[0]` (the actual executable path, not `$0` which can be the symlink)
2. while the path is a symlink (`-L` test), follows it with `readlink`
3. handles relative symlinks by prepending the current dir
4. after loop exits, `SCRIPT_DIR` is the real directory

**relation to wish**: this is the exact pattern we need to copy into `bin/rhx`.

## pattern 2: naive dir lookup in bin/rhx [REPLACE]

**location**: `bin/rhx:7`

```sh
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
```

**what it does**:
- uses `$0` which is the symlink path when invoked via symlink
- does NOT follow the symlink
- `SCRIPT_DIR` ends up as `node_modules/.bin/` instead of `actual/path/bin/`

**relation to wish**: this is the bug. replace with pattern 1.
