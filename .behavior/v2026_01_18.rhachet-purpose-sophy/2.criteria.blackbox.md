# blackbox criteria: rhachet purpose sophy

experience bounds required to fulfill the wish of rhachet philosophy formalization.

---

# usecase.1 = understand the dual ratchet metaphor

```
given('a user who wants to understand why rhachet is named after "ratchet"')
  when('they read the readme.md')
    then('they find a clear explanation of the dual ratchet metaphor')
      sothat('they grasp that rhachet operates at two levels')
    then('the first ratchet is explained: slipless iterative improvement of thought routes')
      sothat('they understand how roles/briefs/skills enable incremental capability improvement')
    then('the second ratchet is explained: guarantee the distributors postlabor future')
      sothat('they understand how brain-build capability distribution secures collective abundance')
  when('they want deeper context on either ratchet')
    then('the readme links to briefs for further exploration')
      sothat('they can dive deeper without readme clutter')
```

# boundary cases

```
given('a user who only cares about the tool, not the philosophy')
  when('they read the readme')
    then('the philosophy section is clearly labeled and skippable')
      sothat('practical users are not blocked by philosophical content')

given('a user who is skeptical of the optimistic frame')
  when('they read the briefs')
    then('they find logical arguments, not emotional appeals')
      sothat('skeptics can engage with the logic')
    then('they find source citations for key claims')
      sothat('skeptics can verify independently')

given('a user who is already familiar with "Why Nations Fail" or similar')
  when('they read the briefs')
    then('they find the rhachet-specific synthesis, not just summaries')
      sothat('they gain new insight from the connection to distributed brains')
```

# non-goals (out of scope for blackbox)

- implementation details of how briefs are structured
- specific file paths or directory organization
- internal name conventions
- mechanism contracts between components
- build/test/deployment processes

these are whitebox concerns, not experience bounds.
