# research: product root cause reflection

## summary

this document drills to the true root cause of why envvar-boot-roles is needed via 5 whys analysis.

---

## the symptom

**users cannot customize which roles load when they spawn a clone.**

today: `claude` boots with whatever roles are in `.claude/settings.json`. no runtime customization.

---

## 5 whys analysis

### why #1: why can't users customize roles at spawn time?

**because** the role hooks are baked into `.claude/settings.json` at `rhx init` time.

### why #2: why are role hooks baked at init time?

**because** brain hooks must be in config files before the brain boots — brains read config at startup, not at runtime.

### why #3: why don't we generate config dynamically at spawn time?

**because** we have no spawn-time hook — users invoke `claude` directly, which reads its own config.

### why #4: why do users invoke `claude` directly?

**because** there's no rhachet wrapper command that intercepts spawn to inject dynamic config.

### why #5: why is there no spawn wrapper?

**because** the initial design assumed roles would be static per-repo, not dynamic per-session.

---

## root cause

**the initial design assumed roles are a repo-level concern, not a session-level concern.**

roles were treated like project settings — set once, apply everywhere. the design lacked a session-level customization layer.

---

## root cause tree

```
symptom: no per-session role customization
    │
    └─ why: hooks baked at init
           │
           └─ why: brain reads config at startup
                  │
                  └─ why: no dynamic config at spawn
                         │
                         └─ why: no spawn wrapper
                                │
                                └─ ROOT: design assumed roles are repo-level, not session-level
```

---

## layer analysis

| layer | what exists | what's absent |
|-------|-------------|---------------|
| repo-level | `.claude/settings.json` with all linked roles | — |
| session-level | — | dynamic config per spawn |
| spawn-time | direct `claude` invocation | wrapper to inject config |

the absent piece: **a spawn-time mechanism that generates session-scoped config**.

---

## candidate solutions by layer

| layer | solution | tradeoff |
|-------|----------|----------|
| repo-level | multiple settings files per role combo | combinatorial explosion, not scalable |
| session-level | env var read by brain | requires brain to support env var (not controllable) |
| spawn-time | wrapper generates dynamic config | requires wrapper command, but fully controllable |

---

## recommendation

**address at spawn-time layer via wrapper command.**

rationale:
- repo-level: not scalable (N roles = 2^N combos)
- session-level: depends on brain support (out of our control)
- spawn-time: fully controllable, one implementation, all combos

the wrapper (`rhx enroll`) intercepts spawn, generates session-scoped config, then invokes brain. this is the only layer we fully control.

---

## verification

the root cause is correct if:
1. wrapper command can generate dynamic config — yes, filesystem is controllable
2. dynamic config can override defaults — yes, settings.local.json has precedence
3. brain can boot with dynamic config — yes, standard config paths work

all three are true. the root cause is validated.
