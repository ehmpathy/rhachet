# research: product audience reflection

## summary

this document identifies who benefits from envvar-boot-roles and why they care.

---

## who is the audience?

### primary: developers who spawn task-focused clones

**profile**: developers who use `rhx` or `claude` to spawn ai assistants for specific tasks.

**context**:
- work on repos with multiple roles linked (mechanic, driver, ergonomist, architect)
- want focused assistance for a specific task
- frustrated by irrelevant briefs that clutter context

**examples**:
- vlad spawns a mechanic clone to fix tests — doesn't need driver briefs
- sarah spawns an ergonomist clone for ux review — doesn't need mechanic briefs
- alex spawns architect + mechanic for a refactor — wants exactly these two

### secondary: repo maintainers who set up default roles

**profile**: developers who configure which roles are linked in a repo.

**context**:
- set up `.agent/` with linked roles
- want sensible defaults for the team
- want individual developers to customize without repo changes

**examples**:
- repo maintainer links mechanic, driver, ergonomist as defaults
- team members can subtract roles they don't need per-session
- no need to modify `.claude/settings.json` manually

### tertiary: rhachet contributors who extend brain support

**profile**: developers who add support for new brains (beyond claude-code).

**context**:
- implement BrainHooksAdapter for new brains
- benefit from pattern established by `rhx enroll`
- reuse roles spec syntax and computation logic

## why do they care?

### pain relieved

| audience | pain | how relieved |
|----------|------|--------------|
| primary | context clutter from irrelevant briefs | load only needed roles |
| primary | no way to customize roles per-session | `--roles` flag enables per-session control |
| secondary | manual edit of settings.json | dynamic config generation |
| tertiary | no pattern for role customization | reusable spec syntax |

### goals enabled

| audience | goal | how enabled |
|----------|------|-------------|
| primary | focused ai assistance | fewer roles = more relevant context |
| primary | faster iteration | less noise = clearer conversation |
| secondary | team flexibility | individuals customize without repo changes |

### frustrations eliminated

| audience | frustration | eliminated by |
|----------|-------------|---------------|
| primary | "why does it offer to create prs when i just want code fixed?" | subtract driver role |
| primary | "i have to edit settings.json every time" | `--roles` flag at spawn |
| secondary | "everyone wants different role combos" | per-session customization |

## what do they care about?

### success criteria

| audience | criteria |
|----------|----------|
| primary | "does `--roles mechanic` boot with only mechanic briefs?" |
| primary | "can i append architect to defaults without a full role list?" |
| primary | "are my other flags passed through to claude?" |
| secondary | "does default behavior stay the same when `--roles` is omitted?" |

### acceptable tradeoffs

| tradeoff | primary accepts? | secondary accepts? |
|----------|------------------|-------------------|
| wrapper command vs env var | ✅ yes (explicit is fine) | ✅ yes |
| ephemeral config (not persisted) | ✅ yes (per-session is the point) | ✅ yes |
| fail-fast on typo | ✅ yes (prefer clarity over silent failure) | ✅ yes |

### delighted vs disappointed

| outcome | delighted | disappointed |
|---------|-----------|--------------|
| `--roles mechanic` boots with only mechanic | ✅ | |
| typo gives helpful suggestion | ✅ | |
| passthrough args work seamlessly | ✅ | |

## how will they experience it?

### touchpoints

1. **discover**: read docs or see example in team workflow
2. **first use**: try `rhx enroll claude --roles mechanic`
3. **verify**: notice only mechanic briefs in session
4. **iterate**: try `--roles -driver` to subtract from defaults
5. **error recovery**: typo gives "did you mean?" suggestion

### friction points

| friction | mitigation |
|----------|------------|
| "what's the syntax again?" | clear help text, examples |
| "which roles are available?" | `rhx roles list` shows linked roles |
| "did it work?" | boot message shows loaded roles |
| "typo — role not found" | levenshtein suggestion |

### smooth experience

```bash
# vlad wants to fix tests — mechanic only
$ rhx enroll claude --roles mechanic
# 🎭 enrolled with roles: mechanic
# claude boots with mechanic briefs only

# vlad made a typo
$ rhx enroll claude --roles mechnic
# ❌ role 'mechnic' not found, did you mean 'mechanic'?

# vlad wants defaults minus driver
$ rhx enroll claude --roles -driver
# 🎭 enrolled with roles: mechanic, ergonomist (driver removed)
```
