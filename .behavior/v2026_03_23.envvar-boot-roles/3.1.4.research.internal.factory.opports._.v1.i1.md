## the constraint

### the bottleneck

**brain spawn verification** is the singular bottleneck.

the enroll command must:
1. parse roles spec ✅ (unit testable)
2. compute final roles ✅ (unit testable)
3. generate config file ✅ (integration testable)
4. spawn brain with config ❌ (requires actual brain to run)

step 4 creates friction:
- tests cannot easily verify spawn behavior
- manual verification adds ~5s per iteration
- mock `child_process.spawn` adds complexity and brittleness

### if fixed

`--mode plan` would:
- output generated config to stdout
- output computed spawn args to stdout
- exit without spawn

tests can then assert on output directly — no mock, no brain, no wait.

## flow and queues

### verification queues

| queue | wait time | controllable? |
|-------|-----------|---------------|
| unit tests | ~100ms | ✅ yes |
| integration tests | ~500ms | ✅ yes |
| brain spawn | ~5s | ❌ no (external process) |

### what must wait

brain spawn waits for:
- claude-code CLI startup
- model selection
- credential verification

none of these are controllable from test code.

### work-in-progress limit

can iterate freely on parser and config generation. stuck when need to verify end-to-end with actual brain.
