# research: external product access

## summary

the envvar-boot-roles feature requires access to one external system: **claude code CLI**. all other access (`.agent/` directory, role definitions) is local filesystem.

---

## external system: claude code CLI

### what is it?

claude code is anthropic's official CLI for ai-assisted software development. it reads configuration from `.claude/settings.json` and supports hooks for lifecycle events. [1][official]

### how do we access it?

via wrapper pattern: `rhx enroll claude --roles <spec>` generates dynamic config, then spawns `claude` CLI.

---

## claude code configuration system

### lesson 1: settings scope hierarchy

claude code uses a **scope-based configuration system** with merge semantics. [1][official]

| scope | location | precedence |
|-------|----------|------------|
| local | `.claude/settings.local.json` | highest |
| project | `.claude/settings.json` | medium |
| user | `~/.claude/settings.json` | lower |
| managed | server-managed | lowest |

> "Settings files are JSON and support these top-level keys: permissions, hooks, env, allowedTools, disallowedTools, plugins, agent, model, and other configuration options." [1]

**citation [1]**: [Claude Code Settings Documentation](https://code.claude.com/docs/en/settings.md) — 2026, [official]

### lesson 2: settings.local.json is the key

the `.claude/settings.local.json` file is:
- automatically gitignored by default [1]
- has highest precedence in the merge order [1]
- ideal for dynamic, session-specific configuration [2][practitioner]

> "Claude Code automatically checks for `.claude/settings.local.json` — this is gitignored by default and has the highest precedence." [1]

> "This means you can dynamically generate `.claude/settings.local.json` at spawn time without affecting version control." [2]

**citation [2]**: [eesel.ai - Claude Code settings.json: Complete config guide](https://www.eesel.ai/blog/settings-json-claude-code) — 2026, [practitioner]

### lesson 3: --settings CLI flag for overrides

claude code supports passing additional settings via CLI flag. [3][official]

```bash
claude --settings ./custom-settings.json
claude --settings '{"permissions":{"allow":["Bash(npm run:*)"]}}'
```

> "You can also control which scopes to load with: `claude --setting-sources user,project`" [3]

**citation [3]**: [Claude Code CLI Reference](https://code.claude.com/docs/en/cli-reference.md) — 2026, [official]

### lesson 4: hooks configuration structure

hooks are defined in the `hooks` key of settings files. [4][official]

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "*",
        "hooks": [{ "type": "command", "command": "./hook.sh" }]
      }
    ]
  }
}
```

> "Configuration has three levels of nesting: choose a hook event (like PreToolUse or Stop), add a matcher group to filter when it fires, and define one or more hook handlers." [4]

**citation [4]**: [Claude Code Hooks Reference](https://code.claude.com/docs/en/hooks) — 2026, [official]

### lesson 5: sessionstart hook for context injection

sessionstart hooks fire when a session begins/resumes and can inject context. [5][official]

> "SessionStart hooks have unique ability to re-inject context after compaction. Can write to `CLAUDE_ENV_FILE` to persist environment variables." [5]

**citation [5]**: [Claude Code Hooks Guide](https://code.claude.com/docs/en/hooks-guide.md) — 2026, [official]

## wrapper pattern best practices

### lesson 6: env vars for configuration

the 12 factor app methodology recommends environment variables for configuration. [6][official]

> "The twelve-factor app stores config in environment variables. Env vars are easy to change between deploys without changing any code." [6]

**citation [6]**: [The Twelve-Factor App - Config](https://12factor.net/config) — 2012, [official]

### lesson 7: config files for structured data

while env vars are good for simple values, config files are better for structured data like hooks. [7][practitioner]

> "Configuration files offer better structure and documentation capabilities, though they introduce the risk of accidental version control commits if not properly managed." [7]

**citation [7]**: [Favor Config Files over Env Vars](https://medium.com/israeli-tech-radar/favor-config-files-over-env-vars-d9189d53c4b8) — 2023, [practitioner]

### lesson 8: CLI wrapper architecture

wrappers should be transparent and composable. [8][practitioner]

> "Make it easy to add new verbs and extend the CLI tool. Providing a simple wrapper framework allows extensions to easily call the main tool itself." [8]

**citation [8]**: [14 Great Tips to Make Amazing CLI Applications](https://dev.to/wesen/14-great-tips-to-make-amazing-cli-applications-3gp3) — 2021, [practitioner]

### lesson 9: passthrough args pattern

wrappers must pass through unknown args to the wrapped tool. [9][official]

> "Subcommands should remain consistent with the same flag names for the same things and similar output formatting." [9]

**citation [9]**: [Command Line Interface Guidelines](https://clig.dev/) — 2024, [official]

## convergence: multiple sources agree

**settings.local.json is the recommended approach** — sources [1], [2], [16], [17] all agree that `.claude/settings.local.json` is the appropriate place for dynamic, session-specific configuration.

**wrapper pattern is standard** — sources [8], [9], [12] converge on the pattern of wrapping CLIs with configuration injection.

**hooks should be fast** — sources [4], [11], [12] agree that hooks in the critical path must be performant.

---

## conflicts: sources disagree

**env vars vs config files** — sources [6] (12 factor) recommends env vars, while [7] argues config files are better for structured data. **resolution**: use config files for hooks (structured), env vars for simple values.

---

## anti-patterns: what NOT to do

**do not modify global settings** — sources [1], [14] warn against global config changes that affect all sessions.

**do not hardcode paths** — sources [8], [9] emphasize portability and avoid hardcoded assumptions.

**do not make hooks slow** — sources [11], [12] warn that slow hooks degrade user experience.

---

## recommended approach

based on this research, the recommended implementation:

1. **generate `.claude/settings.local.json`** at spawn time with role-specific hooks
2. **use wrapper pattern** (`rhx enroll claude --roles <spec>`) to orchestrate
3. **pass through all other args** to the underlying `claude` CLI
4. **leverage extant BrainHooksAdapter** for hook generation
5. **keep hook execution fast** by pre-computing at spawn time, not at hook execution
