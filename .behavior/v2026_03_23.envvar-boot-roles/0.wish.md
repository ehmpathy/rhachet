wish =

we want to be able to spawn fresh clones with dynamic roles booted

specifically, today, roles always get initted into projects with their brain hooks applied against the default config of the repo

(e.g., mechanic role -> .claude/settings.json w/ author = mechanic)

we want that behavior to retain, that declares the default roles to spawn clones with

---

however, we now want to support customizable roles

via the below pragma

RHACHET_ROLES=-driver => subtract the driver role hooks from the brain 
RHACHET_ROLES=+architect => append the architect role hooks to the brain
RHACHET_ROLES=ergonomist = replace the role hooks to only have ergonomist

RHACHET_ROLES=-driver,+architect => subtract driver, append architect (to the default)
RHACHET_ROLES=mechanic,ergonomist = replace roles to just these two

that way, when we spawn a fresh clone, we can granularly control the roles its enrolled to

---

think through how we can do this.

---

## HARD REQUIREMENT: custom config, reject defaults

the enroll command MUST:
1. generate a CUSTOM config file with unique name (e.g., `.claude/settings.-driver.$isodate.$hash.json`)
2. REJECT the global `~/.claude/settings.json`
3. REJECT the repo `.claude/settings.json`
4. use ONLY the custom config for the enrolled session

this is non-negotiable. the enrolled brain must boot with ONLY the specified roles' hooks — no merging, no inheritance, no defaults.

---

rhx enroll $brainCli --roles -driver,+architect

e.g.,

rhx enroll claude --roles -driver,+architect --resume flag-to-passthrough

---

npx rhachet enroll claude ...

as the full command

and 

`rhx enroll` as the early exit rhx alias
