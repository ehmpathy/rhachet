# research: external product claims

## claim categories

### 1. multi-agent role specialization

#### claim 1.1 [FACT]
**specialized agents perform better than generalist agents**

> "most real-world deployments rely on multiple specialized agents working together, each handling a specific role within a larger workflow. one agent plans, another executes, a third validates."
> — [analytics vidhya, ai agents trends 2026](https://www.analyticsvidhya.com/blog/2026/01/ai-agents-trends/) [practitioner, 2026]

#### claim 1.2 [OPIN]
**role overlap causes redundancy and confusion**

> "assign each agent a distinct role based on the tasks it should perform, avoid overlapping functionalities to reduce redundancy and confusion"
> — [talkdesk, ai agent platform best practices](https://support.talkdesk.com/hc/en-us/articles/39096730105115-AI-Agent-Platform-Best-Practices) [official, 2026]

#### claim 1.3 [SUMP]
**personas improve agent behavior when using "you are" statements**

> "use 'you are…' statements to help the agent assume a specific persona or mindset"
> — [medium, building ai agents with personas](https://medium.com/@leviexraspk/building-ai-agents-with-personas-goals-and-dynamic-memory-6253acacdc0a) [practitioner, 2026]

#### claim 1.4 [FACT]
**context quality determines agent quality**

> "agents are only as good as the context they operate in"
> — [analytics vidhya, ai agents trends 2026](https://www.analyticsvidhya.com/blog/2026/01/ai-agents-trends/) [practitioner, 2026]

#### claim 1.5 [KHUE]
**too much irrelevant context degrades performance**

> "providing too much irrelevant data, especially without proper context, can confuse the agent and lead to less accurate or off-topic responses"
> — [talkdesk, ai agent platform best practices](https://support.talkdesk.com/hc/en-us/articles/39096730105115-AI-Agent-Platform-Best-Practices) [official, 2026]

### 2. configuration injection patterns

#### claim 2.1 [FACT]
**environment variables are injected before subprocess spawn**

> "if env is not None, it must be a mapping that defines the environment variables for the new process; these are used instead of the default behavior of inheriting the current process' environment"
> — [python docs, subprocess module](https://docs.python.org/3/library/subprocess.html) [official, 2026]

#### claim 2.2 [FACT]
**wrapper libraries transform cli commands to class methods**

> "cli wrapper uses subprocess to wrap external cli tools and present an interface that looks more like a python class, with cli commands becoming methods on the class"
> — [pypi, cli-wrapper](https://pypi.org/project/cli-wrapper/) [official, 2026]

#### claim 2.3 [FACT]
**environment variables can be layered on top of parent process**

> "environment variables configured using WithEnvironmentVariables(...) are applied on top of those inherited from the parent process"
> — [github, cliwrap](https://github.com/Tyrrrz/CliWrap) [official, 2026]

#### claim 2.4 [FACT]
**mcp clients inject env vars from config before spawn**

> "for mcp-compliant clients, environment variables are defined in a configuration file... the client reads this mapping before executing the command and ensures these values are present in the spawned process's environment block"
> — [apxml, mcp environment variables](https://apxml.com/courses/getting-started-model-context-protocol/chapter-4-debugging-and-client-integration/managing-environment-variables) [tutorial, 2026]

### 3. twelve-factor app configuration

#### claim 3.1 [FACT]
**twelve-factor stores config in environment variables**

> "the twelve-factor app stores config in environment variables. env vars are easy to change between deploys without changing any code; unlike config files, there is little chance of them being checked into the code repo accidentally"
> — [12factor.net, config](https://12factor.net/config) [official, 2012]

#### claim 3.2 [OPIN]
**env vars should be granular and orthogonal**

> "in a twelve-factor app, env vars are granular controls, each fully orthogonal to other env vars. they are never grouped together as 'environments'"
> — [12factor.net, config](https://12factor.net/config) [official, 2012]

#### claim 3.3 [KHUE]
**environment grouping does not scale**

> "sometimes apps batch config into named groups (often called 'environments') named after specific deploys... this method does not scale cleanly"
> — [12factor.net, config](https://12factor.net/config) [official, 2012]

### 4. config files vs environment variables

#### claim 4.1 [OPIN]
**config files provide better structure and visibility**

> "configuration files are more structured and provide better visibility"
> — [medium, favor config files over env vars](https://medium.com/israeli-tech-radar/favor-config-files-over-env-vars-d9189d53c4b8) [practitioner, 2023]

#### claim 4.2 [FACT]
**env vars are limited to strings, no structures**

> "environment variables are limited to strings—no lists or structures such as json, and no numbers. this means you need to handle conversions in the code"
> — [medium, favor config files over env vars](https://medium.com/israeli-tech-radar/favor-config-files-over-env-vars-d9189d53c4b8) [practitioner, 2023]

#### claim 4.3 [SUMP]
**hybrid approach is optimal**

> "a hybrid approach is recommended: have most configurations committed and tracked in code in different files for development, production, etc., and use environment variables or a secure key store only for configurations that can't be committed to code"
> — [dzone, configuration files](https://dzone.com/articles/configuration-files) [practitioner, 2018]

#### claim 4.4 [KHUE]
**variable cross-contamination risk**

> "there's a high chance of variable cross contamination by accidentally naming one variable the same as another which is unknowingly used for a different purpose"
> — [medium, favor config files over env vars](https://medium.com/israeli-tech-radar/favor-config-files-over-env-vars-d9189d53c4b8) [practitioner, 2023]

### 7. prompt injection security

#### claim 7.1 [FACT]
**system prompts are meant to be immutable**

> "system prompts serve as the foundation of llm security and behavior control. they define how a model should behave... these prompts are meant to be immutable instructions"
> — [promptfoo, system prompt override plugin](https://www.promptfoo.dev/docs/red-team/plugins/system-prompt-override/) [official, 2026]

#### claim 7.2 [FACT]
**llm providers typically allow system messages in any request**

> "some llm deployments have a critical vulnerability: the system prompt can be modified by users. this security flaw exists because llm providers typically allow system messages to be included in any request"
> — [promptfoo, system prompt override plugin](https://www.promptfoo.dev/docs/red-team/plugins/system-prompt-override/) [official, 2026]

#### claim 7.3 [OPIN]
**scope limitations in system prompts are a defense**

> "defense strategies include including clear scope limitations in the system prompt, adding behavioral reminders, and repeating core instructions after user input to re-anchor the model"
> — [owasp, llm prompt injection prevention](https://cheatsheetseries.owasp.org/cheatsheets/LLM_Prompt_Injection_Prevention_Cheat_Sheet.html) [official, 2026]

### 9. race conditions and concurrency

#### claim 9.1 [FACT]
**race conditions cause collisions when concurrent requests modify same data**

> "race conditions... occur when websites process requests concurrently without adequate safeguards. this can lead to multiple distinct threads interacting with the same data at the same time, resulting in a 'collision'"
> — [portswigger, race conditions](https://portswigger.net/web-security/race-conditions) [official, 2024]

#### claim 9.2 [FACT]
**file locking prevents concurrent write conflicts**

> "when a process wants to write into a file, it first asks the kernel to lock that file or a part of it, and as long as the process keeps the lock, no other process can ask to lock the same part of the file"
> — [geeksforgeeks, race condition vulnerability](https://www.geeksforgeeks.org/race-condition-vulnerability/) [tutorial, 2024]

#### claim 9.3 [OPIN]
**immutable objects are inherently thread-safe**

> "adopting immutable objects, which, once created, cannot be modified... makes them inherently safe in concurrent environments"
> — [qwiet.ai, race conditions in concurrent programming](https://qwiet.ai/navigating-race-conditions-unraveling-security-concerns-in-concurrent-programming/) [practitioner, 2024]

## convergence signals

### multiple sources agree

**local override pattern is standard**:
- sources [5.1], [5.2], [5.3], [5.4] all describe local config overriding global config
- this is a universal pattern across salesforce cli, backstage, vs code, and dhis2

**wrapper pattern for environment injection**:
- sources [2.1], [2.2], [2.3], [6.1], [6.3] converge on wrappers as the way to inject configuration
- the pattern: wrap the target command, inject env/config, spawn with modified environment

**idempotency is required for configuration management**:
- sources [8.1], [8.2], [8.3] agree that configuration operations should be idempotent
- defines desired state, maintains it, handles repeated application gracefully

---

## conflicts detected

### env vars vs config files

- source [3.1] (12factor) strongly advocates environment variables
- source [4.1] argues config files provide better structure
- **resolution**: use config files for structured data (hooks), env vars for simple values

### immutability vs state modification

- source [7.1] says system prompts should be immutable
- our feature explicitly modifies what prompts/briefs are loaded
- **resolution**: we're modifying which prompts load, not the prompts themselves — this is config, not prompt mutation

---

## anti-patterns identified

**from research, what NOT to do:**

1. **do not use flat env vars for structured configuration** [4.2]
   - hooks require structured json, not flat strings

2. **do not batch configs into named "environments"** [3.3]
   - avoid `-prep`, `-production` file variants
   - instead: one base, local overrides via standard mechanism

3. **do not process concurrent requests without locking** [9.1], [9.2]
   - if multiple terminals spawn sessions, file collisions are possible
   - need per-session config files or process-level coordination

4. **do not ignore active sessions during cleanup** [10.3]
   - dynamic config files must persist for session duration

5. **do not provide silent failures on typos** [11.3]
   - "did you mean?" suggestions required for pit of success

## summary for implementation

### confirmed facts

1. **wrapper pattern is standard** — wrap command, inject config, spawn process
2. **local config override is universal** — `.local` files override base config
3. **idempotency is required** — operations should define and maintain desired state
4. **fuzzy matching improves ux** — "did you mean?" prevents frustration
5. **file locking prevents race conditions** — concurrent sessions need coordination

### open questions from research

1. **[KHUE]** how to handle concurrent sessions in same repo? (file locking vs per-session files)
2. **[KHUE]** when to cleanup dynamic config? (session end vs timeout vs never)
3. **[KHUE]** how much context is too much? (role count vs context window)

### recommended implementation path

based on convergence across sources:
1. use wrapper pattern (`rhx enroll claude --roles <spec>`)
2. generate `.claude/settings.local.json` at spawn time
3. include typo suggestions via levenshtein distance
4. make role resolution idempotent (+present = no-op, -absent = no-op)
5. consider per-session config files for concurrency safety
