# research: internal production codepaths

## summary

this document analyzes the production codepaths relevant to envvar-boot-roles. patterns are marked with `[REUSE]`, `[EXTEND]`, or `[REPLACE]` to indicate the implementation strategy.

---

## relevant patterns

### 1. brain hooks adapter pattern [REUSE]

**file**: `src/domain.objects/BrainHooksAdapter.ts`

**what**: adapter contract that wraps DAO with slug identifier

```typescript
export interface BrainHooksAdapter {
  slug: string;
  dao: BrainHooksAdapterDao;
}
```

**relation to wish**: the adapter pattern already enables brain-agnostic hook application. `rhx enroll` will use this to write hooks for the specified roles.

**strategy**: [REUSE] — no changes needed. the adapter interface supports dynamic config generation.

---

### 2. claude code hooks adapter [REUSE/EXTEND]

**file**: `src/_topublish/rhachet-brains-anthropic/src/hooks/genBrainHooksAdapterForClaudeCode.ts`

**what**: creates claude code brain hooks adapter that reads/writes `.claude/settings.json`

key capabilities:
- `dao.get.one()` — find hook by unique key (author, event, command)
- `dao.get.all()` — list all hooks, optionally filtered
- `dao.set.findsert()` — idempotent insert
- `dao.set.upsert()` — insert or update
- `dao.del()` — remove hook

**relation to wish**: adapter already handles hook CRUD. for dynamic roles, we need to:
1. generate hooks for only the requested roles
2. write to `settings.local.json` instead of `settings.json`

**strategy**:
- [REUSE] the adapter structure and CRUD methods
- [EXTEND] config.dao to support `settings.local.json` as output path

---

### 3. config file I/O [EXTEND]

**file**: `src/_topublish/rhachet-brains-anthropic/src/hooks/config.dao.ts`

**what**: reads/writes claude code settings from `.claude/settings.json`

```typescript
export const readClaudeCodeSettings = async (input: {
  from: string;
}): Promise<ClaudeCodeSettings>

export const writeClaudeCodeSettings = async (input: {
  settings: ClaudeCodeSettings;
  to: string;
}): Promise<void>
```

**relation to wish**: needs extension to support `settings.local.json` for dynamic configs

**strategy**: [EXTEND] — add variant that writes to `.claude/settings.local.json`:
- new function `writeClaudeCodeSettingsLocal()` or param `{ toLocal: true }`
- local file has highest precedence per claude code docs
- local file is gitignored by default

### 4. role discovery from .agent/ [REUSE]

**file**: `src/domain.operations/invoke/findUniqueRoleDir.ts`

**what**: finds exactly one role directory by slug, with optional repo filter

key capabilities:
- scans `.agent/repo=*/role=*` structure
- normalizes `this` → `.this`
- handles disambiguation when multiple repos have the same role
- fail-fast with helpful errors

**relation to wish**: perfect fit for lookup of role names from `--roles` spec

**strategy**: [REUSE] — use directly. already handles:
- unique role lookup
- repo filter via `slugRepo` param
- `ifPresent` mode for soft failures

### 5. linked roles with hooks discovery [REUSE/EXTEND]

**file**: `src/domain.operations/brains/getLinkedRolesWithHooks.ts`

**what**: discovers linked roles and loads their full Role objects with hooks

key capabilities:
- scans `.agent/repo=*` directories (skips `repo=.this`)
- imports packages dynamically to get full Role objects
- filters to roles that have `hooks.onBrain` declared

**relation to wish**: we need this to get the hooks for specified roles

**strategy**:
- [REUSE] the discovery logic
- [EXTEND] to accept a filter for specific role slugs (not all linked roles)

### 6. hooks sync orchestration [REUSE/EXTEND]

**file**: `src/domain.operations/init/syncHooksForLinkedRoles.ts`

**what**: syncs brain hooks for linked roles to brain configs

flow:
1. discover linked roles with hooks
2. prune orphan hooks from brains
3. sync all role hooks into each brain repl

**relation to wish**: this is the current "apply all hooks" flow. for `rhx enroll`, we need:
- hooks for only the specified roles
- no orphan prune (we create dynamic config, not modify default)

**strategy**: [EXTEND] — extract core sync logic that accepts explicit role list:
- new function `genHooksForRoles({ roles, brainSlug })` that returns hooks config
- reuse in both `syncHooksForLinkedRoles` and new `rhx enroll` command

## implementation approach

### new components needed

1. **roles spec parser** — parse `--roles` syntax:
   - bare names → replace mode
   - `+name` → append to defaults
   - `-name` → subtract from defaults
   - validate no conflicts

2. **role computer** — compute final role list:
   - load default roles from `.agent/`
   - apply spec operations
   - validate role names exist (with typo suggestions)

3. **config generator** — generate dynamic config:
   - call `genBrainHooksAdapterForClaudeCode` variant
   - write to `.claude/settings.local.json`
   - include only hooks for specified roles

4. **brain spawner** — spawn brain CLI:
   - exec claude with passthrough args
   - inherit stdio for interactive session

### extension points

| component | current | extension needed |
|-----------|---------|------------------|
| config.dao | writes `settings.json` | add `settings.local.json` support |
| getLinkedRolesWithHooks | returns all linked roles | accept filter for specific slugs |
| syncHooksForLinkedRoles | syncs all roles | extract `genHooksConfig` for role subset |

## gaps identified

### 1. no default roles concept

currently, there's no explicit "default roles" list. roles are linked, and all linked roles get their hooks applied.

**options**:
- derive defaults from all roles with hooks in `.agent/`
- add explicit config (e.g., `.agent/defaults.yml`)
- use all linked roles as implicit defaults

**recommendation**: use all linked roles with hooks as implicit defaults. this matches current behavior and requires no new config.

### 2. no local settings support

config.dao currently only handles `settings.json`. need extension for `settings.local.json`.

**implementation**:
```typescript
export const writeClaudeCodeSettingsLocal = async (input: {
  settings: ClaudeCodeSettings;
  to: string;
}): Promise<void> => {
  const settingsPath = path.join(input.to, '.claude', 'settings.local.json');
  // ... same write logic
};
```

### 3. no session cleanup

dynamic `settings.local.json` files may accumulate. need cleanup strategy.

**options**:
- overwrite on each `rhx enroll` (simplest, file is gitignored)
- timestamped files with cleanup (complex, may not be needed)
- single file per repo (current approach)

**recommendation**: single file approach. `settings.local.json` is overwritten each spawn. gitignored by default. no cleanup needed.
