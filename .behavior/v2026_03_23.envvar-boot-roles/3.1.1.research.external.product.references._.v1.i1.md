## reference categories

### 1. cli design standards

#### reference 1 [official]
**command line interface guidelines (clig.dev)**

> "design for humans first: traditionally, unix commands were written under the assumption they were going to be used primarily by other programs. it's time to shed some of this baggage: if a command is going to be used primarily by humans, it should be designed for humans first."
> — [clig.dev](https://clig.dev/) [official, 2024]

**concepts established**: human-first design, composability, consistency

#### reference 2 [official]
**posix utility conventions**

> "guidelines are established for the naming of utilities and for the specification of options, option-arguments, and operands. the getopt() function in the system interfaces volume of posix.1-2017 assists utilities in handling options and operands that conform to these guidelines."
> — [opengroup.org](https://pubs.opengroup.org/onlinepubs/9699919799/basedefs/V1_chap12.html) [official, 2017]

**concepts established**: option syntax (`-o`, `--option`), option-arguments, operands

#### reference 3 [official]
**gnu coding standards - command-line interfaces**

> "define long-named options that are equivalent to the single-letter unix-style options. this is easy to do with the gnu function getopt_long. one of the advantages of long-named options is that they can be consistent from program to program."
> — [gnu.org](https://www.gnu.org/prep/standards/html_node/Command_002dLine-Interfaces.html) [official, 2024]

**concepts established**: long options, consistency across programs

#### reference 4 [official]
**microsoft .net command-line design guidance**

> "commands should be consistent with the conventions of the operating system and shell they run in. options can appear in any order."
> — [microsoft.com](https://learn.microsoft.com/en-us/dotnet/standard/commandline/design-guidance) [official, 2026]

**concepts established**: platform consistency, order independence

### 2. unix philosophy

#### reference 5 [academic]
**unix philosophy - wikipedia**

**concepts established**: single responsibility, composability, text streams

**concepts established**: small tools, rapid prototype, portability

### 3. argument parse standards

#### reference 8 [official]
**getopt - posix specification**

> "the getopt() function is a command-line parser that follows utility syntax guidelines."
> — [opengroup.org](https://pubs.opengroup.org/onlinepubs/009696799/functions/getopt.html) [official, 2017]

**concepts established**: standard option parse, optstring format

#### reference 9 [official]
**getopt(3) - linux man page**

> "an option is a hyphen followed by a single alphanumeric character. options that do not require arguments can be grouped after a hyphen, so -lst is equivalent to -t -l -s. the -- argument terminates options."
> — [man7.org](https://man7.org/linux/man-pages/man3/getopt.3.html) [official, 2024]

**concepts established**: grouped options, `--` terminator, option-argument bind

#### reference 10 [official]
**python getopt module**

> "if a character is followed by a colon in the optstring, the option takes an argument. long options which require an argument should be followed by an equal sign ('=')."
> — [python.org](https://docs.python.org/3/library/getopt.html) [official, 2026]

**concepts established**: colon convention, long option = syntax

### 6. string match algorithms

#### reference 17 [academic]

> "in information theory, linguistics, and computer science, the levenshtein distance is a string metric for measure the difference between two sequences, defined as the minimum number of single-character edits (insertions, deletions or substitutions) required to change one word into the other. it is named after soviet mathematician vladimir levenshtein, who defined the metric in 1965."
> — [wikipedia](https://en.wikipedia.org/wiki/Levenshtein_distance) [academic, 2024]

**concepts established**: edit distance, string similarity metric

## concepts for implementation

### from cli standards [1-4, 8-10]

- `--roles` as long option (gnu compatible)
- syntax: `--roles <spec>` where spec follows role grammar
- passthrough: all non-rhx args pass to base command
- fail fast on invalid syntax

### from role theory [11-16]

- roles as permission bundles (briefs + skills)
- multiple roles per session (many-to-many)
- specialization improves output quality

### from string match [17-19]

- suggest closest match on typo (levenshtein)
- threshold: distance <= 2 for suggestions
- "did you mean 'mechanic'?" pattern

### from context management [20-22]

- fewer roles = less context = better focus
- effective context < advertised context
- role selection manages context budget

### from design principles [23-27]

- fail fast on parse errors
- pit of success: defaults work, deviations explicit
- human-first: `--roles mechanic` over `RHACHET_ROLES=mechanic`
