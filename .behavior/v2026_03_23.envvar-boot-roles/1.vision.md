# vision: envvar-boot-roles

## the outcome world

### before

to spawn a clone means to accept whatever roles the repo defaults to:

```bash
# vlad spawns a mechanic clone to fix tests
claude  # boots with mechanic, driver, ergonomist — all defaults

# the clone has driver briefs that clutter its context
# the clone has ergonomist briefs when vlad just needs code fixed
# no way to customize without manual edits to .claude/settings.json
```

### after

spawn a clone with the exact roles needed via `rhx enroll`:

```bash
# vlad spawns a focused mechanic clone
rhx enroll claude --roles mechanic  # boots with ONLY mechanic

# sarah spawns a clone for ux work
rhx enroll claude --roles ergonomist  # boots with ONLY ergonomist

# alex wants the default but without driver overhead
rhx enroll claude --roles -driver  # default minus driver

# dev spawns architect + mechanic for a refactor
rhx enroll claude --roles mechanic,architect  # exactly these two

# resume a prior session with custom roles
rhx enroll claude --roles mechanic --resume
```

### the "aha" moment

> "wait, i can just... say which roles i want? and it boots with exactly those briefs and skills?"

the value clicks when:
1. **control**: you spawn a mechanic clone that stays in mechanic mindset — no driver workflow suggestions when you just need code
2. **focus**: the context window has fewer tokens of irrelevant briefs — only what's needed for the task
3. **speed**: boot is faster because fewer roles means fewer briefs to load

## user experience

### usecases

| goal | command | outcome |
|------|---------|---------|
| focused code work | `rhx enroll claude --roles mechanic` | only mechanic briefs/skills |
| ux review | `rhx enroll claude --roles ergonomist` | only ergonomist perspective |
| architecture deep-dive | `rhx enroll claude --roles architect` | only architect briefs |
| default minus noise | `rhx enroll claude --roles -driver` | defaults without driver |
| default plus specialist | `rhx enroll claude --roles +architect` | defaults plus architect |
| multi-role combo | `rhx enroll claude --roles mechanic,ergonomist` | exactly these two |
| resume with roles | `rhx enroll claude --roles mechanic --resume` | resume prior session |

### contract

**input**: `rhx enroll <brain> --roles <spec>`

**syntax** (for `--roles`):
- `role` — replace defaults with this role
- `role1,role2` — replace defaults with these roles
- `+role` — append to defaults
- `-role` — subtract from defaults
- `+role1,-role2` — append one, subtract another (from defaults)

**output**: brain boots with computed roles' hooks applied via unique config file

**hard requirement**: the enrolled brain boots with ONLY the specified roles' hooks — no merge, no inheritance, no defaults.

### timeline

```
t=0   human runs `rhx enroll claude --roles -driver`
t=1   rhx parses --roles spec (required)
t=2   rhx computes: default_roles - driver
t=3   rhx generates unique config (e.g., .claude/settings.enroll.$hash.json)
t=4   rhx spawns: claude --bare --settings <unique-config>
t=5   brain boots with ONLY the specified roles' hooks (no defaults loaded)
```

## mental model

### how users describe it

> "it's like a shell profile but for ai roles. set the env var, get a differently-configured clone."

> "think of it like docker's `-e` flag but for which hats the ai wears."

### analogies

| familiar concept | rhachet equivalent |
|------------------|-------------------|
| `PATH` customizes which binaries are found | `RHACHET_ROLES` customizes which roles are booted |
| docker `-e VAR=val` sets container config | `RHACHET_ROLES=x` sets clone config |
| shell aliases per-project | role combos per-task |

## evaluation

### how well does it solve the goals?

| goal | proposed | notes |
|------|----------|-------|
| spawn with dynamic roles | ✓ goal | wrapper at spawn time |
| reject default configs | ✓ goal | --bare rejects ~/.claude and .claude defaults |
| explicit roles required | ✓ goal | --roles is required, no implicit defaults |
| granular control | ✓ goal | +/- syntax for delta from linked roles |
| replace entirely | ✓ goal | bare names = replace |

### pros

- **low friction**: just add `--roles <spec>` to the command
- **composable**: +/- syntax builds on defaults
- **explicit**: no hidden state, roles visible in command
- **shell-native**: works with any shell, any wrapper

### cons

- **ephemeral**: env var doesn't persist (feature or bug?)
- **no validation at set time**: typo in role name fails at boot
- **syntax curve**: +/- vs bare requires comprehension

### edgecases and pit of success

| edgecase | handle |
|----------|--------|
| typo in role name | fail fast with helpful error: "role 'mechnic' not found, did you mean 'mechanic'?" |
| `-` a role not in defaults | no-op (idempotent) |
| `+` a role already in defaults | no-op (idempotent) |
| absent `--roles` | error: "--roles is required" |
| empty `--roles ""` | error: "--roles is empty, specify roles to enroll" |
| ops that conflict `+foo,-foo` | error: "cannot both add and remove 'foo'" |
| no `.agent/` directory | error: "no .agent/ found, run rhx init first" |

## open questions & assumptions

### assumptions

1. **[research, superseded] sessionstart hook can read env vars** — lower priority. wrapper approach bypasses hooks.
2. **[answered] hooks can modify settings dynamically** — no. wisher confirmed wrapper approach is mandatory.
3. **[answered] roles are discoverable at boot time** — yes, `.agent/` extant after `rhx init`. feature requires initialized repo.
4. **[research] one syntax fits all brains** — scope to claude-code first. other brains are future work.

### questions for wisher

1. **[wisher]** should `RHACHET_ROLES` persist in any way, or purely ephemeral?
2. **[wisher]** should we support `RHACHET_ROLES=@file` to read from a file? (in scope or future?)
3. **[answered]** what's the expected behavior if a role has no hooks? → silent skip. role may contribute briefs without hooks.
4. **[answered]** should this work for `rhx ask`/`rhx act` too? → no, spawned brains only. sdk usage passes roles via code.

### external research needed

1. **[research, superseded]** how does claude-code expose env vars to sessionstart hooks? (lower priority — wrapper bypasses hooks)
2. **[answered]** can we dynamically generate/swap settings.json at boot time? → yes, via `claude --bare --settings <path>`. `--bare` rejects all defaults, `--settings` loads custom config only.
3. **[research]** what's the performance cost of role computation at every boot? (measure after prototype)
4. **[research]** what happens to dynamic config files after session ends? (cleanup strategy)
5. **[answered]** how to handle multiple concurrent sessions in the same repo? → unique config naming `.claude/settings.enroll.$hash.json` prevents collision

## what is awkward?

### friction points

1. **wrapper required** — `rhx enroll claude` is slightly more verbose than `RHACHET_ROLES=x claude`. but it's explicit and reliable.

2. **dynamic config files** — we generate `.claude/settings.enroll.$hash.json`. these need cleanup or gitignore rules.

3. **passthrough args** — `rhx enroll claude --roles mechanic --resume` must pass `--resume` to claude. arg parse must be clean.

### tradeoffs that feel uncomfortable

| tradeoff | discomfort |
|----------|------------|
| wrapper vs env var | wrapper is reliable but less elegant |
| dynamic config vs file-per-combo | files are explicit but may clutter |
| fail-fast vs graceful degrade | strict is safer but may annoy users |

### where design fights mental model

- users might expect `RHACHET_ROLES=x claude` to work directly
- reality: claude-code reads settings.json at startup, we must intercept via wrapper
- `rhx enroll` is the explicit, reliable path — document this clearly
