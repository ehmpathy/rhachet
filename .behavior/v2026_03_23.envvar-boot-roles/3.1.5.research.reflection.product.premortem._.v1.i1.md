# research: product premortem reflection

## summary

this document surfaces risks and failure modes for envvar-boot-roles via premortem analysis.

---

## premortem: imagine we shipped and it failed

project forward: we shipped this, and it failed badly. why?

### most plausible failure scenarios

1. **settings.local.json doesn't merge correctly** — claude-code reads settings.json but ignores settings.local.json, so generated config has no effect. users see default roles despite `--roles` flag.

2. **passthrough args cause parse conflicts** — `--resume` or other claude flags collide with `--roles` parse. users can't combine custom roles with other claude features.

3. **concurrent sessions corrupt each other** — two terminals spawn clones in same repo, both write settings.local.json, second overwrites first. first session suddenly has wrong roles mid-conversation.

---

## what obvious signs did we ignore?

look back from the imagined failure — what signs were present that we dismissed?

- settings.local.json behavior was assumed from docs but never verified manually
- passthrough arg pass-through was assumed to work but never tested with actual claude flags
- concurrency scenario was noted as "low likelihood" without mitigation
- no acceptance test verifies actual brain behavior (only config generation)

---

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| settings.local.json has highest precedence | config has no effect, default roles load | low (docs are clear) |
| claude-code auto-reads settings.local.json | must pass explicit flag to read it | medium |
| passthrough args don't conflict with --roles | parse breaks on certain flag combos | low |
| single settings.local.json per repo is fine | concurrent sessions corrupt each other | medium |
| users understand +/- syntax | unclear UX, wrong role combos, frustration | low |

---

## what is the nightmare scenario?

### worst plausible outcome

**users spawn clones that ignore their role selection.**

- user runs `rhx enroll claude --roles mechanic`
- config is generated correctly
- but claude-code boots with default settings.json, not local
- user sees all roles, not just mechanic
- user thinks feature is broken
- trust in rhachet erodes

### hardest to recover from

**concurrent session corruption** — if session A's config is overwritten by session B mid-conversation, session A suddenly behaves unpredictably. this is hard to diagnose and erodes trust.

---

## what mitigations to consider?

| risk | mitigation | cost of mitigation |
|------|------------|--------------------|
| settings.local.json not read | verify behavior manually before implementation | 30 min |
| settings.local.json not read | research if explicit flag needed (e.g., `--settings`) | 1 hour |
| passthrough conflicts | test with common claude flags (--resume, --model, etc) | 1 hour |
| concurrent corruption | use unique config file per session (hash of timestamp + pid) | 2 hours |
| concurrent corruption | document "one session per repo" constraint | 30 min |
| syntax unclear | clear help text with examples | 1 hour |
| syntax unclear | helpful error messages for common mistakes | 1 hour |

---

## recommended actions

### before implementation

1. **verify settings.local.json behavior** — manual test: create settings.local.json, run claude, confirm it reads local config
2. **research --settings flag** — if local isn't auto-read, determine correct mechanism

### in implementation phase

3. **test passthrough args** — integration tests with `--resume`, `--model`, etc
4. **add unique session config** — generate `settings.local.{pid}.json` or similar

### after implementation

5. **acceptance test with actual claude** — one test that actually spawns claude and verifies roles
6. **clear documentation** — examples for all syntax variants
