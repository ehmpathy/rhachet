## section 5: other blockers

### 5.1 claude code background task behavior

**previous concern:** background task output may not wake idle sessions

**citation** [external, 3.1.1.research.external.product.references]
> "When run commands with run_in_background: true, the automatic completion notification does not appear in the conversation and does NOT activate claude from idle state."

**⚠️ RESOLVED — EMPIRICAL VALIDATION [2026-04-11]:**
we ran a sanity check: `sleep 600 && echo "did you get me?"` as a background task. after 10 minutes of complete idle (no user input, no activity), the task completed and **woke Claude Code CLI from idle state**. the message was delivered successfully.

**conclusion:** citation appears to be either outdated, context-specific, or applies to Claude Desktop (not CLI). our empirical test proves background tasks DO wake idle Claude Code CLI sessions. this validates the poll-via-background-task mechanism.

**status:** RESOLVED — not a blocker

### 5.2 stop_hook_active infinite loop guard

**potential concern:** onStop hook could cause infinite loops

**mitigation:** poll command must check stop_hook_active field and exit early if true

**citation** [external, 3.1.1.research.external.product.references]
> "Your Stop hook command needs to check whether it already triggered a continuation. Parse the stop_hook_active field from the JSON input and exit early if it's true."

**status:** addressed in design, not a blocker

### 5.3 role filter complexity

**potential concern:** role filter parse and match logic

**assessment:** simple enough to implement inline
- syntax: `+role`, `-role`, `role`, comma-separated
- no external library needed
- unit testable with data-driven cases

**status:** not a blocker
