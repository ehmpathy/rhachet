# research.reflection.product.premortem: brain-hook-on-push

## summary

premortem analysis for brain-hook-on-push. surfaces risks and failure modes before implementation.

---

## section 1: premortem — imagine we shipped and it failed

**scenario:** we shipped onPush hooks and they failed badly.

### most plausible failure scenarios

1. **infinite loop disaster** — onStop hook triggers continuation, poll runs, finds message, invokes hook, hook completes, onStop fires again, loop forever. clone burns tokens, never completes.

2. **silent message drop** — poll runs but messages are consumed by one clone before another polls. race condition causes some clones to miss messages they should receive.

3. **role filter mismatch** — clone with roles `[mechanic, driver]` receives message for `+foreman` because filter logic is inverted. wrong clones process wrong messages.

4. **poll never restarts** — background task completes, but onStop guard is too aggressive, prevents restart. clone goes idle, never polls again.

5. **message validation rejects valid messages** — strict schema validation rejects legitimate messages with optional fields absent. producers must retry with correct format.

---

## section 2: what obvious signs did we ignore?

**signs we might dismiss:**

- "stop_hook_active is documented, we'll handle it" — but if we read the field wrong or edge case occurs, loop happens anyway
- "file system is atomic, no race conditions" — but multiple clones poll same directory, whoever reads first wins
- "role filter syntax is simple" — but edge cases like empty filter or invalid syntax could cause unexpected matches
- "poll interval of 60s is fine" — but compounded with other delays, messages may arrive too late

---

## section 3: what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| stop_hook_active field is reliable | infinite loop burns tokens | low |
| file delete is atomic acknowledgment | race condition, double process | medium |
| poll interval acceptable for all usecases | time-sensitive messages missed | low |
| role filter syntax is intuitive | producers format messages wrong | medium |
| JSON parse never fails on valid input | edge cases crash poll | low |
| background task always restarts | clone goes idle permanently | low |

---

## section 4: what is the nightmare scenario?

**worst plausible outcome:** infinite loop + cost explosion

**scenario:**
1. clone receives message via onPush hook
2. hook command completes successfully
3. onStop fires, stop_hook_active check fails (bug in check logic)
4. poll re-invokes, finds no new messages, but still triggers
5. loop continues indefinitely
6. clone burns thousands of dollars in tokens
7. user's monthly budget exhausted in hours
8. trust in rhachet hooks destroyed

**why this is the nightmare:**
- financial damage is immediate and significant
- user loses trust in automation
- recovery requires manual intervention
- could affect multiple users before detected

---

## section 5: what mitigations to consider?

| risk | mitigation | cost of mitigation |
|------|------------|--------------------|
| infinite loop | guard counter: max 10 onStop per session | low: simple counter |
| infinite loop | message deduplication: track processed ids | low: set of uuids |
| race condition | claim file before process: rename to `.claimed` | low: atomic rename |
| role filter mismatch | unit tests with full case matrix | low: data-driven tests |
| poll never restarts | health check: verify poll task active | medium: additional hook |
| validation rejects | permissive parse with clear error messages | low: zod with defaults |
| cost explosion | circuit breaker: stop after N invocations | low: counter check |

### recommended mitigations for v1

1. **guard counter** — track onStop invocations per session, hard stop at 10
2. **message deduplication** — track processed message ids, skip duplicates
3. **atomic claim** — rename message file before process to prevent double-process
4. **comprehensive filter tests** — data-driven test matrix for all role patterns
