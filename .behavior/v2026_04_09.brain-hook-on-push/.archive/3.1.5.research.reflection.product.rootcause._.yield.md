# research.reflection.product.rootcause: brain-hook-on-push

## summary

root cause analysis for brain-hook-on-push. drills to the core reason this capability is needed.

---

## section 1: what is the symptom?

### observable problem

**what is broken or absent:**
- brain CLI clones cannot receive notifications while idle
- no mechanism to wake a clone on demand
- peer clones cannot coordinate without human intervention

**how it manifests:**
- clone completes task, goes idle
- notification arrives (slack, github, peer)
- notification sits unprocessed until human starts new session
- coordination stalls, workflows fragment

**when it occurs:**
- any time a clone finishes work and waits
- whenever external event should trigger clone action
- in multi-clone orchestration scenarios

---

## section 2: 5 whys tree

### branch 1: no wake mechanism

1. **why can't clones receive notifications while idle?**
   → no hook event fires when clone is idle

2. **why no hook event for idle clones?**
   → extant hooks only fire on boot, tool use, or stop events

3. **why only boot/tool/stop events?**
   → brain CLIs designed for session lifecycle, not push events

4. **why session lifecycle, not push events?**
   → traditional CLI model: human starts, AI responds, session ends

5. **why traditional CLI model?**
   → **ROOT CAUSE: brain CLIs (claude code) designed for human-initiated interaction, not autonomous coordination**

### branch 2: no peer communication

1. **why can't peers communicate directly?**
   → no shared communication channel

2. **why no shared channel?**
   → each clone runs in isolated session

3. **why isolated sessions?**
   → security boundary: clones should not access each other's state

4. **why not a mediated channel?**
   → no message queue or pub/sub built in

5. **why no built-in message queue?**
   → **ROOT CAUSE: brain CLIs not designed for multi-agent coordination**

### branch 3: file system as shared channel

1. **why use file system for notifications?**
   → only shared state accessible to multiple clones

2. **why not use native channels?**
   → native channels not exposed for programmatic delivery

3. **why use background tasks?**
   → **VALIDATED**: background task output DOES wake idle CLI sessions (empirical test 2026-04-11)

4. **why poll via background task?**
   → background task can poll `.comms/`, wake on message, invoke hook

5. **why this approach works?**
   → **SOLUTION: poll via background task, wake on completion, invoke hook command**

---

## section 3: root cause summary

| layer | issue | is root cause? |
|-------|-------|----------------|
| symptom | clones cannot receive notifications while idle | no |
| hooks | no onPush event in hook lifecycle | no |
| brain CLI | designed for human-initiated sessions | **yes** |
| multi-agent | no built-in coordination mechanism | **yes** |
| solution | poll via background task, wake on completion | **validated** |

**root causes identified:**
1. brain CLIs designed for human-initiated interaction
2. no built-in multi-agent coordination mechanism

---

## section 4: recommendation

### address symptom (appropriate for this scope)

**why symptom treatment is appropriate:**
- root cause (brain CLI design) is outside our control
- cannot modify claude code architecture
- workaround via poll + hooks achieves the goal
- acceptable latency (~30s average)

**symptom-level fix:**
- add onPush hook event type to BrainHook
- implement poll adapter for `.agent/.comms/`
- use onStop exit(2) to force wake behavior
- filter messages by role

### root cause is out of scope

**why:**
- requires brain CLI vendor changes (Anthropic)
- would need native push event in claude code
- workaround achieves 80% of value with 20% of effort

**follow-up if root cause addressed later:**
- if claude code adds native push/channel wake: migrate from poll
- if claude code adds multi-agent primitives: integrate directly
- tracked in: future behavior route (not this one)

---

## conclusion

**root cause:** brain CLIs designed for human-initiated sessions, not autonomous coordination

**approach:** poll via background task, validated to wake idle CLI sessions (empirical test 2026-04-11)

**tradeoffs accepted:**
- poll latency (~30s) vs instant push
- file system based delivery vs native channel

**note:** onStop exit(2) mechanism is no longer required for wake behavior — background task completion alone wakes the CLI. onStop may still be useful for other guards (e.g., ensure poll task is active).
