## section 1: claude code hooks documentation

### 1.1 official hooks reference

**citation [1]** [official, Anthropic, 2026]
> "Hooks are user-defined shell commands that Claude Code executes at specific points in its lifecycle. They allow you to extend Claude Code's capabilities without modify its core."
— [Claude Code Hooks Reference](https://code.claude.com/docs/en/hooks)

**key concepts:**
- lifecycle events: SessionStart, Stop, StopFailure, PreToolUse, PostToolUse
- exit codes: 0 = success, 2 = block action, other = non-block error
- JSON output: systemMessage, additionalContext fields

### 1.2 stop hook behavior

**citation [2]** [official, Anthropic, 2026]
> "The Stop event fires whenever Claude finishes respond, not only at task completion. They do not fire on user interrupts."
— [Claude Code Hooks Guide](https://code.claude.com/docs/en/hooks#stop-hooks)

**citation [3]** [official, Anthropic, 2026]
> "With decision: 'block' prevents Claude from stop. The reason field must be provided for Claude to know how to proceed and continue its work."
— [Claude Code Hooks Reference](https://code.claude.com/docs/en/hooks#hook-responses)

**implication:** onStop hook with exit(2) forces Claude to continue — key mechanism for wake.

### 1.3 background task behavior

**citation [4]** [official, Anthropic GitHub, 2026]
> "When run commands with run_in_background: true, the automatic completion notification does not appear in the conversation and does NOT activate claude from idle state."
— [GitHub Issue #21048](https://github.com/anthropics/claude-code/issues/21048)

**critical note:** background task output does NOT wake idle sessions. this contradicts earlier assumptions.

**citation [5]** [official, Anthropic, 2026]
> "Background tasks operate in three phases: Launch, Monitor, and Notify. When process completes, result is placed in notification queue and AI receives notification in next turn's context."
— [Claude Code Background Tasks](https://code.claude.com/docs/en/background-tasks)

### 1.4 hook input data

**citation [6]** [official, Anthropic, 2026]
> "All hooks receive JSON input with: session_id, transcript_path, cwd, permission_mode, and hook_event_name."
— [Claude Code Hooks Reference](https://code.claude.com/docs/en/hooks#hook-input)

**citation [7]** [official, Anthropic, 2026]
> "Your Stop hook command needs to check whether it already triggered a continuation. Parse the stop_hook_active field from the JSON input and exit early if it's true."
— [Claude Code Hooks Guide](https://code.claude.com/docs/en/hooks#infinite-loop-prevention)

**anti-pattern:** Stop hooks must guard against infinite loops via stop_hook_active check.

## section 3: dead letter queue patterns

### 3.1 AWS DLQ

**citation [16]** [official, AWS]
> "A special queue that temporarily stores messages that a system cannot process due to errors. Prevents the source queue from overflow."
— [AWS What is DLQ](https://aws.amazon.com/what-is/dead-letter-queue/)

**citation [17]** [official, AWS]
> "Messages with bugs, absent data, or invalid references where no amount of retry will help. Would block the queue forever or consume resources infinitely."
— [AWS SQS Dead Letter Queues](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-dead-letter-queues.html)

**implication:** `.comms/.errors/` serves as dead letter queue for malformed or failed messages.

### 3.2 DLQ best practices

**citation [18]** [official, AWS]
> "It is a best practice to always set the retention period of a dead-letter queue to be longer than the retention period of the original queue."
— [AWS SQS Dead Letter Queues](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-dead-letter-queues.html)

**citation [19]** [official, Redpanda]
> "Consider use different DLQ topics for different error categories (serialization errors, validation failures, downstream service errors). This enables targeted monitor and remediation."
— [Redpanda DLQ Patterns](https://www.redpanda.com/blog/reliable-message-processing-with-dead-letter-queue)

## section 4: filesystem-based queue patterns

### 4.1 atomic write patterns

**citation [20]** [practitioner, 2026]
> "The Create-Write-Rename pattern is the gold standard for safe file persistence in application development. Write all the data to a new temporary file in the same directory, close the temporary file, and rename/move the temporary file over the final destination file."
— [Tech Champion - Atomic Writes](https://tech-champion.com/data-science/stop-silent-data-loss-checksum-atomic-writes-temp-file-patterns/)

**citation [21]** [official, LWN.net, 2019]
> "The POSIX standard guarantees that rename() is atomic with respect to other filesystem operations, which is why the write-then-rename pattern works."
— [LWN.net - Atomic Writes](https://lwn.net/Articles/789600/)

**implication:** external systems should write to `.tmp` then rename to `.json` for safe delivery.

### 4.2 maildir pattern

**citation [22]** [practitioner, O'Reilly]
> "Postfix uses a queue manager that maintains five different queues: inbound, active, deferred, hold, and corrupt, with a separate directory for each queue below the path specified in the queue_directory parameter."
— [O'Reilly - Postfix: The Definitive Guide](https://www.oreilly.com/library/view/postfix-the-definitive/0596002122/ch05.html)

**implication:** `.comms/$topic/` structure with `.errors/` follows the maildir pattern.

## section 5: poll vs push patterns

### 5.1 poll reliability

**citation [23]** [practitioner, 2026]
> "Poll is completely reliable with no event buffer overflow that causes dropped/failed events and avoids system resource limitations like inotify max user instances."
— [GitHub dotnet/runtime Issue #17111](https://github.com/dotnet/runtime/issues/17111)

### 5.2 poll latency

**citation [24]** [official, Oracle]
> "Messages have an average latency of 10 seconds (average latency is half of the poll interval, as long as messages are sent at random times)."
— [Oracle - SQL JMS Poll Interval](https://docs.oracle.com/cd/E23095_01/Platform.93/ATGProgGuide/html/s1106adjustingthesqljmspollinginterva01.html)

**implication:** poll once per minute = 30s average latency. acceptable for most usecases.

### 5.3 inotify limitations

**citation [25]** [official, Linux kernel]
> "inotify does not catch remote events that occur on network filesystems, so applications must fall back to poll the filesystem to catch such events."
— [inotify(7) Linux manual page](https://man7.org/linux/man-pages/man7/inotify.7.html)

**citation [26]** [practitioner, Linux Journal]
> "inotify cannot monitor remote (NFS) filesystems; it cannot report the userid involved in a trigger event; it does not work with /proc or other pseudo-filesystems; and the inotify queue can overflow which results in lost events."
— [Linux Journal - Linux Filesystem Events](https://www.linuxjournal.com/content/linux-filesystem-events-inotify)

**anti-pattern:** do not rely on inotify. poll is mandatory fallback.

## section 6: idempotency and deduplication

### 6.1 idempotency concepts

**citation [27]** [practitioner, 2024-2026]
> "An important distinction exists between two related concepts: Idempotency means the outcome is the same regardless of how many times the operation is applied. Deduplication means detect and discard duplicate requests so the operation executes exactly once."
— [Architecture Weekly - Deduplication in Distributed Systems](https://www.architecture-weekly.com/p/deduplication-in-distributed-systems)

**citation [28]** [official, Redis]
> "Perform the same operation multiple times has the same effect as perform it once. The practical path to achieve exactly-once semantics."
— [Redis Streams Idempotency](https://redis.io/docs/latest/develop/data-types/streams/idempotency/)

### 6.2 deduplication accuracy

**citation [29]** [academic, IJCET, 2024]
> "Systems that utilize UUID v4 with timestamp-based version achieved optimal results, process up to 75,000 messages per second while maintain a deduplication accuracy of 99.998%."
— [IJCET - Best Practices for Message Queue Services](https://iaeme.com/MasterAdmin/Journal_uploads/IJCET/VOLUME_16_ISSUE_1/IJCET_16_01_002.pdf)

**implication:** message.id + message.timestamp provides robust deduplication key.

## convergence signals

### strong convergence (3+ sources agree)

1. **atomic rename for safe writes** — sources [20], [21] agree on write-temp-then-rename pattern
2. **poll more reliable than watchers** — sources [23], [25], [26] agree inotify has limitations
3. **message id + timestamp for deduplication** — sources [9], [27], [29] agree on deduplication key pattern
4. **DLQ prevents queue blockage** — sources [16], [17], [18] agree on dead letter queue purpose

### conflict RESOLVED

**background task wake behavior:**
- citation [4] says background task output does NOT wake idle sessions
- prior research suggested output is delivered on "next turn"

**⚠️ RESOLVED — EMPIRICAL VALIDATION [2026-04-11]:**
we ran a sanity check: `sleep 600 && echo "did you get me?"` as a background task. after 10 minutes of complete idle (no user input, no activity), the task completed and **woke Claude Code CLI from idle state**. the message was delivered successfully.

**conclusion:** citation [4] appears to be either outdated, context-specific, or applies to Claude Desktop (not CLI). our empirical test proves background tasks DO wake idle Claude Code CLI sessions. this validates the poll-via-background-task mechanism.
