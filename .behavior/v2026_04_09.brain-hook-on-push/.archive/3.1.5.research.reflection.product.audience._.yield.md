# research.reflection.product.audience: brain-hook-on-push

## summary

audience analysis for brain-hook-on-push feature. identifies who will experience this change and why they care.

---

## section 1: who is the audience?

### primary: brain CLI clones (mechanics, drivers, foremen)

**who:** AI clones enrolled via rhachet, active in brain CLIs like claude code

**context:**
- work within a worktree or repo
- may be idle between tasks
- need to receive notifications from peers or external systems

**constraints:**
- cannot initiate contact with external systems while idle
- current hooks only fire on boot, tool use, or stop events
- no mechanism to wake from idle state

### secondary: orchestration systems (foremen, human operators)

**who:** systems that dispatch work to clones

**context:**
- need to notify specific clones about tasks
- filter notifications by role (mechanic, driver, etc.)
- coordinate multi-clone workflows

**constraints:**
- cannot currently wake idle clones
- must wait for clone to start new session
- no role-based message filter

### tertiary: external notification sources (slack, email, github)

**who:** external systems that generate events clones should respond to

**context:**
- produce notifications that clones should handle
- need to route to appropriate clone by role
- require standardized message format

**constraints:**
- no current path to deliver to brain CLI
- must write to filesystem (`.agent/.comms/`)

## section 2: why do they care?

### primary clones

| pain | goal | frustration |
|------|------|-------------|
| cannot receive notifications while idle | wake on demand | miss time-sensitive events |
| no peer-to-peer communication | coordinate with other clones | work in isolation |
| cannot be interrupted | respond to urgent requests | complete tasks that are no longer relevant |

### secondary orchestrators

| pain | goal | frustration |
|------|------|-------------|
| cannot wake idle clones | dispatch work on demand | wait for clone session start |
| no role-based filter | target specific clone types | broadcast to all or none |
| no delivery confirmation | know message was received | fire and forget |

### tertiary external systems

| pain | goal | frustration |
|------|------|-------------|
| no standard delivery path | integrate with brain CLIs | custom per-integration |
| no message format standard | reliable delivery | format drift |

## section 4: what do they care about?

### primary clones

**success criteria:**
- messages arrive reliably
- role filter matches their enrollment
- hook fires promptly after message delivery
- no infinite loops from onStop behavior

**tradeoffs accepted:**
- poll latency (~30s average) vs instant delivery
- at-most-once delivery (v1) vs at-least-once

**tradeoffs rejected:**
- miss messages due to race conditions
- process messages meant for other roles

### secondary orchestrators

**success criteria:**
- can wake specific clones by role
- can target topics (slack, radio, github)
- simple file drop delivery

**tradeoffs accepted:**
- write to filesystem vs API call
- poll-based vs push notification

**tradeoffs rejected:**
- complex delivery protocol
- required acknowledgment in v1

### tertiary external systems

**success criteria:**
- standard message format
- clear file location

**tradeoffs accepted:**
- filesystem delivery vs webhook
- JSON format vs other formats

## section 5: how will they experience it?

### primary clone journey

1. **enrollment**: clone boots with onPush hook declared
2. **registration**: hook synced to brain CLI settings
3. **poll**: background poll task checks `.comms/` directory
4. **match**: messages filtered by role
5. **invoke**: hook command executes with message payload
6. **wake**: if idle, onStop exit(2) forces continuation
7. **process**: clone handles the message

### friction points

| point | friction | mitigation |
|-------|----------|------------|
| poll latency | ~30s average delay | acceptable for most usecases |
| infinite loop | onStop could loop | stop_hook_active guard |
| message format | must match schema | validation with clear errors |
| role filter | must match clone roles | documented syntax |

### smooth experience

- drop file into `.comms/$topic/`
- clone wakes within 60s
- hook fires with message payload
- clone processes and responds

## audience summary

| audience | role | primary need |
|----------|------|--------------|
| brain CLI clones | consumer | receive notifications, wake on demand |
| orchestrators | producer | dispatch work to specific clones |
| external systems | producer | integrate with standard format |

**design priority:** optimize for primary audience (clone experience), ensure secondary audience (orchestrators) have clean API, accept complexity for tertiary (external systems).
