# research.external.product.access: brain-hook-on-push

## summary

for brain-hook-on-push, "external access" is minimal:
- **local filesystem**: `.agent/.comms/` directory (no remote APIs)
- **claude code CLI internals**: background tasks, hooks, lifecycle events

the primary research need is understanding Claude Code's behavior around background tasks and hooks to implement the poll-based wake mechanism.

---

## lesson 1: background task output delivery

### findings

background task output is delivered to Claude on the "next turn" — meaning when Claude next has an opportunity to process input.

**citation [1]** [official, 2026]
> "When a background process exits, if the hook produced a JSON response with a systemMessage or additionalContext field, that content is delivered to Claude as context on the next conversation turn."
— Claude Code hooks reference documentation

**citation [2]** [official, 2026]
> "Hook output is delivered on the next conversation turn. If the session is idle, the response waits until the next user interaction."
— Claude Code documentation

**convergence:** sources [1] and [2] agree that output delivery is turn-based.

**potential conflict with wisher input:** wisher confirmed "background task output does occur" when Claude is idle. need to validate whether output truly wakes Claude or waits for user input.

## lesson 2: stop hooks can force continuation

### findings

stop hooks fire when Claude finishes a response. they can block Claude from stopping and force it to continue.

**citation [3]** [official, 2026]
> "The `Stop` event fires whenever Claude finishes responding, not only at task completion. They do not fire on user interrupts."
— Claude Code hooks guide

**citation [4]** [official, 2026]
> "With `"decision": "block"` prevents Claude from stopping. The reason field must be provided for Claude to know how to proceed and continue its work. This provides a powerful mechanism for ensuring complex tasks are fully completed."
— Claude Code hooks reference documentation

**citation [5]** [official, 2026]
> "Exit 0: Success, parses JSON output; Exit 2: Blocking error, prevents action; Other codes: Non-blocking errors"
— Claude Code hooks reference documentation

**convergence:** sources [3], [4], [5] confirm that Stop hooks with exit(2) or decision: block force Claude to continue.

### implication for onPush

the onStop + exit(2) mechanism in our vision is validated. when poll task detects notification:
1. onStop hook fires (Claude about to stop)
2. hook checks heartbeat, sees notification pending
3. hook returns exit(2) or decision: block
4. Claude is forced to continue and process the notification

## lesson 3: async hooks run without block

### findings

hooks can run asynchronously without block Claude's execution.

**citation [6]** [official, January 2026]
> "Add async: true to run hooks in the background without blocking Claude's execution."
— Claude Code hooks guide

**citation [7]** [official, 2026]
> "Async hooks cannot block or control Claude's behavior: response fields like decision, permissionDecision, and continue have no effect, because the action they would have controlled has already completed."
— Claude Code hooks reference documentation

**citation [8]** [official, 2026]
> "Only type: 'command' supports async. Prompt and agent hooks cannot run asynchronously."
— Claude Code hooks reference documentation

**anti-pattern:** async hooks cannot be used for block/control. our poll task lifecycle needs synchronous hooks (onStop) for the exit(2) mechanism.

## lesson 4: hook lifecycle events

### findings

Claude Code has comprehensive lifecycle events including Stop and PostToolUse.

**citation [9]** [official, 2026]
> "As of March 2026, Claude Code Hooks has grown into a full-fledged automation platform with 21 lifecycle events, 4 handler types, async execution, and JSON structured output."
— SmartScope Claude Code Hooks guide

**citation [10]** [official, 2026]
> "Hook Event Stages: Session level: SessionStart, SessionEnd; Per-turn level: UserPromptSubmit, Stop, StopFailure; Agentic loop level: PreToolUse, PostToolUse, PermissionRequest, PermissionDenied"
— Claude Code hooks reference documentation

**citation [11]** [official, 2026]
> "PostToolUse hooks fire after a tool has already executed successfully. The input includes both tool_input, the arguments sent to the tool, and tool_response, the result it returned."
— Claude Code hooks reference documentation

### implication for onPush

we can use:
- **SessionStart**: spawn poll background task
- **PreToolUse/PostToolUse**: track poll task state (started, heartbeat)
- **Stop**: check if poll task active, force restart if stale

## lesson 5: hook input data model

### findings

all hooks receive JSON input with session context.

**citation [12]** [official, 2026]
> "All hooks receive JSON input with: session_id (current session identifier), transcript_path (path to conversation JSON), cwd (current working directory), permission_mode (current permission mode), and hook_event_name (name of the firing event)."
— Claude Code hooks reference documentation

**citation [13]** [official, 2026]
> "Your Stop hook command needs to check whether it already triggered a continuation. Parse the stop_hook_active field from the JSON input and exit early if it's true"
— Claude Code hooks guide documentation

**anti-pattern from [13]:** Stop hooks must guard against infinite loops. our onStop hook needs to check if it already forced restart.

---

## lesson 6: background task lifecycle

### findings

background tasks run independently with stdout/stderr captured.

**citation [14]** [official, 2026]
> "Background tasks operate in three phases: Launch (tool call starts process in daemon thread, AI receives confirmation instantly), Monitor (background process runs independently with stdout/stderr captured), and Notify (when process completes, result is placed in notification queue and AI receives notification in next turn's context)."
— Claude Code hooks reference documentation

**citation [15]** [official, 2026]
> "Background tasks keep long-running processes like dev servers active without blocking Claude Code's progress on other work."
— Anthropic Claude Code docs

**citation [16]** [official, v2.0.60]
> "Background tasks were introduced in v2.0.60 and enable true multitasking—fire off an agent, let it work while you do other things, then it hooks back in with results when complete."
— Anthropic product announcements

**convergence:** sources [14], [15], [16] confirm background tasks are designed for long-lived processes.
