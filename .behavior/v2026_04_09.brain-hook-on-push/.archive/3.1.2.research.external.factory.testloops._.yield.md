# research.external.factory.testloops: brain-hook-on-push

## summary

research on test loop infrastructure for brain-hook-on-push implementation. focus on rapid verification feedback for CLI + package project type, including filesystem-based message testing, stdin/stdout simulation, and extant test patterns in rhachet.

## section 4: CLI output testing

### 4.1 execa for subprocess execution

**assessment:** recommended — already in recommended OSS stack

**citation [10]** [official, npm]
> "execa: Process execution for humans. Returns promise with stdout, stderr, and exitCode."
— [npm execa](https://www.npmjs.com/package/execa)

**citation [11]** [practitioner, GitHub]
> "execa captures stdout and stderr as strings by default. Use buffer: false for streaming or inputFile for stdin."
— [execa README](https://github.com/sindresorhus/execa)

**usage for hook command testing:**
```typescript
import { execa } from 'execa';

const result = await execa('poll.sh', {
  input: JSON.stringify(message),
  timeout: 30_000,
  reject: false,
});

expect(result.exitCode).toBe(0);
expect(result.stdout).toContain('processed');
```

### 4.2 native child_process.spawn

**assessment:** acceptable — zero dependencies

**citation [12]** [official, Node.js]
> "child_process.spawn: Launches a new process with a given command. Returns ChildProcess with stdin, stdout, stderr streams."
— [Node.js child_process](https://nodejs.org/api/child_process.html)

**usage:**
```typescript
import { spawn } from 'node:child_process';

const child = spawn('poll.sh', { stdio: 'pipe' });
child.stdin.write(JSON.stringify(message));
child.stdin.end();

let stdout = '';
child.stdout.on('data', (chunk) => { stdout += chunk; });
await new Promise(r => child.on('close', r));
```

```typescript
import { mkdtemp, writeFile, readdir } from 'node:fs/promises';
import { join } from 'node:path';

// create temp comms directory

  join(commsDir, 'slack', 'msg-001.json'),
  JSON.stringify(testMessage),
);

// verify poll reads it

expect(files).toContain('msg-001.json');
```

## section 9: integration test patterns

### 9.1 real filesystem integration

**citation [21]** [practitioner, ehmpathy]
> "Integration tests must cross remote boundaries. Unit tests must not. The distinction is what makes unit tests fast and deterministic."
— rule.forbid.remote-boundaries.md

**onPush integration test pattern:**
```typescript
describe('pollCommsDirectory', () => {
  given('[case1] message file extant', () => {
    const tempDir = useBeforeAll(async () => {
      const dir = await mkdtemp(join(tmpdir(), 'comms-'));
      await mkdir(join(dir, 'slack'), { recursive: true });
      await writeFile(
        join(dir, 'slack', 'msg-001.json'),
        JSON.stringify(testMessage),
      );
      return dir;
    });

    when('[t0] poll is invoked', () => {
      const result = useThen('poll succeeds', async () =>
        pollCommsDirectory({ commsDir: tempDir, topics: ['slack'] }),
      );

      then('message is found', () => {
        expect(result.messages).toHaveLength(1);
      });

      then('message is deleted after processing', async () => {
        const files = await readdir(join(tempDir, 'slack'));
        expect(files).not.toContain('msg-001.json');
      });
    });
  });
});
```

## section 12: onPush specific test scenarios

### 12.1 message format validation

**test cases:**

| scenario | expected |
|----------|----------|
| valid message | parsed successfully |
| absent id | BadRequestError |
| absent topic | BadRequestError |
| invalid roles syntax | BadRequestError |
| malformed JSON | move to .errors/ |

### 12.2 role filter matching

**test cases:**

| filter | clone roles | matches |
|--------|-------------|---------|
| `+mechanic` | `['mechanic', 'driver']` | yes |
| `+mechanic` | `['driver']` | no |
| `-foreman` | `['mechanic']` | yes |
| `-foreman` | `['mechanic', 'foreman']` | no |
| `mechanic` | `['mechanic']` | yes |
| `mechanic` | `['mechanic', 'driver']` | no |
| `+mechanic,+driver` | `['mechanic', 'driver']` | yes |
| `+mechanic,-foreman` | `['mechanic']` | yes |

### 12.3 poll lifecycle

**test cases:**

| scenario | expected |
|----------|----------|
| empty directory | no messages processed |
| single message | message processed, file deleted |
| multiple messages | all processed in order |
| partial file (write in progress) | skip, retry next poll |
| hook command fails | message moved to .errors/ |
| hook command timeout | message moved to .errors/ |

### 12.4 hook exit code semantics

**citation [27]** [official, Anthropic]
> "With decision: 'block' prevents Claude from stop. The reason field must be provided for Claude to know how to proceed and continue its work."
— Claude Code Hooks Reference

**test cases:**

| exit code | expected behavior |
|-----------|------------------|
| 0 | success, continue normally |
| 2 | block, force re-engage |
| other | error logged, continue |
