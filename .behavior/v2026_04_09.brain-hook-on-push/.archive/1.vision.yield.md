# vision: brain-hook-on-push

## v1 scope

**onPush only responds to NEW messages after the poll starts.**

| in scope (v1) | out of scope (v1) |
|---------------|-------------------|
| wake clone from idle when new message arrives | process messages that existed before poll started |
| filter by topic and role | state management across sessions |
| broadcast to multiple consumers | per-consumer acknowledgment |
| detect files added after poll started | persist/restore which messages were seen |

no state management, no consumer considerations — just filters and "what's new?".

subsequent PR will cover: notify while active (clone not idle).

## the outcome world

### what does a day-in-the-life look like with this in place?

**before:**
vlad starts a mechanic clone to work on a feature. the clone works autonomously. meanwhile, vlad receives a slack message: "hey, that PR needs changes - reviewers left comments." vlad must manually interrupt the clone, paste context, and redirect its focus. if the clone is deep in a task, vlad waits. if the clone finished and exited, vlad starts a new session and re-explains all context.

**after:**
vlad starts a mechanic clone to work on a feature. the clone works autonomously. meanwhile, slack notifications flow into `.agent/.comms/slack/`. a driver clone that monitors PR reviews receives the push: "PR #247 has new comments." the driver wakes up, fetches the comments, and either handles them directly or broadcasts to the mechanic: "pause current work — reviewer feedback on your PR." the mechanic receives the push, acknowledges, and pivots.

no human intervention. no context loss. peer-to-peer coordination.

### what's the before/after contrast?

| before | after |
|--------|-------|
| clones are deaf — they only respond to human prompts | clones listen — external events trigger responses |
| humans are the message bus | `.agent/.comms/` is the message bus |
| one clone per task, no coordination | multiple clones coordinate via push |
| clone exits = context lost | clone on standby = ready to wake |
| slack notification → human reads → human types | slack notification → `.comms` → clone acts |

### what's the "aha" moment where the value clicks?

the aha moment is when you realize **you can leave**.

start a clone on standby: "wait for PR review feedback, then address it." walk away. the clone receives the push when reviewers comment, wakes up, addresses the feedback, and pushes the fix. you come back to a green CI.

or: start a foreman clone that orchestrates three driver clones. each driver works independently. when driver A finishes, it pushes to the foreman: "task complete." the foreman dispatches the next task. no poll loop. no human dispatch. event-driven orchestration.

## user experience

### what usecases do folks fulfill?

1. **slack bridge** — slack messages to a channel flow into `.agent/.comms/slack/`. clones with `+mechanic` role filter receive relevant messages.

2. **PR review notifications** — github webhook → `.agent/.comms/github/`. driver clone with review-watcher role receives "review submitted" events, wakes, processes.

3. **peer clone coordination** — mechanic A finishes a subtask, drops a message into `.agent/.comms/radio/`. mechanic B (different workzone) receives, picks up the baton.

4. **subconscious intervention** — a "supervisor" clone monitors for anomalies. when detected, pushes to active clones: "stop — potential issue." clones receive, pause, await guidance.

5. **remote approval** — clone needs human sign-off. pushes question to `.agent/.comms/phone/`. human's phone buzzes. human approves. approval flows back.

### what contract inputs & outputs do they leverage?

**declare push handlers (in role hooks):**

```typescript
export const hooks: RoleHooksOnBrain = {
  onPush: [
    {
      filter: { topic: 'slack', roles: '+mechanic' },  // slack messages for mechanics
      command: 'npx rhachet run --skill slack.notification.get --when hook.onPush',
      timeout: 'PT30S',
    },
    {
      filter: { topic: 'radio', roles: '+driver,-foreman' },  // radio for drivers, not foremen
      command: 'npx rhachet run --skill radio.broadcast.get --when hook.onPush',
      timeout: 'PT10S',
    },
  ],
};
```

note: `filter` matches against message's `for: { topic, roles }`. both topic and roles can be filtered.

**push message shape (in `.agent/.comms/$topic/$id.json`):**

```json
{
  "id": "msg_abc123",
  "for": {
    "topic": "slack",
    "roles": "+mechanic"
  },
  "timestamp": "2026-04-09T14:32:00Z",
  "payload": {
    "channel": "#eng-alerts",
    "user": "alice",
    "text": "PR #247 needs changes"
  }
}
```

note: messages declare recipients via `for: { topic, roles }`. the hook's `filter: { roles }` further narrows delivery. both work together — sender addresses, subscriber filters.

**receive a push (via hook command):**

the poll adapter detects a new message in `.comms/slack/`, checks the filter, and invokes the hook command. the command receives the payload via stdin or file arg:

```bash
# hook command invoked with message path
npx rhachet run --skill slack.notification.get --when hook.onPush --message .agent/.comms/slack/msg_abc123.json
```

note: the poll adapter only invokes the hook if the session's filter matched the message's `for` field. filter happens before delivery.

### what would it look like to leverage them?

**step 1: configure role with onPush hook**

```typescript
// in rhachet-roles-ehmpathy/src/roles/mechanic/index.ts
export const mechanicRole = new Role({
  slug: 'mechanic',
  hooks: {
    onBrain: {
      onPush: [
        {
          filter: { topic: 'slack', roles: '+mechanic' },
          command: 'npx rhachet run --skill slack.notification.get --when hook.onPush',
        },
      ],
    },
  },
});
```

**step 2: sync hooks**

```bash
rhx init --hooks
# writes onPush hooks to brain config via BrainHooks DAO
# translates to native brain hooks (PreToolUse, Stop, etc)
```

**BrainHooks DAO writes onPush to brain config:**

each onPush hook is stored in the brain's native config (e.g., `.claude/settings.json`). the poll task reads declarations back FROM the config via BrainHooks DAO. zero `.agent/.hooks/` directory needed.

**translated hooks:**

```bash
# Stop hook — forces poll to start if not active
rhx hook brain.onPush --when hook.onStop
# checks heartbeat, exit(2) if poll not active → forces re-engage

# PreToolUse hook — validates poll was started in background
rhx hook brain.onPush --when hook.onTool
# exit(2) if poll not started in background

# background task — does the actual poll
rhx hook brain.onPush --when bash.onBackground &
# reads onPush hooks via BrainHooks DAO, polls, invokes commands
```

**`--when hook.onStop` behavior:**
1. checks heartbeat (is poll active?)
2. if active: exit 0 (all good, stay idle)
3. if not active: exit 2 (forces clone to re-engage → runs `--start poll`)

**`--when hook.onTool` behavior:**
1. checks if `--when bash.onBackground` was started in background
2. if yes: exit 0
3. if no (foreground): exit 2 (must run poll in background)

**`--when bash.onBackground` behavior:**
1. starts heartbeat (so `--when hook.onStop` knows poll is active)
2. reads clone's enrolled roles from session context
3. reads ALL onPush hooks via BrainHooks DAO
4. records start time T0
5. poll loop detects NEW files (after T0) in ALL `.agent/.comms/$topic/` directories
6. for each new file:
   - looks up hook for that topic
   - parses message's `for.roles` filter
   - checks if clone's roles satisfy BOTH filters (hook + message)
7. if match, invokes command from hook with `--message $path` appended
8. outputs to stdout (wakes clone from idle)
9. on exit: clears heartbeat

**key insight:** ONE poller command, reads ALL onPush hooks via BrainHooks DAO. clone's enrolled roles read from session context. per-enrollment configs are respected. zero `.agent/.hooks/` directory needed.

**match logic:** for each new file, poll parses message's `for.roles` (e.g., `+mechanic`) and checks if clone's roles satisfy the filter. if yes, invokes the command from the hook config.

**step 3: clone session starts**

1. claude starts session
2. onToolUse fires → starts `getPushNotification` background task
3. background task reads hook config, polls `.comms/slack/`, waits for messages

**step 4: push arrives**

external system (slack bot, github webhook, peer clone) writes to `.agent/.comms/slack/msg_123.json`. the background task detects the file, validates role filter, reads command from hook config, and invokes it.

**step 5: clone responds**

the `slack.notification.get` skill runs:
```bash
#!/usr/bin/env bash
# read the push payload from file arg
# parse, decide action, respond
echo "ack slack push..."
```

**step 6: clone goes idle**

1. claude finishes work, stops (waits at prompt)
2. onStop fires → checks if background task active
3. if not active → exit(2) → claude re-engages → onToolUse restarts task

### what timelines do they go through?

1. **setup (once)**: declare onPush in role hooks → `rhx init --hooks` → hooks translated
2. **session start**: onToolUse fires → background poll task starts → records "start time"
3. **runtime (continuous)**: external events → `.comms/` → background task detects NEW files (after start time) → invokes hook
4. **response (per-push)**: hook command runs → processes payload → clone acts
5. **idle check**: onStop verifies task active → exit(2) if not → forces restart

**scope (v1):** onPush only responds to messages that arrive AFTER the poll task starts. no state management, no consumer considerations — just filters and "what's new?".

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| push to brain cli | ✅ | via background task poll on `.comms/` |
| filter by role | ✅ | `+mechanic`, `-driver` syntax |
| support slack/email/peer | ✅ | each is a topic |
| enable standby clone | ✅ | clone waits for push, wakes |
| minimal brain contract change | ✅ | onPush is new event, same hook shape |

### what are the pros? the cons?

**approach: background task poll on `.comms/`**

| pros | cons |
|------|------|
| works with any brain REPL | poll latency (not instant, up to poll interval) |
| no dependency on native channels | must ensure poll task stays active |
| fully controlled by us | |
| simple: just watch a directory | |
| low power: poll once a minute is near-zero cost | |

**how it wakes claude:**

the translation produces hooks that enforce poll lifecycle:
1. **Stop hook** — runs `rhx hook brain.onPush --when hook.onStop`
   - checks heartbeat (is poll active?)
   - if not active, returns exit(2) to force clone to start poll
2. **PreToolUse hook** — runs `rhx hook brain.onPush --when hook.onTool`
   - validates poll was started in background mode
   - exit(2) if attempted in foreground
3. **background task** — started via `rhx hook brain.onPush --when bash.onBackground &`

the **poll background task** (`--when bash.onBackground`):
1. starts heartbeat (so Stop hook knows poll is active)
2. reads ALL onPush hooks via BrainHooks DAO
3. polls `.comms/$topic/` directories for each registered topic
4. matches role filter against message's `for.roles` via clone's enrolled roles
5. invokes hook command with message path
6. outputs to stdout (wakes clone from idle)
7. on exit: clears heartbeat

**heartbeat lifecycle:**
- `--when bash.onBackground` starts heartbeat on entry
- `--when bash.onBackground` clears heartbeat on exit
- `--when hook.onStop` checks heartbeat to decide exit(0) vs exit(2)
- if no heartbeat for N minutes, assume inactive

**why not native channels?** poll on `.comms/` is universal, simple, and works with any brain REPL.

**decision:** poll-only. background task watches `.comms/`, outputs when notification found, claude wakes and acts.

### what edgecases exist and how do our contracts keep users in a pit of success?

| edgecase | pit of success |
|----------|----------------|
| push arrives while clone idle | background task detects new file, wakes clone |
| push for role clone doesn't have | role filter prevents delivery |
| malformed payload | schema validation in skill, moved to `.comms/.errors/` |
| too many pushes | rate limit in poll adapter (v1.1) |
| multiple clones same role | each receives (broadcast semantics) |
| push arrives before poll started | out of scope — onPush only responds to NEW messages after poll starts |
| push arrives while clone not active | out of scope — v1 covers wake-from-idle only, not session restore |

## open questions & assumptions

### assumptions we've made

1. **file-based `.comms/` is sufficient** — assumes local filesystem, not distributed
2. **role filter syntax is expressive enough** — `+role,-role` covers most cases
3. **one topic = one handler** — may need multiple handlers per topic later
4. **poll latency is acceptable** — sub-second poll interval keeps latency low enough

### questions answered (from prior research)

1. **[answered] what's the message delivery guarantee?** — fire-and-forget at poll level. v1 is "what's new?" — poll detects files added after it starts. no state management, no consumer acknowledgment.

2. **[answered] how do we handle offline/disconnected sessions?** — out of scope for v1. onPush only responds to NEW messages after poll starts. pre-poll messages are ignored. subsequent PR may add session restore.

3. **[n/a] native channels research** — decided against native channels. poll-only approach is simpler — no MCP dependency.

### wisher decisions (resolved)

1. **[resolved] channels vs poll** — poll-only. universal approach that works with any brain REPL. simple: watch `.comms/`, wake brain when filter-matched message drops.

2. **[resolved] role filter semantics** — `+role,-role` is sufficient for v1. no need for AND/OR expressions.

3. **[resolved] topic scope** — per-workzone (implicit). topics live in `.agent/.comms/$topic/` which is per-worktree. no global namespace concern.

4. **[resolved] message persistence** — gitignored (ephemeral). findsert `.comms/` into `.gitignore`. messages are transient. v1 does NOT track which messages were seen — poll just detects "what's new?" (files after start).

5. **[resolved] two-way communication** — one-way only for v1 (`onPush`). roles can build reply tools/skills on top of this primitive.

6. **[resolved] filter design** — hybrid. message has `for: { topic, roles }` (sender addresses). hook has `filter: { topic, roles }` (subscriber filters on `for` attributes). both work together.

### what must we research externally?

1. **[answered] how to wake a stopped Claude session**

   the scenario:
   1. user runs `claude`
   2. has a conversation
   3. claude finishes response and **stops** — waiting at the prompt for user input
   4. session is live but claude is idle
   5. notification drops into `.comms/`
   6. **how do we wake claude to process it?**

   **answer:** background task output wakes claude. when a background task writes to stdout, claude sees it and responds. so:
   - background poll task watches `.comms/`
   - finds notification, outputs it
   - claude wakes, processes notification, invokes hook command

2. **[answered] poll task lifecycle**

   **constraint:** can't programmatically start a background task from outside Claude. must make Claude itself start it via hooks.

   **solution: Stop check + PreToolUse validate**

   translate ALL onPush hooks to poll lifecycle hooks:

   ```bash
   # Stop hook — checks heartbeat, forces poll start if not active
   rhx hook brain.onPush --when hook.onStop
   # returns exit(2) if heartbeat absent → forces clone to start poll

   # PreToolUse hook — validates poll was started in background
   rhx hook brain.onPush --when hook.onTool
   # exit(2) if poll not started in background

   # background poll — started via --start poll
   rhx hook brain.onPush --when bash.onBackground &
   # reads ALL onPush hooks via BrainHooks DAO, runs poll loop, invokes commands
   ```

   **state machine:**
   - Stop: if heartbeat absent → exit(2) → forces `--start poll`
   - `--start poll`: starts `--when bash.onBackground &`
   - `--when bash.onBackground`: starts heartbeat → poll loop → invoke hooks → on exit → clears heartbeat
   - PreToolUse: validates poll was started in background

   **heartbeat:** poll task writes heartbeat on start, clears on exit. if no heartbeat for N minutes, assume inactive.

note: distributed comms (multi-machine `.comms/`) is out of scope for v1. poll watches local filesystem only.

## what is awkward?

### what feels off or forced?

1. **file-based message bus** — `.agent/.comms/` as a message bus is simple but crude. no order guarantees, no acknowledgment. but: v1 scope is just "what's new?" — poll detects files after start. simple is good.

### where does the design fight the user's mental model?

1. **"push" but it's poll** — poll is to sums as push is to integrals — behaviorally the same at different scales. poll once a minute is acceptable latency for most usecases, low power. true push may come later if needed.

2. **topics vs hooks** — two concepts that overlap. "topic" is the message category, "hook" is the handler. but: matches how onStop/onSessionStart work.

### what tradeoffs feel uncomfortable?

1. **local vs distributed** — `.comms/` works on one machine. multi-machine clones need external storage (s3? redis?). but: out of scope for v1.
