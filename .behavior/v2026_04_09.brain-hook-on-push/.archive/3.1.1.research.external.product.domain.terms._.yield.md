## section 3: role filter syntax research

### 3.1 established filter syntax patterns

| pattern | source | example |
|---------|--------|---------|
| **+/- prefix** | Git refs, Google search | `+include -exclude` |
| **selector expressions** | CSS, jQuery, AMQP | `role.mechanic` |
| **glob patterns** | Unix, Redis | `sports:*` |
| **attribute conditions** | AWS SNS | `{"role": ["mechanic"]}` |

**citation [14]** [official, Git]
> "refspecs use +/- prefix to indicate force push or exclude patterns"
— [Git Refspec Documentation](https://git-scm.com/book/en/v2/Git-Internals-The-Refspec)

**citation [15]** [official, Redis]
> "PSUBSCRIBE allows subscribe to channels that match glob-style patterns such as 'news.*'"
— [Redis Pub/Sub](https://redis.io/docs/latest/develop/pubsub/)

### 3.2 our role filter syntax

proposed syntax aligns with established patterns:

| pattern | semantic | example |
|---------|----------|---------|
| `+role` | has role | `+mechanic` (has mechanic) |
| `-role` | lacks role | `-foreman` (lacks foreman) |
| `role` | exactly role | `mechanic` (only mechanic) |
| `,` | and | `+mechanic,+driver` |

**rationale:**
- `+/-` prefix is well-established (Git, Google)
- comma separation is universal
- minimal syntax, maximum clarity

## section 4: term options for domain objects

### 4.1 message entity

| option | etymology | pros | cons |
|--------|-----------|------|------|
| **BrainHookPushMessage** | push + message | aligns with industry (APNs, FCM) | none |
| **Hail** | Old Norse *heill* | evocative, short | less recognized |
| **Signal** | Latin *signum* | military feel | overloaded term |
| **Dispatch** | Latin *dispactum* | industrial feel | verbose |

**recommendation:** **BrainHookPushMessage** — maximizes recognition, minimizes ambiguity.

### 4.2 message route

| option | etymology | pros | cons |
|--------|-----------|------|------|
| **topic** | Greek *topos* — place | industry standard (Kafka, SNS) | none |
| **channel** | Latin *canalis* | common (Redis, Slack) | overloaded |
| **queue** | Latin *cauda* — tail | precise (SQS) | implies FIFO |
| **berth** | nautical | evocative | unfamiliar |

**recommendation:** **topic** — maximizes recognition, aligns with Kafka, SNS, Pub/Sub.

### 4.3 subscription entity

| option | etymology | pros | cons |
|--------|-----------|------|------|
| **onPush hook** | on + push | aligns with onStop, onToolUse | already our term |
| **subscription** | Latin *subscribere* | industry standard | verbose |
| **listener** | listen + -er | simple | overloaded in JS |
| **watcher** | watch + -er | simple | overloaded in file systems |

**recommendation:** **onPush hook** — consistent with our hook pattern (onStop, onToolUse).

### 4.4 role filter

| option | etymology | pros | cons |
|--------|-----------|------|------|
| **filter** | filter | industry standard | generic |
| **selector** | Latin *seligere* | precise | AMQP-specific feel |
| **matcher** | match + -er | action-oriented | verbose |
| **predicate** | Latin *praedicatum* | formal | academic |

**recommendation:** **filter** — simple, universal, aligns with AWS SNS filter policy.

### 4.5 poll task

| option | etymology | pros | cons |
|--------|-----------|------|------|
| **poll task** | poll + task | descriptive | verbose |
| **watcher** | watch + -er | simple | overloaded |
| **daemon** | Greek *daimon* — spirit | Unix tradition | less accessible |
| **sentinel** | Latin *sentire* — to watch | evocative | unfamiliar |

**recommendation:** **poll task** — descriptive, unambiguous.

### 4.6 wake action

| option | etymology | pros | cons |
|--------|-----------|------|------|
| **wake** | PIE *weg-* — be lively | nautical, evocative | none |
| **activate** | Latin *activus* | formal | verbose |
| **resume** | Latin *resumere* | accurate | generic |
| **hail** | Old Norse *heill* | maritime | less familiar |

**recommendation:** **wake** — short, evocative, shares root with "vigil".

## section 5: term recommendations

### final recommendations

| concept | recommended term | rationale |
|---------|-----------------|-----------|
| message entity | **BrainHookPushMessage** | industry standard |
| message route | **topic** | Kafka, SNS alignment |
| subscription | **onPush hook** | pattern consistency |
| role filter | **filter** | universal |
| poll process | **poll task** | descriptive |
| activate clone | **wake** | evocative etymology |
| message store | **.comms/** | short, clear |
| error store | **.comms/.errors/** | standard DLQ pattern |

### etymology alignment

| term | root | resonance |
|------|------|-----------|
| **wake** | PIE *weg-* (be lively) | return from idle to active |
| **topic** | Greek *topos* (place) | destination for messages |
| **filter** | Germanic *filtrum* (felt) | let through only what matches |
| **poll** | Middle English *pol* (head) | count, sample repeatedly |

## anti-patterns identified

| term to avoid | why avoid | use instead |
|---------------|-----------|-------------|
| **listener** | overloaded in JS (event listeners) | onPush hook |
| **watcher** | overloaded (file system watchers) | poll task |
| **callback** | overloaded in JS | hook |
| **handler** | generic | hook |
| **subscriber** | verbose | hook with filter |

## convergence signals

### strong convergence (3+ sources agree)

1. **push + notification** — universal across Apple, Google, AWS, Firebase
2. **topic** — Kafka, SNS, Pub/Sub all use this term
3. **filter** — AWS SNS, EventBridge, RabbitMQ all use this concept
4. **poll** — universal concept for query-based updates

1. **wake/vigil** — both from PIE *weg-* (be lively)
2. **hail** — Old Norse *heill* (be whole) — a call that demands acknowledgment
3. **manifest** — Latin *manifestus* (clear) — naval: a list of cargo
