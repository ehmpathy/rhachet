# research.external.product.domain: brain-hook-on-push

## summary

domain terminology research for brain-hook-on-push feature. maps established message queue and pub/sub domain concepts to our onPush hook design.

---

## section 1: message domain objects

### message.entity

the core data structure that moves through the system.

**established terminology:**

| term | definition | source |
|------|------------|--------|
| **Message** | data that moves through a message service, contains payload and metadata | AWS SQS [official] |
| **MessageId** | application-defined value that uniquely identifies the message | Azure Service Bus [official] |
| **Body / Payload** | the message contents, handled as opaque binary block | Azure Service Bus [official] |
| **MessageAttributes** | optional metadata separate from body (name, type, value) | AWS SQS [official] |
| **CorrelationId** | context for the message for correlation (e.g., reply-to context) | Azure Service Bus [official] |
| **TimeToLive (TTL)** | relative duration after which message expires | Azure Service Bus [official] |

**citation [1]** [official, AWS]
> "The data that moves through the message service. Contains a payload and metadata (key-value pairs that describe the payload and give handle instructions)."
— [AWS SQS API Reference](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/APIReference/API_Message.html)

**citation [2]** [official, Azure]
> "An application-defined value that uniquely identifies the message and its payload. A free-form string that can reflect a GUID or an identifier derived from the application context."
— [Azure Service Bus Messages](https://learn.microsoft.com/en-us/azure/service-bus-messaging/service-bus-messages-payloads)

**onPush map:**
- `message.id` = MessageId
- `message.payload` = Body/Payload
- `message.for` = MessageAttributes (topic, roles)
- `message.timestamp` = standard attribute

### message.route (topic)

the named destination for messages.

**established terminology:**

| term | definition | source |
|------|------------|--------|
| **Topic** | named entity that represents a feed of messages, multi-producer multi-subscriber | Apache Kafka [official] |
| **Channel** | named conduit through which messages are published and subscribed | Redis Pub/Sub [official] |
| **Queue** | FIFO storage for messages, single-consumer or compete pattern | RabbitMQ [official] |
| **Exchange** | route entity that receives messages and distributes to queues | RabbitMQ AMQP [official] |

**citation [3]** [official, Kafka]
> "A named entity that represents a feed of messages. Topics are always multi-producer and multi-subscriber."
— [Apache Kafka Introduction](https://kafka.apache.org/intro/)

**citation [4]** [official, Redis]
> "The named conduit through which messages are published and to which clients subscribe."
— [Redis Pub/Sub](https://redis.io/docs/latest/develop/pubsub/)

**onPush map:**
- `message.for.topic` = Topic (e.g., 'slack', 'github', 'radio')
- `.agent/.comms/$topic/` = Topic directory

**established terminology:**

| term | definition | source |
|------|------------|--------|
| **Filter Policy** | JSON object with properties that define which messages subscriber receives | AWS SNS [official] |
| **Event Pattern** | filters to match events based on metadata or property values | AWS EventBridge [official] |
| **Route Key** | message attribute used by exchanges to determine destinations | RabbitMQ [official] |
| **Pattern Subscription** | subscribe to channels that match glob-style patterns | Redis Pub/Sub [official] |

**citation [5]** [official, AWS SNS]
> "A JSON object that contains properties to define which messages the subscriber receives. Can act on message attributes or message body."
— [AWS SNS Subscription Filter Policies](https://docs.aws.amazon.com/sns/latest/dg/sns-subscription-filter-policies.html)

**citation [6]** [official, Redis]

**onPush map:**
- `hook.filter.topic` = topic filter
- `hook.filter.roles` = attribute filter ('+mechanic,-foreman' pattern)
- filter matches against `message.for`

## section 2: subscription domain objects

### subscription.entity

the named interest in received messages.

**established terminology:**

| term | definition | source |
|------|------------|--------|
| **Subscription** | named entity that represents interest in received messages on a topic | Google Cloud Pub/Sub [official] |
| **Consumer** | application that receives and processes messages from queues | RabbitMQ [official] |
| **Consumer Group** | set of consumers that cooperate to consume data from topics | Confluent Kafka [official] |
| **Bind** | rule that connects an exchange to a queue for message route | RabbitMQ [official] |

**citation [7]** [official, Google Cloud]
> "A named entity that represents an interest in receive messages on a particular topic. Each subscription independently receives a complete copy of all topic messages."
— [Google Cloud Pub/Sub Basics](https://docs.cloud.google.com/pubsub/docs/pubsub-basics)

**citation [8]** [official, Confluent]
> "A set of consumers that cooperate to consume data from topics. Partitions are assigned proportionally among group members."

**onPush map:**
- `onPush hook` = Subscription
- multiple clones with same role = Consumer Group
- `hook.filter` = Subscription filter

## section 3: delivery domain objects

### delivery.semantics

the guarantees around message delivery.

**established terminology:**

| term | definition | source |
|------|------------|--------|
| **At-Most-Once** | messages delivered once, may be lost on failure, "fire and forget" | Confluent Kafka [official] |
| **At-Least-Once** | messages delivered one or more times, never lost but may duplicate | Confluent Kafka [official] |
| **Exactly-Once** | each message delivered once and only once, achieved through transactions | Confluent Kafka [official] |

**citation [10]** [official, Confluent]
> "Messages are delivered once, and if there is a system failure, messages may be lost and are not redelivered. Uses 'fire and forget' send."
— [Confluent Kafka Delivery Semantics](https://docs.confluent.io/kafka/design/delivery-semantics.html)

**citation [11]** [official, Confluent]
> "Messages are delivered one or more times. If there is a system failure, messages are never lost, but they may be delivered more than once."
— [Confluent Kafka Delivery Semantics](https://docs.confluent.io/kafka/design/delivery-semantics.html)

**onPush map:**
- v1 = at-most-once (fire-and-forget, delete on consume)
- v2 = at-least-once (with acknowledgment)

---

### delivery.acknowledgment

confirmation that message was processed.

**established terminology:**

| term | definition | source |
|------|------------|--------|
| **Acknowledgment (Ack)** | confirmation that message process succeeded, broker removes message | RabbitMQ [official] |
| **Negative Acknowledgment (Nack)** | causes broker to redeliver message immediately | Google Cloud Pub/Sub [official] |
| **Acknowledgment Deadline** | initial deadline after which unacknowledged message is sent again | Google Cloud Pub/Sub [official] |

**citation [12]** [official, RabbitMQ]
> "A confirmation where consumers notify the broker that message process succeeded. The broker only removes a message from a queue when it receives acknowledgment."
— [RabbitMQ AMQP Concepts](https://www.rabbitmq.com/tutorials/amqp-concepts)

**citation [13]** [official, Google Cloud]
> "The initial deadline after which an unacknowledged message is sent again. Can be extended per-message via ModifyAckDeadline requests."
— [Google Cloud Subscription Properties](https://docs.cloud.google.com/pubsub/docs/subscription-properties)

**onPush map:**
- v1: implicit ack on successful hook command (delete message)
- v2: explicit ack mechanism

## section 4: error handle domain objects

### error.deadletter

isolation for failed messages.

**established terminology:**

| term | definition | source |
|------|------------|--------|
| **Dead Letter Queue (DLQ)** | special queue that stores messages system cannot process due to errors | AWS [official] |
| **Poison Message** | messages with bugs or invalid data where retry will not help | AWS SQS [official] |
| **Max Receive Count** | number of times consumer can receive message before move to DLQ | AWS SQS [official] |

**citation [14]** [official, AWS]
> "A special queue that temporarily stores messages that a system cannot process due to errors. Prevents the source queue from overflow."
— [AWS What is DLQ](https://aws.amazon.com/what-is/dead-letter-queue/)

**citation [15]** [official, AWS SQS]
> "Messages with bugs, absent data, or invalid references where no amount of retry will help. Would block the queue forever or consume resources infinitely."
— [AWS SQS Dead Letter Queues](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-dead-letter-queues.html)

**onPush map:**
- `.comms/.errors/` = Dead Letter Queue
- malformed messages → `.errors/`
- hook command failures → `.errors/`

## section 6: idempotency domain objects

### idempotency.key

duplicate prevention mechanism.

**established terminology:**

| term | definition | source |
|------|------------|--------|
| **Idempotency** | same operation multiple times has same effect as once | Redis [official] |
| **MessageDeduplicationId** | token to prevent duplicate delivery within deduplication window | AWS SQS FIFO [official] |

**citation [20]** [official, Redis]
> "Perform the same operation multiple times has the same effect as perform it once. The practical path to achieve exactly-once semantics."
— [Redis Streams Idempotency](https://redis.io/docs/latest/develop/data-types/streams/idempotency/)

**citation [21]** [official, AWS SQS]
> "A token used in Amazon SQS FIFO queues to prevent duplicate message delivery within a 5-minute deduplication window."
— [AWS SQS Message Deduplication](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/using-messagededuplicationid-property.html)

**onPush map:**
- `message.id` = deduplication key
- poll adapter tracks consumed ids to prevent reprocess

## domain object treestruct

```
message
├── message.id              # unique identifier (MessageId)
├── message.for             # route attributes
│   ├── message.for.topic   # topic name (Topic/Channel)
│   └── message.for.roles   # role filter (Filter Policy)
├── message.timestamp       # creation time
└── message.payload         # arbitrary data (Body/Payload)

hook
├── hook.filter             # subscription filter
│   ├── hook.filter.topic   # topic match
│   └── hook.filter.roles   # role pattern match
├── hook.command            # target invocation
└── hook.timeout            # execution deadline

delivery
├── delivery.semantics      # at-most-once, at-least-once
└── delivery.acknowledgment # ack/nack mechanism

error
├── error.deadletter        # .comms/.errors/ (DLQ)
└── error.poison            # unprocessable messages
```

---

## domain operation treestruct

```
message operations
├── getMessage              # retrieve from .comms/$topic/
├── setMessage              # write to .comms/$topic/$id.json
└── delMessage              # remove after consume

filter operations
├── matchFilter             # check message.for against hook.filter
└── parseBrainHookPushFilter         # parse '+role,-role' syntax

hook operations
├── invokeHook              # execute hook command
├── timeoutHook             # kill after deadline
└── failHook                # move to .errors/

poll operations
├── scanTopic               # list files in .comms/$topic/
├── processMessage          # match + invoke + delete
└── trackConsumed           # deduplication window
```

## relationship diagram

```
external system
    │
    ▼ writes
.agent/.comms/$topic/$id.json  ←── message (entity)
    │
    ▼ poll adapter scans
poll task (background)
    │
    ▼ matches filter
hook.filter (subscription)
    │
    ▼ invokes
hook.command (target)
    │
    ├── success → delete message
    └── failure → move to .errors/ (DLQ)
```
