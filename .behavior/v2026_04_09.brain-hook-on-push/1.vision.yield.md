# vision: brain-hook-on-push

## .goal

enable push notifications into brain clis (like claude code) so external systems can wake idle clones and inject messages.

## .problem

brain clis are repl-style: they exit when idle. there's no way to:
- wake a terminated session
- inject messages into an active session from outside
- coordinate multiple clones via push

existing approaches (hooks, channels) only work during active tool use вҖ” they can't wake idle sessions.

the original vision proposed a poll-based approach with hooks. research revealed a better path.

## .pivot: from poll to pty wrapper

**original approach (poll-based):**
- background task polls `.comms/` directory
- hooks enforce poll lifecycle
- exit(2) forces restart when poll inactive

**problem with poll:**
- can't programmatically inject into active claude session
- can't wake terminated session (only idle-at-prompt)
- poll latency adds delay

**new approach (pty wrapper + daemon + ipc socket):**
- `rhx enroll claude` wraps claude in a pty
- wrapper holds pty master вҶ’ can write to stdin
- daemon connects via socket вҶ’ writes to wrapper вҶ’ wrapper writes to pty вҶ’ claude receives

see `refs/pivot.todaemon.proposal.md` for full architectural details.

## .how it works

1. **pty wrapper** (`rhx enroll claude`):
   - spawns claude via node-pty (or equivalent)
   - holds pty master, with write access to stdin
   - creates ipc socket at `.agent/.sockets/for.actor=$xyz.sock`
   - passes through stdin/stdout normally

2. **daemon** (separate process):
   - watches `.agent/.comms/$topic/` directories
   - matches messages against role filters
   - connects to actor socket
   - writes message content вҶ’ socket вҶ’ pty master вҶ’ claude stdin

3. **message flow**:
   - external system drops file into `.agent/.comms/support/msg-001.json`
   - daemon sees new file, checks role filter
   - daemon connects to socket, writes message
   - pty wrapper receives on socket, writes to pty master
   - claude receives input on stdin, wakes from idle

## .message format

```json
{
  "for": {
    "roles": "+mechanic,-driver"
  },
  "from": {
    "actor": "foreman-001",
    "system": "slack"
  },
  "payload": {
    "type": "feedback",
    "content": "please review the pr"
  }
}
```

## .role filter syntax

- `+mechanic` = has mechanic role
- `-driver` = excludes driver role
- `mechanic` = exactly only mechanic
- `+mechanic,+driver` = has both roles
- `+mechanic,-driver` = has mechanic but not driver

## the outcome world

### what does a day-in-the-life look like with this in place?

**before:**
vlad starts a mechanic clone to work on a feature. the clone works autonomously. meanwhile, vlad receives a slack message: "hey, that PR needs changes." vlad must manually interrupt the clone, paste context, and redirect its focus. if the clone is deep in a task, vlad waits. if the clone finished and exited, vlad starts a new session and re-explains all context.

**after:**
vlad runs `rhx enroll claude` to start a mechanic clone. the clone works autonomously. meanwhile, slack notifications flow into `.agent/.comms/slack/`. the daemon matches the message to the clone's role filter and injects it via the socket. the clone receives the message on stdin, acknowledges, and pivots.

no human intervention. no context loss. immediate delivery.

### what's the before/after contrast?

| before | after |
|--------|-------|
| clones are deaf вҖ” they only respond to human prompts | clones listen вҖ” external events inject via socket |
| humans are the message bus | `.agent/.comms/` + daemon is the message bus |
| clone exits = context lost | pty wrapper can resurrect with --resume |
| poll latency (seconds) | socket latency (milliseconds) |

### what's the "aha" moment where the value clicks?

the aha moment is when you realize **you can inject at any time**.

start a clone via `rhx enroll claude`. walk away. external events (slack, github, peer clones) drop messages into `.comms/`. the daemon delivers immediately via socket. the clone receives on stdin, acts, responds. you come back to completed work.

or: the foreman daemon coordinates three driver clones. when driver A finishes, it writes to `.comms/radio/`. the daemon delivers to driver B via socket. no poll interval. no human dispatch. sub-millisecond coordination.

## why this works

| aspect | poll-based | pty wrapper |
|--------|-----------|-------------|
| wake idle | exit(2) restarts | socket write to stdin |
| inject active | not possible | socket write to stdin |
| latency | poll interval (seconds) | socket (milliseconds) |
| complexity | hooks + state tracking | wrapper + daemon |
| brain coupling | brain-specific hooks | brain-agnostic pty |

the key insight: **pty master has write access to the slave's stdin**. by holding the master, we can inject at any time.
