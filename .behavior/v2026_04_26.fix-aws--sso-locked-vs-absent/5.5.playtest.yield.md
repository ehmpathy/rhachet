# playtest: aws.config locked vs absent

## prerequisites

- rhachet cli available (`rhx` or `npx rhachet`)
- ehmpath keyrack initialized (`rhx keyrack init --owner ehmpath`)
- aws cli available for sso login (`aws sso login --profile ...`)

## sandbox

all keyrack operations operate on user home directories (`~/.rhachet/`, `~/.aws/config`). no file operations target the repo.

## happy paths

### journey 1: full lifecycle (set → unlock → relock → del)

this is the primary journey that demonstrates the locked vs absent distinction.

#### step 1: verify absent status (before set)

```bash
rhx keyrack status --owner ehmpath --env test | grep -A5 AWS_PROFILE
```

**expected outcome:**
- status shows `absent` (not `locked`)
- tip shows `set` command

```
└─ ehmpath.test.AWS_PROFILE
   ├─ status: absent 🫧
   └─ tip: rhx keyrack set --key AWS_PROFILE ...
```

**acceptance test**: `blackbox/cli/keyrack.status.acceptance.test.ts` case4:t0

---

#### step 2: set the key

```bash
rhx keyrack set --key AWS_PROFILE --env test --vault aws.config --owner ehmpath
```

(follow the prompts to configure aws sso profile)

**expected outcome:**
- success message with profile name
- `~/.aws/config` now contains the profile

**acceptance test**: `blackbox/cli/keyrack.status.acceptance.test.ts` case4:t1

---

#### step 3: verify locked status (after set, before unlock)

```bash
rhx keyrack status --owner ehmpath --env test | grep -A5 AWS_PROFILE
```

**expected outcome:**
- status shows `locked` (not `absent`)
- tip shows `unlock` command

```
└─ ehmpath.test.AWS_PROFILE
   ├─ status: locked 🔐
   └─ tip: rhx keyrack unlock --owner ehmpath --env test
```

**this is the key fix**: after set, status is `locked` because we know the key exists but isn't unlocked.

**acceptance test**: `blackbox/cli/keyrack.status.acceptance.test.ts` case4:t1

---

#### step 4: unlock the key

```bash
rhx keyrack unlock --owner ehmpath --env test
```

**expected outcome:**
- prompts for aws sso login (if not cached)
- success message with unlocked keys

**acceptance test**: `blackbox/cli/keyrack.status.acceptance.test.ts` case4:t2

---

#### step 5: verify granted status (after unlock)

```bash
rhx keyrack status --owner ehmpath --env test | grep -A5 AWS_PROFILE
```

**expected outcome:**
- status shows `granted` (key is accessible)
- tip shows `relock` command

```
└─ ehmpath.test.AWS_PROFILE
   ├─ status: granted ✅
   └─ tip: rhx keyrack relock --owner ehmpath --env test
```

**acceptance test**: `blackbox/cli/keyrack.status.acceptance.test.ts` case4:t2

---

#### step 6: relock the key

```bash
rhx keyrack relock --owner ehmpath --env test
```

**expected outcome:**
- success message confirms relock

**acceptance test**: `blackbox/cli/keyrack.status.acceptance.test.ts` case4:t3

---

#### step 7: verify locked status (after relock)

```bash
rhx keyrack status --owner ehmpath --env test | grep -A5 AWS_PROFILE
```

**expected outcome:**
- status shows `locked` (NOT `absent`)
- tip shows `unlock` command

```
└─ ehmpath.test.AWS_PROFILE
   ├─ status: locked 🔐
   └─ tip: rhx keyrack unlock --owner ehmpath --env test
```

**this is the key fix**: after relock, status returns to `locked`, not `absent`. the key still exists.

**acceptance test**: `blackbox/cli/keyrack.status.acceptance.test.ts` case4:t3

---

#### step 8: delete the key

```bash
rhx keyrack del --key AWS_PROFILE --env test --owner ehmpath
```

**expected outcome:**
- success message confirms deletion

**acceptance test**: `blackbox/cli/keyrack.status.acceptance.test.ts` case4:t4

---

#### step 9: verify absent status (after del)

```bash
rhx keyrack status --owner ehmpath --env test | grep -A5 AWS_PROFILE
```

**expected outcome:**
- status shows `absent` (key was deleted)
- tip shows `set` command

```
└─ ehmpath.test.AWS_PROFILE
   ├─ status: absent 🫧
   └─ tip: rhx keyrack set --key AWS_PROFILE ...
```

**acceptance test**: `blackbox/cli/keyrack.status.acceptance.test.ts` case4:t4

## edgey paths

### edge 1: profile added externally (not via keyrack)

if a profile exists in `~/.aws/config` but was not added via `rhx keyrack set`, status should show `absent` (not tracked by keyrack).

**why**: keyrack tracks what it manages via inventory. external profiles are unknown to keyrack.

**acceptance test**: `blackbox/cli/keyrack.status.acceptance.test.ts` case5

---

### edge 2: orphan inventory (vault deleted externally)

if the vault file is deleted externally but the inventory entry remains:
1. status shows `locked` (inventory says it exists)
2. unlock reports the key as "lost"
3. after unlock, status shows `absent` (inventory cleaned up)

**acceptance test**: `blackbox/cli/keyrack.status.acceptance.test.ts` case6

---

## pass/fail criteria

### ✓ pass if

- step 3 shows `locked` (not `absent`) after set
- step 7 shows `locked` (not `absent`) after relock
- step 9 shows `absent` after del
- no "absent" status appears between set and del

### ✗ fail if

- any step shows `absent` when the key exists but is locked
- relock causes status to revert to `absent`
- status tip does not match the status
