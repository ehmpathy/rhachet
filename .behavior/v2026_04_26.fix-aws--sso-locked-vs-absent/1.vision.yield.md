# vision: aws sso locked vs absent

## the outcome world

### before

vlad sets up AWS_PROFILE via keyrack:
```sh
rhx keyrack set --key AWS_PROFILE --env prep --vault aws.config --exid ahbode-engineer
```

later, the SSO session expires. vlad runs status:
```
🔐 keyrack
   └─ ehmpathy.prep.AWS_PROFILE
      ├─ status: absent 🫧
      └─ tip: rhx keyrack set --key AWS_PROFILE --env prep --vault aws.config
```

vlad: "wait, i just set it? did it not save?"

confusion ensues. vlad runs `keyrack set` again, which works but wastes time.

### after

same scenario. vlad runs status after SSO expires:
```
🔐 keyrack
   └─ ehmpathy.prep.AWS_PROFILE
      ├─ status: locked 🔒
      └─ tip: rhx keyrack unlock --env prep --key AWS_PROFILE
```

vlad: "oh, i just need to re-auth"

runs unlock, browser opens, signs in, done. no confusion.

### the "aha" moment

the value clicks when you realize:
- **absent** = never configured (run `keyrack set`)
- **locked** = configured but needs re-auth (run `keyrack unlock`)

the tip tells you exactly what to do. no guesswork.

## user experience

### usecases

| scenario | status shown | action |
|----------|--------------|--------|
| never configured AWS_PROFILE | absent 🫧 | `keyrack set` |
| configured, SSO session expired | locked 🔒 | `keyrack unlock` |
| configured, SSO session valid | granted 🔑 | (ready to use) |

### contract inputs & outputs

no new CLI flags needed. the changes are internal:

1. `keyrack set` (any vault) creates an empty inventory entry at `~/.rhachet/keyrack/inventory/owner={owner}/{slug-hash}.stocked`
2. `inferKeyrackKeyStatusWhenNotGranted` checks if stocked → locked; absent → absent

### what it looks like

```sh
# check status
rhx keyrack status --key AWS_PROFILE --env prep

# before (shows wrong status)
🔐 keyrack
   └─ ehmpathy.prep.AWS_PROFILE
      ├─ status: absent 🫧                  # incorrect!
      └─ tip: rhx keyrack set ...

# after (accurate)
🔐 keyrack
   └─ ehmpathy.prep.AWS_PROFILE
      ├─ status: locked 🔒                  # accurate!
      └─ tip: rhx keyrack unlock ...
```

### timeline

1. user runs `keyrack set --vault aws.config --exid myprofile`
2. keyrack stores profile name in host manifest
3. **new:** keyrack creates empty inventory entry at `~/.rhachet/keyrack/inventory/owner={owner}/{slug-hash}.stocked`
4. time passes, SSO session expires
5. user runs `keyrack status` or `keyrack get`
6. keyrack checks: is this slug stocked? → yes → locked
7. user sees `locked 🔒` with correct tip

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| no surprise when "absent" but configured | ✅ yes |
| accurate status for aws.config vault | ✅ yes |
| correct tip for next action | ✅ yes |

### pros

- eliminates confusion
- correct action guidance
- minimal implementation (generic inventory system)
- vault-agnostic — works for all vaults uniformly

### cons

- adds ~1ms to status check (inventory entry existence check)
- extant keys need re-run of `keyrack set` to create inventory entry (one-time migration)

### edgecases

| edge case | behavior |
|-----------|----------|
| no inventory entry (never ran keyrack set) | absent |
| stocked, not in daemon | locked (needs unlock) |
| stocked, in daemon | granted |
| profile in config but not stocked (added via AWS CLI, not keyrack) | absent (correct) |
| keyrack del invoked | entry deleted (last), status becomes absent |
| orphan entry (del failed midway) | locked → unlock fails → user runs set (recoverable) |

key insight: inventory lifecycle is in lockstep with keyrack set/del. entry created on set, deleted on del (last).

## open questions & assumptions

### assumptions

1. **inventory entry is existence-only**: inventory entry is empty. its existence alone signals "this key was set on this host". no content, no exid, no lookup info.
2. **no host manifest decryption needed**: inventory existence is sufficient — no need to read the exid.
3. **inventory is vault-agnostic**: a single inventory system at `~/.rhachet/keyrack/inventory/` tracks all keys. owned vaults (os.secure, os.direct) already have implicit entries (the file/entry), but the generic inventory system works for all vaults uniformly.
4. **unsalted slug-hash**: consistent with os.secure pattern. salted hashes would require os.secure migration too — a separate behavior if desired.

### security: no key→persistence lookup leaked

**critical security stance:** inventory entries must never reveal what a key points to.

the key→persistence relationship is itself a privilege escalation. if an attacker learns "slug X resolves to profile Y in ~/.aws/config", they can target their next step.

therefore:
- inventory entry is empty (zero bytes)
- slug-hash is one-way — reveals no info about the key name
- no exid, no profile name, no vault path in the entry
- attacker with ~/.rhachet/ access learns only "N keys were configured" — not which keys or where they point
- inventory directory is chmod 700, inventory entrys are chmod 600 (same as os.secure)

we say "does the host know? yes|no" — never where to find the persisted key.

### deletion order: inventory entry last

on `keyrack del`, delete inventory entry last (after host manifest entry). if del fails midway:
- stocked but key is gone → status shows "locked" (inaccurate but safe)
- user runs unlock → fails with clear error → user runs set

this is better than deleting entry first:
- entry gone but key still exists → status shows "absent" (misleads user)

"locked but gone" is recoverable. "absent but extant" misleads the user.

### questions

1. **[answered]** should we check session validity (via `aws sts get-caller-identity`) to distinguish "locked but session valid" from "locked and expired"?
   - **answer:** no. the simpler "stocked → locked" is sufficient. unlock will handle session refresh.

2. **[resolved]** should other vaults (1password) have similar checks?
   - **answer:** yes, via the generic inventory system. all vaults now use the same inventory path. no vault-specific logic needed.

3. **[answered]** should slug-hashes be salted to prevent key name enumeration?
   - **answer:** no. stay consistent with os.secure pattern (unsalted). salted hashes would require os.secure migration — a separate behavior if desired.

### research needed

none — the API is internal (inventory entry check only).

## what is awkward?

### the host manifest problem

the current design avoids decryption of the host manifest to check status. but aws.config keys are tracked in the host manifest (exid = profile name).

**problem with naive approach:** we can't just check "does profile exist in ~/.aws/config" because that would leak the key→persistence lookup (a privilege escalation). we also don't know WHICH profile to check without decrypting the host manifest.

**solution: inventory entry**

when `keyrack set` runs (any vault), create an empty inventory entry:
```
~/.rhachet/keyrack/inventory/owner={owner}/{slug-hash}.stocked
```

the .stocked file is empty (zero bytes). this is intentional:
- we never leak key→persistence lookups
- inventory existence alone signals "this key was set"
- no exid, no profile name, no content

then `inferKeyrackKeyStatusWhenNotGranted` can:
1. compute slug hash
2. check if inventory entry exists → locked; absent → absent

this is consistent with os.secure (which also uses slug-hash based file lookup).

### generic inventory replaces vault-specific checks

previously, `inferKeyrackKeyStatusWhenNotGranted` had vault-specific checks (os.secure file, os.direct json). the generic inventory system simplifies this:

- one inventory path for all vaults
- no vault-specific logic needed
- `keyrack set` stocks entry, `keyrack del` removes entry
- `inferKeyrackKeyStatusWhenNotGranted` just checks inventory existence

this is cleaner than per-vault `isConfigured` methods.

## summary

the fix requires:
1. create empty inventory entry in `keyrack set` (zero bytes, chmod 600, directory chmod 700) at `~/.rhachet/keyrack/inventory/owner={owner}/{slug-hash}.stocked`
2. delete inventory entry in `keyrack del` (delete entry last, after host manifest)
3. add inventory existence check to `inferKeyrackKeyStatusWhenNotGranted`
4. stocked → locked; not stocked → absent
5. test that aws.config keys show "locked" when stocked but not in daemon

**security principle:** we say "does the host know? yes|no" — never where to find the persisted key.

the user experience improves significantly: no more confusion between "never set up" and "session expired".
